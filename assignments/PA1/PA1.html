Purpose

This assignment reinforces the fundamentals of dynamic memory allocation, pointer manipulation, and copy-constructor / Rule-of-Three design by implementing two different linked-list structures. You will also (1) practice compiling template classes in C++, (2) build data structures whose internals are encapsulated behind a stable public interface, and (3) measure what you built using the Week-1 analysis toolkit — a doubling experiment for time and a bytes-per-node analysis for space.

The linked lists themselves are review (CSS 142/342). The new work is doing the C++ memory management cleanly and leak-free, and then measuring the result instead of just asserting its Big-O.

Variations of linked lists

Linked lists come in many internal forms:

  • singly vs doubly linked (a next pointer only, vs prev and next);
  • with or without a dummy (sentinel) header node;
  • linear (the last node's next is nullptr) vs circular (the last node links back to the header / first node).

Although these differ internally, they can expose the same external behavior. A user of the list — your driver program — should not be able to tell which internal form it is using. That is the point of data abstraction and encapsulation: the same public interface, two different implementations behind it. This PA makes that concrete by giving the two classes below identical public methods over different internals.

Part 1 — Singly linked list, no dummy header (NDLList)

Design and implement a singly linked list without a dummy header, named NDLList. Implement it as a template class so it can store any type. head points at the first real node. Files: NDLList.h, NDLList.cpp.

template <class Object>
struct LListNode {                 // list node
    Object item;
    LListNode* next;
};

template <class Object>
class NDLList {
public:
    NDLList();                                   // constructor: head = nullptr
    NDLList(const NDLList& rhs);                  // copy constructor (deep copy)
    ~NDLList();                                  // destructor (free all nodes)
    const NDLList& operator=(const NDLList& rhs); // assignment operator (deep copy)

    bool   isEmpty() const;                      // is the list empty?
    int    size() const;                         // number of nodes
    void   clear();                              // remove all nodes
    void   insert(const Object& obj, int index); // insert at 0-based index; 0 = at head
    int    find(const Object& obj) const;        // index of obj, or -1 if not found
    void   remove(const Object& obj);            // remove first occurrence of obj
    Object retrieve(int index) const;            // data at index
private:
    LListNode<Object>* head;                      // first real node (no dummy header)
};

You may adapt the dummy-header list from lab, removing the sentinel to meet this requirement.

Part 2 — Circular doubly linked list, dummy header (CDLList)

Design and implement a circular, doubly linked list with a dummy header, named CDLList, also a template class. Same public interface as NDLList — different internals. Files: CDLList.h, CDLList.cpp.

Structural requirements:

  • a dummy header (sentinel) node;
  • each node has item, prev, next;
  • the header's next → the first data node, the header's prev → the last data node;
  • the last data node's next → the header (circular).
template <class Object>
struct DLListNode {
    Object item;
    DLListNode* prev;
    DLListNode* next;
};

template <class Object>
class CDLList {
public:
    CDLList();                                    // constructor: create the dummy header
    CDLList(const CDLList& rhs);                   // copy constructor (deep copy)
    ~CDLList();                                   // destructor (free all nodes incl. header)
    const CDLList& operator=(const CDLList& rhs);  // assignment operator (deep copy)

    bool   isEmpty() const;
    int    size() const;
    void   clear();
    void   insert(const Object& obj, int index);  // index 0 = first data node
    int    find(const Object& obj) const;
    void   remove(const Object& obj);
    Object retrieve(int index) const;
private:
    DLListNode<Object>* header;                    // dummy header (sentinel)
};
Template compilation — the one rule Because these are template classes, the implementation must be visible at the point of instantiation. End each header with the matching .cpp — #include "NDLList.cpp" at the bottom of NDLList.h, #include "CDLList.cpp" at the bottom of CDLList.h — and do not compile the .cpp files separately. You compile only your driver, which #includes the headers. (The Computing Environment handout, §4, walks through this with a worked build.)

Part 3 — Functional testing (your own driver.cpp)

Write your own driver.cpp that verifies the correctness of both classes. (starter/sample_driver.cpp is a minimal example — it does not test everything.)

Testing goals:

  • #include both headers (NDLList.h, CDLList.h).
  • Test every public method of both classes.
  • Test with multiple data types (e.g. int and string) — they are templates.
  • Cover edge cases: empty list, insert at head / middle / tail, remove a missing element, retrieve at the boundaries, and a deep-copy test (copy or assign a list, mutate the original, confirm the copy is unchanged — proving you did not alias the nodes).
g++ -std=c++17 driver.cpp -o driver && ./driver

Part 4 — Performance comparison: the doubling experiment

Both classes implement the same methods with the same Big-O for most operations — yet their actual runtime differs because of their internals. Write your own compare_driver.cpp (extend starter/doubling_compare.cpp) to measure and compare them.

Comparison goals:

  • #include both headers.
  • Compare the public operations of NDLList and CDLList.
  • Use multiple data types.
  • For each operation, run it at N = 1000, 2000, 4000, 8000, …, time it, and print the ratio of each time to the previous N.

Read the order of growth off the ratio, exactly as in L01:

ratio → 2 ⇒ linear Θ(n) · ratio → 4 ⇒ quadratic Θ(n²) · ratio → 1 ⇒ constant Θ(1).

Most operations are the same Θ on both lists, so the interesting findings are (a) the constant-factor difference and (b) the operations where the two genuinely differ in order — e.g. inserting at the tail is an O(n) walk in NDLList but O(1) via the header's prev in CDLList.

g++ -std=c++17 -O2 compare_driver.cpp -o compare && ./compare

Memory management & Valgrind (required)

Because this assignment leans heavily on pointers, your classes must correctly manage all dynamically allocated memory: every new must be matched by a delete, handled in your constructors, destructor, clear(), and assignment operator. Your program must not leak.

We grade memory with Valgrind; develop with it yourself:

g++ -std=c++17 -g compare_driver.cpp -o compare && valgrind --leak-check=full ./compare

If any leaks or memory errors are detected, no points are awarded for the Memory criterion. (Alternatively, the Address/Undefined-Behavior Sanitizer catches many of the same bugs: g++ -std=c++17 -fsanitize=address,undefined ….) See the Valgrind handout for how to read the report (clean run vs. a real leak, with annotated transcripts).
