Purpose

In the ICAs you built each tree in isolation. In PA2 you build them as an engineer does: one BinTree class whose balancing policy is chosen at construction — Mode::VANILLA (plain BST), Mode::AVL (rotations — port your ICA 04 machinery), and Mode::REDBLACK (Sedgewick's left-leaning red-black insert, from the §3.3 reading) — then run the same adversarial input through all three and measure what each policy buys. Your class carries its own instruments (rotations(), colorFlips()), so the S4/S5 theory becomes numbers your own code produces: sorted 1..1023 gives heights 1023 vs ~10 vs ~10, at a price of ~1013 rotations.

The second half is an operation none of the ICAs touched — and this time the algorithm is yours to design: mergeWith, combining two trees into one. We give you a contract (correct union, a height bound, other untouched) and a public leaderboard: every submission is benchmarked on four standard workloads, ranked on time and on peak memory, against five instructor baselines — from the naive insert-loop to split/join merging, the algorithm inside modern log-structured storage engines (RocksDB, Cassandra). No baseline wins everywhere; where does yours land?

A production system would put all this behind a database — and the index that database builds is a B-tree (S5), these structures' disk-scale cousin. Where the network hop is too slow (order books, schedulers, std::map itself — a red-black tree), this code runs in memory, as you'll write it.

Statement of work

Implement BinTree (int keys) in bintree.h / bintree.cpp. The public interface is fixed; add your own private recursive helpers. displayTree is given as a debugging aid.

  1. Rule of Three + equality — deep copy constructor, destructor, self-assignment-safe operator= (mode and counters copy too); operator==/!= mean same keys AND same structure (mode is not compared — so the same keys inserted into VANILLA and AVL trees give !=).
  2. Core queries — isEmpty, size, contains, treeHeight() (# nodes on the longest path; empty = 0).
  3. insert, per mode — duplicates return false in every mode; then the mode's fix-up: VANILLA nothing · AVL heights + the four rotation cases (your ICA 04 code, ported) · REDBLACK the left-leaning insert — attach red, then rotate-left / rotate-right / flip-colors per the three ifs of §3.3, root forced black.
  4. remove — VANILLA only (Hibbard, successor promotion, no leaks). In balanced modes it returns false and changes nothing. The two refusals differ: AVL deletion is tractable — some of you built it as the ICA 04 stretch — but red-black deletion is genuinely hard, and one method that works in one mode and breaks the invariant in another is worse than an honest refusal. Your report addresses this.
  5. Instrumentation — rotations() (every single rotation; a double counts 2) and colorFlips(), bumped inside your helpers.
  6. toSortedArray + mergeWith — your algorithm. The contract, checked by our grader and the leaderboard harness: the tree ends up holding the union of the two key sets (shared keys once); other unchanged; self-merge safe; the result is a valid BST with height ≤ 2·⌈log₂(n+1)⌉ that stays internally consistent for its mode; and you instrument it yourself — set mergeAuxBytes() to the peak heap bytes your merge allocated (vectors, stacks, every new node); the harness audits it with a counting allocator, within 2× passes. How you meet the contract is the design exercise — insert-loop, flatten-merge-rebuild, streaming iterators, in-place vine rotations, split/join, or your own invention. Predict its time and space before you measure.
Worked example — why "mode-consistent" bites

Merging by flatten → union → rebuild a balanced tree is a great way to hit the height bound, and for VANILLA it's the whole job. But the rebuilt nodes carry fields the mode depends on, and the union + height checks won't notice if they're wrong:

  • AVL: every node's height must equal its true subtree height. Leave it stale and the next insert computes a bogus balance factor and rotates the wrong way — the Θ(log n) guarantee silently breaks.
  • REDBLACK: the rebuilt nodes need a valid coloring (no two reds stacked, equal black-height on every path, root black). All-black won't do — the next insert's fix-up misfires.

So a merge can pass "union" and "height ≤ 2·⌈log₂(n+1)⌉" yet still fail the mode-consistency clause — and the grader's post-merge inserts are what catch it. Fix: after rebuilding, recompute height (AVL) or assign a valid coloring (RB) — or merge by a method that maintains those fields as it goes. Either way, test a few inserts after a merge in your mydriver.cpp.

Measurement (for your report)
  1. The head-to-head (the driver prints it): heights and rotation counts for sorted 1..1023 in all three modes — what did each policy buy, and what did it cost?
  2. Your merge, predicted and measured: state your algorithm and its predicted time/space complexity before measuring; reconcile against your own timings and your mergeAuxBytes() accounting. Disagreement between prediction and measurement is the interesting part — explain it.
  3. Regime analysis: on which workloads should your algorithm be strongest and weakest, and why? (The baselines disagree with each other — W2 rewards m ≪ n specialists, W4 rewards concatenation.)
  4. Two sentences on remove: AVL deletion is doable (ICA 04's stretch goal!) yet PA2 refuses it — defend or attack that interface decision, mentioning the red-black case. One sentence: which of your operations CREATE INDEX + SQL would replace in a database-backed design.
