Purpose

You have already written the algorithms: BFS/DFS in ICA 08, heap-based Dijkstra in ICA 09. PA3 is about what those exercises hid: the representation, and the engineering around it. You build the same graph twice:

  • GraphM — an adjacency matrix, running all-pairs Dijkstra in the O(V²) linear-scan form (no priority queue — deliberately a different algorithm than your ICA 09), with full path reconstruction;
  • GraphL — an adjacency list of raw, heap-allocated EdgeNodes, with iterative DFS (explicit stack) and BFS (queue), and a destructor that frees every node;

— both built from a real file format with named vertices and multiple graphs per file. Part 4 races your two Dijkstras — matrix scan vs your ICA 09 heap version — on dense vs sparse generated graphs. One of them loses on each family; the asymptotics say which, and your measurements will agree.

The file format

First line: the vertex count n. Then n name lines (names contain spaces — getline), then one from to cost triple per line, terminated by a line starting with 0. A file may hold several graphs back-to-back — buildGraph must consume exactly one and leave the stream ready for the next. Vertices are numbered 1..n (slot 0 unused). See simple_graph.txt (two graphs) and complex_graph.txt (10 vertices).

Part 1 — GraphM (matrix + all-pairs Dijkstra)

Internals (declared in the given graphm.h): vector<string> vertices, vector<vector<int>> adjM (cell = weight, INF = no edge; a repeated edge in the file simply overwrites the cell), and vector<vector<TableType>> pathM where TableType { visited, dist, prev_node } — one row of tables per source, because you compute shortest paths from every vertex.

Required: constructor · buildGraph · insertEdge/removeEdge (validate range, self-loops, negative cost → −1) · findShortestPath() — for each source, repeat V times: linear-scan the nearest unsettled vertex, settle it, relax its out-edges, recording prev_node only when the distance improves · getSize/getDist/getPath (your class is a component, not just a printer) · displayAllPaths/displayPath — formats pinned by expected-output.txt.

The subtle deliverable is path reconstruction: the table stores only each vertex's predecessor; getPath follows prev_node backward and prints forward (recursion makes this pleasant).

Part 2 — GraphL (list + iterative traversals)

Raw structures, exactly as declared: each GraphNode heads a singly linked list of heap-allocated EdgeNodes. Every new edge is inserted at the head — an O(1) insert whose visible consequence is that each list holds its edges in reverse input order. That is not a bug: it makes every DFS/BFS order deterministic, and expected-output.txt is computed under this rule.

Required: constructor/destructor (every EdgeNode freed — and buildGraph on an already-built object frees the old graph first) · buildGraph (weights read and ignored) · getSize · displayGraph · edgeList(id) · DFSorder(start=1) — iterative, explicit stack; recursion is not accepted · BFSorder(start=1) — queue, marking visited on enqueue.

Copying: the class declares GraphL(const GraphL&) = delete; — with raw pointers, the compiler's copy would double-free. PA1/PA2 answered the Rule of Three by implementing deep copies; deleting the operations is the other legitimate answer. Keep the = delete and say why in a comment, or implement a real deep copy — your choice, documented.

Part 3 — your own testing (driver.cpp)

The given sample_driver.cpp is the floor: it never feeds a malformed file, never inserts an invalid edge, never starts a traversal anywhere but vertex 1, and never rebuilds an existing object. Your driver.cpp does all of that, on both provided files.

Part 4 — the representation race Generate the fixed-seed families with the given gen_graphs.cpp: dense_V.txt (E ≈ V²/2) and sparse_V.txt (E ≈ 4V), V = 200…1600. Time all-pairs shortest paths two ways: GraphM::findShortestPath() (O(V³) total) vs your ICA 09 heap Dijkstra looped over every source. Doubling table per family, then the explanation: on which family does the "dumb" linear scan win or tie, and why does the heap pay exactly there? This is the rare experiment where the asymptotically worse-looking code wins a fair race.
