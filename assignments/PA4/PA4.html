Purpose

Every time you run git diff, a dynamic program runs: comparing two files is the longest-common-subsequence problem, and the diff you read is the traceback. In PA4 you build that tool — mydiff — end to end. It is the capstone of the DP arc: the 2-D table (S14), the traceback as the product rather than a by-product (S15), a second DP nested inside the first, and the space-optimization lesson forced on you by arithmetic — two 30,000-line files need a 3.6 GB table unless you know better.

Two phases: design first (PA4a, graded), then build (PA4b) and reconcile what changed. That is how non-trivial software actually gets made — and the design work is where the DP decisions live.

The tool

./mydiff fileA fileB           unified diff, 3 context lines
./mydiff -u N fileA fileB      unified diff, N context lines
./mydiff --stat fileA fileB    counts only — the ROLLING-ROW mode; survives
                               files far too large for the full table
./mydiff --word i j fileA fileB   within-line markup for one line pair

The CLI (mydiff.cpp) is given. You implement the four functions behind it, declared in the given diff.h:

  1. lcsLines — the LCS value over file lines with two rolling rows (O(min(n,m)) memory). This is --stat, and it must survive the generated 30,000-line pair. (Port your ICA 14 lcs; keys change from characters to lines, and the memory discipline is new.)
  2. diffLines — the full edit script: strip the common prefix and suffix first (that keeps real inputs inside the table budget), build the LCS table over what remains, then traceback into ' '/'-'/'+' edits. Convention: within a changed run, deletions precede insertions — and mind that a backward walk reverses your tie-break.
  3. unifiedDiff — group the script into hunks (changes plus context common lines), each headed @@ -start,count +start,count @@. Check against expected-unified.txt.
  4. markLine — the same DP one level down: character-level comparison of one changed line pair, rendered wdiff-style — [-deleted-] and {+inserted+}.

PA4a — the design document (12 pts, due Thu Aug 6)

  1. Overview & data-structure choices, with reasons — which tables, dimensions, cell meaning; where rolling rows apply and why they can't naively support traceback (the design problem of the assignment); how trimming bounds the table.
  2. Component diagram — the four functions' data flow; who owns which memory.
  3. Function descriptions — recurrences written out, base cases, your traceback tie-break.
  4. Test plan + the greedy counterexample — construct, by hand, a file pair where greedy line matching produces a non-minimal diff, and show the minimal one. (S12 taught you why greedy needs a proof; here's a place it fails.)

Your design is graded on its own terms; changing course during implementation costs nothing — PA4b's report explains the delta.

PA4b — implementation (28 pts, due Tue Aug 18)

g++ -std=c++17 -O2 diff.cpp mydiff.cpp -o mydiff
./mydiff sampleA.txt sampleB.txt | diff - expected-unified.txt   # exact with the stated tie-break
valgrind --leak-check=full ./mydiff sampleA.txt sampleB.txt      # on the CSS lab
g++ -std=c++17 -O2 gen_corpus.cpp -o gen && ./gen
./mydiff --stat big_A_30000.txt big_B_30000.txt                  # seconds and small memory — or your
                                                                 # lcsLines built the full table
Measurement (report) A doubling table over the bench_* pairs (1000 → 8000 lines) for the full diff — time and peak memory — with the O(n·m) reading; the big-pair --stat run as evidence the rolling rows work; and your PA4a greedy counterexample revisited with the real tool's output.
The video (≤ 10 min, required) Compile and run on the lab, walk each rubric feature to where it lives in your code, and disclose known bugs or limitations. Your program is graded following your video's instructions — unclear steps cost points.

What to submit

Grading policy: we compile your diff.cpp against our own main(). The autograder pins no particular alignment: edit scripts are checked by reconstruction and optimality (changes = n+m−2·LCS); unified output by patching A into B. Any correct tie-break passes. Doesn't compile ⇒ 0 for correctness.
