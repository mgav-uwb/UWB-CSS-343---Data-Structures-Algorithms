Chapter 1 — Analysis of Algorithms: Time

CSS 343 — Data Structures, Algorithms & Discrete Mathematics II. Course text, primary reading for Lecture 1.
How long does a program take, and how do we answer scientifically rather than by guessing? We fix a cost model and a basic operation, count how often it runs, and read off the order of growth — the function of \(n\) the cost is proportional to. Two independent routes reach it: a doubling experiment that measures the exponent from running-time ratios, and a counting argument that derives the same exponent from the code. Tilde notation keeps the leading term and discards the rest. The 3-SUM problem runs throughout, first as a \(\Theta(n^3)\) triple loop and then, once analysis tells us where the waste is, as a \(\Theta(n^2)\) sort-and-scan. With worked traces, verified numbers, code in four languages, and exercises.

Every choice this course makes — which tree, which sort, which graph algorithm — is ultimately a conversation about two costs: time and space. This chapter builds the tools for the first. The trouble is that a program’s running time seems to depend on everything: the machine, the compiler, the input, the phase of the moon. The escape is to stop asking “how many seconds?” and ask instead “how does the time grow as the input grows?” That question has a clean, machine-independent answer — the order of growth — and we reach it two ways that must agree: by measuring a running program and by counting a single operation in its inner loop. When they agree we trust the result; when they disagree, one of them is a bug. Along the way, the analysis will hand us a faster algorithm for free. The formal \(O\), \(\Omega\), and \(\Theta\) machinery that makes “order of growth” precise is developed in the Asymptotic Notation reading; here we stay close to the running time itself.

1. Cost models and the basic operation

We analyze running time the way a scientist studies any natural process: observe (measure the time), hypothesize (a model consistent with the data), predict (a new time from the model), and verify (measure again and compare). Two ground rules keep it honest: experiments must be reproducible, and the model must be falsifiable — one missed prediction sinks it. We never prove a running-time model correct, only consistent with everything we have seen so far. The lucky part is that our experiment is free: to gather data we just run the program.

A stopwatch, though, measures the wrong thing. Wall-clock time folds in the processor, the language, the compiler flags, and whatever else the machine is doing at that instant. To talk about the algorithm rather than one run of one implementation, we count operations under a fixed cost model.

Definition (cost model, basic operation) A cost model declares which operations we count and charges each a constant. The classic choice is the RAM model, in which arithmetic, comparisons, and array accesses each cost \(O(1)\). A basic operation is a single representative operation whose repetition dominates the total work; we count how many times it runs as a function of the input size \(n\), and take that count as a proxy for time.

The art is picking a basic operation whose count has the same order of growth as the true running time, so that counting it and timing the program tell the same story. If several operations run in lockstep — say each loop iteration does one comparison and three array reads — any one of them serves, because they differ only by a constant factor, and constant factors are exactly what order-of-growth analysis discards.

Why counting beats timing An operation count is exact and machine-independent: rerun the program on a phone or a server and the count is identical, while the seconds differ. Sedgewick & Wayne make counting the primary tool and use timing only to validate the count. We will do the same: count when we can, time to check.

2. A first algorithm: brute-force 3-SUM

We need a concrete problem to practice on, and one problem anchors this whole chapter.

Definition (the 3-SUM problem) Given an array of \(n\) integers, count the unordered triples \((i,j,k)\) with \(i \lt j \lt k\) and \(a[i] + a[j] + a[k] = 0\).

As stated it is a toy, but the shape is real: testing whether any three of \(n\) points are collinear reduces to 3-SUM, and 3-SUM is the textbook example of a problem for which no strongly sub-quadratic algorithm is known — a genuine open question we return to in §6. For now it is a clean target for analysis.

Example (a small instance) On the eight-element array 
  −40  −20  −10   0   5  10  30  40
exactly four triples sum to zero: \((\,{-}40, 10, 30\,)\), \((\,{-}40, 0, 40\,)\), \((\,{-}20, {-}10, 30\,)\), and \((\,{-}10, 0, 10\,)\). Each is easy to check — \(-40 + 10 + 30 = 0\), and so on — and a short search confirms there are no others, so the answer is \(4\). This is the count our algorithms must reproduce.

The direct idea is to enumerate every triple: one loop picks \(i\), a nested loop picks \(j \gt i\), a third picks \(k \gt j\), and we test the sum. The constraint \(i \lt j \lt k\) counts each triple exactly once — it is what the loop bounds i+1 and j+1 encode. 

// Brute-force 3-SUM: count unordered triples i<j<k with a[i]+a[j]+a[k]==0.
// The inner-loop body runs C(n,3) ~ n^3/6 times -> Theta(n^3).
long count3(const vector<int>& a) {
    int n = a.size();
    long cnt = 0, ops = 0;
    for (int i = 0;   i < n; i++)
      for (int j = i+1; j < n; j++)
        for (int k = j+1; k < n; k++) {
            ops++;                         // one tick per triple tested
            if (a[i] + a[j] + a[k] == 0)
                cnt++;
        }
    return cnt;                            // ops ends at n(n-1)(n-2)/6
}

The code is correct almost by inspection: it literally examines every triple. The interesting question is its cost, and here the choice of basic operation is easy. Nearly all the work happens in the innermost loop body, which runs once per triple; a single ops++ there captures the total work, and the three array reads it performs are a constant multiple of that. So we count inner-loop executions, one tick per triple.

Cost depends on \(n\), not the values Run count3 on different arrays of the same size and the running time barely moves: the loops execute the same number of times no matter which integers are inside, because the values affect only the final count, not how many triples are examined. So for this algorithm there is one question — how does the cost grow with \(n\)? (Not every algorithm is so input-insensitive; best-, worst-, and average-case analysis, where they differ, is taken up with the formal notation.)

3. Observe: the doubling experiment

Before deriving anything, we measure. Run count3 on a random input, then double \(n\) and run again, and again, recording the operation count each time along with its ratio to the previous line. The ratios are the whole point.

The operation ratio converges cleanly on \(8\). What does an \(8\) mean? Suppose the running time follows a power law \(T(n) = a\,n^b\). Then doubling the input multiplies the time by a fixed factor that reveals \(b\): \[ \frac{T(2n)}{T(n)} = \frac{a\,(2n)^b}{a\,n^b} = 2^b . \] The constant \(a\) cancels exactly, and the exponent falls out of the ratio. A ratio of \(2 = 2^1\) means linear, \(4 = 2^2\) means quadratic, and \(8 = 2^3\) means cubic: the ratio is the exponent in disguise. Our measured \(8\) points straight at \(T(n) \sim a\,n^3\).

Example (predict, then verify) With \(b = 3\), the next doubling should again multiply operations by \(8\): from \(n = 2000\) we predict \(n = 4000\) will cost about \(8 \times 1{,}331{,}334{,}000 = 10{,}650{,}672{,}000\) operations. That is a falsifiable forecast — run it and if the count misses, the cubic model is wrong. (It does not miss; the exact value is \(\binom{4000}{3} = 10{,}658{,}668{,}000\), within about \(0.08\%\) of the prediction, the tiny gap coming from the lower-order terms we quantify next.)

Why a log-log plot exposes the exponent

The ratio test and its graphical cousin are the same fact seen twice. Take logarithms of the power law \(T(n) = a\,n^b\) (write \(\lg\) for \(\log_2\)): \[ \lg T(n) = b\,\lg n + \lg a . \] That is the equation of a straight line in the coordinates \((\lg n, \lg T)\) with slope \(b\). So if you plot the measured times on log-log axes and see a straight line, you have strong evidence of a power law, and its slope is the order of growth — discovered with no reasoning about the algorithm at all. Reading the slope off the plot and reading the exponent off the doubling ratio are two ways to recover the same \(b\).

Measurement has now told us the exponent and given us a forecast — but it cannot tell us why the exponent is \(3\). For that we leave the stopwatch and count.

4. Model: counting the basic operation

Knuth’s framing of running time is a sum over the statements of a program: \[ \text{total time} \;=\; \sum_{\text{statements}} (\text{cost}) \times (\text{frequency}) . \] The cost of a statement is a property of the machine and compiler; the frequency — how often it runs — is a property of the algorithm and its input. The costs are fixed constants we do not care about; the frequency of the most-executed statement is where the analysis actually lives.

For count3 that statement is the inner-loop body, and its frequency is a counting question straight out of discrete mathematics: how many triples \(i \lt j \lt k\) are there? It is the number of ways to choose \(3\) of \(n\) indices, \[ \binom{n}{3} = \frac{n(n-1)(n-2)}{6} = \frac{n^3}{6} - \frac{n^2}{2} + \frac{n}{3} . \] No measurement, no fitting — pure combinatorics. And it matches the experiment to the digit: at \(n = 1000\), \[ \frac{10^9}{6} - \frac{10^6}{2} + \frac{1000}{3} = 166{,}666{,}666.\overline{6} - 500{,}000 + 333.\overline{3} = 166{,}167{,}000, \] exactly the operation count the experiment printed for \(n = 1000\).

Tilde: keep the leading term

The formula \(\frac{n^3}{6} - \frac{n^2}{2} + \frac{n}{3}\) is exact but cluttered. At \(n = 1000\) the two trailing terms contribute \(-500{,}000 + 333 \approx -499{,}667\), a rounding error beside the leading \(\frac{n^3}{6} \approx 1.667 \times 10^8\). We formalize “the rest is negligible” with tilde notation.

Definition (tilde notation) We write \(f(n) \sim g(n)\), read “\(f\) is asymptotic to \(g\),” when \(\displaystyle \lim_{n \to \infty} \frac{f(n)}{g(n)} = 1\). Tilde keeps a function’s leading term and discards everything that vanishes in proportion to it.

The condition is exactly met here, because dividing the exact count by its leading term gives \[ \frac{\,n^3/6 - n^2/2 + n/3\,}{n^3/6} = 1 - \frac{3}{n} + \frac{2}{n^2} \;\longrightarrow\; 1 \] as \(n \to \infty\). So \(\binom{n}{3} \sim \frac{n^3}{6}\). Tilde is stronger than order-of-growth: it keeps the constant \(\frac{1}{6}\), reporting not just the shape but the coefficient of the leading term.

The correction predicts the wobble The discarded \(-3/n\) term is not merely small — it explains the experiment’s approach to \(8\). The exact doubling ratio is \(\dfrac{\binom{2n}{3}}{\binom{n}{3}} = \dfrac{4(2n-1)}{n-2}\), which gives \(8.048,\ 8.024,\ 8.012\) at \(n = 250, 500, 1000\) — the table’s \(8.05, 8.02, 8.01\) to the digit, closing on \(8\) from above exactly as the \(-3/n\) correction predicts. The model explains even the imperfection of the measurement.

Order of growth

Drop the constant as well and keep only the shape — the function of \(n\) the cost is proportional to — and you have the order of growth:

exact frequencytildeorder of growth
\(n(n-1)(n-2)/6\)\(\sim n^3/6\)\(n^3\)
\(n^2/2 - n/2\)\(\sim n^2/2\)\(n^2\)
\(\lg n + 1\)\(\sim \lg n\)\(\lg n\)

Order of growth strips both the machine constant and the lower-order terms, leaving the most portable description of cost there is — independent of language, machine, and compiler. It is what lets a running-time analysis from decades ago still apply to today’s hardware.

Choosing a cost model whose count grows like the running time lets us state a fact about the algorithm, not one implementation of it.

Proposition (3-SUM cost) Under the array-access cost model, brute-force 3-SUM makes \(\sim n^3/2\) array accesses, and its running-time order of growth is \(n^3\).
Proof. The inner-loop body runs \(\binom{n}{3} \sim n^3/6\) times, and each execution performs three array reads \(a[i], a[j], a[k]\), for \(\sim 3 \cdot n^3/6 = n^3/2\) accesses. Since every other statement (loop bookkeeping, the counter) runs at most a constant multiple of that, the total work is \(\Theta(n^3)\), so the running time is proportional to \(n^3\). ∎

Two independent routes have now reached the same place: the experiment measured a log-log slope of \(3\) and a doubling ratio of \(8\); the count derived \(\sim n^3/6\) triples from the code alone. Neither used the other, and they agree — which is exactly the corroboration the scientific method asks for. The order of growth of 3-SUM is \(n^3\), and it depends on the algorithm (it examines all triples), not on the language or the machine.

5. The order-of-growth classes

Because we build programs from a few primitives — straight-line code, loops, nesting, halving, recursion — the cost is almost always one of a small handful of functions. This table is the vocabulary of the entire course.

Each shape comes from a structural feature of the code. Work that does not loop over the input is constant. A step that throws away a constant fraction of the data — binary search halving its range — is logarithmic. Touching each element a constant number of times is linear. Splitting the input in half, recursing on the pieces, and combining is linearithmic. Then the expensive ones: every pair is quadratic, every triple is cubic, every subset is exponential. Nesting is what drives the exponent up, because nested loops multiply their counts (§6).

What the classes mean in seconds

The exponent is not an academic nicety — it decides feasibility. Take an input of \(n = 10^6\) on a machine doing \(10^9\) operations per second:

Same machine, same language, every row: the difference between a millisecond and seventeen minutes is a difference of algorithm, not hardware. This is why the first question about a slow program should be “is there a better order of growth?” before “can I buy a faster machine?” At \(n = 10^6\) the quadratic and cubic algorithms are already unusable, which is precisely the motivation to make 3-SUM faster.

Where this becomes precise “Proportional to,” “dominates,” and “order of growth” are made rigorous by the \(O\), \(\Omega\), and \(\Theta\) set-definitions — \(O\) for an upper bound, \(\Omega\) for a lower bound, \(\Theta\) for a matching pair — developed in the Asymptotic Notation reading. In this chapter “order of growth \(n^3\)” and “\(\Theta(n^3)\)” mean the same thing; there we prove the equivalence.

6. Design: a faster 3-SUM

Analysis is not only a report card; it is a design tool. We know 3-SUM is \(n^3\) because it examines every triple. The instinct “you have to check every triple, so cubic is unavoidable” is worth challenging, because it is wrong. To see why, shrink the problem.

The warm-up: 2-SUM

Counting pairs that sum to zero is 3-SUM with one loop removed — a double loop, order \(n^2\). But look at what the inner loop is really doing: for each \(a[i]\), it asks “is \(-a[i]\) somewhere in the array?” That is a search, and search is slow on unordered data but fast on sorted data. If we sort once up front (\(n\lg n\)), each “is \(-a[i]\) present?” becomes a binary search in \(\lg n\), and the whole thing is \(n\lg n + n \cdot \lg n = 2\,n\lg n\), which is \(n\lg n\): we spent a sort to knock 2-SUM from quadratic down to linearithmic.

To add these costs correctly we need two rules, and they are the whole of practical analysis.

The sum rule and the product rule Sequential steps add, and the larger dominates: \(f + g\) has the order of growth of whichever of \(f, g\) is larger. Nested work multiplies: an outer loop of \(n\) iterations each doing \(g(n)\) work costs \(n \cdot g(n)\).

The sum rule is why a cheap preprocessing pass is “free” next to a dominant loop: \(n\lg n + n^2\) is just \(n^2\), and \(n + n^2\) is \(n^2\). The product rule is why nesting drives the exponent: \(n\) iterations of \(\lg n\)-work is \(n\lg n\), and \(n^2\) iterations of \(\lg n\)-work is \(n^2\lg n\). Getting these two straight — sequential adds, nested multiplies — is most of what analysis asks of you.

Lifting the idea to 3-SUM

The same lever works one dimension up. Sort the array; then for each pair \(i \lt j\), binary-search for the third value \(-(a[i] + a[j])\). By the rules this is \(n\lg n + n^2 \cdot \lg n = n^2\lg n\) — already down from \(n^3\). But we can do better still, and remove the log, by replacing the search with a two-pointer scan. Fix \(a[i]\); place lo just after \(i\) and hi at the end, and walk them toward each other, watching the sum \(s = a[i] + a[\texttt{lo}] + a[\texttt{hi}]\):

  • if \(s \lt 0\), the sum is too small, so advance lo;
  • if \(s \gt 0\), the sum is too big, so retreat hi;
  • if \(s = 0\), count the triple and move both inward.
// Fast 3-SUM: sort (n log n), then for each i sweep two pointers over the
// tail. Every step retires an index, so each sweep is <= n -> Theta(n^2).
long count3fast(vector<int> a) {
    sort(a.begin(), a.end());              // n log n preprocessing
    int n = a.size();
    long cnt = 0;
    for (int i = 0; i < n; i++) {          // fix a[i] ...
        int lo = i + 1, hi = n - 1;        // ... two pointers on the tail
        while (lo < hi) {
            int s = a[i] + a[lo] + a[hi];
            if      (s < 0) lo++;          // too small: advance lo
            else if (s > 0) hi--;          // too big:   retreat hi
            else { cnt++; lo++; hi--; }    // hit 0 (distinct values)
        }
    }
    return cnt;
}

Why is the inner while loop only \(O(n)\)? Because each iteration permanently retires an index. This is the first of many “each step discards a candidate, so the loop is linear” arguments in the course, and it deserves a proof.

Lemma (the two-pointer scan is correct and linear) For a fixed \(i\), the scan counts every pair \((\texttt{lo}, \texttt{hi})\) with \(a[i] + a[\texttt{lo}] + a[\texttt{hi}] = 0\) and \(i \lt \texttt{lo} \lt \texttt{hi}\), and performs at most \(n\) iterations.
Proof. The array is sorted. Suppose \(s \lt 0\). Then for every index \(h' \le \texttt{hi}\) we have \(a[\texttt{lo}] + a[h'] \le a[\texttt{lo}] + a[\texttt{hi}] \lt -a[i]\), so no pair using the current lo can reach \(-a[i]\); the current lo can be discarded forever, and advancing it loses no solution. The case \(s \gt 0\) is the mirror image, retiring hi. When \(s = 0\) the pair is a genuine solution, counted, after which both a smaller-partner-for-hi and a larger-partner-for-lo are impossible (values are distinct), so both pointers move. Every iteration moves lo up or hi down by one, and they start \(n\) apart at most, so the loop runs at most \(n\) times and terminates when they cross. ∎

Now add it up. The sort is \(n\lg n\). The outer loop runs \(n\) times, each firing a scan of at most \(n\) iterations, so the scanning work is \(n \cdot n = n^2\) by the product rule. By the sum rule \(n\lg n + n^2 = n^2\): the sort vanishes beside the scans, and fast 3-SUM is \(\Theta(n^2)\).

Example (the scan on the small instance) The array \(-40, -20, -10, 0, 5, 10, 30, 40\) is already sorted. Fix \(i = 0\) (\(a[i] = -40\)), start \(\texttt{lo} = 1\), \(\texttt{hi} = 7\), and walk:

Same answer, same triples — only the work differs. The whole ladder:

algorithmideaorderdoubling ratio
triple looptest every triple\(n^3\)\(\times 8\)
sort + binary searchsearch for the third value\(n^2\lg n\)\(\times 4\) and a bit
sort + two pointersretire an index per step\(n^2\)\(\times 4\)

The middle rung doubles by “\(4\) and a bit” because its \(\lg n\) factor grows slowly with \(n\); the two-pointer rung is clean \(n^2\), so it doubles by exactly \(4\). We turned a cubic algorithm into a quadratic one not by cleverness at the keyboard but by letting the analysis point at the waste.

Knowing when to stop: lower bounds A lower bound is a floor no algorithm can beat. For 2-SUM, \(n\lg n\) is a proven floor in the comparison model (Chapter 13 proves a lower bound of this kind). For 3-SUM, our \(n^2\) is essentially the floor: only log-factor shavings are known (Grønlund–Pettie, 2014), and the 3-SUM conjecture holds that no \(n^{2-\varepsilon}\) algorithm exists. When you can prove you are standing on the floor, stop optimizing — that is what a lower bound buys you.

7. The doubling method, in practice

The doubling experiment of §3 is worth naming as a reusable tool, because it extracts an algorithm’s order of growth with almost no analysis. Run the program at sizes \(n, 2n, 4n, \dots\), form the ratio of consecutive running times, and read the exponent: a ratio settling on \(2^b\) means \(T(n) \sim a\,n^b\). The derivation is the one line from §3 — the constant \(a\) cancels in the ratio and the lower-order terms fade — and it needs no knowledge of the code’s internals.

The same exponent decides whether faster hardware can rescue a slow algorithm. Moore’s Law historically doubled speed every couple of years. For an \(n^b\) algorithm, a \(2\times\) machine grows the feasible input size by only \(2^{1/b}\), and to double the input you need a \(2^b\times\) speedup, i.e. \(b\) doublings of the hardware:

\(n^2\) — quadratic\(4\)\(\times\sqrt{2} \approx 1.41\)\(4\times\)

Linear rides Moore’s Law one for one; cubic falls steadily behind; and exponential is immune — a machine twice as fast lets you add a single element to the input, never double it. The exponent you read off the doubling ratio is literally the number of hardware generations needed to keep pace, which is why a better order of growth beats a faster machine.

Pitfalls of the method The doubling method has two failure modes worth guarding against. First, the input range may be too small for the asymptotics to show — at tiny \(n\) the lower-order terms still matter and the ratio has not settled. Second, you may accidentally measure a lucky best-case input rather than a representative one, reporting an order of growth the algorithm does not have in general. A ratio that keeps climbing instead of settling is a useful alarm: the model is wrong, or there is a performance bug. Always drive the experiment to sizes large enough that consecutive ratios agree.

8. Exercises (with solutions)

  1. A program’s measured running times at \(n = 1000, 2000, 4000, 8000\) are \(0.5, 4.0, 32.0, 256.0\) seconds. What is its order of growth, and how do you know?
    solution Each doubling of \(n\) multiplies the time by \(8\) (\(4.0/0.5 = 8\), \(32/4 = 8\), \(256/32 = 8\)). A doubling ratio of \(8 = 2^3\) means \(T(n) \sim a\,n^3\): the order of growth is \(n^3\) (cubic), since \(T(2n)/T(n) = 2^b\) forces \(b = 3\).
  2. Using the exact frequency \(\binom{n}{3}\), compute the operation count of brute-force 3-SUM at \(n = 500\) and confirm it against the doubling table.
    solution \(\binom{500}{3} = \dfrac{500 \cdot 499 \cdot 498}{6} = \dfrac{124{,}251{,}000}{6} = 20{,}708{,}500\), exactly the operation count the experiment printed for \(n = 500\). The intermediate \(500 \cdot 499 = 249{,}500\) and \(249{,}500 \cdot 498 = 124{,}251{,}000\).
  3. Show that \(\dfrac{n^3}{6} - \dfrac{n^2}{2} + \dfrac{n}{3} \sim \dfrac{n^3}{6}\) directly from the definition of tilde.
    solution Divide by the leading term: \(\dfrac{n^3/6 - n^2/2 + n/3}{n^3/6} = 1 - \dfrac{3}{n} + \dfrac{2}{n^2}\). As \(n \to \infty\) both \(3/n\) and \(2/n^2\) tend to \(0\), so the ratio tends to \(1\), which is exactly the condition \(f \sim g\). Hence the two are asymptotic.
  4. Give the order of growth of each expression, naming the rule you used: (a) \(n\lg n + n^2\); (b) \(n \cdot \lg n\) (a loop of \(n\) iterations, each a binary search); (c) \(n^2 + 100n + 5000\).
    solution (a) \(n^2\), by the sum rule (sequential steps add; the larger term \(n^2\) dominates \(n\lg n\)). (b) \(n\lg n\), by the product rule (nested work multiplies: \(n\) iterations times \(\lg n\) each). (c) \(n^2\), by the sum rule again — the linear and constant terms are dominated, and constant factors are dropped.
  5. Trace the two-pointer scan of fast 3-SUM for \(i = 1\) (\(a[i] = -20\)) on the sorted array \(-40, -20, -10, 0, 5, 10, 30, 40\). Which triple(s) does it find, and how many iterations does it take?
    solution Start \(\texttt{lo} = 2\ ({-}10)\), \(\texttt{hi} = 7\ (40)\). (1) \(s = -20 - 10 + 40 = 10 \gt 0\): hi\(-\!-\) \(\to 6\ (30)\). (2) \(s = -20 - 10 + 30 = 0\): count \((-20, -10, 30)\); lo++\(\to 3\ (0)\), hi\(-\!-\)\(\to 5\ (10)\). (3) \(s = -20 + 0 + 10 = -10 \lt 0\): lo++\(\to 4\ (5)\). (4) \(s = -20 + 5 + 10 = -5 \lt 0\): lo++\(\to 5\). Now \(\texttt{lo} = \texttt{hi} = 5\), the loop stops. It finds the single triple \((-20, -10, 30)\) in four iterations.
  6. On a \(10^9\) ops/sec machine, roughly how long does an \(n^2\) algorithm take at \(n = 10^6\), and an \(n^3\) algorithm at the same \(n\)? Why does this favor improving the order of growth over buying faster hardware?
    solution \(n^2 = 10^{12}\) operations \(/\,10^9\) per second \(= 10^3\) seconds \(\approx 17\) minutes. \(n^3 = 10^{18}/10^9 = 10^9\) seconds \(\approx 31\) years. Buying a \(1000\times\) faster machine would cut the cubic case only from \(31\) years to about \(11\) days, while switching to the \(n^2\) algorithm cuts it to \(17\) minutes on the original hardware: a better exponent dwarfs any realistic constant-factor speedup at scale.
  7. True or false, with justification: “An algorithm that does \(1000\,n\) operations is always faster than one that does \(n^2\) operations.”
    solution False. Order of growth is an asymptotic statement: \(1000\,n\) is \(\Theta(n)\) and beats \(\Theta(n^2)\) for all sufficiently large \(n\), but for small \(n\) the constant matters. Here \(1000\,n \lt n^2\) only when \(n \gt 1000\); for \(n \le 1000\) the quadratic algorithm does fewer operations. Order of growth predicts the winner eventually, not at every size — which is why constant factors and input ranges still deserve attention in practice.
  8. Why do we count inner-loop executions (or array accesses) instead of timing the program with a stopwatch, given that the stopwatch measures the thing we actually care about?
    solution The operation count is exact, reproducible, and machine-independent: it is identical on a laptop, a phone, or a server, and it isolates the algorithm from the implementation, language, and hardware. Wall-clock time folds in all of those plus interference from other processes, so it wobbles (the table’s \(7.83, 8.47, 7.24\)) even when the underlying count is clean (\(8.05, 8.02, 8.01\)). We count to get a precise, portable answer and time only to validate that the count predicts reality — observe, then confirm.
