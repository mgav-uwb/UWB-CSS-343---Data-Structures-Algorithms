Chapter 2 — Space Complexity

CSS 343 — Data Structures, Algorithms & Discrete Mathematics II. Course text, primary reading for Lecture 2.
The second half of the analysis toolkit: a program’s memory cost. We count actual bytes — primitive sizes, struct layout, alignment and padding, the price of a pointer — then separate where memory lives (stack vs heap), watch space grow as a function of \(N\), pay the hidden rent of recursion (call-stack depth), distinguish in-place from auxiliary space, and trade time against space. Order-of-growth notation (\(O,\Omega,\Theta\)) is used only in passing here; its formal definitions and proofs live in the companion Asymptotic Notation reading.

Chapter 1 measured and modelled a program’s running time. Every program also occupies memory, and that cost is just as real: a structure that does not fit in RAM cannot run at any speed. This chapter analyses space exactly the way Chapter 1 analysed time — count first, then keep the order of growth — but it starts one level lower, at the actual bytes. How big is a node? Why is a linked list of \(N\) integers four times heavier than an array of the same \(N\) integers? What does a recursive call cost even when it returns a single number? And when can you spend memory to buy time, or the reverse? These are the questions every data-structure choice in the rest of the course quietly answers.

1. Counting bytes: primitives and sizeof

A data structure occupies memory; the first question of space analysis is simply how much? To answer it we start from the sizes of the primitive types, because everything larger is built from them. In C++ the operator sizeof(x) returns the number of bytes a value or type occupies — a compile-time fact the compiler knows exactly. On a typical 64-bit machine (the LP64 model, which we assume throughout):

typebytestypebytes
char1long8
bool1double8
int4any pointer8

The one that surprises people is the pointer. A pointer is a memory address, and on a 64-bit machine an address is 8 bytes regardless of what it points at: a char*, an int*, and a Node* are all 8 bytes. That single fact drives the cost of every linked structure in the course — each link is an 8-byte tax.

Sizes are a contract with the machine The C++ standard fixes only minimums and orderings (sizeof(char) == 1 by definition; char ≤ short ≤ int ≤ long). The concrete numbers above are what mainstream 64-bit compilers (LP64: Linux, macOS) actually produce, and are what the course’s assignments assume. On a 32-bit target a pointer is 4 bytes; on Windows LLP64 a long is 4. When it matters, do not guess — ask sizeof.

2. Object layout: alignment and padding

A struct is a bundle of fields, so a first guess at its size is the sum of its fields’ sizes. That guess is usually wrong. Consider the linked-list node we will use all term: 

  struct Node {
      int   data;    // 4 bytes
      Node* next;    // 8 bytes
  };

The naive sum is \(4 + 8 = 12\). The real answer is sizeof(Node) == 16. The extra four bytes are padding, inserted to satisfy alignment.

Definition (alignment, padding) A type of size \(s\) usually has an alignment requirement: its address must be a multiple of its alignment (typically \(s\) itself for primitives). To honour every field’s alignment, the compiler inserts unused padding bytes between fields, and pads the whole struct up to a multiple of its largest field’s alignment so that arrays of it stay aligned.

In Node, the pointer next requires an 8-byte alignment, so it cannot start at offset 4 (right after data); the compiler pushes it to offset 8 and fills the gap with four padding bytes:

Field order changes the size

When a struct has several small fields, the padding often can be reduced — for free — just by reordering the declarations. The same three fields in two orders give two different sizes: 

  struct A { char a; int b; char c; };   // sizeof(A) == 12
  struct B { int b; char a; char c; };   // sizeof(B) == 8

The habit is worth keeping: declare fields large-to-small to group same-sized fields and minimise interior padding. It costs nothing and, on a structure you allocate by the million, saves real memory. (It does not rescue Node, whose single int and single pointer cannot pack below the 8-byte slot.) The program below prints all of these numbers, and a Rust and Python view of the same idea: 

// Count the bytes: primitive sizes, and how field ORDER changes a struct's size.
// On a typical 64-bit machine. Compile: g++ -std=c++17 struct-layout.cpp
#include <cstddef>   // offsetof
#include <iostream>

struct Node { int data; Node* next; };      // 4 + 4 pad + 8  = 16
struct A    { char a; int b; char c; };      // 1 +3pad +4 +1 +3pad = 12
struct B    { int b; char a; char c; };      // 4 +1 +1 +2pad = 8  (large->small)

int main() {
    std::cout << "int="    << sizeof(int)    << " ptr=" << sizeof(void*)
              << " double="<< sizeof(double) << "\n";           // 4 8 8
    std::cout << "Node="   << sizeof(Node)                       // 16
              << " (next at offset " << offsetof(Node, next) << ")\n";  // 8
    std::cout << "A=" << sizeof(A) << "  B=" << sizeof(B) << "\n";      // 12  8
}
// Java hides layout: there is no sizeof. Every object carries a HEADER
// (~12-16 bytes) before its fields, and a reference is 4 bytes (compressed
// oops) or 8. So a Node's real footprint is larger than "4 + 8".
class Node {
    int data;        // 4 bytes of field
    Node next;       // a REFERENCE (4 or 8), not an inlined struct
}
// Approximate footprint of one Node on a 64-bit JVM with compressed oops:
//   12 (header) + 4 (int data) + 4 (ref next) = 20, padded up to 24.
// The point: managed runtimes add per-object overhead that C++/Rust do not,
// so an object-per-element list is even costlier than the raw 16 bytes.
// (Measure precisely with the JOL tool: ClassLayout.parseClass(Node.class).)
# Python has no structs and no fixed sizes: everything is a heap object with a
# header. sys.getsizeof reports one object's bytes (not what it references).
import sys

print(sys.getsizeof(0))          # a small int: ~28 bytes, not 4
print(sys.getsizeof(3.14))       # a float:     ~24 bytes, not 8

# A "node" is a class instance; each attribute is a REFERENCE to another object.
class Node:
    __slots__ = ("data", "next")   # without __slots__, add a per-instance __dict__
    def __init__(self, data):
        self.data = data
        self.next = None

# One Node object plus the int it points at costs far more than 16 bytes.
# Lesson: in a managed language you cannot "count bytes" the way sizeof does in
# C++/Rust; the runtime charges a header per object and boxes every primitive.
// Rust exposes layout like C++: size_of and align_of are compile-time facts.
// #[repr(C)] forces C's field order/padding; the default repr may reorder
// fields to shrink padding automatically (a nicety C++ does NOT have).
use std::mem::{size_of, align_of};

#[repr(C)]
struct Node { data: i32, next: *const Node }   // 4 + 4 pad + 8 = 16

#[repr(C)] struct A { a: u8, b: i32, c: u8 }    // 12  (like C++ struct A)
#[repr(C)] struct B { b: i32, a: u8, c: u8 }    // 8   (large -> small)

fn main() {
    println!("i32={} ptr={}", size_of::<i32>(), size_of::<*const Node>()); // 4 8
    println!("Node={} align={}", size_of::<Node>(), align_of::<Node>());   // 16 8
    println!("A={}  B={}", size_of::<A>(), size_of::<B>());                 // 12 8
}

Managed languages hide — and inflate — the count Only C, C++, and Rust let you count bytes this directly. In Java and Python there is no sizeof, and every object carries a runtime header: a Java object header is 12–16 bytes before any field, and a Python int is roughly 28 bytes, not 4. So the per-node overhead we compute here is a floor — on a managed runtime an object-per-element list is heavier still. This is why bulk numeric work in those languages leans on packed arrays (Java primitive arrays, NumPy) rather than object graphs. See Sedgewick & Wayne §1.4 for the JVM accounting.

3. Stack vs heap, and the price of a pointer

Knowing a struct’s size is half the story; the other half is where those bytes live. A program has two very different pools of memory, and dynamic structures depend on the difference. 

  Node  a;              // on the STACK: automatic, freed at scope exit
  Node* p = new Node;   // the node is on the HEAP: freed by delete p
Definition (stack, heap) The stack holds a function’s local variables; it grows and shrinks automatically as calls enter and leave, and its storage is freed the instant a variable goes out of scope. The heap is the pool that new / delete (C++), malloc / free (C), or a garbage collector manage; a heap object lives until it is explicitly (or automatically) freed. Dynamic structures — linked lists, trees, hash tables — grow on the heap.

A linked list makes the split concrete. The list variable itself is one 8-byte head pointer on the stack; every node is a separate heap allocation, one new apiece. Forgetting to delete the nodes leaks the heap memory (a leak the course’s Valgrind checks catch), while the head pointer on the stack cleans itself up.

Each pointer is overhead

Return to the 16-byte Node holding one int of payload. Of its 16 bytes, only 4 carry data; the other 12 — four padding plus the 8-byte next pointer — are pure structural overhead:

part of a Nodebytesrole
data4useful payload
padding4alignment waste
next8the link (overhead)
total1675% overhead

Three quarters of a linked-list-of-int is machinery, not data. The lesson is not “never use linked lists” — a larger payload amortises the pointer, and lists buy \(O(1)\) splicing — but that the structure you choose has a memory price, not only a time price. We now watch that price grow with the number of elements.

4. Space as a function of N

Space, like time, is a function of the input size \(N\), and we summarise it the same way: count the bytes, then keep the order of growth, dropping constants and lower-order terms. Take two ways to hold \(N\) integers — a contiguous array, and a linked list of \(N\) nodes:

structurebytes for \(N\) intsorder
array int a[N]\(4N\)\(\Theta(N)\)
linked list of \(N\) Nodes\(16N\)\(\Theta(N)\)

Both grow linearly, so both are \(\Theta(N)\): the order of growth cannot tell them apart. Yet the array uses one quarter of the memory, because its constant is 4 bytes per element against the list’s 16 — exactly the padding-plus-pointer overhead from §3, now multiplied by \(N\). This is the recurring theme of the chapter: same order, different constant. (The array also wins a second, unmodelled prize — its elements are contiguous, so it is far friendlier to the cache; §6 returns to this.)

Notation, in passing We write memory bounds with the same \(O\) (upper), \(\Omega\) (lower), and \(\Theta\) (tight) that Chapter 1 used for time; a \(\Theta(N)\) space claim means the byte count is squeezed between two constant multiples of \(N\) for large \(N\). The formal set-definitions, the proofs, and the worked exercises are in the companion Asymptotic Notation reading — this chapter uses the notation but does not re-derive it.

The vocabulary scales past linear. A single scalar is \(\Theta(1)\); an array or list of \(N\) elements is \(\Theta(N)\); an \(N \times N\) matrix — an adjacency matrix for a graph, say — is \(\Theta(N^2)\). That quadratic will matter in the graph unit: a matrix stores a cell for every possible edge and so wastes space on a sparse graph, where an adjacency list keeps only the edges that exist, at \(\Theta(V+E)\) (Chapter 8).

memory usedexampleorder
one variablea loop counter, an accumulator\(\Theta(1)\)
proportional to \(N\)array or list of \(N\) items\(\Theta(N)\)
proportional to \(N^2\)\(N \times N\) adjacency matrix\(\Theta(N^2)\)

Input space vs auxiliary space

When we analyse an algorithm (rather than a structure), we usually do not charge it for the input it was handed — that memory belongs to the caller. What we count is the extra memory the algorithm allocates on top:

Definition (auxiliary space, in-place) The auxiliary space of an algorithm is the memory it uses beyond its input: temporaries, recursion stack, and any allocated scratch. An algorithm is in-place if its auxiliary space is \(\Theta(1)\) — a constant number of extra cells, independent of \(N\) (recursion-stack space, in the strict definition, may be \(\Theta(\log N)\)).

The distinction is why an algorithm can be slow in time yet cheap in space. A summation reads all \(N\) inputs but allocates only one accumulator, so it is \(\Theta(1)\) auxiliary space: 

  long sum(const vector<int>& a) {
      long s = 0;               // one accumulator — no allocation
      for (int x : a) s += x;
      return s;                 // Theta(1) auxiliary space
  }

Chapter 1’s brute-force 3-sum is the striking case: \(\Theta(N^3)\) time but only \(\Theta(1)\) extra space — three loop counters and nothing allocated. Time and space are independent budgets, a point §6 makes with a table. First, though, the one source of auxiliary space that is easy to forget because you never wrote a new: the recursion itself.

5. The space cost of recursion

A recursive function allocates no heap memory, yet it is not free in space. Each call in progress needs its own stack frame — room for its parameters, local variables, and return address — and those frames coexist until the calls return. The number of frames stacked at the deepest moment is the recursion depth, and it is the algorithm’s auxiliary space. 

  long fact(int n) {
      if (n <= 1) return 1;      // base case
      return n * fact(n - 1);    // one frame stays live per level
  }

Calling fact(n) stacks \(n\) frames before the base case returns, even though the result is a single number. Its auxiliary space is \(\Theta(n)\), all of it on the stack.

The general rule is recursion depth \(=\) auxiliary stack space, and the depth depends on the structure being recursed over. A linear recursion over \(N\) items is \(\Theta(N)\) deep. A traversal of a tree recurses to the tree’s height, so its stack is \(\Theta(h)\): \(\Theta(\log N)\) for a balanced tree of \(N\) nodes, but \(\Theta(N)\) for a degenerate one that has collapsed to a chain — a motivation for the balanced trees of Chapter 4. A divide-and-conquer recursion that halves its input each level is only \(\Theta(\log N)\) deep; mergesort’s recursion (§6) is exactly this.

Deep recursion can overflow the stack The call stack is a bounded region (often a few megabytes), so unbounded depth is not merely slow — it crashes with a stack overflow. Recursing \(\Theta(N)\) deep on a million-element chain can exhaust it, whereas an iterative version, or a balanced structure that caps depth at \(\Theta(\log N)\), is safe. Some compilers convert a tail call (where the recursive call is the last action) into a loop, reclaiming the frame and restoring \(\Theta(1)\) space; C++ does not guarantee it, so do not rely on it. Erickson’s Algorithms is a good reference for reasoning about recursion depth as a resource.

6. Auxiliary space and time–space tradeoffs

With input, auxiliary, and stack space defined, we can read off the memory cost of two representative algorithms and then set space against time.

\(\Theta(N)\) auxiliary space: mergesort

Mergesort sorts by splitting the array in two, sorting each half recursively, and merging the two sorted halves. Splitting is free, but merging cannot be done in place with a simple linear scan: it needs a scratch buffer to hold the merged output before copying it back. 

// Merge two adjacent sorted runs a[lo..mid] and a[mid+1..hi] into sorted order,
// using aux[] as scratch. Every element is copied once: Theta(n).
void merge(int a[], int lo, int mid, int hi, int aux[]) {
    for (int k = lo; k <= hi; k++) aux[k] = a[k];    // copy into scratch
    int i = lo, j = mid + 1;
    for (int k = lo; k <= hi; k++) {
        if (i > mid)              a[k] = aux[j++];    // left run exhausted
        else if (j > hi)          a[k] = aux[i++];    // right run exhausted
        else if (aux[j] < aux[i]) a[k] = aux[j++];    // take the smaller front
        else                      a[k] = aux[i++];    // ties take left: STABLE
    }
}

// Top-down mergesort: sort each half, then merge. T(n) = 2T(n/2) + Theta(n).
void mergesort(int a[], int lo, int hi, int aux[]) {
    if (hi <= lo) return;                 // base case: 0 or 1 element
    int mid = lo + (hi - lo) / 2;
    mergesort(a, lo, mid, aux);           // sort the left half
    mergesort(a, mid + 1, hi, aux);       // sort the right half
    merge(a, lo, mid, hi, aux);           // combine
}

        return

    merge(a, lo, mid, hi, aux)            # combine
// Merge sorted runs a[lo..=mid] and a[mid+1..=hi] using aux as scratch. O(n).
fn merge(a: &mut [i32], lo: usize, mid: usize, hi: usize, aux: &mut [i32]) {
    aux[lo..=hi].copy_from_slice(&a[lo..=hi]);       // copy into scratch
    let (mut i, mut j) = (lo, mid + 1);
    for k in lo..=hi {
        if i > mid {                                 // left run exhausted
            a[k] = aux[j]; j += 1;
        } else if j > hi {                           // right run exhausted
            a[k] = aux[i]; i += 1;
        } else if aux[j] < aux[i] {                  // take the smaller front
            a[k] = aux[j]; j += 1;
        } else {                                     // ties take left: STABLE
            a[k] = aux[i]; i += 1;
        }
    }
}

// Top-down mergesort. T(n) = 2T(n/2) + O(n).
fn mergesort(a: &mut [i32], lo: usize, hi: usize, aux: &mut [i32]) {
    if hi <= lo { return; }               // base case: 0 or 1 element
    let mid = lo + (hi - lo) / 2;
    mergesort(a, lo, mid, aux);           // sort the left half
    mergesort(a, mid + 1, hi, aux);       // sort the right half
    merge(a, lo, mid, hi, aux);           // combine
}

The aux buffer holds one copy of the range being merged; sized once to the full array, it is \(4N\) bytes for \(N\) ints, so mergesort’s auxiliary space is \(\Theta(N)\). The recursion adds a stack of depth \(\lceil \log_2 N \rceil\), i.e. \(\Theta(\log N)\), which the \(\Theta(N)\) buffer dominates. Two contrasts are worth drawing out. First, the time is \(\Theta(N \log N)\) — many merges — while the peak memory is only \(\Theta(N)\): the merges happen one at a time as the recursion unwinds, so the buffers do not coexist. Second, quicksort (Chapter 13) partitions in place and so uses only \(\Theta(\log N)\) space — a genuine space advantage that is one reason it is often preferred despite mergesort’s cleaner worst-case time.

The lecture’s mergesort visualiser makes the two costs visible side by side: the recursion brackets nest to depth \(\log_2 N\) (the \(\Theta(\log N)\) stack), while an auxiliary-bytes gauge rises and falls with the live buffer, peaking at \(4N\) at the top-level merge — time \(N\log N\), but peak space only \(N\).

\(\Theta(n^2)\) space: grid exploration

Breadth-first search over an \(n \times n\) grid (flood-fill, maze shortest-paths; Chapter 8) expands outward in waves from a start cell, keeping a queue of the current frontier and a visited grid so it never processes a cell twice. The queue holds only the frontier — on a grid, an \(O(n)\) wavefront — but the visited grid is a full \(n \times n\) array, allocated up front regardless of the maze, and it dominates: 

  vector<vector<char>> visited(n, vector<char>(n, 0));  // n*n bytes: Theta(n^2)
  queue<pair<int,int>> frontier;                       // O(n) wavefront

A deliberate choice here is char per cell rather than vector<bool>, which would bit-pack to \(n^2/8\) bytes and hide the cost. What you store per cell decides the constant — but not the order, which stays \(\Theta(n^2)\):

stored per cellbytestotalbuys you
visited flag (char)1\(n^2\)reachable?
distance (int)4\(4n^2\)hops to each cell
predecessor id (int)4\(4n^2\)the actual path
came-from direction (char)1\(n^2\)the path, 4× cheaper

This is the array-versus-list lesson again — same \(\Theta\), different constant — now inside a single algorithm. And it warns of a real trap: a char distance overflows at 127, but a snaking corridor can make a cell’s distance approach \(n^2\), so distance needs the wider int. Quadratic space is also where memory binds before time does: a \(100{,}000 \times 100{,}000\) grid is \(10^{10}\) cells, which at 4 bytes each (an int per cell) is \(4 \times 10^{10}\) bytes — about 40 GB, beyond most machines, no matter how fast the processor. Dynamic-programming tables (Chapters 14–15) hit the same wall.

Time and space are independent budgets

A cost is not “efficient” until you say which resource. The two columns below vary independently: fast can be memory-hungry, and slow can be memory-thrifty.

algorithmtimeauxiliary space
array sum\(\Theta(N)\)\(\Theta(1)\)
brute-force 3-sum\(\Theta(N^3)\)\(\Theta(1)\)
mergesort\(\Theta(N \log N)\)\(\Theta(N)\)
quicksort\(\Theta(N \log N)\) avg\(\Theta(\log N)\)
BFS on an \(n \times n\) grid\(\Theta(n^2)\)\(\Theta(n^2)\)

The 3-sum row is the extreme for time (cubic) with trivial space; the last row is the extreme for space — a table with one entry per subset of \(N\) items has \(2^N\) entries, so its memory is exponential, which is what makes such exact methods intractable past \(N \approx 20\) regardless of speed.

Trading one budget for the other

Because the budgets are separate, you can often spend one to save the other. The choice is a recurring design lever in the rest of the course:

tradeoffspendto savewhere
hash tableextra spacelookup time (to \(O(1)\))Chapter 7

Constants matter for memory in practice Order of growth decides feasibility at scale, but for memory the constant often decides real speed, through the cache. A contiguous array is read in cache-friendly runs; a pointer-chasing list can miss the cache on every hop. Two structures with identical \(\Theta(N)\) space can differ several-fold in wall-clock time for this reason alone — the reason std::vector usually beats std::list. Sedgewick & Wayne and Cormen et al. (CLRS) both stress that the memory access pattern, not just the byte count, drives performance.

7. Exercises (with solutions)

  1. Give sizeof for each on a 64-bit LP64 machine, with the padding shown: struct P { char a; double b; }; and struct Q { double b; char a; };.
    solution Both are 16. In P: a at offset 0, then 7 padding bytes so the double lands on an 8-aligned offset 8, giving \(1 + 7 + 8 = 16\). In Q: the double fills 0–7, a takes byte 8, then 7 tail-padding bytes bring the size to a multiple of 8 (the struct’s alignment): \(8 + 1 + 7 = 16\). Reordering does not help here because the 8-byte alignment forces a full 8-byte tail either way — the same reason Node cannot shrink.
  2. A program stores \(N = 10^6\) integers. Give the total bytes for (a) an int array and (b) a singly linked list of Nodes, and the ratio.
    solution (a) \(4N = 4{,}000{,}000\) bytes \(\approx 3.8\) MB. (b) \(16N = 16{,}000{,}000\) bytes \(\approx 15.3\) MB. The ratio is \(16N / 4N = 4\): the list costs four times as much for the same data, entirely from the per-node padding (4) plus pointer (8). Both are \(\Theta(N)\); the factor of 4 is the constant the order of growth discards.
  3. State the auxiliary space of each, with a one-line reason: (a) iterative fact(n) using a loop; (b) recursive fact(n); (c) an in-order traversal of a balanced binary search tree of \(N\) nodes; (d) the same traversal of a tree degenerated to a chain.
    solution (a) \(\Theta(1)\) — one accumulator, one frame. (b) \(\Theta(n)\) — \(n\) live stack frames at the deepest point. (c) \(\Theta(\log N)\) — the recursion goes as deep as the height, which is \(\Theta(\log N)\) when balanced. (d) \(\Theta(N)\) — the height is \(N-1\), so the stack is as deep as the chain is long (and may overflow). Same code as (c); the input’s shape sets the space.
  4. True or false, with justification: “Any algorithm that runs in \(\Theta(N^2)\) time must use \(\Theta(N^2)\) space.”
    solution False. Time and space are independent budgets. Selection sort runs in \(\Theta(N^2)\) time but is in-place, \(\Theta(1)\) auxiliary space; brute-force 3-sum is even \(\Theta(N^3)\) time with \(\Theta(1)\) space. Time bounds the number of operations, not the memory allocated. (A partial converse does hold: an algorithm that allocates and touches \(\Theta(N^2)\) space must spend at least \(\Theta(N^2)\) time doing so — you cannot use more memory than you have time to address.)
  5. You have a struct with fields, in this order: char flag; double value; int id; char kind;. Give its size as written, then reorder the fields to minimise it, and give the new size.
    solution As written: flag at 0; 7 pad; value at 8–15; id at 16–19; kind at 20; then 3 tail pad to a multiple of 8, total \(1+7+8+4+1+3 = \mathbf{24}\). Reordered large-to-small — double value; int id; char flag; char kind; — gives value 0–7, id 8–11, flag 12, kind 13, then 2 tail pad, total \(8+4+1+1+2 = \mathbf{16}\). A 33% reduction with no change to the data, exactly the A-vs-B effect of §2.
  6. Mergesort is \(\Theta(N \log N)\) time and \(\Theta(N)\) auxiliary space, yet the recursion goes \(\Theta(\log N)\) levels deep and every level does \(\Theta(N)\) work on \(\Theta(N)\) total buffer. Why is the peak memory \(\Theta(N)\) and not \(\Theta(N \log N)\)?
    solution Peak memory counts what is live simultaneously, not the total ever allocated. The merges execute one at a time as the recursion unwinds; a merge’s scratch buffer is freed (or reused) before the next merge at that level begins, so at any instant only one buffer of total size \(\le N\) is live, plus the \(\Theta(\log N)\) stack of frames. Time sums over all levels (\(\Theta(N)\) per level \(\times \log N\) levels), but space is a high-water mark, so the two need not match.
  7. BFS on an \(n \times n\) grid, as written, uses \(\Theta(n^2)\) space for its fixed visited array. A colleague claims you can make it use only \(\Theta(n)\). Under what condition is that possible, and what do you give up?
    solution You can drop the fixed \(n^2\) array if you only need to know reachability or distance, not the path: BFS only ever steps between adjacent layers, so keeping just the previous and current frontiers (Korf’s frontier search) avoids revisiting, and on a grid a frontier is \(O(n)\) wide. What you give up is the path: reconstructing it needs a predecessor per reached cell, which is \(\Theta(n^2)\) again in the worst case. So space complexity here is a property of the implementation and the question asked, not of “BFS” in the abstract — the same order-of-growth / constant / requirement layering as the rest of the chapter.
