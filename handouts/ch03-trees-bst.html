Chapter 3 — Trees & Binary Search Trees

CSS 343 — Data Structures, Algorithms & Discrete Mathematics II. Course text, primary reading for Lecture 3.
The first real data structure of the term. Tree vocabulary (root, leaf, depth, height, subtree) and the recursive definition that drives every algorithm here; binary trees, their shapes, and their traversals, with expression trees as a first payoff; then the binary search tree as an ordered symbol table — search, insert, the order-based operations (min/max, floor/ceiling, rank/select, range), and Hibbard deletion — ending in the one analysis that governs them all: every operation costs \(\Theta(\text{height})\), a random tree is shallow (\(\sim 1.39\log_2 n\)) but a sorted insert is a \(\Theta(n)\) path. Chapters 1 and 2 built an analysis toolkit — \(O\), \(\Theta\), \(\Omega\), and the accounting of time and space. This chapter spends it on a real structure. A tree stores data not in a line but in a branching hierarchy, and a binary search tree keeps that hierarchy ordered so that a lookup follows a single root-to-leaf path instead of scanning. That is the whole promise: the work of any BST operation is the length of one path, so the height of the tree is the entire story. The catch, and the reason Chapters 4 and 5 exist, is that the same keys can build a tree as short as \(\log_2 n\) or as tall as \(n-1\), and the shape is decided by nothing more than the order the keys arrive. We finish by measuring exactly how tall a tree you should expect.

1. Trees: vocabulary and the recursive definition

A tree is a set of nodes joined by links (edges): each link either is null or points to another node, and the structure has no cycles. One node, the root, has no parent; every other node has exactly one parent. The picture is a family tree drawn upside down — the root at the top, links fanning downward.

Definition (tree vocabulary) In a tree, a node’s children are the nodes its links point to, and it is their parent; nodes with the same parent are siblings. A node with no children is a leaf (or external node); a node with at least one child is internal. The ancestors of a node are itself, its parent, its parent’s parent, and so on to the root; the descendants are itself and everything below it. A subtree is a node together with all of its descendants — a link “points to a subtree.”

The definition that actually drives the code is recursive, mirroring the structure itself.

Definition (a tree, recursively) A tree is either empty, or a root node together with zero or more disjoint subtrees, each of which is itself a tree. The root’s links point to the roots of those subtrees.

Almost every algorithm in this chapter has the same shape as that definition: do something to the node, then recurse on its subtrees. When a structure is defined recursively, recursion is the natural way to walk it.

Depth, level, and height

Definition (depth and height) The depth (or level) of a node is the number of links on the path from the root to it; the root has depth \(0\). The height of a tree is the maximum depth of any node — the number of links on the longest root-to-leaf path. An empty tree has height \(-1\); a single node has height \(0\).

Conventions differ — some texts put the root at level \(1\), and some measure height in nodes rather than links. We fix root depth \(0\), height in links and hold to it. The choice matters only in that you must state it and stay consistent; the reason to care is that height is the quantity that determines every BST operation’s worst-case cost (§7). Keep an eye on it from here on.

Trees are everywhere in this book The tree is the backbone of the whole middle of the course. The search trees of this chapter become balanced in Chapters 4 (AVL) and 5 (2–3, red-black, B-trees); the complete trees of §2 become heaps and priority queues in Chapter 6; Huffman coding builds an optimal tree in Chapter 12; and the recursion-tree method for analyzing divide-and-conquer (Chapter 13) draws a tree to count work. Learn the vocabulary once here and it pays off five more times.

General trees

Nothing so far limited the number of children a node may have. A general tree lets a node hold any number of children, stored in a list, and its algorithms recurse over that list — exactly the recursive definition, made concrete: 

struct TreeNode {
    Object            item;
    vector<TreeNode*> children;   // 0, 1, 2, ... many
};

int countNodes(TreeNode* t) {
    if (!t) return 0;
    int n = 1;
    for (TreeNode* c : t->children) n += countNodes(c);   // recurse on each subtree
    return n;
}

Variable-length child lists are flexible but awkward. A standard trick re-encodes any general tree with just two links per node — firstChild (go down to the leftmost child) and nextSibling (go across to the next sibling). Reading firstChild as “left” and nextSibling as “right,” any general tree becomes a binary tree. That is one reason the rest of the chapter, and much of the course, studies the binary case: it loses no generality and gains a fixed, two-pointer node.

2. Binary trees and their shapes

A binary tree restricts every node to at most two children, distinguished as the left and right child (the distinction matters even when a node has only one child). Its node is the workhorse of the chapter: 

struct Node {
    Object item;
    Node*  left;
    Node*  right;
};

The same three-field node underlies expression trees (§3), binary search trees (§4 on), heaps (Chapter 6), and Huffman trees (Chapter 12). A few shape names recur often enough to be worth fixing.

termdefinition
fullevery node has 0 or 2 children (never exactly one)
perfectfull and every leaf at the same depth
completeevery level filled except possibly the last, which is packed to the left
balancedat every node, the two subtree heights differ by at most 1

These are nested but not identical. Every perfect tree is complete, and every complete tree is balanced — but not conversely: a balanced tree need not be complete, and a complete tree need not be perfect. The subtle implication, complete \(\Rightarrow\) balanced, is what lets a heap (a complete tree, Chapter 6) guarantee logarithmic height for free.

How tall can a binary tree be?

Fix the number of nodes \(n\). The same \(n\) nodes can be arranged into wildly different heights, and the two extremes bracket everything this chapter cares about.

Proposition (height bounds) A binary tree on \(n \ge 1\) nodes has height \(h\) with \[ \lfloor \log_2 n \rfloor \;\le\; h \;\le\; n-1. \] The lower bound is achieved by a perfect/complete tree; the upper bound by a degenerate path in which every node has one child.
Proof. A tree of height \(h\) has at most \(2^d\) nodes at depth \(d\), so \(n \le 2^0 + 2^1 + \cdots + 2^h = 2^{h+1}-1\). Rearranging, \(2^{h+1} \ge n+1\), hence \(h \ge \log_2(n+1) - 1\), which for integer \(h\) gives \(h \ge \lfloor \log_2 n \rfloor\); a perfect tree fills every level and attains it. For the upper bound, each of the \(h+1\) depths \(0,1,\ldots,h\) on the longest path holds at least one node, so \(n \ge h+1\), i.e. \(h \le n-1\); a path where each node has exactly one child puts one node at each depth and attains it. ∎

This is the whole lecture in one line: a binary tree’s height ranges from \(\log n\) to \(n\), and which you get — set by the tree’s shape, not by \(n\) — decides the cost of every operation. Everything from §4 onward is an effort to land near the short end.

Storing a tree in a flat array A binary tree can live in an array with no pointers at all, addressed by position: the root at index \(0\), and node \(i\)’s children at \(2i+1\) and \(2i+2\), its parent at \(\lfloor (i-1)/2 \rfloor\). Contiguous and cache-friendly — but the array must reach the largest index used, so a complete tree fills \(0 \ldots n-1\) exactly while a path scatters nodes to indices \(0,1,3,7,\ldots\), needing \(\sim 2^n\) slots. The layout pays off only for (near-)complete trees, which is precisely the heap of Chapter 6.

3. Traversals and expression trees

To traverse a tree is to visit every node exactly once, in some order. Three orders come straight from the recursive definition and differ only in when a node is visited relative to its two subtrees:

  • pre-order — node, then left, then right;
  • in-order — left, then node, then right;
  • post-order — left, then right, then node.

A fourth, level-order, visits the tree top-to-bottom and left-to-right using a queue rather than recursion — it is exactly the breadth-first search we return to with graphs in Chapter 8. The three recursive orders are one piece of code with the visit line slid to three positions: 

// The three recursive traversals are ONE recursion; only the line where you
// visit(t) moves. In-order of a BST visits the keys in SORTED order.
void inorder(Node* t) {
    if (!t) return;
    // visit(t);            // PRE-order:  node BEFORE its subtrees
    inorder(t->left);
    visit(t);               // IN-order:   node BETWEEN the subtrees (sorted!)
    inorder(t->right);
    // visit(t);            // POST-order: node AFTER its subtrees
}
// The three recursive traversals are ONE recursion; only the line where you
// visit(t) moves. In-order of a BST visits the keys in SORTED order.
void inorder(Node t) {
    if (t == null) return;

    inorder(t.left);
    visit(t);               // IN-order:   node BETWEEN the subtrees (sorted!)
    inorder(t.right);
    // visit(t);            // POST-order: node AFTER its subtrees
}

Expression trees

A first use of a binary tree is to represent an arithmetic expression: operators are internal nodes, operands are leaves, and the shape encodes precedence. Because the shape carries the meaning, the two ways of grouping \(2+3\times 4\) are two different trees with different values — parentheses and precedence rules exist only to pick the tree, and once the tree is fixed there is nothing left to interpret.

Evaluation is then a post-order computation: a leaf is its own value, and an operator combines the already-evaluated values of its two subtrees. 

double eval(Node* t) {
    if (isLeaf(t)) return t->value;     // operand
    double a = eval(t->left);           // evaluate the subtrees first
    double b = eval(t->right);
    return apply(t->op, a, b);          // then combine: a (op) b
}

No precedence rules appear in the code — the structure already encoded them. On the right tree above, eval computes \(3\times 4 = 12\) at the lower node, then \(2+12 = 14\) at the root: values bubble up, children before parents, which is exactly post-order.

4. The binary search tree: an ordered symbol table

A symbol table is a collection of (key, value) pairs supporting insert and search-by-key — a map. A binary search tree is a symbol table kept in a binary tree under one ordering rule, which additionally makes it ordered: it can answer questions about the ranking of keys, not just membership.

Definition (BST ordering invariant) A binary search tree is a binary tree whose nodes carry comparable, distinct keys (each with an associated value), such that at every node \(x\): every key in \(x\)’s left subtree is less than \(x\)’s key, and every key in its right subtree is greater than \(x\)’s key.

The word every is the whole point. The invariant is recursive: it holds not just for the root but at each node in every subtree. That is strictly stronger than “each node’s immediate children are on the correct side,” and it is what makes search a guided descent — at each node one comparison eliminates an entire subtree — and what makes in-order traversal emit the keys in sorted order.

Following Sedgewick, we give each node a size field: the number of nodes in the subtree rooted there. It costs one integer per node, is trivially maintained on the way up during insert and delete, and it is what powers the rank and select operations of §5. 

// A BST node (Sedgewick's ordered symbol table): key, value, two links, and a
// subtree-size count. get returns a pointer to the value or null; put inserts.
struct Node {
    Key   key;   Value val;
    Node* left  = nullptr;
    Node* right = nullptr;
    int   size  = 1;                    // # nodes in this subtree
};
int size(Node* t) { return t ? t->size : 0; }      // null-safe

Value* get(Node* t, const Key& key) {
    if (!t) return nullptr;             // miss: fell off a null link
    if (key == t->key) return &t->val;  // hit
    return key < t->key ? get(t->left, key) : get(t->right, key);
}
Node* put(Node* t, const Key& key, const Value& val) {
    if (!t) return new Node{key, val};  // grow at the null link
    if      (key < t->key) t->left  = put(t->left,  key, val);
    else if (key > t->key) t->right = put(t->right, key, val);
    else t->val = val;                  // equal key: update the value
    t->size = 1 + size(t->left) + size(t->right);   // recount on the way up
    return t;
}
// A BST node: key, value, two links, and a subtree-size count. get returns the
// value or null; put inserts, or updates the value on an equal key.
class Node {
    Key key; Value val; Node left, right; int size = 1;
    Node(Key k, Value v) { key = k; val = v; }
}
int size(Node t) { return t == null ? 0 : t.size; }   // null-safe

Value get(Node t, Key key) {
    if (t == null) return null;                 // miss: fell off a null link
    int c = key.compareTo(t.key);
    if (c == 0) return t.val;                    // hit
    return c < 0 ? get(t.left, key) : get(t.right, key);
}
Node put(Node t, Key key, Value val) {
    if (t == null) return new Node(key, val);    // grow at the null link
    int c = key.compareTo(t.key);
    if      (c < 0) t.left  = put(t.left,  key, val);
    else if (c > 0) t.right = put(t.right, key, val);
    else t.val = val;                            // equal key: update
    t.size = 1 + size(t.left) + size(t.right);   // recount on the way up
    return t;
}
# A BST node: key, value, two links, and a subtree-size count. get returns the
# value or None; put inserts, or updates the value on an equal key.
class Node:
    def __init__(self, key, val):
        self.key, self.val = key, val
        self.left = self.right = None
        self.size = 1                       # nodes in this subtree

def size(t):
    return t.size if t else 0               # null-safe

def get(t, key):
    if t is None:    return None            # miss: fell off a null link
    if key == t.key: return t.val           # hit
    return get(t.left, key) if key < t.key else get(t.right, key)

def put(t, key, val):
    if t is None:    return Node(key, val)  # grow at the null link
    if   key < t.key: t.left  = put(t.left,  key, val)
    elif key > t.key: t.right = put(t.right, key, val)
    else:             t.val = val           # equal key: update
    t.size = 1 + size(t.left) + size(t.right)   # recount on the way up
    return t
// A BST node with a subtree-size count. Links are Option<Box<Node>>, so the
// tree owns its children and drops them automatically (no manual free).
struct Node {
    key: i64, val: String,
    left: Option<Box<Node>>, right: Option<Box<Node>>,
    size: usize,
}
fn size(t: &Option<Box<Node>>) -> usize { t.as_ref().map_or(0, |n| n.size) }

fn get<'a>(t: &'a Option<Box<Node>>, key: i64) -> Option<&'a str> {
    match t {
        None => None,                               // miss: null link
        Some(n) if key == n.key => Some(&n.val),    // hit
        Some(n) if key <  n.key => get(&n.left, key),
        Some(n) => get(&n.right, key),
    }
}
fn put(t: Option<Box<Node>>, key: i64, val: String) -> Option<Box<Node>> {
    let mut n = match t {
        None => return Some(Box::new(               // grow at the null link
            Node { key, val, left: None, right: None, size: 1 })),
        Some(n) => n,
    };
    if      key < n.key { n.left  = put(n.left.take(),  key, val); }
    else if key > n.key { n.right = put(n.right.take(), key, val); }
    else { n.val = val; }                           // equal key: update
    n.size = 1 + size(&n.left) + size(&n.right);
    Some(n)
}

get walks down comparing keys: equal is a hit, and falling off a null link is a miss. put searches the same way and, on a miss, hangs a new node at the null link where the search ended; on the way back up it resets each parent’s child link and recomputes each size. An equal key updates the value in place — keys stay unique. Each is \(\Theta(\text{depth})\): the work is one root-to-leaf path.

5. Ordered operations

Keeping the keys ordered buys far more than membership. The operations below are what make a BST an ordered symbol table — and what a hash table (Chapter 7), for all its \(O(1)\) lookups, cannot do. Each is a short recursion running in \(\Theta(\text{height})\).

  • min / max — follow left (resp. right) links to the end.
  • floor(k) — the largest key \(\le k\); and ceiling(k) — the smallest key \(\ge k\).
  • select(i) — the key of rank \(i\) (the \(i\)-th smallest, \(0\)-based); and rank(k) — the number of keys strictly less than \(k\). These use the size field.
  • range(lo, hi) — all keys in \([lo, hi]\), in sorted order.

floor is the instructive one, because the answer is not always on the search path. If \(k\) equals the node’s key, that is the floor. If \(k\) is smaller, the floor must lie in the left subtree. If \(k\) is larger, the current node is a candidate, but a still-larger key \(\le k\) might sit in the right subtree, so we look there and keep the node only if the right subtree offers nothing. 

Node* floor(Node* t, const Key& k) {   // largest key <= k
    if (!t) return nullptr;
    if (k == t->key) return t;
    if (k < t->key)  return floor(t->left, k);   // answer is strictly left
    Node* r = floor(t->right, k);               // a closer key on the right?
    return r ? r : t;                           // else this node is the floor
}

select and rank are inverses, and both steer by subtree sizes. Let \(s\) be the size of the node’s left subtree. For select(i): if \(i < s\) the answer is the \(i\)-th key on the left; if \(i > s\) it is the \((i-s-1)\)-th key on the right; if \(i = s\) it is this node. For rank(k): recurse left if \(k\) is smaller, return \(s\) on a match, and return \(s + 1 + \texttt{rank}(\text{right}, k)\) if \(k\) is larger. Without the size field these would be linear; with it they are \(\Theta(\text{height})\).

range(lo, hi) is an in-order traversal with pruning: recurse left only if \(lo\) could lie there, emit the node if it is in range, and recurse right only if \(hi\) could lie there. It runs in time proportional to the height plus the number of keys reported — the one operation whose cost is not height alone.

Example (ordered queries on the §4 tree) On the ten-key tree (in-order \(20,30,35,40,50,60,65,70,80,90\)): min = 20, max = 90. floor(45): from \(50\) go left to \(30\); \(45 > 30\) so \(30\) is a candidate, look right into \(\{40, 35\}\); at \(40\), \(45 > 40\) with an empty right, so \(40\) wins — floor(45) = 40. Symmetrically ceiling(45) = 50. Since the sorted order is \(20,30,35,40,50,\ldots\), select(3) = 40 and rank(60) = 5 (the five keys \(20,30,35,40,50\) precede it). Finally range(35, 65) reports \(35, 40, 50, 60, 65\).

6. Deletion (Hibbard)

Removing a key is the one operation with a genuine complication. A leaf is simply detached. A node with one child is spliced out: the parent’s link jumps to the lone child, which carries its subtree along. The trouble is a node with two children — detaching it leaves two dangling subtrees and only one link in the parent to reattach them.

Deleting the minimum is the easy building block: walk left until a node has no left child (that is the minimum), free it, and splice in its right child. The general two-child case reduces to it.

Definition (Hibbard deletion) To delete a node \(x\) with two children (Hibbard, 1962), replace it with its in-order successor \(s\) — the smallest key in \(x\)’s right subtree, which by construction has no left child. Concretely: copy \(s\)’s key and value up into \(x\)’s node, then deleteMin the duplicate \(s\) out of the right subtree.

Order is preserved because no key lies strictly between \(x\)’s key and its successor: \(s\) is larger than everything in the left subtree (which was all below \(x\)) and smaller than everything remaining in the right subtree, so it belongs exactly where \(x\) sat. The predecessor — the largest key in the left subtree — would serve equally well by symmetry. 

// Hibbard deletion. 0/1-child: splice in the lone child. 2-child: copy the
// in-order successor (min of the right subtree) up, then deleteMin it out.
Node* deleteMin(Node* t) {
    if (!t->left) { Node* r = t->right; delete t; return r; }   // free the min
    t->left = deleteMin(t->left);
    t->size = 1 + size(t->left) + size(t->right);
    return t;
}
Node* erase(Node* t, const Key& key) {
    if (!t) return nullptr;
    if      (key < t->key) t->left  = erase(t->left,  key);
    else if (key > t->key) t->right = erase(t->right, key);
    else {                                   // found the node to remove
        if (!t->left || !t->right) {         // 0 or 1 child
            Node* c = t->left ? t->left : t->right;
            delete t;                        // free the node (no leak)
            return c;                        // splice in the lone child
        }
        Node* s = t->right;                  // in-order successor:
        while (s->left) s = s->left;         //   min of the right subtree
        t->key = s->key; t->val = s->val;    // copy it up
        t->right = deleteMin(t->right);      // remove (and free) the duplicate
    }
    t->size = 1 + size(t->left) + size(t->right);
    return t;
}
// Hibbard deletion. 0/1-child: splice the lone child. 2-child: copy the
// in-order successor (min of the right subtree) up, then deleteMin it out.
Node deleteMin(Node t) {

    t.left = deleteMin(t.left);
    t.size = 1 + size(t.left) + size(t.right);
    return t;
}
Node erase(Node t, Key key) {
    if (t == null) return null;

    if      (c < 0) t.left  = erase(t.left,  key);
    else if (c > 0) t.right = erase(t.right, key);
    else {                                   // found the node to remove
        if (t.left  == null) return t.right; // 0 or 1 child
        if (t.right == null) return t.left;
        Node s = t.right;                    // in-order successor:
        while (s.left != null) s = s.left;   //   min of the right subtree
        t.key = s.key; t.val = s.val;        // copy it up
        t.right = deleteMin(t.right);        // remove the duplicate
    }
    t.size = 1 + size(t.left) + size(t.right);
    return t;
}

def delete_min(t):

        return t.right                       # the min; spliced out here
    t.left = delete_min(t.left)
    t.size = 1 + size(t.left) + size(t.right)
    return t

def erase(t, key):
    if t is None:
        return None
    if   key < t.key: t.left  = erase(t.left,  key)
    elif key > t.key: t.right = erase(t.right, key)
    else:                                    # found the node to remove
        if t.left  is None: return t.right   # 0 or 1 child
        if t.right is None: return t.left

        while s.left: s = s.left             #   min of the right subtree
        t.key, t.val = s.key, s.val          # copy it up
        t.right = delete_min(t.right)        # remove the duplicate

// Hibbard deletion. 0/1-child: splice the lone child. 2-child: copy the
// in-order successor (min of the right subtree) up, then delete_min it out.
fn delete_min(mut t: Box<Node>) -> Option<Box<Node>> {
    match t.left.take() {
        None => t.right.take(),                        // the min; dropped here
        Some(l) => {
            t.left = delete_min(l);
            t.size = 1 + size(&t.left) + size(&t.right);
            Some(t)
        }
    }
}
fn min_kv(t: &Box<Node>) -> (i64, String) {            // successor key + value
    let mut n = t;
    while let Some(l) = &n.left { n = l; }
    (n.key, n.val.clone())
}
fn erase(t: Option<Box<Node>>, key: i64) -> Option<Box<Node>> {
    let mut n = t?;                                     // None stays None
    if      key < n.key { n.left  = erase(n.left.take(),  key); }
    else if key > n.key { n.right = erase(n.right.take(), key); }
    else {                                             // found it
        if n.left.is_none()  { return n.right.take(); }    // 0 or 1 child
        if n.right.is_none() { return n.left.take(); }
        let (sk, sv) = min_kv(n.right.as_ref().unwrap());
        n.key = sk; n.val = sv;                        // copy successor up
        n.right = delete_min(n.right.take().unwrap()); // remove the duplicate
    }
    n.size = 1 + size(&n.left) + size(&n.right);
    Some(n)
}

In C++ each removed node must be deleted exactly once, which the code does at every case: the 0/1-child branch frees the node and returns its lone child; the two-child branch defers to deleteMin, which frees the duplicated successor. No leaks — the standard the first programming assignment grades. (Java and Python lean on garbage collection; the Rust version drops owned Boxes automatically.)

Hibbard’s asymmetry wart Always taking the successor biases the tree: after many interleaved random inserts and deletes, always-successor Hibbard deletion drives the expected path length to \(\sim\!\sqrt{n}\) rather than \(\log n\) (Sedgewick & Wayne report this long-standing result). Alternating successor and predecessor, or always taking from the taller subtree, mitigates it; the balanced trees of Chapters 4–5 remove the concern entirely, which is one more reason to want them.

7. How tall is a BST?

Every operation in this chapter walks one root-to-leaf path (delete walks two: down to the node, then down to its successor), doing constant work per node. So each costs \(\Theta(\text{depth})\), and in the worst case \(\Theta(\text{height})\). This is worth stating on its own.

Proposition (cost is height) In a BST of height \(h\), search, insert, delete, and every ordered operation of §5 run in \(O(h)\) time in the worst case (range queries in \(O(h + m)\) for \(m\) keys reported). Every question about BST performance therefore reduces to a single one: how tall is the tree?

By the height bounds of §2, \(h\) lies between \(\lfloor \log_2 n \rfloor\) and \(n-1\), and the shape is set by insertion order. Sorted (or reverse-sorted) input is the disaster: each key is larger than all before it, so every insertion goes right, and the tree degenerates into a path of height \(n-1\) — a linked list with extra pointers, on which search is \(\Theta(n)\).

The average case: a random BST is shallow

The worst case is real but pessimistic. If the \(n\) keys are inserted in random order (every permutation equally likely), the tree is almost always close to balanced. The precise result is one of the prettiest in the subject.

Proposition (average search cost, Sedgewick Prop. C) In a BST built by inserting \(n\) distinct keys in random order, a search hit costs on average \(\sim 2\ln n \approx 1.39\log_2 n\) comparisons; search misses and insertions cost \(\sim 1.39\log_2 n\) as well (Prop. D).

The comparisons for a search hit equal one plus the node’s depth, so the average hit cost is one plus the average depth — the mean depth over all nodes. Here is a self-contained proof of the bound, avoiding the quicksort recurrence Sedgewick solves.

Proof (ancestor probabilities). Number the keys \(1, 2, \ldots, n\) by rank. A node’s depth is its number of proper ancestors, so the total depth summed over all nodes equals the number of pairs \((i, j)\) for which \(j\) is an ancestor of \(i\). Fix two ranks \(i < j\) (the case \(j < i\) is symmetric) and consider the block of keys ranked \(i, i+1, \ldots, j\) — there are \(j - i + 1\) of them. Whichever key of this block is inserted first becomes an ancestor of every other key in the block, and no key of the block is an ancestor of another unless it was inserted first: once two of them are split into different subtrees they can never become ancestor and descendant. Hence \(j\) is an ancestor of \(i\) exactly when \(j\) is the first of the block to be inserted, which by randomness has probability \(\dfrac{1}{j - i + 1}\).
Summing over all \(j > i\), the expected number of ancestors of \(i\) that are larger than it is \[ \sum_{j > i} \frac{1}{j - i + 1} = \frac12 + \frac13 + \cdots + \frac{1}{n - i + 1} \;\le\; \ln(n-i) + 1, \] a harmonic sum, and symmetrically the smaller ancestors contribute \(\le \ln(i-1) + 1\). Adding the two sides, each key has expected depth at most \(\ln(n-i) + \ln(i-1) + 2 \le 2\ln n + 2\), and averaging over \(i\) keeps the leading term: the mean depth is \(\sim 2\ln n\). Since \(2\ln n = (2\ln 2)\log_2 n \approx 1.39\log_2 n\), the average hit costs \(\sim 1.39\log_2 n\) comparisons. ∎

The single fact doing the work — among a contiguous block of ranks, the first inserted becomes everyone’s ancestor — is the same harmonic-number argument that governs quicksort (Chapter 13) and the length of hash-table probe sequences (Chapter 7); it is worth knowing once and reusing. The BST is in fact dual to quicksort: the root plays the role of the first pivot, splitting the keys into a smaller-left and larger-right part, each built recursively, so a random BST and a random-pivot quicksort have the same \(\sim 2\ln n\) analysis.

Average depth is not average height Prop. C bounds the average depth (\(\sim 1.39\log_2 n\)), the mean over all nodes. The height — the maximum depth — of a random BST is larger, \(\sim 2.99\log_2 n\) (Robson; Devroye), still logarithmic but taller. Both are averages over random insertion orders and neither is a guarantee: a single sorted run destroys them, as the path above shows. This gap between “shallow on average” and “shallow always” is exactly the gap that balanced trees close.

The scorecard, then, is the following, comparing a BST against the two elementary symbol tables.

The BST is the only one of the three with logarithmic insertion on average: a sorted array can binary-search in \(\Theta(\log n)\) but must shift \(\Theta(n)\) elements to insert. The price is the bad worst case — a random tree is a hope, not a guarantee. The balanced search trees of Chapter 4 (AVL) and Chapter 5 (2–3, red-black, B-trees) buy the guarantee back, forcing height \(\Theta(\log n)\) for any insertion order — which is why std::map and std::set are balanced BSTs. That is the cliffhanger this chapter hands to the next.

8. Exercises (with solutions)

  1. Insert the keys \(50, 30, 70, 20, 40, 60, 80\) into an empty BST in that order, then also into a second empty BST in the order \(20, 30, 40, 50, 60, 70, 80\). Give the height of each and the in-order traversal of each.
    solution First order: \(50\) is the root; \(30\) and \(70\) its children; \(20, 40\) under \(30\) and \(60, 80\) under \(70\) — a perfect tree of height \(2\). Second order (sorted): every key exceeds the previous, so each attaches as a right child, giving a path \(20 \to 30 \to \cdots \to 80\) of height \(6\). Both have the same in-order traversal, \(20, 30, 40, 50, 60, 70, 80\) — in-order always yields sorted order regardless of shape.
  2. A binary tree has pre-order \(H, C, A, E, R, P, X\) and in-order \(A, C, E, H, P, R, X\). Reconstruct the tree and give its post-order.
    solution Pre-order’s first key \(H\) is the root; in in-order, \(A, C, E\) precede \(H\) (the left subtree) and \(P, R, X\) follow it (the right). Recursing on the left with pre-order \(C, A, E\): root \(C\), with \(A\) left and \(E\) right. On the right with pre-order \(R, P, X\): root \(R\), with \(P\) left and \(X\) right. The tree is the §3 figure; its post-order (left, right, node) is \(A, E, C, P, X, R, H\).
  3. On the ten-key tree of §4 (in-order \(20,30,35,40,50,60,65,70,80,90\)), compute floor(68), ceiling(68), rank(65), and select(6).
    solution The sorted keys are \(20,30,35,40,50,60,65,70,80,90\) at ranks \(0,\ldots,9\). floor(68) is the largest key \(\le 68\), namely \(65\); ceiling(68) is the smallest \(\ge 68\), namely \(70\). rank(65) counts keys \(< 65\): \(20,30,35,40,50,60\), so \(6\). select(6) is the rank-\(6\) key, which is \(65\) — confirming rank and select are inverses.
  4. Delete the root \(50\) from the ten-key tree of §4 using Hibbard (successor) deletion. Which key replaces it, and what are the new root and its right subtree?
    solution The in-order successor of \(50\) is the minimum of its right subtree. That subtree is rooted at \(70\) with left child \(60\); since \(60\) has no left child (its only child, \(65\), is on the right), \(60\) is the minimum. Copy \(60\) up to the root, then deleteMin removes the old \(60\), splicing its right child \(65\) into \(60\)’s place. The new root is \(60\); its right subtree is \(70\) with left child \(65\) and right child \(80\) (\(90\) still under \(80\)). In-order is now \(20,30,35,40,60,65,70,80,90\) — \(50\) gone, still sorted.
  5. Prove that the in-order traversal of a binary search tree visits the keys in increasing order.
    solution Induct on the number of nodes. The empty tree emits nothing, vacuously sorted. For a tree rooted at \(x\), in-order emits (in-order of the left subtree), then \(x\), then (in-order of the right subtree). By the induction hypothesis the left output is sorted and, by the BST invariant, consists entirely of keys \(< x\); likewise the right output is sorted and all \(> x\). Concatenating a sorted run of keys below \(x\), then \(x\), then a sorted run above \(x\) yields one sorted sequence. ∎
  6. True or false, with justification: “If every node of a binary tree has its left child’s key less than its own and its right child’s key greater than its own, the tree is a binary search tree.”
    solution False. The BST invariant constrains whole subtrees, not just immediate children. Consider root \(20\) with left child \(10\) and right child \(30\), and give \(30\) a left child \(5\): locally \(5 < 30\) is fine, but \(5\) sits in \(20\)’s right subtree while \(5 < 20\), violating the invariant. In-order would emit \(10, 20, 5, 30\) — not sorted — and a search for \(5\) starting at \(20\) would wrongly go right and miss it. The correct test bounds each node against an \((lo, hi)\) interval inherited from its ancestors.
  7. You must repeatedly find, in a growing set of timestamps, the most recent event at or before a query time, and occasionally list all events in a date range. Would you use a hash table or a BST, and which operations does each query map to?
    solution A BST (in practice a balanced one). “Most recent at or before a query time” is exactly floor(t), and “all events in a range” is a range(lo, hi) query — both \(\Theta(\log n)\) (plus output size for the range) on a balanced BST. A hash table gives \(O(1)\) exact lookups but has no notion of order, so it cannot answer floor or range at all without scanning every key. Ordered queries are precisely where a BST beats a hash table.
  8. Explain why a random BST’s average depth (\(\sim 1.39\log_2 n\)) can be smaller than its average height (\(\sim 2.99\log_2 n\)) even though both are logarithmic, and why neither protects you against a bad input.
    solution Depth is averaged over all nodes, so shallow nodes near the root pull the mean down; height is the single deepest node, an extreme value that a random tree pushes to about \(2.99\log_2 n\). A tree can have small average depth while one lucky branch runs deep, so the two constants differ. Both statements are averages over random insertion orders and say nothing about a specific order: feed the keys sorted and the height is \(n-1\) with certainty. Guaranteeing logarithmic height for every order is the job of the balanced trees of Chapters 4–5.
