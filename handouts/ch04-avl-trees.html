Chapter 4 — Balanced Trees I: AVL Trees

CSS 343 — Data Structures, Algorithms & Discrete Mathematics II. Course text, primary reading for Lecture 4.
Chapter 3 left a cliffhanger: a binary search tree is fast on average but can degrade to a \(\Theta(n)\) path on sorted input. This chapter fixes it with one local invariant — every node’s two subtrees differ in height by at most one — restored after every update by \(O(1)\) rotations. We prove that this single rule forces height \(\le 1.44\log_2 n\) (a Fibonacci argument), giving the term’s first worst-case \(\Theta(\log n)\) guarantee for search, insert, and delete. With the four rotation cases drawn and verified, worked insertion traces, links to the interactive demos, and exercises.

Chapter 3 built the binary search tree and proved a single equation about it: every operation walks one root-to-leaf path, so its cost is \(\Theta(\text{height})\). That was good news and bad news at once. A random tree is about \(1.39\log_2 n\) tall, but height is not something the BST controls, and the worst case is a disaster: feed it keys in sorted order and each new key hangs off the rightmost node, building a \(\Theta(n)\) path — a linked list wearing a tree’s clothes. An AVL tree refuses to let that happen. It adds one local rule that each node can check on its own, restores the rule after every insertion or deletion with a constant-time pointer move called a rotation, and in exchange turns \(\Theta(\log n)\) from an average into a promise that holds for every input, adversarial ones included.

1. From average to guarantee

Recall the one fact Chapter 3 rested everything on: in a binary search tree, search, insert, and delete each follow a single path from the root down, so all three cost \(\Theta(h)\), where \(h\) is the tree’s height. A binary tree on \(n\) nodes has height anywhere from \(\lfloor\log_2 n\rfloor\) (bushy and balanced) to \(n-1\) (a single path). The entire game is to keep \(h\) small.

The BST does not play that game. Insert the keys \(1,2,3,4,5\) in order: each is the largest so far, so it becomes the right child of the previous one, and the tree is a downward staircase. 

  1
   \
    2
     \
      3        height = n − 1
       \       search / insert / delete = Θ(n)
        4
         \
          5

Sorted or reverse-sorted input is not exotic — it is exactly what you get from a database returning rows in key order, or from a stream of timestamps, or from IDs handed out in sequence. On any such input the plain BST collapses to a linked list, and its logarithmic average means nothing. We proved in Chapter 3 that a random BST is shallow, but you rarely control the insertion order, and Hibbard deletions skew even a good tree toward \(\sqrt n\) depth over time. A structure you ship has to survive its worst input, not just its average one.

The ideal would be a perfectly balanced tree, every leaf at depth \(\lfloor\log_2 n\rfloor\). But perfect balance is too rigid to maintain: one insertion can force a \(\Theta(n)\) rebuild of a large subtree, and one insert should not cost that. AVL’s insight is to relax perfect balance by just a little — a bounded wobble at each node — so that the tree is cheap to repair after every update yet still provably \(\Theta(\log n)\) tall.

Interactive plain BST vs. AVL. Insert the same keys \(1\dots16\) into both trees in lockstep and watch them diverge: the plain BST grows a height-\(15\) path while the AVL stays flat at height \(4\). Step key-by-key, or sweep the whole sequence at once. The gap you see is exactly the guarantee this chapter builds.

2. The height-balance invariant

AVL is named for Adelson-Velsky and Landis, who in 1962 gave the first self-balancing binary search tree. Its rule is a small strengthening of the ordinary tree: keep every node’s two subtrees close in height.

Definition (balance factor, AVL invariant) For a node \(x\), define its balance factor \[ \mathrm{bf}(x) = \text{height}(x.\text{left}) - \text{height}(x.\text{right}), \] with the convention \(\text{height}(\text{null}) = -1\) and a leaf having height \(0\). A binary search tree is AVL-balanced when \(\mathrm{bf}(x) \in \{-1,0,+1\}\) at every node — equivalently, \(|\mathrm{bf}(x)| \le 1\) everywhere.

The balance factor is just the node’s lean: positive means the left subtree is taller, negative means the right is taller, zero means they tie. The invariant allows a lean of one but forbids a lean of two. This is precisely the balanced shape of Chapter 3 (heights differing by at most one at every node), now maintained as a live invariant rather than merely observed. The textbooks that call the property height-balanced mean the same thing. 

   bf = 0        bf = +1        bf = +2  ← illegal
     o             o               o
    / \           /               /
   o   o         o               o
                                 /
                                o

The rightmost tree leans left by two: its left subtree has height \(1\) and its right subtree is empty (height \(-1\)), so \(\mathrm{bf} = 1 - (-1) = +2\). That is the situation an AVL tree must never rest in; whenever an update produces it, a rotation (§3) repairs it before the operation returns.

AVL is not perfectly balanced — it is merely balanced. The \(\pm 1\) of slack is deliberate: it is loose enough to restore in \(O(1)\) work per level, yet (as §7 proves) tight enough to force height \(\Theta(\log n)\). Perfect balance would be stricter than we can afford; this is exactly as strict as we need.

Storing the height

Rather than recompute subtree heights on demand, each node caches its own height in one extra integer field, updated in \(O(1)\) whenever its children change. Two null-safe helpers do all the arithmetic: 

  struct Node {
      int   key;
      Node* left  = nullptr;
      Node* right = nullptr;
      int   height = 0;          // leaf = 0, null = -1
  };
  int  height(Node* t){ return t ? t->height : -1; }
  int  bf(Node* t){ return height(t->left) - height(t->right); }
  void fix(Node* t){ t->height = 1 + max(height(t->left), height(t->right)); }

One extra int per node is the entire space cost of balance. fix recomputes a node’s height from its children’s cached heights, and bf reads their difference; both are \(O(1)\) and are called on the way back up an update. Verifying that a whole tree is AVL is the recursive “act on the node, recurse on the subtrees” pattern of Chapter 3: 

  bool isAVL(Node* t) {
      if (!t) return true;                  // empty is AVL
      if (abs(bf(t)) > 1) return false;      // this node violates
      return isAVL(t->left) && isAVL(t->right);
  }

Only the update path can break

The reason repair is cheap: an insertion or deletion changes subtree heights only along the single root-to-leaf path it walked. Every subtree hanging off to the side of that path is untouched, so its balance factors cannot change. Only nodes on the path can reach \(\mathrm{bf} = \pm 2\), and so we need only re-check and repair on the way back up that one path — never anywhere else. That is what makes the fix \(O(\text{height})\) rather than \(O(n)\), and it is why a single upward pass suffices.

Why a local rule is enough It is not obvious that a rule each node checks about only its own two children can control the height of the whole tree. That it does is the small miracle of balanced search trees, and it recurs for every family in the course: red-black trees, 2-3 and B-trees (Chapter 5) all pick a different local invariant and restore it bottom-up. Open Data Structures (Morin) calls this the “template” for balance; AVL is its cleanest instance because the invariant is simply a bound on a height difference.

3. Rotations: the order-preserving move

A rotation re-hangs a parent and one of its children, changing their heights (which is how it repairs balance) while leaving the left-to-right order of every key unchanged (which is how it keeps the tree a valid BST). It is the only structural move an AVL tree ever makes, and it is \(O(1)\): a few pointer writes and two height updates.

Definition (right and left rotation) A right rotation at \(y\) takes \(y\)’s left child \(x\) and lifts it above \(y\); \(x\)’s right subtree \(B\) re-hangs as \(y\)’s new left subtree. A left rotation at \(x\) is the mirror image, lifting the right child. In both, only the middle subtree changes parent.

The order-preservation is worth seeing explicitly. In both trees above, an in-order traversal visits \(A\), then \(x\), then \(B\), then \(y\), then \(C\). The rotation moves structure, never order — so search still works after any rotation, which is exactly why rotations are the legal move. The one subtree that changes parent is \(B\), the keys that lie between \(x\) and \(y\); in code this is the single line y.left = x.right, and the whole subtree follows that one pointer. 

// AVL rotations. Each re-hangs one parent/child pair, updates the two heights,
// and returns the new subtree root. O(1) work; the in-order sequence is
// unchanged, so the tree stays a valid BST. (Node stores an int height field.)
int height(Node* t) { return t ? t->height : -1; }   // null has height -1
void fix(Node* t) {                                   // recompute from children
    t->height = 1 + max(height(t->left), height(t->right));
}
Node* rotateRight(Node* y) {      // LL fix: left child x rises, y sinks
    Node* x = y->left;
    y->left = x->right;           // B (keys between x and y) re-hangs under y
    x->right = y;
    fix(y); fix(x);               // y is now lower, so recompute its height first
    return x;                     // x is the new root of this subtree
}
Node* rotateLeft(Node* x) {       // RR fix: the exact mirror of rotateRight
    Node* y = x->right;
    x->right = y->left;           // B re-hangs under x
    y->left = x;
    fix(x); fix(y);
    return y;
}

// AVL rotations over an owned-subtree link. Each re-hangs one parent/child pair,
// updates the two heights, and returns the new subtree root. O(1); the in-order
// sequence is unchanged. (Node has left, right: Link and height: i32.)
type Link = Option<Box<Node>>;

fn height(t: &Link) -> i32 { t.as_ref().map_or(-1, |n| n.height) }   // null: -1
fn fix(n: &mut Node) {                             // recompute from children
    n.height = 1 + height(&n.left).max(height(&n.right));
}
fn rotate_right(mut y: Box<Node>) -> Box<Node> {   // LL fix: left child x rises
    let mut x = y.left.take().unwrap();
    y.left = x.right.take();                        // B re-hangs under y
    fix(&mut y);                                    // y is now lower: fix first
    x.right = Some(y);
    fix(&mut x);
    x                                               // x is the new subtree root
}
fn rotate_left(mut x: Box<Node>) -> Box<Node> {    // RR fix: mirror of the above
    let mut y = x.right.take().unwrap();
    x.right = y.left.take();                        // B re-hangs under x
    fix(&mut x);
    y.left = Some(x);
    fix(&mut y);
    y
}

The order of the two fix calls matters: after the rotation \(y\) sits below \(x\), so we must recompute \(y\)’s cached height first, then \(x\)’s, which reads it. Each routine returns the new root of the rotated subtree so the caller can re-link it into place. The left rotation is the exact mirror — swap every left/right — and it repairs the symmetric RR case, where a node and its right child both lean right.

4. The four rebalancing cases

When an update drives some node \(z\) to \(\mathrm{bf} = \pm 2\), the shape of the imbalance — and hence the repair — is named by the direction of the two steps from \(z\) down toward the taller side. Follow the taller child (the sign of the balance factor), then follow its taller child; the two directions give the case.

casethe two steps below \(z\)trigger (\(z\) then its child)fix
LLleft, then left\(\mathrm{bf}(z)=+2\), \(\mathrm{bf}(z.\text{left})\ge 0\)single right rotation at \(z\)
RRright, then right\(\mathrm{bf}(z)=-2\), \(\mathrm{bf}(z.\text{right})\le 0\)single left rotation at \(z\)
LRleft, then right\(\mathrm{bf}(z)=+2\), \(\mathrm{bf}(z.\text{left})\lt 0\)double: left at \(z.\text{left}\), then right at \(z\)
RLright, then left\(\mathrm{bf}(z)=-2\), \(\mathrm{bf}(z.\text{right})\gt 0\)double: right at \(z.\text{right}\), then left at \(z\)

A straight lean (LL or RR) is fixed by one rotation, as the two figures of §3 showed. A kinked lean (LR or RL) — the node leans one way but its taller child leans the other — cannot be fixed by a single rotation, which would only move the kink to the other side. First rotate the child to straighten the kink into an LL or RR, then finish with the single rotation. Two \(O(1)\) rotations in all.

Example (LR: a double rotation, with subtrees) Start from the balanced tree \(30(10(5,20),\,40)\) and insert \(25\). It descends \(30 \to 10 \to 20\) and becomes \(20\)’s right child, which pushes \(30\) to \(\mathrm{bf}=+2\) while its left child \(10\) leans right (\(\mathrm{bf}=-1\)) — a left-then-right kink. A left rotation at \(10\) straightens it into an LL, and a right rotation at \(30\) finishes.
Example (RL: the mirror double) Symmetrically, from \(30(20,\,50(40,55))\) insert \(35\): it lands as \(40\)’s left child, driving \(30\) to \(\mathrm{bf}=-2\) with its right child \(50\) leaning left (a right-then-left kink). A right rotation at \(50\) straightens it into RR; a left rotation at \(30\) finishes.

All four cases are dispatched by one small function. The sign of \(\mathrm{bf}(z)\) picks the heavy side; the sign of the heavy child’s balance factor distinguishes the single case from the double and, if double, straightens the kink first: 

// AVL insert: descend like a plain BST, then on the way back up recompute each
// height and, at the first node whose balance factor reaches +/-2, rotate. The
// balance factor bf = height(left) - height(right); positive means left-heavy.
int bf(Node* t) { return height(t->left) - height(t->right); }

Node* rebalance(Node* t) {
    fix(t);
    if (bf(t) > 1) {                      // left-heavy: LL or LR
        if (bf(t->left) < 0)              // LR: left child leans right -> straighten
            t->left = rotateLeft(t->left);
        return rotateRight(t);            // now a plain LL
    }
    if (bf(t) < -1) {                     // right-heavy: RR or RL
        if (bf(t->right) > 0)             // RL: right child leans left -> straighten
            t->right = rotateRight(t->right);
        return rotateLeft(t);             // now a plain RR
    }
    return t;                             // |bf| <= 1: already balanced
}
Node* insert(Node* t, int key) {
    if (!t) return new Node(key);         // new leaf, height 0
    if      (key < t->key) t->left  = insert(t->left,  key);
    else if (key > t->key) t->right = insert(t->right, key);
    else return t;                        // duplicate key: nothing to do
    return rebalance(t);                  // fix heights + rotate on the way up
}

5. Insertion and rebalancing

AVL insertion is ordinary BST insertion plus one call on the way back up. The recursion gives the upward pass for free: the return trip up the call stack is the walk back up the search path.

  1. descend and place the new key at a leaf, exactly as a plain BST would;
  2. returning up the path, fix each node’s cached height;
  3. at the first node whose balance factor reaches \(\pm 2\), apply the one rotation (single or double) that its case calls for.

That is the whole algorithm; the insert function in the tabbed listing above is Chapter 3’s recursive BST insert with return rebalance(t) in place of the plain return.

Proposition (one rotation suffices for insert) A single insertion into an AVL tree provokes at most one rebalancing rotation (counting a double rotation as one repair).
Proof. Inserting a leaf increases the height of a subtree by at most \(1\), and only along the search path. Let \(z\) be the lowest node that becomes unbalanced (\(\mathrm{bf}=\pm 2\)); below \(z\) everything is still AVL. Before the insert, \(z\)’s subtree had some height \(H\); the insert made it \(H+1\). The rotation at \(z\) — as the figures of §3–4 show — produces a balanced subtree of height exactly \(H\) again, its pre-insert height. Every ancestor of \(z\) therefore sees the same subtree height it saw before the insertion, so no ancestor’s balance factor changed, and none can be violated. One rotation restores the entire tree. ∎

This height-restoration is the crucial AVL fact, and it is special to insertion; deletion (§6) behaves differently. It is also why AVL insertion is genuinely cheap: \(O(\log n)\) to descend and retrace, plus \(O(1)\) for the single rotation.

6. Deletion

Deletion begins exactly as in Chapter 3’s BST. To remove a node:

  • with zero or one child, splice it out (replace it by its lone child, or by null);
  • with two children, copy its in-order successor’s key up into it, then delete that successor from the right subtree (the successor has at most one child, reducing to the easy case).

Then, on the way back up, fix heights and rebalance each node with the very same four-case dispatch as insertion. The erase code is Chapter 3’s leak-safe deletion with return rebalance(t) replacing the plain return.

Deletion can rotate at every level Here is the one real difference from insertion. A rotation on insert restores the subtree’s pre-insert height, so the fix stops at the first violator. A deletion instead shortens a subtree, and the rebalancing rotation can shorten it further, which may unbalance the parent, which may then need its own rotation, and so on up the path. A single deletion can therefore trigger up to \(O(\log n)\) rotations — one per level — rather than just one. It is still \(O(\log n)\) total, because the path is only \(O(\log n)\) tall, so the worst-case cost of delete matches insert; only the rotation count differs.
Example (a delete that rotates) In the AVL tree \(3(1,\,4(\_,5))\), delete the leaf \(1\). The right side is now two taller than the empty left side, so \(3\) reaches \(\mathrm{bf}=-2\) with its right child \(4\) leaning right — an RR case. A single rotateLeft(3) restores balance, giving \(4(3,5)\). 
     3              3 (bf −2)        4
    / \      →        \        →     / \
   1   4                4              3   5
        \                \
         5                5
In a taller tree the shortening could cascade further up; here it stops after one rotation.

7. Why the height is \(\Theta(\log n)\)

Everything so far keeps the invariant cheaply; now we cash it in for the height bound that is the point of the whole construction. The lower bound is immediate — any binary tree on \(n\) nodes has height at least \(\log_2 n\), since \(k\) levels hold at most \(2^{k+1}-1\) nodes. The work is the upper bound: how tall can the \(\pm 1\) slack let an AVL tree grow?

Turn the question around. Instead of asking the most height for \(n\) nodes, ask the fewest nodes that a legal AVL tree of a given height \(h\) can have — call it \(N(h)\). If even the thinnest height-\(h\) tree needs many nodes, then a tree with few nodes cannot be tall.

Lemma (minimum-size AVL tree) The fewest nodes in an AVL tree of height \(h\) is \[ N(h) = 1 + N(h-1) + N(h-2), \qquad N(-1)=0,\; N(0)=1. \]
Proof. A height-\(h\) tree has a root plus two subtrees, at least one of which must have height \(h-1\) to make the whole tree height \(h\). To use as few nodes as possible, make that tall subtree a minimal tree of height \(h-1\) (using \(N(h-1)\) nodes) and make the other subtree as short as the invariant permits. The invariant allows the two subtree heights to differ by at most \(1\), so the shortest the other may be is \(h-2\), using \(N(h-2)\) nodes. Adding the root gives \(N(h)=1+N(h-1)+N(h-2)\). The bases: an empty tree (height \(-1\)) has \(0\) nodes and a single node (height \(0\)) has \(1\). ∎

That recurrence is the Fibonacci recurrence in disguise. Tabulate it: 

  h      :  0   1   2   3    4    5    6    7
  N(h)   :  1   2   4   7   12   20   33   54
  N(h)+1 :  2   3   5   8   13   21   34   55   ← Fibonacci

The pesky \(+1\) is the only thing hiding the pattern. Set \(M(h) = N(h)+1\). Then \[ M(h) = N(h)+1 = \bigl(N(h-1)+N(h-2)+1\bigr)+1 = \bigl(N(h-1)+1\bigr)+\bigl(N(h-2)+1\bigr) = M(h-1)+M(h-2), \] pure Fibonacci, with \(M(0)=2\) and \(M(1)=3\). Matching against \(F_1=F_2=1,\,F_3=2,\,F_4=3,\dots\) gives \(M(h)=F_{h+3}\), so \(N(h)=F_{h+3}-1\). What we actually need is a clean lower bound on \(N(h)\), and Fibonacci grows like the golden ratio \(\varphi = \tfrac{1+\sqrt5}{2}\approx 1.618\).

Theorem (AVL height bound) An AVL tree with \(n\) nodes has height \[ h \;\le\; \log_\varphi n \;=\; \frac{\log_2 n}{\log_2 \varphi} \;\approx\; 1.44\,\log_2 n, \] and therefore height \(\Theta(\log n)\).
Proof. First, \(N(h) \ge \varphi^{\,h}\), by induction on \(h\). Bases: \(N(0)=1=\varphi^0\) and \(N(1)=2\ge\varphi\). For the step, using \(\varphi^2=\varphi+1\), \[ N(h) = 1 + N(h-1) + N(h-2) \;\ge\; 1 + \varphi^{\,h-1} + \varphi^{\,h-2} = 1 + \varphi^{\,h-2}(\varphi+1) = 1 + \varphi^{\,h} \;\gt\; \varphi^{\,h}. \] Now any AVL tree of height \(h\) has at least \(N(h)\) nodes, so \(n \ge N(h) \ge \varphi^{\,h}\). Taking \(\log_\varphi\) of both sides, \(h \le \log_\varphi n\); converting the base, \(\log_\varphi n = \log_2 n / \log_2 \varphi\) and \(1/\log_2\varphi = 1/0.694\ldots \approx 1.44\). Combined with the universal lower bound \(h \ge \log_2 n\), the height is pinned to \(\log_2 n \le h \le 1.44\log_2 n\), i.e. \(\Theta(\log n)\). ∎
Where the \(1.44\) comes from The constant \(1/\log_2\varphi \approx 1.4405\) is the price of the \(\pm 1\) wobble: a perfectly balanced tree would achieve \(1.00\log_2 n\). Knuth named the minimum-size trees Fibonacci trees for exactly the recurrence above; Binet’s formula sharpens the estimate to \(N(h)\approx 1.894\,\varphi^{\,h}\), but the crude \(N(h)\ge\varphi^{\,h}\) is all the height bound needs. Every extra unit of height multiplies the minimum node count by another factor of \(\varphi\).

The consequence is the guarantee promised in §1. Because each of search, insert, and delete walks one path, and the path is now at most \(1.44\log_2 n\), all three are \(\Theta(\log n)\) in the worst case — for any sequence of operations, adversarial input included.

operationplain BST (worst)AVL (worst)
search\(\Theta(n)\)\(\Theta(\log n)\)
insert\(\Theta(n)\)\(\Theta(\log n)\)
delete\(\Theta(n)\)\(\Theta(\log n)\)

Chapter 3 gave \(\Theta(\log n)\) only on average; AVL gives it as a worst-case promise. To feel the difference concretely, take \(n = 10^9\): a plain BST fed sorted keys can take a billion comparisons for one lookup, while an AVL tree takes at most \(1.44\log_2 10^9 \approx 43\) — a few dozen nodes touched, guaranteed, no matter the input. The price is modest: a single extra integer per node, one rotation per insert (up to \(O(\log n)\) per delete), and a handful of \(O(1)\) checks on the way up.

The template for every balanced tree AVL is one instance of a universal recipe: (1) pick a local invariant that bounds height; (2) after each update, restore it with \(O(1)\) rotations, bottom-up; (3) prove the invariant forces \(\Theta(\log n)\) height. Chapter 5’s 2-3, B-, and red-black trees change only the invariant — a red-black tree admits height up to \(2\log_2 n\), looser than AVL’s \(1.44\log_2 n\) but with cheaper updates, which is why std::map uses red-black while read-mostly indexes lean toward the stricter AVL. Same goal, different local rule. And it is balanced trees like these that make the ordered-map and priority structures the rest of the course relies on run in guaranteed logarithmic time.

8. Exercises (with solutions)

  1. Give the balance factor of every node in the tree \(8(4(2,6),\,9)\) (root \(8\), left child \(4\) with children \(2,6\), right child \(9\)), and say whether it is AVL.
    solution Leaves \(2,6,9\) have height \(0\) and balance factor \(0\). Node \(4\) has two height-\(0\) children, so height \(1\), \(\mathrm{bf}=0\). Node \(8\) has left subtree (rooted at \(4\)) of height \(1\) and right child \(9\) of height \(0\), so \(\mathrm{bf}(8) = 1 - 0 = +1\). Every balance factor is in \(\{-1,0,+1\}\), so the tree is AVL. (Extend the left side one node deeper — say insert \(1\) below \(2\) — and \(\mathrm{bf}(8)\) tips to \(+2\), triggering a rotation.)
  2. Insert \(50,25,10\) in that order into an empty AVL tree. Name the case, the rotation, and the resulting tree.
    solution \(50\) is the root; \(25\) becomes its left child; \(10\) becomes \(25\)’s left child. Now \(\mathrm{bf}(50)=+2\) and its left child \(25\) leans left (\(\mathrm{bf}=+1\)): a straight LL case. A single rotateRight(50) lifts \(25\), giving \(25(10,50)\), balanced with height \(1\). This is the sorted-descending mirror of the \(1,2,3\) example.
  3. Insert \(10,20,15\) into an empty AVL tree. Identify the case and give the two rotations and the final tree.
    solution \(10\) root, \(20\) its right child, then \(15\) goes right of \(10\) and left of \(20\), landing as \(20\)’s left child. Now \(\mathrm{bf}(10)=-2\) while its right child \(20\) leans left (\(\mathrm{bf}=+1\)): a kinked RL case. First rotateRight(20) straightens it (making \(15\) the right child of \(10\) with \(20\) below), then rotateLeft(10) finishes, giving \(15(10,20)\). In-order \(10,15,20\) throughout.
  4. What is the maximum height of an AVL tree with \(20\) nodes? With \(n=54\) nodes?
    solution Use the minimum-size table \(N(h)=1,2,4,7,12,20,33,54\) for \(h=0,\dots,7\). A tree of height \(h\) needs at least \(N(h)\) nodes, so the tallest tree with \(n\) nodes has the largest \(h\) with \(N(h)\le n\). For \(n=20\): \(N(5)=20\le 20\) but \(N(6)=33\gt 20\), so the maximum height is \(5\). For \(n=54\): \(N(7)=54\le 54\) but \(N(8)=88\gt 54\), so the maximum height is \(7\). (A \(54\)-node tree of height \(7\) is exactly the minimum-size tree of that height.)
  5. Prove that AVL insertion needs at most one rotation but AVL deletion may need one at every level of the path. What structural fact distinguishes the two?
    solution The distinguishing fact is what the rotation does to the subtree’s height. On insertion, the rotation at the lowest violator restores the subtree to its pre-insert height (§5), so every ancestor sees an unchanged child height and none can become unbalanced — one rotation ends it. On deletion, the removal shortens a subtree, and the rebalancing rotation shortens it further; this reduced height can propagate to the parent, unbalancing it, and so on up the path. Each level may thus need its own rotation, for up to \(O(\log n)\) rotations, though the total cost stays \(O(\log n)\) since the path is that tall.
  6. A balance factor is stored as a small integer. Some implementations store only \(\mathrm{bf}\in\{-1,0,+1\}\) per node instead of a full height. What is saved, and what becomes harder?
    solution Storing only the balance factor needs two bits per node instead of a full int height, saving space. The cost is bookkeeping: after a rotation you must update balance factors from case analysis (the several sub-cases of how the inserted key’s subtree changed) rather than simply recomputing 1 + max(height(left), height(right)) from cached child heights. The height-field version in this chapter trades a little space for markedly simpler, less error-prone code, which is why it is the teaching default.
  7. Explain why the height bound uses \(\varphi^{\,h}\) rather than, say, \(2^{h}\). What would go wrong if AVL required perfect balance instead?
    solution The minimum-size AVL tree is deliberately lopsided — subtree heights \(h-1\) and \(h-2\) — so its node count grows like \(\varphi^{\,h}\), not \(2^{h}\); a perfectly balanced tree of height \(h\) would have \(\sim 2^{h}\) nodes, giving the better constant \(1.00\log_2 n\). But requiring perfect balance would make the invariant impossible to maintain in \(O(\log n)\): a single insert could force a \(\Theta(n)\) rebuild of a large subtree to keep every leaf at the same depth. AVL accepts the weaker \(\varphi^{\,h}\) growth (constant \(1.44\)) precisely to buy \(O(1)\)-per-level, one-rotation repair. The slack is what makes it maintainable.
  8. True or false, with justification: “If a binary search tree has height \(\le 1.44\log_2 n\), then it is an AVL tree.”
    solution False. The height bound is a consequence of the AVL invariant, not equivalent to it: a tree can respect the global height bound yet still contain a node whose two subtrees differ in height by \(2\), violating the invariant. Take \(8\bigl(4(2,6),\,12\bigr)\) where the right child \(12\) has a right child \(14\), which has a right child \(16\): seven nodes, height \(3\), and \(3 \le 1.44\log_2 7 \approx 4.04\), so the height bound holds. But \(\mathrm{bf}(12) = \text{height}(\text{null}) - \text{height}(14\text{'s subtree}) = -1 - 1 = -2\), so the tree is not AVL. The invariant is a per-node condition; a global height bound cannot imply it.
