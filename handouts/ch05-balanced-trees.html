Chapter 5 — Balanced Trees II: 2-3, B-Trees & Red-Black Trees

CSS 343 — Data Structures, Algorithms & Discrete Mathematics II. Course text, primary reading for Lecture 5.
A second route to a height-balanced search tree, taken not by rotating a binary tree back into shape but by letting a node hold more than one key. The 2-3 tree and its perfect-balance invariant; insertion by split-and-promote (why a 2-3 tree grows at the root and therefore never needs a rotation); deletion by borrow-and-merge; the exact correspondence that turns a 2-3 tree into a red-black BST — one node type plus one colour bit — and the local moves (rotateLeft, rotateRight, flipColors) that maintain it; and finally B-trees and B+ trees, the same split sized to a disk block, which is why databases and filesystems are built on them. With worked traces, figures, a correctness proof, links to the interactive demos, and exercises.

Chapter 4 won a \(\Theta(\log n)\) guarantee for the binary search tree of Chapter 3 by keeping it nearly balanced — every node’s balance factor pinned to \(\{-1,0,+1\}\), and any violation repaired by a rotation. That is one strategy, not the only one. This chapter reaches the same guarantee by a completely different mechanism: let a node hold one or two keys, keep every leaf at the same depth, and grow the tree upward by splitting. That idea is the 2-3 tree; a red-black BST is how we actually code it (a 2-3 tree wearing a binary-tree costume, with a single colour bit remembering where the multi-key nodes were); and B-trees are how databases scale it to disk, where the keys do not fit in memory and every node access is an expensive block read. Three machines, one idea — split a full node and send its middle key up.

1. From rotations to fatter nodes

The AVL tree of Chapter 4 keeps a binary tree balanced: one invariant (balance factor \(\in\{-1,0,+1\}\)), restored after each update by rotations, buying a height of at most \(1.44\log_2 n\) — a guarantee. It works, but the rotations are fiddly: four cases on insert (LL, RR, LR, RL), and a deletion can cascade rotations all the way up the path. AVL fixes imbalance after it happens. What if a tree could be built so that imbalance never happens at all?

The shift is to allow a node to hold more than one key and have more than two children. Then a new key can be absorbed into an existing node rather than grown onto a lopsided path, and balance becomes a property we get by construction instead of one we repair.

Definition (2-3 tree) A 2-3 tree is a search tree in which every internal node is one of two kinds. A 2-node holds one key and has two children (an ordinary BST node). A 3-node holds two keys \(a \lt c\) and has three children: the subtree of keys \(\lt a\), the subtree of keys strictly between \(a\) and \(c\), and the subtree of keys \(\gt c\). Every leaf is required to lie at the same depth. 
   2-node (1 key)            3-node (2 keys, a < c)
       [ b ]                    [ a | c ]
       /   \                   /    |    \
    < b     > b            < a   a..c    > c

A 3-node is the genuinely new thing: two keys guarding three doors. Reading one just extends the BST rule — instead of “left or right,” it is “left, middle, or right” depending on how the search key compares to the two stored keys.

Invariant (perfect balance) A 2-3 tree keeps every leaf at the same depth, at all times — no “\(\pm 1\) slack” as in AVL. Consequently its height \(h\) satisfies \(\log_3 n \le h \le \log_2 n\), so a 2-3 tree is \(\Theta(\log n)\) tall, guaranteed, for any sequence of insertions.
Proof (of the height bounds). A tree of height \(h\) with every leaf at depth \(h\) has between \(2^{h+1}-1\) nodes (all 2-nodes, branching factor 2) and, packing every node with two keys and three children, at most \(3^{h+1}-1\) key-slots. Turning that around on the number of keys \(n\): the sparsest 2-3 tree of \(n\) keys is all 2-nodes, a perfect binary tree of height \(\ge \log_2(n+1)-1\); the densest is all 3-nodes, of height \(\le \log_3(n+1)-1\). Both bounds are \(\Theta(\log n)\). ∎

This is a stronger balance condition than AVL’s, yet — remarkably — it is easier to maintain, because of how insertion works (§3). The one-sentence reason, worth planting now: a 2-3 tree grows at the root, not the leaves. A BST adds a new leaf, so one side can sink deeper and the tree tilts; a 2-3 tree pushes growth upward, adding a level at the very top, which lowers every leaf by one simultaneously. The “same depth” invariant is preserved by construction, never patched.

A family, not one tree Sedgewick & Wayne (§3.3) treat the 2-3 tree chiefly as a stepping stone: it is conceptually clean but awkward to code directly (two node types, much pointer surgery), so its real value is that it explains the red-black BST of §5, which is the balanced tree real libraries ship. Everything in this chapter — 2-3, red-black, B-trees — is the one mechanism “split a full node, promote its middle key,” tuned to a different node width and a different level of the memory hierarchy.

2. Searching a 2-3 tree

Search is the easy half, and it is essentially BST search with one more door to choose from at a 3-node. Compare the target against the key(s) in the current node and descend into the one child whose key range contains it; stop when you match a stored key or fall off at a leaf.

Search cost is one comparison group per level, so it is \(\Theta(h) = \Theta(\log n)\). The catch versus a plain BST is that a 3-node may cost two key comparisons rather than one; the win is not raw comparisons but that balance is automatic and stays that way under updates, which is the subject of the next two sections.

3. Insertion: split and promote

Search down for the new key as usual; since it is not present, the search ends at a leaf. We insert there — but never as a new, deeper leaf (that would break “all leaves same depth”). Instead we absorb the key into the leaf. Two cases follow from whether that leaf has room.

Case 1 — into a 2-node: just grow it

A 2-node has a free slot, so it simply becomes a 3-node. Nothing moves and no height changes: inserting \(15\) into the leaf \([10]\) yields \([10\,|\,15]\). Half of all leaf insertions are this trivial.

Case 2 — into a 3-node: it overflows, so split

A 3-node is already full. Adding a key makes a momentary 4-node (three keys), which the tree does not allow, so we split it: the middle of the three keys moves up into the parent, and the node breaks into two 2-nodes holding the smaller and the larger key. The middle is the only choice that preserves order — everything below it goes left, everything above it goes right, and it becomes their guard in the parent.

Because the promoted key joins the parent, the parent may itself overflow, split, and promote again — a cascade that can ripple to the root. This is the crux of the whole design:

Proposition (growth is only at the root) A 2-3 tree gains height only when the root splits: the promoted middle key becomes a brand-new root with the two halves as its children. Since the new level is added at the top, every leaf drops one level together, so all leaves remain at a common depth. This is the only way a 2-3 tree grows taller, and it needs no rotation.
Proof. A split at a non-root node moves one key up but leaves the number of levels below unchanged and touches no leaf’s depth. The only split with nowhere to promote is a split of the root; there we create a new root whose two children are the halves, adding exactly one level above all existing nodes. Every root-to-leaf path lengthens by one, so if the leaves were at a common depth before, they are at a common (one greater) depth after. Any other insertion (Case 1, or a split that stops below the root) leaves all leaf depths unchanged. By induction the invariant holds after every insertion. ∎
Example (a worked insertion sequence) Insert \(50, 30, 70, 10, 20\) into an empty 2-3 tree: 
  50  →  [50]
  30  →  [30|50]
  70  →  [30|50|70]  overflow ⇒ promote 50
             ⇒ root [50] over leaves [30] [70]
  10  →  into [30] ⇒ [10|30]
  20  →  [10|20|30]  overflow ⇒ promote 20
             ⇒ root [20|50] over [10] [30] [70]
The two splits show both cases: \(70\) splits a leaf whose promotion creates the very first second level, and \(20\) splits a leaf and promotes into a root that had room, widening it to \([20\,|\,50]\). All three leaves finish at depth 1.
Example (a cascade to a new root) Insert \(25\) into a tree whose root is already full: 
  before:        [ 30 | 60 ]
                /      |      \
          [10|20]  [40|50]  [70|80]

  25 → leaf [10|20] overflows: temp [10|20|25], promote 20
  root [30|60] now overflows: temp [20|30|60], promote 30

  after:            [ 30 ]
                  /        \
              [ 20 ]      [ 60 ]
             /     \      /     \
          [10]    [25]  [40|50] [70|80]
One insertion, two splits: the leaf splits (promoting \(20\)), the root then splits (promoting \(30\) into a new root), and the tree grows from two levels to three — every leaf still landing at depth 2. Note that \([40\,|\,50]\) and \([70\,|\,80]\) never moved: only the path from the changed leaf up to the root was touched, which is why an insertion is \(\Theta(\log n)\) even when the tree “grew.”

In every case the cost is a search down (\(\Theta(\log n)\)) plus at most one split per level on the way back up, each split being \(O(1)\) pointer work. Insertion is \(\Theta(\log n)\), and we never wrote a rotation. This split-and-promote rule is exactly what recurs, wider, in B-trees (§6), and, encoded with a colour bit, in red-black trees (§5).

4. Deletion: borrow and merge

Deletion is the mirror image of insertion: merge and demote in place of split and promote. There is more casework but no new idea, and it rests on the same discipline — act only at a leaf.

Step 1: get down to a leaf. If the key sits in an internal node, swap it with its in-order predecessor (the rightmost key of its left subtree, always a leaf), and delete that copy instead — the same trick every BST-family delete uses (Chapter 3’s Hibbard delete reached for the successor; either side works). Removal therefore always happens at a leaf.

Step 2: fix underflow. Removing a key from a leaf that held only one key leaves it empty (an underflow). Two repairs, chosen by what the siblings can spare:

  • Borrow if a sibling has two keys to spare: rotate a key through the parent. Deleting \(10\) from \([10]\) under a parent \([30]\) whose other child \([40\,|\,50]\) is a 3-node, the parent’s \(30\) drops into the empty node and the sibling’s \(40\) rises to replace it — a single, local move, and the tree’s shape above is untouched.
  • Merge if every sibling is already at the minimum (one key): pull the parent’s separating key down and fuse it with a sibling into a single 3-node. The parent loses a key, which may underflow the parent in turn, cascading upward. If the merge empties the root, the root’s only child becomes the new root and the tree shrinks a level — the exact mirror of a root split.

Both moves are \(O(1)\), applied along a single root-to-leaf path, so deletion is \(\Theta(\log n)\) like insertion. The full case analysis (which sibling, which side, cascading merges) is why 2-3 code is fussy to write by hand — precisely the fussiness the red-black encoding of the next section tames into a handful of uniform recolour-and-rotate moves.

5. Red-black BSTs: a 2-3 tree in binary

The 2-3 tree is elegant but painful to implement: two node types, split logic, promotion, new roots — many cases and much pointer surgery. In real code we would rather have one node type. The idea, due to Sedgewick, is to represent a 2-3 tree as an ordinary binary tree, using a single colour bit on each link to remember where the 3-nodes were.

Definition (left-leaning red-black BST) A left-leaning red-black BST is a BST whose links are coloured red or black so that: (1) red links lean left (no right-leaning red link); (2) no node has two red links in a row; and (3) black balance holds — every root-to-null path crosses the same number of black links. By convention a node’s stored colour is the colour of the link to its parent, and the root link is black.

A red link is the “glue” inside a 3-node. A 3-node \([a\,|\,c]\) becomes two BST nodes joined by a red link; a 2-node is a plain black-linked node. Read a red link as “these two nodes are really one 3-node,” and the three invariants become the 2-3 rules in disguise: “no two reds in a row” is “no node has a third key” (the 4-node we must split), and “equal black-height” is “all 2-3 leaves at the same depth.”

The three restoring moves

Insertion is a normal BST insert with the new link coloured red (a new key always tries to join an existing node as a 3-node), followed by a walk back up applying whichever of three constant-time fix-ups restores the invariants: 

  right child red, left child black   →  rotateLeft
  left red AND its left child red     →  rotateRight
  both children red                   →  flipColors

rotateLeft and rotateRight are the order-preserving pointer surgery of Chapter 4, with one extra line to carry the colour; flipColors is the encoding of a 2-3 split. A node with two red children is a temporary 4-node, and flipping all three colours pushes the middle key up (the node turns red, “joining its parent”) while its two children become independent black 2-nodes — precisely “promote the middle and split,” expressed in colour. If the parent now has two reds of its own, the flip cascades upward exactly as the 2-3 split did. 

// Left-leaning red-black BST: the three constant-time moves that maintain a
// 2-3 tree's shape. A node's `red` flag = the colour of the link to its PARENT;
// a null link counts as black.
struct Node { int key; Node *left = nullptr, *right = nullptr; bool red = true; };

bool isRed(Node* h) { return h && h->red; }    // null links are black

Node* rotateLeft(Node* h) {        // h's right link leans red; lean it left
    Node* x = h->right;
    h->right = x->left;
    x->left  = h;
    x->red = h->red;               // x inherits h's colour
    h->red = true;                 // h is now glued under x by a red link
    return x;                      // x is the new subtree root
}

Node* rotateRight(Node* h) {       // mirror image of rotateLeft
    Node* x = h->left;
    h->left  = x->right;
    x->right = h;
    x->red = h->red;
    h->red = true;
    return x;
}

void flipColors(Node* h) {         // two red children = a temporary 4-node
    h->red        = !h->red;       // push the middle key UP (h "promotes")
    h->left->red  = !h->left->red; // the two children become black 2-nodes
    h->right->red = !h->right->red;
}
// Left-leaning red-black BST: the three constant-time moves. A node's `red`
// flag = the colour of the link to its PARENT; a null link counts as black.
static class Node {
    int key; Node left, right; boolean red = true;
    Node(int k) { key = k; }
}

static boolean isRed(Node h) { return h != null && h.red; }

static Node rotateLeft(Node h) {      // right link leans red -> lean it left
    Node x = h.right;
    h.right = x.left;
    x.left  = h;
    x.red = h.red;                     // x inherits h's colour
    h.red = true;                      // h is now glued under x by a red link
    return x;
}

static Node rotateRight(Node h) {     // mirror image of rotateLeft
    Node x = h.left;
    h.left  = x.right;
    x.right = h;
    x.red = h.red;
    h.red = true;
    return x;
}

static void flipColors(Node h) {      // two red children = a temporary 4-node
    h.red       = !h.red;              // push the middle key UP (h "promotes")
    h.left.red  = !h.left.red;         // children become black 2-nodes
    h.right.red = !h.right.red;
}
# Left-leaning red-black BST: the three constant-time moves. A node's `red`
# flag = the colour of the link to its PARENT; a missing link counts as black.
class Node:
    def __init__(self, key):
        self.key = key
        self.left = self.right = None
        self.red = True

def is_red(h):                       # None links are black
    return h is not None and h.red

def rotate_left(h):                  # right link leans red -> lean it left
    x = h.right
    h.right = x.left
    x.left = h
    x.red, h.red = h.red, True       # x inherits h's colour; h turns red
    return x                         # x is the new subtree root

def rotate_right(h):                 # mirror image of rotate_left
    x = h.left
    h.left = x.right
    x.right = h
    x.red, h.red = h.red, True
    return x

def flip_colors(h):                  # two red children = a temporary 4-node
    h.red = not h.red                # push the middle key UP (h "promotes")
    h.left.red = not h.left.red      # children become black 2-nodes
    h.right.red = not h.right.red
// Left-leaning red-black BST moves. `red` = colour of the link to the PARENT;
// a None link counts as black. Children are Option<Box<Node>>, so we take/put
// the boxes to move subtrees around without aliasing.
type Link = Option<Box<Node>>;
struct Node { key: i32, left: Link, right: Link, red: bool }

fn is_red(h: &Link) -> bool { h.as_ref().map_or(false, |n| n.red) }

fn rotate_left(mut h: Box<Node>) -> Box<Node> {   // right link leans red
    let mut x = h.right.take().unwrap();
    h.right = x.left.take();
    x.red = h.red;                 // x inherits h's colour
    h.red = true;                  // h is now glued under x by a red link
    x.left = Some(h);
    x                              // x is the new subtree root
}

fn rotate_right(mut h: Box<Node>) -> Box<Node> {  // mirror image
    let mut x = h.left.take().unwrap();
    h.left = x.right.take();
    x.red = h.red;
    h.red = true;
    x.right = Some(h);
    x
}

fn flip_colors(h: &mut Box<Node>) {               // two red children = 4-node
    h.red = !h.red;                               // push the middle key up
    if let Some(l) = h.left.as_mut() { l.red = !l.red; }
    if let Some(r) = h.right.as_mut() { r.red = !r.red; }
}

Only nodes move in a rotation; the in-order sequence is unchanged, exactly as in Chapter 4. “Left-leaning” is what keeps the case count at three rather than the classic red-black tree’s six — Sedgewick’s simplification — and it is why the whole of insertion is BST-insert plus these three \(O(1)\) moves applied bottom-up.

Interactive Red-black BST demo. Red links are drawn thick and red — each is a glued 3-node. Insert an ascending run \(10, 20, 30, \dots\) to trigger every fix-up: a rotateLeft leans a red link left, a flipColors splits a 4-node and pushes a red up. Every root-to-null path keeps the same number of black links, and the tree never exceeds \(2\log_2 n\) tall no matter the insertion order. (The demo’s Delete button works too; deletion is the same three moves applied both on the way down — borrowing a red onto a black-linked child before descending — and on the way back up, which is why it is roughly twice the casework of insert yet uses no new primitive.)

Why the height is \(\Theta(\log n)\)

Theorem (red-black height) A red-black BST on \(n\) keys has height at most \(2\log_2 n\), so search, insert, and delete are \(\Theta(\log n)\) for any insertion order.
Proof. Let \(b\) be the number of black links on a root-to-null path — the same on every path, by black balance (invariant 3). Collapse each red link into its parent: two nodes glued by a red link merge into one 2-3 node, recovering the very 2-3 tree this encoding came from. The black links are exactly that 2-3 tree’s edges, so \(b\) equals the 2-3 tree’s height, which by §1 is at most \(\log_2 n\); hence \(b \le \log_2 n\). Now count all links on a path. Between any two black links invariant 2 permits at most one red (two in a row is the violation flipColors removes), so the total number of links is at most \(2b\). Therefore the height is at most \(2b \le 2\log_2 n\). ∎

The bound is looser than AVL’s \(1.44\log_2 n\) — a red-black tree can be up to twice as tall — but still \(\Theta(\log n)\), and it is reached with far fewer restructuring operations per update. That trade is why a red-black BST, being just one node type plus one colour bit, is the balanced tree real libraries ship: C++’s std::map and std::set, Java’s TreeMap and TreeSet, and the Linux kernel’s scheduler among them.

The correspondence is the whole point “2-3 trees are balanced” (§1) and “red-black trees are \(\Theta(\log n)\)” (here) are the same fact, seen through the collapse-the-red-links map. This is Sedgewick & Wayne’s central insight: do not analyse the red-black tree directly — recognise it as a 2-3 tree in a costume and inherit everything already proved. Cormen et al. develop the classic (not-left-leaning) red-black tree from five colour axioms instead; the two agree on the \(\Theta(\log n)\) guarantee and differ mainly in how many cases the code must handle.

6. B-trees and B+ trees: sized for the disk

A 2-3 node holds one or two keys. Why stop at two? A B-tree of order \(M\) lets each node hold up to \(M-1\) keys (and thus up to \(M\) children), keeps every leaf at the same depth, and splits a node the moment it reaches \(M\) keys — the identical split-and-promote, just with a wider node. A 2-3 tree is exactly a B-tree of order 3.

Definition (B-tree of order \(M\)) A search tree in which every node holds a sorted array of at most \(M-1\) keys and at most \(M\) child pointers, every non-root node has at least \(\lceil M/2 \rceil\) children (so at least \(\lceil M/2\rceil - 1\) keys), the root has at least two children unless it is a leaf, and all leaves lie at the same depth. The “at least half full” lower bound is what bounds the height at \(\log_{\lceil M/2\rceil} n\) and keeps the tree shallow.

The reason to want fat nodes is not comparison count — it is the disk. Reading from disk or SSD is not \(\approx 1\times\) the cost of a memory access; it is \(\approx 10^5\times\) slower, and it arrives a whole block (page) at a time. So the cost that matters is the number of node accesses = disk reads, not the number of comparisons. 

   register  ~1 ns
   RAM       ~100 ns
   SSD       ~100 µs      // ~1000x RAM
   disk      ~10 ms       // ~100000x RAM

Size a node to fill exactly one block, and each node holds hundreds of keys, so the tree is only a few levels deep. One block read brings a whole key array into memory, where a single in-RAM binary search picks the child — one disk read per level, and the height is \(\log_M n\).

Insertion is the 2-3 rule at width \(M\), as the demo shows.

Example (a B-tree split, \(M=5\)) Insert \(55\) into a full leaf \([60\,|\,70\,|\,80\,|\,90]\): 
  temp [55|60|70|80|90]   // 5 keys — overflow at M
              ↓  split, promote the middle (70)
        parent gains 70
       [ ... | 70 | ... ]
        /             \
   [55|60]          [80|90]
Split at \(M\) keys, promote the middle, and two half-full nodes remain — the same rule as a 2-3 split, only the threshold (5 keys instead of 3) and the halves (two keys each instead of one) differ. A root split still grows the height by one, at the top.
Proposition (few disk reads) A B-tree of order \(M\) on \(n\) keys has height at most \(\log_{\lceil M/2\rceil} n\), so a lookup costs \(O(\log_M n)\) disk reads.
Proof. Every non-root node has at least \(\lceil M/2\rceil\) children, so a tree of height \(h\) has at least \(2\lceil M/2\rceil^{\,h-1}\) leaves, hence at least that many keys. Solving \(n \ge 2\lceil M/2\rceil^{\,h-1}\) for \(h\) gives \(h \le 1 + \log_{\lceil M/2\rceil}(n/2) = O(\log_M n)\). Each level costs exactly one node access (one block read); the in-node binary search is free once the block is in RAM. ∎

The contrast with a binary tree is the whole argument for B-trees. A balanced binary tree over \(10^9\) keys is about \(\log_2 10^9 \approx 30\) levels — up to 30 disk reads, roughly \(0.3\) s on a spinning disk. The same keys in an order-1000 B-tree take 3 reads, roughly \(30\) ms: the branching factor does all the work. In RAM, 30 versus 3 comparisons is nothing; against the disk it is the difference between a snappy query and a slow one. The lesson is to match the node size to the unit of I/O — historically the original motivation (Bayer & McCreight, 1972).

B+ trees: values in the leaves

Databases use a variant, the B+ tree, that pushes all values down to the leaves. Internal nodes hold only keys (pure routing information); every value lives in a leaf, and the leaves are linked into a sorted list. Value-free internal nodes fan out even wider — each block routes to more children — so the tree is even shorter, and the linked leaves make range scans a single descent followed by a walk along the chain. 

  index on age, order 3 (routers upstairs, records in the leaves):

  internal:          [ 30 | 60 ]
                    /      |      \
  leaves:   [10|20] → [30|45] → [60|90] → ∅
             (each leaf key carries its own record; a key such
              as 30 or 60 appears twice — once as a router,
              once in a leaf)

Because the leaves form a sorted linked list, a range query such as SELECT * WHERE age BETWEEN 20 AND 40 descends once to find \(20\), then follows leaf links until it passes \(40\). ORDER BY, BETWEEN, and pagination are all just sequential leaf walks — which is why B+ trees, not plain B-trees, index essentially every relational database and back most modern filesystems (NTFS, ext4, APFS).

Choosing the order \(M\) The order is dictated by hardware, not taste: pick \(M\) so a node fills one block. With a \(4096\)-byte block, \(8\)-byte keys, and \(8\)-byte pointers, an internal node needs about \(M\cdot 8 + (M-1)\cdot 8 \approx 4096\), giving \(M \approx 256\). Then three levels hold roughly \(1 \times 256 \times 256^2 \times 255 \approx 16\) million keys in three reads; pushing to \(M = 1000\) and one more level reaches billions in about four. Those figures assume full nodes, though. The order property only guarantees half-full, and the root is exempt even from that — right after any root split the new root holds a single key. Counting the guaranteed minimum (\(\lceil M/2\rceil\) children) instead of the maximum gives far fewer keys at the same depth, but because capacity grows exponentially in the fan-out, one extra disk read makes it up. “A few disk reads for the largest real datasets” is the sound takeaway (Open Data Structures, Morin, Ch. 14) — just do not quote the best case as if it were the guarantee. This block-size arithmetic is the disk exercise you will see on the homework.
Interactive B-tree demo. Insert into an order-5 B-tree and watch the same split-and-promote as a 2-3 tree, only wider: a node splits when it reaches five keys, its middle rises, and the fan-out stays high so the tree hardly deepens. Notice how many keys sit at each level even as the tree fills — the high fan-out is the entire benefit.

7. One goal, several tools

Every structure in this chapter and the last delivers \(\Theta(\log n)\) search, insert, and delete. They differ in the mechanism that keeps them balanced and in which level of the memory hierarchy they optimise for.

The choice between AVL and red-black is a choice of constant, not of asymptotic class. AVL keeps a tighter tree (height \(1.44\log_2 n\) versus \(2\log_2 n\)), so it searches a little faster, and it does at most one rotation per insertion; but a deletion can rotate at every level. Red-black does slightly more work per insertion yet keeps deletion to \(O(1)\) rotations (recolours may cascade, but they are cheaper than rotations). So AVL optimises the search and wins when reads dominate; red-black optimises the update and wins when writes dominate — which is why a general-purpose library map is red-black.

In a red-black tree the “promote” is a flipColors passing a red link up; in a B-tree it is the same middle-key promotion, widened to fill a disk block. Balance is a goal, reached by whichever mechanism suits the machine, not a single algorithm. A blunt decision guide: 

   in RAM, read-heavy        →  AVL
   in RAM, general/mutable   →  red-black   (std::map)
   on disk / a database      →  B+ tree
   learning / interviews     →  2-3         (it explains red-black)

You will rarely hand-roll any of these: you reach for std::map (red-black) in memory or trust your database’s B+ tree on disk. But knowing which is running tells you the cost and the gotchas — and the split-and-promote idea returns, transformed, when we leave comparison-based structures for hashing in Chapter 7.

8. Exercises (with solutions)

  1. Insert \(60, 20, 80, 10, 40, 50\), in that order, into an initially empty 2-3 tree. Draw the final tree and give every leaf’s depth.
    solution \(60 \to [60]\); \(20 \to [20\,|\,60]\); \(80 \to [20\,|\,60\,|\,80]\) overflows, promote \(60\) to root \([60]\) over leaves \([20]\) and \([80]\). \(10 \to [10\,|\,20]\). \(40\): descend to leaf \([10\,|\,20]\), overflow \([10\,|\,20\,|\,40]\), promote \(20\) into the root, which becomes \([20\,|\,60]\); its children are \([10], [40], [80]\). \(50\): \(20 \lt 50 \lt 60\) so descend to the middle child \([40]\), a 2-node, giving \([40\,|\,50]\). Final: 
            [ 20 | 60 ]
       /     |     \
    [10]  [40|50]  [80]
    All three leaves are at depth 1.
  2. Convert the 2-3 tree whose root is the 3-node \([40\,|\,80]\) with children \([20], [60], [100]\) into a left-leaning red-black BST. Give each node’s colour and its parent link.
    solution The root 3-node \([40\,|\,80]\) becomes \(80\) (black, since it is the root) with a red left link to \(40\). The three subtrees hang under the two keys by range: \(20\) (\(\lt 40\)) is \(40\)’s left child, \(60\) (\(40\!-\!80\)) is \(40\)’s right child, and \(100\) (\(\gt 80\)) is \(80\)’s right child; all three are black. In-order \(20,40,60,80,100\) matches the 2-3 tree, and collapsing the one red link \(40\!-\!80\) recovers the original 3-node.
  3. Explain, in your own words, why a 2-3 tree grows only at the root and why that single fact keeps all leaves at the same depth — and contrast it with why a plain BST can become unbalanced.
    solution A split promotes its middle key to the parent, which either has room (the split stops) or overflows and splits in turn. The only split with nowhere to promote is a split of the root, which manufactures a new root one level above everything else. Adding a level at the top lengthens every root-to-leaf path by exactly one, so a common leaf depth stays common. A BST, by contrast, only ever adds a new leaf, deepening one path while leaving the others untouched; repeated one-sided inserts then tilt it toward a chain of height \(n-1\). Grow-up preserves balance; grow-down does not.
  4. Prove that a left-leaning red-black BST on \(n\) keys has height at most \(2\log_2 n\), using the correspondence with 2-3 trees.
    solution Let \(b\) be the number of black links on any root-to-null path; black balance makes \(b\) the same on every path. Collapsing each red link into its parent merges each red-glued pair into one 2-3 node and turns the black links into the edges of a 2-3 tree, so \(b\) equals that 2-3 tree’s height, which is at most \(\log_2 n\) (the all-2-nodes worst case of §1); thus \(b \le \log_2 n\). Between two consecutive black links at most one red link can appear (invariant 2 forbids two reds in a row), so a path has at most \(2b\) links total. Hence height \(\le 2b \le 2\log_2 n\).
  5. A B-tree has order \(M = 256\) and three levels (so a lookup is three disk reads). Assuming every node is full, how many keys can it hold? Repeat for order \(M\) and \(h\) levels in general.
    solution A full node has \(M\) children and \(M-1\) keys. With \(h\) levels the number of nodes is \(1 + M + M^2 + \dots + M^{h-1} = (M^h - 1)/(M - 1)\), and each holds \(M-1\) keys, for a total of \(M^h - 1\) keys. For \(M = 256\), \(h = 3\): \(256^3 - 1 \approx 1.67 \times 10^7\), about 16.7 million keys in three reads. (This is the best case; the order property only guarantees half-full nodes, so the guaranteed capacity at three levels is far smaller, but one more level restores “millions.”)
  6. Why do databases prefer B+ trees over plain B-trees for queries such as SELECT * WHERE age BETWEEN 20 AND 40?
    solution In a B+ tree all records live in the leaves, and the leaves are linked into a sorted list. A range query descends once to the first key in range, then walks the leaf chain sequentially until it leaves the range — no re-traversal of the tree. In a plain B-tree records sit at every level, so a range scan must wander up and down the tree, revisiting internal nodes. B+ trees also keep internal nodes value-free, so each block routes to more children (higher fan-out, shorter tree). The cost is that some keys appear twice (as a router and in a leaf), a small price for cheap range and ordered scans.
  7. For each situation, name the structure from this chapter (or Chapter 4) you would reach for: (a) an in-memory dictionary that is queried far more often than it is updated; (b) the general-purpose ordered map in a standard library; (c) an index for a database table on disk that must support ordered range scans.
    solution (a) an AVL tree — its tighter height (\(1.44\log_2 n\)) makes reads fastest. (b) a red-black BST — fewer rotations per update suit a mix of inserts and erases, which is why std::map/TreeMap use it. (c) a B+ tree — wide nodes minimise disk reads and the linked leaves make range scans a single descent plus a leaf walk. All three are \(\Theta(\log n)\); the machine and the workload decide.
  8. True or false, with justification: “AVL, red-black, 2-3, and B-trees all guarantee \(\Theta(\log n)\) search regardless of the order the keys are inserted.”
    solution True. Each maintains a height bound that is \(\Theta(\log n)\) under every insertion order: AVL keeps \(h \le 1.44\log_2 n\) via the balance-factor invariant; red-black keeps \(h \le 2\log_2 n\) via black balance and the no-double-red rule; a 2-3 tree keeps \(\log_3 n \le h \le \log_2 n\) by perfect balance; and a B-tree of order \(M\) keeps \(h \le \log_{\lceil M/2\rceil} n\) by the half-full rule. Search visits one node per level, so it is \(\Theta(h) = \Theta(\log n)\) in every case. (A plain BST would make the claim false — adversarial inserts give height \(n-1\) — which is exactly the problem these structures were built to solve.)
