Chapter 6 — Heaps, Priority Queues & Heapsort

CSS 343 — Data Structures, Algorithms & Discrete Mathematics II. Course text, primary reading for Lecture 6.
A queue serves oldest-first; a priority queue serves the most important item first, and it must do so while items arrive and depart continuously. The binary heap — a complete binary tree flattened into a plain array, with no pointers — makes both insert and delMax cost \(\Theta(\log n)\). We derive its two invariants and everything they force, build the two primitives (swim and sink) that every operation reduces to, prove that a heap can be assembled from \(n\) keys in \(\Theta(n)\) time (a genuinely surprising bound), and turn the whole machine into heapsort, the only \(\Theta(n\log n)\) sort that is guaranteed and in place. With worked traces, both proofs, links to the interactive demo, and exercises.

A stack serves the newest item and a queue the oldest; both order by when a thing arrived. A priority queue orders by a key you choose and serves the most important item next — the highest-priority thread, the most critical patient, the earliest pending event. The challenge is that both ends of the workload must be fast: inserting a new item and removing the best one, over and over, as the collection churns. Neither a sorted nor an unsorted array can make both cheap. This chapter builds the structure that can, the binary heap, which keeps just enough order — each parent at least its children — to serve the maximum instantly and restore order after any change by touching a single root-to-leaf path. It is the simplest structure in the course with a genuinely surprising analysis, a linear-time build we will actually prove, and it is the engine behind Dijkstra (Chapter 9), Prim (Chapter 11), and Huffman coding (Chapter 12).

1. The priority-queue ADT

A stack orders by insertion time and returns the newest item; a queue returns the oldest. A priority queue keeps a collection of keyed items and returns the one whose key is best — largest, for a max-priority-queue. The rule is a comparison on the data itself rather than on arrival order, which is exactly what a great many algorithms need: an operating-system scheduler runs the highest-priority ready thread, an emergency room treats the most critical patient first, a discrete-event simulator processes the earliest pending event next.

Definition (max-priority-queue) A max-priority-queue over comparable keys supports insert(x) (add a key), max() (return the largest key), delMax() (remove and return the largest key), together with isEmpty() and size(). A min-priority-queue is the mirror image, with min() and delMin(); we build the max version and everything flips symmetrically.

The defining workload is that insert and delMax interleave. A discrete-event simulator is the canonical case: 

  PQ = min-PQ keyed by event time
  insert the first few events
  while PQ not empty:
      e = delMin()          // the earliest remaining event
      process e             // may schedule NEW future events -> insert

New events appear in the middle of processing, out of time order, yet delMin must always return the chronologically next one. A sorted array would answer delMin in \(\Theta(1)\) but pay \(\Theta(n)\) to absorb each mid-run insertion; an unsorted array flips the costs. Each simple choice makes one operation cheap by making the other linear:

implementationinsertdelMax
unordered array / list\(\Theta(1)\)\(\Theta(n)\)
ordered array / list\(\Theta(n)\)\(\Theta(1)\)
binary heap\(\Theta(\log n)\)\(\Theta(\log n)\)

The unsorted array must scan to find the max; the sorted array must shift to make room. We want both operations fast, and the heap gets there by keeping the data only partially ordered — strong enough to find the max instantly, loose enough to insert cheaply.

Isn’t a balanced tree already a priority queue? It is. An AVL tree (Chapter 4) does insert and delete-max in \(\Theta(\log n)\) — the max is the rightmost node. So why a new structure? Because the heap does less and therefore costs less: it stores keys in a plain array with no per-node pointers (Chapter 2’s arithmetic: a pointer node costs several times the memory of an array slot, and scatters across the heap), it needs no rotations, its code is about twenty lines, and it can be built from \(n\) keys in \(\Theta(n)\) rather than \(\Theta(n\log n)\). A balanced tree can answer “is key \(x\) present?” and “what is the \(k\)-th smallest?”; a heap cannot, and it charges nothing for the answers it declines to give. The heap is not the only priority queue — it is the cheapest one for this exact job. Section 7 widens the design space.

2. The binary heap: a complete tree in an array

Definition (binary max-heap) A binary max-heap is a binary tree that is both complete — every level is full except possibly the last, which is filled left-to-right with no gaps — and heap-ordered — every node’s key is \(\ge\) both of its children’s keys.

The two conditions do two separate jobs, and it pays to keep them distinct. Completeness controls the shape: no gaps, so the tree fits an array and stays logarithmically shallow. Heap order controls the values: the maximum is on top. Every operation below temporarily breaks one condition and restores it.

No pointers: store the tree in an array

Because a complete tree has no gaps, we can number its nodes level by level, left to right, from \(1\) to \(n\), and store them in an array a[1..n]. The tree structure then becomes pure arithmetic. We use the 1-based convention throughout this chapter, leaving a[0] unused so the root sits at index \(1\) and the index formulas stay clean: 

  parent(k) = k / 2       // integer division, floor
  left(k)   = 2k
  right(k)  = 2k + 1

Going to a child is a multiply by two; going to a parent is a divide by two. There is no dereferencing, no pointer to follow, nothing to allocate per node. (A 0-based array works too, with \(\texttt{parent}(k)=\lfloor(k-1)/2\rfloor\), \(\texttt{left}(k)=2k+1\), \(\texttt{right}(k)=2k+2\); the course, its assignments, and every snippet here use the 1-based form.)

This is the whole reason the heap is the default priority queue: a complete tree maps perfectly onto a contiguous array, so we get a tree’s structure with an array’s cache-friendliness and zero pointer overhead. A classic space-saver stores the size \(n\) in the otherwise-wasted a[0].

Height, and why every operation is \(\Theta(\log n)\)

Proposition (height) A complete binary tree on \(n\) nodes has height \(h = \lfloor \log_2 n \rfloor\).
Proof. Every level except the last is completely full, so levels hold \(1, 2, 4, \dots, 2^{h-1}\) nodes and then between \(1\) and \(2^h\) nodes on the last. A tree of height \(h\) therefore has at least \(1 + 2 + \dots + 2^{h-1} + 1 = 2^h\) nodes and at most \(2^{h+1}-1\) nodes, so \(2^h \le n \le 2^{h+1}-1\), which gives \(h \le \log_2 n \lt h+1\), i.e. \(h = \lfloor \log_2 n \rfloor\). ∎

Completeness thus guarantees logarithmic height for free — no balancing, no rotations, ever, in contrast to the work Chapters 4 and 5 spent forcing balance. Any operation that walks a single root-to-leaf path costs \(O(\log n)\), which is where the heap’s \(\Theta(\log n)\) insert and delete come from. The maximum, meanwhile, is always a[1], read in \(\Theta(1)\).

Heap is not a BST A heap and a binary search tree draw the same picture — a binary tree — under different invariants, and confusing them is the first classic error. A BST orders \(\text{left} \lt \text{node} \lt \text{right}\) and can find any key in order; a heap orders \(\text{parent}\ge\text{children}\) and can find only the max. A BST can be lopsided; a heap is always complete. You cannot binary-search a heap for an arbitrary key — that is a \(\Theta(n)\) scan. The heap gives up general search to make “find the max” and “stay balanced” trivial.

3. What the invariants imply

Two local conditions — parent-versus-child, level-by-level — force a surprising amount of structure. Extracting it now pays off in every proof that follows; getting into the habit of squeezing an invariant before touching code is itself the transferable skill.

Every subtree is a heap

Heap order is checked one parent at a time, so it still holds when we restrict attention to any subtree; and a subtree of a complete tree is itself complete. Hence every subtree of a heap is a heap — a heap is heaps all the way down. This is the recursion that the correctness proofs of sink and of heapify both lean on.

Max on top, min at a leaf

Along any downward path the keys only shrink, so by transitivity the root is the global maximum. Conversely any node that has a child is \(\ge\) that child and so cannot be the minimum; the minimum must sit at a leaf. But no rule says which leaf, so finding the minimum of a max-heap is a \(\Theta(n)\) scan — the sharpest way to see both what partial order buys (one end of the order is instant) and what it does not (the other end is as slow as an unsorted array). Assuming distinct keys the minimum is unique; with duplicates, a minimum sits at a leaf.

The leaves are the back half of the array

Node \(k\) has a child exactly when \(2k \le n\). So the internal nodes are \(k = 1, \dots, \lfloor n/2 \rfloor\) and the leaves are \(k = \lfloor n/2\rfloor + 1, \dots, n\) — at least half of any heap is leaves. 

  internal nodes:  k = 1 .. n/2
  leaves:          k = n/2 + 1 .. n

  a: [ _ | 90  80  70 | 30  60  50 ]     n = 6
          |internal|  |-- leaves --|

Two payoffs. The \(\Theta(n)\) scan for the minimum need only look in the back half. And heapify (Section 5) will start at \(\lfloor n/2\rfloor\) precisely because everything past it is a leaf, hence already a one-node heap needing no work — half the build is free before it begins, the first hint that the build might beat \(n\log n\).

Paths are sorted; levels are not

Every root-to-leaf path is non-increasing, but nothing at all is guaranteed across subtrees. A key deep in one subtree can exceed a key one level higher in another: 

        [ 90 ]
       /      \
    [30]      [80]
    /  \      /
 [10] [20] [75]        75 > 30, yet 75 is deeper

Only ancestor–descendant pairs are ordered; siblings and cousins are not. This is the second classic error to avoid: a heap’s array is not roughly sorted, and level \(2\) does not dominate level \(3\). The consequence is that a heap answers “who is on top?” but cannot answer rank, \(k\)-th-largest, or range queries without paying \(\Theta(n)\) or dismantling itself — and it is exactly this “far less information than fully sorted” that lets the \(\Theta(n)\) build coexist with the \(\Omega(n\log n)\) sorting bound (Section 5).

4. The two primitives: swim and sink

Both priority-queue operations follow one pattern: do the easy structural thing that keeps the tree complete, which may break heap order along a single path, then repair that one path. Repairing upward is swim; repairing downward is sink. Everything else in the chapter — build, heapsort, Dijkstra, Huffman — is these two in wrappers. 

// Restore heap order along ONE path. 1-based array a[1..n]; a[0] is unused.
// swim: a[k] may now exceed its parent -- lift it up. Cost Theta(log n).
void swim(int a[], int k) {
    while (k > 1 && a[k / 2] < a[k]) {     // parent smaller: heap order broken
        std::swap(a[k / 2], a[k]);         // lift the bigger key up
        k /= 2;                            // move up to the parent
    }
}
// sink: a[k] may now be smaller than a child -- push it down among a[1..n].
void sink(int a[], int k, int n) {
    while (2 * k <= n) {                   // while a[k] has a left child
        int j = 2 * k;                     // left child
        if (j < n && a[j] < a[j + 1]) j++; // pick the LARGER of the two children
        if (a[k] >= a[j]) break;           // already >= both children: done
        std::swap(a[k], a[j]);             // demote a[k] under the larger child
        k = j;                             // move down
    }
}

        int t = a[k / 2]; a[k / 2] = a[k]; a[k] = t;   // lift the bigger key up
        k /= 2;                            // move up to the parent
    }
}

    }
}

# sink: a[k] may now be smaller than a child -- push it down among a[1..n].
def sink(a, k, n):
    while 2 * k <= n:                      # while a[k] has a left child
        j = 2 * k                          # left child
        if j < n and a[j] < a[j + 1]:
            j += 1                         # pick the LARGER of the two children
        if a[k] >= a[j]:
            break                          # already >= both children: done
        a[k], a[j] = a[j], a[k]            # demote a[k] under the larger child
        k = j                              # move down
// Restore heap order along ONE path. 1-based slice a[1..=n]; a[0] is unused.
// swim: a[k] may now exceed its parent -- lift it up. Cost Theta(log n).
fn swim(a: &mut [i32], mut k: usize) {
    while k > 1 && a[k / 2] < a[k] {       // parent smaller: heap order broken
        a.swap(k / 2, k);                  // lift the bigger key up
        k /= 2;                            // move up to the parent
    }
}
// sink: a[k] may now be smaller than a child -- push it down among a[1..=n].
fn sink(a: &mut [i32], mut k: usize, n: usize) {
    while 2 * k <= n {                     // while a[k] has a left child
        let mut j = 2 * k;                 // left child
        if j < n && a[j] < a[j + 1] { j += 1; }  // pick the LARGER child
        if a[k] >= a[j] { break; }         // already >= both children: done
        a.swap(k, j);                      // demote a[k] under the larger child
        k = j;                             // move down
    }
}

In swim the moving key compares against its one parent and climbs while it is larger. In sink the moving key compares against both children and must swap with the larger one — the single most error-prone line in the whole structure, the if (j < n && a[j] < a[j+1]) j++;. Swap with the smaller child instead and a bigger key is left stranded beneath a smaller one; the code still runs and mostly works, failing only silently. Each loop climbs or descends one level per iteration, so both cost at most \(h = \lfloor\log_2 n\rfloor\) swaps: \(\Theta(\log n)\). (Sedgewick names these swim and sink; Carrano’s text calls them percolate up and percolate down.)

insert: append, then swim

To insert, place the new key at the end (a[++n]), which keeps the tree complete, then swim it: the only edges that can now violate heap order lie on the new node’s path to the root, and swim fixes exactly that path. To remove the max, swap a[1] with the last element and shrink, which parks the old max just past the live array and keeps the tree complete, then sink the new root. 

// Priority-queue operations on a 1-based max-heap: a[0] unused, size n.
// insert appends then swims up; delMax swaps the root out then sinks. Each Theta(log n).
struct MaxHeap { int a[CAP + 1]; int n = 0; };   // a[0] wasted so root sits at 1

int max(MaxHeap& h) { return h.a[1]; }            // the root -- Theta(1)

void insert(MaxHeap& h, int x) {
    h.a[++h.n] = x;                  // append at the end: tree stays complete
    swim(h.a, h.n);                  // restore heap order up one path
}

int delMax(MaxHeap& h) {
    int top = h.a[1];                // the maximum, to be returned
    std::swap(h.a[1], h.a[h.n--]);   // move max to the end, then shrink
    sink(h.a, 1, h.n);               // restore heap order down one path
    return top;                      // old max is now parked at a[n+1]
}

class MaxHeap {
    int[] a; int n = 0;                   // a[0] wasted so the root sits at 1
    MaxHeap(int cap) { a = new int[cap + 1]; }

    int max() { return a[1]; }            // the root -- Theta(1)

    void insert(int x) {
        a[++n] = x;                       // append at the end: tree stays complete
        swim(a, n);                       // restore heap order up one path
    }
    int delMax() {
        int top = a[1];                   // the maximum, to be returned
        int t = a[1]; a[1] = a[n]; a[n] = t;   // move max to the end

struct MaxHeap { a: Vec<i32>, n: usize }

impl MaxHeap {

    fn max(&self) -> i32 { self.a[1] }                  // the root -- Theta(1)

    fn insert(&mut self, x: i32) {
        self.a.push(x);                 // append at the end: tree stays complete
        self.n += 1;
        swim(&mut self.a, self.n);      // restore heap order up one path
    }
    fn del_max(&mut self) -> i32 {
        let top = self.a[1];            // the maximum, to be returned
        self.a.swap(1, self.n);         // move max to the end
        self.n -= 1;                    // shrink (old max parked past the end)
        sink(&mut self.a, 1, self.n);   // restore heap order down one path

    }
}

Example (insert 95) Start from the heap \(a = [\,90\;80\;70\;30\;60\;50\,]\) (\(n=6\)) and insert \(95\). Append at index \(7\), the right child of \(70\); it beats \(70\), then beats \(90\), swimming twice to the root: 
  append:  90 80 70 30 60 50 95        // 95 at a[7], child of a[3]=70
  swap@3:  90 80 95 30 60 50 70        // 95 > parent 70
  swap@1:  95 80 90 30 60 50 70        // 95 > parent 90, now at root
The left subtree \((80, 30, 60)\) never moves — swim touches one path. The displaced keys \(70\) and \(90\) each slide down exactly one level. Result: \([\,95\;80\;90\;30\;60\;50\;70\,]\).

delMax: swap to the end, then sink

Example (delMax undoes the insert) Continue from \(a = [\,95\;80\;90\;30\;60\;50\;70\,]\) (\(n=7\)) and call delMax. The answer is \(a[1]=95\). Swap it with the last key \(70\) and shrink to \(n=6\); the new root \(70\) then sinks, comparing against both children and following the larger each time: 
  swap 95↔70, shrink:  70 80 90 30 60 50        // 95 parked past the end
  70 vs max(80,90)=90:  90 80 70 30 60 50        // swap with the larger child
  70 vs child 50:       90 80 70 30 60 50        // 70 ≥ 50, stop

5. Building a heap, and why it is \(\Theta(n)\)

Given \(n\) keys already sitting in an array in arbitrary order, make the array a heap. The obvious way treats it as \(n\) insertions — swim each element in turn: 

  for i = 1 .. n:  swim(i)          // n successive inserts

Its worst case is ascending input: each new key is the largest so far and swims its full depth. The total is the sum of the nodes’ depths, and in a complete tree most nodes lie near the bottom at depth \(\approx \log n\), so the sum is \(\Theta(n\log n)\). With all the data in hand up front we can do better.

Heapify: sink bottom-up

Leave the keys where they are (a complete tree with the wrong values) and sink every internal node, from the last one down to the root: 

  for k = n/2 down to 1:  sink(k)

Two facts make this work. The nodes past \(\lfloor n/2\rfloor\) are leaves (Section 3), already one-node heaps, so we skip them. And processing bottom-up means that when sink(k) runs, the two subtrees under \(k\) have already been sunk — they are heaps — so one sink merges two working heaps under a new root. Going top-down would fail: sinking the root first pushes it into subtrees that are still garbage, and nothing ever re-fixes the top.

Correctness

Lemma (one sink) If both subtrees rooted at the children of \(k\) are heaps, then sink(k) makes the whole subtree rooted at \(k\) a heap.
Proof. The only possibly-misplaced node is the one at \(k\); everything below it already satisfies heap order. sink promotes the larger child, call it \(c\). Then \(c\) was \(\ge\) its sibling (that is what “larger child” means) and \(\ge\) everything in its own former subtree (it was that subtree’s root, a heap), so after the swap \(c\) dominates both of its new children — the top edge is now correct. The demoted key sits at \(c\)’s old position with two heap subtrees beneath it and is the only node that can still violate order: the identical one-bad-node picture, one level lower. By induction on the height of \(k\) the process ends with \(k\) rooting a heap, either because the key comes to dominate its children or because it reaches a leaf. ∎
Proposition (heapify is correct) After the loop for k = n/2 down to 1: sink(k), the array is a heap.
Proof (loop invariant). Maintain: when the loop is about to process \(k\), every node with index \(\gt k\) roots a valid heap. Initialization (\(k=\lfloor n/2\rfloor\)): the nodes \(\lfloor n/2\rfloor+1, \dots, n\) are leaves, each a one-node heap. Maintenance: the children of \(k\) have indices \(2k, 2k+1 \gt k\), so by the invariant they root heaps; the Lemma then makes \(k\) root a heap, and no node \(\gt k\) was touched, so the invariant holds for \(k-1\). Termination: after \(k=1\), node \(1\) roots a heap — the whole array. ∎

This is the same establish–maintain–harvest shape used for the algorithms of Chapter 1, and it is short precisely because Section 3 did the work: “every subtree is a heap” supplied the right induction unit and “leaves are the back half” supplied the base case.

Why the cost is \(\Theta(n)\)

Here is the surprise. Repeated-insert pays each node its depth (swim rises from a leaf); heapify pays each node its height (sink falls toward the leaves). In a complete tree almost every node lives near the bottom, with large depth but tiny height, so building bottom-up bills every node for the cheap direction.

Count it exactly for a full tree of height \(h\), where \(n = 2^{h+1}-1\). A node at depth \(d\) has height \(h-d\), there are \(2^d\) of them, and its sink costs at most \(h-d\) swaps. Summing over levels,

Theorem (linear build) Building a heap of \(n\) keys by bottom-up sinks costs fewer than \(n\) swaps; heapify runs in \(\Theta(n)\) time.
Proof. Let \(S\) be the total number of swaps on a full tree of height \(h\): \[ S = \sum_{d=0}^{h} 2^d\,(h-d) = h\cdot 1 + (h-1)\cdot 2 + (h-2)\cdot 4 + \dots + 1\cdot 2^{h-1}. \] Double it, which shifts every term one power of two to the right, and line the two rows up by matching powers: 
  S  =  h·1 + (h−1)·2 + (h−2)·4 + ··· + 1·2^(h−1)
  2S =         h·2   + (h−1)·4 + ··· + 2·2^(h−1) + 1·2^h
In each shared power-of-two column the \(2S\) coefficient exceeds the \(S\) coefficient by exactly one, so subtracting collapses the sum: \[ 2S - S = -h\cdot 1 + \big(2 + 4 + \dots + 2^{h}\big) = -h + \big(2^{h+1}-2\big), \] using the geometric sum \(2 + 4 + \dots + 2^h = 2^{h+1}-2\). Since \(2S-S = S\), \[ S = 2^{h+1} - h - 2. \] On a full tree \(2^{h+1} = n+1\), so \(S = (n+1) - h - 2 = n - h - 1 \lt n\): strictly fewer swaps than keys. For an arbitrary complete tree of height \(h\) we have \(2^h \le n\), and \(S \le 2^{h+1}-h-2 \lt 2\cdot 2^h \le 2n\), so the swap count is \(\Theta(n)\); each swap does \(O(1)\) comparisons, and the algorithm must at least inspect each key, so the total is \(\Theta(n)\). ∎

Said in dollars: the root’s one expensive sink is a single node’s bill, while the crowd of shallow-sinking nodes at the bottom rides nearly free.

methodpays per nodecost
repeated insertits depth (swim)\(\Theta(n\log n)\)
heapifyits height (sink)\(\Theta(n)\)

Both yield a heap; heapify wins whenever the keys are all present up front. A live priority queue, whose keys arrive over time, has no choice but to swim them one at a time. (This is why heapifying a live PQ on every insert would be foolish — \(\Theta(n)\) per insert, worse than a single \(\Theta(\log n)\) swim.)

A \(\Theta(n)\) build does not beat the sorting bound Comparison sorting needs \(\Omega(n\log n)\) (Chapter 13), so did a linear-time heap build cheat it? No: a heap is not sorted. Heap order constrains only ancestor–descendant pairs (Section 3), pinning down far less than a full ordering — exponentially many arrays are valid heaps of the same multiset. We paid \(\Theta(n)\) for correspondingly little structure. Extracting the keys in sorted order still costs \(n\) removals of \(\Theta(\log n)\) each — that is heapsort, next. Sedgewick & Wayne make the same point: partial order is cheap to build, full order is not.

6. Heapsort

Selection sort repeatedly finds the maximum of the unsorted part by a \(\Theta(n)\) scan, for \(\Theta(n^2)\) overall. A heap is a find-max machine: replace the scan with delMax and each of the \(n\) rounds drops from \(\Theta(n)\) to \(\Theta(\log n)\). Better yet, delMax already swaps the maximum into the last live slot before shrinking — exactly where a descending selection pass would place it — so the sorted region grows in the same array at no extra cost. That is heapsort: heapify, then sort down. 

// In-place heapsort of a 1-based array a[1..n] (a[0] unused), ascending.
// Theta(n log n), Theta(1) extra space, NOT stable. Uses sink from swim-sink.
void heapsort(int a[], int n) {
    for (int k = n / 2; k >= 1; k--)   // build-heap bottom-up: Theta(n)
        sink(a, k, n);                 // every internal node, last to first
    for (int end = n; end >= 2; end--) {   // sortdown: n-1 delMax steps
        std::swap(a[1], a[end]);       // park the current max at the boundary
        sink(a, 1, end - 1);           // re-heapify the shrunken heap a[1..end-1]
    }
}
// In-place heapsort of a 1-based array a[1..n] (a[0] unused), ascending.
// Theta(n log n), Theta(1) extra space, NOT stable. Uses sink from swim-sink.
static void heapsort(int[] a, int n) {
    for (int k = n / 2; k >= 1; k--)   // build-heap bottom-up: Theta(n)
        sink(a, k, n);                 // every internal node, last to first
    for (int end = n; end >= 2; end--) {   // sortdown: n-1 delMax steps
        int t = a[1]; a[1] = a[end]; a[end] = t;   // park the max at the boundary
        sink(a, 1, end - 1);           // re-heapify the shrunken heap a[1..end-1]
    }
}
# In-place heapsort of a 1-based list a[1..n] (a[0] unused), ascending.
# Theta(n log n), Theta(1) extra space, NOT stable. Uses sink from swim-sink.
def heapsort(a, n):
    for k in range(n // 2, 0, -1):     # build-heap bottom-up: Theta(n)
        sink(a, k, n)                  # every internal node, last to first
    for end in range(n, 1, -1):        # sortdown: n-1 delMax steps
        a[1], a[end] = a[end], a[1]    # park the current max at the boundary
        sink(a, 1, end - 1)            # re-heapify the shrunken heap a[1..end-1]
// In-place heapsort of a 1-based slice a[1..=n] (a[0] unused), ascending.
// Theta(n log n), Theta(1) extra space, NOT stable. Uses sink from swim-sink.
fn heapsort(a: &mut [i32], n: usize) {
    for k in (1..=n / 2).rev() {       // build-heap bottom-up: Theta(n)
        sink(a, k, n);                 // every internal node, last to first
    }
    for end in (2..=n).rev() {         // sortdown: n-1 delMax steps
        a.swap(1, end);                // park the current max at the boundary
        sink(a, 1, end - 1);           // re-heapify the shrunken heap a[1..=end-1]
    }
}

source: code/heap/heapsort.cpp · code/heap/heapsort.java · code/heap/heapsort.py · code/heap/heapsort.rs

The first loop is the \(\Theta(n)\) build of Section 5. The second runs \(n-1\) rounds: swap the root (the current maximum) to the boundary a[end], exclude it by passing end - 1 as the heap size, and sink the new root through the shrunken heap.

Invariant (sortdown) Before each round, a[1..end] is a heap holding the \(end\) smallest keys and a[end+1..n] holds the remaining keys, sorted and in their final positions.

Initialization is the post-heapify state (the whole array is the heap, the tail empty). Each round removes the maximum of the current heap; it is \(\ge\) every key already parked, and it lands immediately to their left, so the sorted tail grows correct by one. At termination (\(end=1\)) the lone remaining key is the smallest, and the array is fully sorted — ascending, even though we used a max-heap, precisely because each maximum is parked at the far right. A min-heap with the same sort-down would produce descending order, a classic gotcha.

Proposition (heapsort cost) Heapsort sorts \(n\) keys in \(O(n\log n)\) time — \(\Theta(n\log n)\) in the worst case — using \(\Theta(1)\) extra space.
Proof. The build is \(\Theta(n)\) (Section 5). The sortdown does \(n-1\) rounds; the round on a heap of size \(m\) sinks the new root through height \(\le \lfloor\log_2 m\rfloor \le \lfloor\log_2 n\rfloor\), so the whole sortdown is \(O(n\log n)\), and with the linear build the total is \(O(n\log n)\). For the matching lower bound, note that after the build the heap’s deepest level holds up to \(\lceil n/2\rceil\) keys; on a worst-case input the key promoted to the root each round is small and sinks essentially the full current height, and the first \(n/2\) rounds each sink through a heap of size \(\gt n/2\) and thus height \(\ge \log_2 n - 1\), giving \(\Omega(n\log n)\) — heapsort’s worst case is \(\Theta(n\log n)\) (Chapter 13). Only a constant number of scalar variables are used beyond the input array, so the extra space is \(\Theta(1)\). ∎

Heapsort is not stable: parking a maximum at a distant slot reorders equal keys. Its niche is clear against its rivals:

sortworst caseextra spacestablein practice
mergesort\(\Theta(n\log n)\)\(\Theta(n)\)yesneeds a buffer
quicksort\(\Theta(n^2)\)\(\Theta(\log n)\)nofastest constants
heapsort\(\Theta(n\log n)\)\(\Theta(1)\)nothe safety net

Heapsort is the only one of the three that is simultaneously guaranteed \(\Theta(n\log n)\) and in place. It loses to quicksort on constant factors and cache behavior — its accesses jump around the array by powers of two rather than sweeping linearly — and it is not stable, so it is nobody’s default. But it is everybody’s fallback: this is exactly why std::sort (introsort) begins with quicksort and switches to heapsort when the recursion goes too deep, buying quicksort’s speed with heapsort’s worst-case guarantee. The full sorting comparison is Chapter 13.

7. The priority-queue landscape

The binary heap is the default priority queue, not the only one. A few variations are worth knowing because two of them show up in later chapters.

Indexed priority queues

Eager forms of Dijkstra (Chapter 9) and Prim (Chapter 11) need to decrease the key of an item already in the queue — “this vertex just got closer.” A plain heap cannot even find that item (Section 3: no search order). An indexed priority queue adds a companion array mapping each item to its current heap position, updated on every swap, so decreaseKey(i, v) can locate item \(i\) in \(O(1)\) and then swim it, and contains(i) is \(O(1)\). The heap operations stay \(\Theta(\log n)\); the cost is a whole second array — the first crack in “the binary heap does everything.” (The lazy alternative, used by default in Chapter 9, sidesteps this: push a duplicate and skip stale entries when they pop.)

Beyond two children: \(d\)-ary heaps

Nothing forces the branching factor to be two. Give each node \(d\) children and the same array trick applies, with children of \(k\) at \(d(k-1)+2, \dots, dk+1\). The tree grows shallower, height \(\log n / \log d\), so swim (and decreaseKey) speeds up; but each sink level now compares \(d\) children, so delMax slows. The common sweet spot is \(d=4\): shallower and cache-friendlier (four children share a cache line), and a fine choice for Dijkstra on dense graphs where decrease-keys vastly outnumber deletions. This is the same “widen the node to lower the height” trade as the B-trees of Chapter 5; and \(d=n\) degenerates to an unsorted array — the very first row of Section 1’s table, so the simple implementations are just the endpoints of the \(d\)-ary family.

† via the indexed-PQ map · ‡ amortized. In practice binary and \(d\)-ary heaps win on constants and cache; the exotic structures buy one specific operation. A leftist heap keeps each node’s right spine short (\(\le \log n\)) and melds two heaps by merging right spines in \(\Theta(\log n)\), after which insert and delMax are both just special cases of meld. A Fibonacci heap (Cormen et al.) procrastinates on cleanup to reach \(\Theta(1)\) amortized decrease-key, which gives Dijkstra its best theoretical bound — yet real Dijkstra still uses a binary or \(4\)-ary heap because the constant factors dominate.

Library priority queues, and the bug that follows You will rarely hand-roll one. C++ std::priority_queue is a max-heap; Java java.util.PriorityQueue and Python heapq are min-heaps (negate keys, or push \(-k\), for max behavior). All are binary heaps: push/pop \(\Theta(\log n)\), build \(\Theta(n)\), peek \(\Theta(1)\). Getting the direction wrong — a min-heap where you meant max — is the single most common heap bug; the others are a sink that fails to pick the larger child, off-by-one errors from running 1-based math on a 0-based array, and doing \(n\) inserts (\(\Theta(n\log n)\)) where heapify (\(\Theta(n)\)) was intended. None of the three libraries ships decreaseKey, which is why Chapter 9’s Dijkstra deletes lazily.
Where this heap reappears The heap is the engine inside three later algorithms. Huffman coding (Chapter 12) builds an optimal prefix code by repeatedly pulling the two least-frequent nodes from a min-heap and pushing their merge, \(n\) symbols in \(\Theta(n\log n)\). Dijkstra (Chapter 9) and Prim (Chapter 11) grow a tree by repeatedly pulling the nearest frontier vertex from a min-heap. In each, a priority queue drives a greedy loop — “take the best remaining” — and tonight’s \(\Theta(\log n)\) operations are what make that loop efficient.

8. Exercises (with solutions)

  1. Starting from an empty max-heap, insert \(5, 15, 10, 20, 8\) in that order. Give the array (1-based) after each insert, and the final tree’s shape.
    solution After \(5\): \([5]\). After \(15\): append then swim past \(5\) ⇒ \([15, 5]\). After \(10\): append at index \(3\); \(10 \lt 15\), no swim ⇒ \([15, 5, 10]\). After \(20\): append at \(4\), swim past \(5\) (index \(2\)) then past \(15\) (index \(1\)) ⇒ \([20, 15, 10, 5]\). After \(8\): append at \(5\); parent is \(a[2]=15 \ge 8\), no swim ⇒ \([20, 15, 10, 5, 8]\). Final tree: root \(20\); children \(15\) and \(10\); \(15\) has children \(5\) and \(8\). Check: \(20\ge15,10\) and \(15\ge5,8\). ✓
  2. Call delMax on the heap \([20, 15, 10, 5, 8]\). Show the returned key and the resulting array.
    solution Return \(a[1]=20\). Swap it with the last key \(8\) and shrink to \(n=4\): \([8, 15, 10, 5]\). Sink \(8\): its children are \(15\) and \(10\), larger is \(15\); \(8 \lt 15\) so swap ⇒ \([15, 8, 10, 5]\). Now \(8\) is at index \(2\) with the single child \(a[4]=5\); \(8 \ge 5\), stop. Result \([15, 8, 10, 5]\), which is a valid heap (\(15\ge8,10\), \(8\ge5\)).
  3. Build a heap from \([4, 7, 3, 9, 1, 6, 2]\) (\(n=7\)) by bottom-up heapify. Show the array after each sink.
    solution Start at \(k=\lfloor 7/2\rfloor=3\). sink 3: \(a[3]=3\), children \(6, 2\), larger \(6\); swap ⇒ \([4, 7, 6, 9, 1, 3, 2]\). sink 2: \(a[2]=7\), children \(9, 1\), larger \(9\); swap ⇒ \([4, 9, 6, 7, 1, 3, 2]\). sink 1: \(a[1]=4\), children \(9, 6\), larger \(9\); swap ⇒ \([9, 4, 6, 7, 1, 3, 2]\); now \(4\) at index \(2\) has children \(7, 1\), larger \(7\); swap ⇒ \([9, 7, 6, 4, 1, 3, 2]\). Final heap: verify \(9\ge7,6\), \(7\ge4,1\), \(6\ge3,2\). ✓
  4. Explain why heapify sinks nodes from \(\lfloor n/2\rfloor\) down to \(1\), and why the reverse order (\(1\) up to \(\lfloor n/2\rfloor\)) does not produce a heap.
    solution The one-sink lemma requires both child subtrees of \(k\) to already be heaps. Processing indices in decreasing order guarantees this: the children \(2k, 2k+1\) exceed \(k\), so they were sunk earlier, and the leaves past \(\lfloor n/2\rfloor\) are heaps to begin with. Going upward breaks the precondition — sinking the root first pushes its key into subtrees that are still unsorted, so the key stops at a wrong place, and no later step revisits the top. The result need not be heap-ordered.
  5. A max-heap holds \(15\) distinct keys in \(a[1..15]\). Where can the second-largest key be? The third-largest? The smallest? At what cost can you verify that an array is a heap?
    solution Every key larger than a given key must be one of its ancestors, and a node at depth \(d\) has exactly \(d\) ancestors. The second-largest has one key above it, so depth \(\le 1\): index \(2\) or \(3\) (a child of the root). The third-largest has depth \(\le 2\): indices \(2\)–\(7\). The smallest is a leaf (any node with a child is larger than it): indices \(8\)–\(15\). Verifying the heap is just checking \(a[k] \ge a[2k]\) and \(a[k] \ge a[2k+1]\) for every internal \(k = 1..7\), i.e. \(\Theta(n)\).
  6. Run two rounds of heapsort sortdown on the heap \([9, 7, 6, 4, 1, 3, 2]\) (\(n=7\)). Show the array after each round and explain why the output ends up ascending.
    solution Round \(end=7\): swap \(a[1]=9\) with \(a[7]=2\) ⇒ \([2,7,6,4,1,3,\mid 9]\); sink \(2\) in \(a[1..6]\): larger child of the root is \(7\), swap ⇒ \([7,2,6,4,1,3\mid 9]\); \(2\) at index \(2\) has larger child \(4\), swap ⇒ \([7,4,6,2,1,3\mid 9]\). Round \(end=6\): swap \(a[1]=7\) with \(a[6]=3\) ⇒ \([3,4,6,2,1\mid 7,9]\); sink \(3\) in \(a[1..5]\): larger child is \(6\), swap ⇒ \([6,4,3,2,1\mid 7,9]\); \(3\) at index \(3\) has no child, stop. After two rounds: \([6,4,3,2,1,7,9]\) with sorted tail \(7, 9\). The order is ascending because each round removes the current maximum and parks it at the right boundary, filling the final positions from largest down to smallest.
  7. True or false, with justification: “A descending-sorted array is always a valid max-heap.”
    solution True. If \(a[1] \ge a[2] \ge \dots \ge a[n]\), then for every \(k\) and any child index \(j \in \{2k, 2k+1\}\) we have \(j \gt k\), so \(a[k] \ge a[j]\) by the sortedness — heap order holds at every node, and the array is complete by construction. (The converse fails: a max-heap need not be sorted, e.g. \([90, 30, 80]\) is a heap but \(30 \lt 80\). And an ascending-sorted array is instead a valid min-heap.)
  8. You build a min-heap and run the identical sortdown loop — swap the root to the current boundary, shrink, sink. Is the output ascending or descending, and what would you change to reverse it?
    solution Descending. Each round now removes the current minimum and parks it at the right end, so the largest keys end up on the left: the array comes out from largest (index \(1\)) to smallest. To get ascending order from a min-heap you would park each extracted minimum at the front instead (growing the sorted region leftward), or simply use a max-heap as in the main text. This direction mismatch is a frequent bug — the sort direction is fixed by the heap’s orientation, not by intent.
