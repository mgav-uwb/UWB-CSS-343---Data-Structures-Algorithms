Chapter 8 — Graphs I: Representation, DFS, BFS & Topological Sort

CSS 343 — Data Structures, Algorithms & Discrete Mathematics II. Course text, primary reading for Lecture 8.
The most general structure in the course: vertices joined by edges. How to store a graph (the adjacency list), two ways to explore it (depth-first and breadth-first) with proofs of what each computes, and the marquee application on directed acyclic graphs (topological sort, two ways). With worked traces, proofs of the \(\Theta(V+E)\) bounds and of BFS shortest paths, links to the interactive demos, and exercises.

Trees were graphs with a restriction: connected and acyclic, exactly one path between any two nodes. This chapter removes the restriction. A graph is any set of vertices connected by edges — cycles allowed, multiple paths allowed, disconnected pieces allowed — and that generality is why graphs model road maps, social networks, the web, task dependencies, and circuits alike. The price of the generality is that we can go in circles, so every algorithm here carries a visited set; and the reward is that two tiny variations of one search — change only the container that holds the frontier — answer strikingly different questions.

1. Vertices, edges, and vocabulary

Definition (graph) A graph \(G = (V, E)\) is a set \(V\) of vertices and a set \(E\) of edges, each edge joining a pair of vertices. In an undirected graph an edge \(\{u,v\}\) connects \(u\) and \(v\) symmetrically; in a directed graph (a digraph) an edge \((u,v)\), written \(u \to v\), goes one way only, and \(u\to v\) does not imply \(v\to u\).

Undirected edges model symmetric relations — friendship, two-way roads; directed edges model one-way relations — web links, one-way streets, “must happen before.” The distinction matters for every algorithm, and topological sort (§6) only makes sense on a digraph. Here is a small undirected graph used throughout this section: 

  0 —— 1       V = { 0, 1, 2, 3 }
  |  / |       E = { 0–1, 0–2, 1–2, 1–3 }
  | /  |
  2    3       4 vertices, 4 edges, a cycle 0–1–2–0

The shared vocabulary for the rest of the unit:

  • adjacent — two vertices joined by an edge;
  • degree of a vertex — the number of edges at it (for a digraph, in-degree and out-degree separately);
  • path — a sequence of edges leading from one vertex to another; its length is the number of edges;
  • cycle — a path that returns to its start;
  • connected (undirected) — a path exists between every pair of vertices.

2. Counting edges: the handshake lemma and trees

Two counting facts about edges anchor the cost analyses later, and place trees within the family of graphs.

Lemma (handshake) In an undirected graph, \(\displaystyle\sum_{v \in V} \deg(v) = 2\,|E|\). In a digraph, \(\displaystyle\sum_{v} \text{in-deg}(v) = \sum_{v} \text{out-deg}(v) = |E|\).
Proof. Count the pairs (vertex, edge incident to it) two ways. Grouping by vertex, vertex \(v\) contributes \(\deg(v)\) pairs, for a total of \(\sum_v \deg(v)\). Grouping by edge, each undirected edge has exactly two endpoints and so contributes exactly \(2\) pairs, for a total of \(2|E|\). The two counts are equal. For a digraph, count (vertex, out-edge) pairs: each edge has one tail, so the out-degree sum is \(|E|\); symmetrically for heads and the in-degree sum. ∎

On the running graph (§3) the total degrees are \(3, 3, 2, 5, 3, 3, 1, 2\), summing to \(22 = 2\cdot 11\). The lemma looks like trivia but it is precisely why “scan every vertex’s entire neighbor list” costs \(V + 2E\), i.e. \(\Theta(V+E)\) — we cash it in for the cost of DFS and BFS. (The average degree \(2E/V\) also measures how sparse a graph is.)

How many edges can a graph have?

In a simple undirected graph (no self-loops, no repeated edges), the edge count ranges from \(0\) to one edge per pair of vertices:

\[ 0 \;\le\; |E| \;\le\; \binom{V}{2} = \frac{V(V-1)}{2} = \Theta(V^2). \]

A graph is sparse when \(|E| = O(V)\) and dense when \(|E| = \Theta(V^2)\). Real graphs are almost always sparse: a road intersection meets a handful of streets, a person has hundreds of friends and not billions. This single fact will choose our representation in §3.

Trees are the sparsest connected graphs

Proposition (trees) A tree is a connected acyclic graph. Every tree on \(V\) vertices has exactly \(V - 1\) edges. Moreover, any two of the three properties {connected, acyclic, \(|E| = V-1\)} imply the third.
Proof that a tree has \(V-1\) edges (induction on \(V\)). A tree with \(V \ge 2\) vertices has a leaf, a vertex of degree \(1\): otherwise every vertex has degree \(\ge 2\), and following unused edges out of vertices can never get stuck, so by the pigeonhole argument of §6 a walk must repeat a vertex and expose a cycle — contradicting acyclicity. Remove a leaf and its one edge: the result is still connected and acyclic, a tree on \(V-1\) vertices, which by induction has \(V-2\) edges. Adding the removed edge back gives \(V-1\). The base case \(V=1\) has \(0\) edges. The “two-of-three” claims follow similarly: adding one edge to a tree creates a cycle, and removing one disconnects it. ∎

So a tree is the minimum connected graph — drop any edge and it falls apart, add any edge and it gains a cycle. All the tree structures of earlier chapters were graphs with these restrictions; this chapter works without them. (The fact previews the minimum spanning tree of Chapter 11: the cheapest set of edges that keeps a graph connected is necessarily a tree.)

3. Representing a graph

Two representations, chosen by whether the graph is dense or sparse.

Adjacency matrix

A \(V \times V\) matrix with \(a[u][v] = 1\) when \(u\)–\(v\) is an edge and \(0\) otherwise: 

       0 1 2 3
    0 [0 1 1 0]      edge 0–1?  → a[0][1] = 1   (O(1) lookup)
    1 [1 0 1 1]
    2 [1 1 0 0]
    3 [0 1 0 0]      space: Theta(V^2)

“Is there an edge \(u\)–\(v\)?” is a single array access. The fatal flaw is space: \(\Theta(V^2)\) regardless of how few edges exist. For a social network of a billion people with a few hundred friends each, the matrix is \(10^{18}\) cells, almost all zero — impossible. Matrices suit only small or dense graphs.

Adjacency list

One list of neighbors per vertex:

  0 → [1, 2]        space: Theta(V + E)
  1 → [0, 2, 3]     iterate 1's neighbors: O(deg 1)
  2 → [0, 1]        edge query 0–2: scan 0's list
  3 → [1]

The list stores only the edges that exist, so its space is \(\Theta(V+E)\) — proportional to the actual graph. Iterating a vertex’s neighbors, the inner loop of every search, is \(O(\deg u)\), which is optimal. Its one weakness is the edge query “is \(u\)–\(v\) an edge?” (scan a list), but graph algorithms rarely ask that — they iterate neighbors. Hence the adjacency list is the default.

4. Depth-first search

The basic search problem: given a start vertex \(s\), visit every vertex reachable from \(s\), each exactly once. Trees never revisit a node because there is one path to each; graphs break that — a cycle would send a naive search around forever, and multiple paths would visit a vertex several times. The fix is a \(\texttt{seen}\) array: mark a vertex the first time you reach it and never process it twice. Both searches in this chapter rely on it.

Depth-first search follows one path as far as it can, then backtracks and tries the next — preorder tree traversal generalized to graphs, with the recursion itself doing the backtracking: 

void dfs(Graph& g, int u, vector<bool>& seen) {
    seen[u] = true;                 // visit u
    for (int v : g.adj[u])          // each neighbor
        if (!seen[v])
            dfs(g, v, seen);        // recurse; backtrack on return
}
void dfs(Graph g, int u, boolean[] seen) {
    seen[u] = true;                 // visit u
    for (int v : g.adj.get(u))      // each neighbor
        if (!seen[v])
            dfs(g, v, seen);        // recurse; backtrack on return
}
def dfs(g, u, seen):
    seen[u] = True                  # visit u
    for v in g.adj[u]:              # each neighbor
        if not seen[v]:
            dfs(g, v, seen)         # recurse; backtrack on return
fn dfs(g: &Graph, u: usize, seen: &mut Vec<bool>) {
    seen[u] = true;                 // visit u
    for &v in &g.adj[u] {           // each neighbor
        if !seen[v] {
            dfs(g, v, seen);        // recurse; backtrack on return
        }
    }
}

Example (a DFS trace) On the directed diamond \(0\to1,\ 0\to2,\ 1\to3,\ 2\to3\) with lists \(0\to[1,2]\): 
  dfs(0): visit 0 → dfs(1): visit 1 → dfs(3): visit 3
          back to 1 (done) → back to 0
          → dfs(2): visit 2 → 3 already seen, SKIP
  visit order:  0  1  3  2
From \(0\) the search dives into \(1\), then \(3\) (a dead end), backs up to \(1\) (done) and to \(0\), then into \(2\); when \(2\) examines its edge to \(3\), that vertex is already visited, so the edge is skipped — the cycle guard at work. The whole \(1\)-branch finishes before \(2\) is touched: that is what “depth-first” means.
If instead \(0\)’s list were \([2,1]\), the order would be \(0\ 2\ 3\ 1\). The set of visited vertices is fixed by the graph; the order depends on how the adjacency lists are stored, so any order some DFS could produce is a valid DFS order.

DFS visits exactly the reachable set

Theorem (DFS = reachability) Calling \(\texttt{dfs}(s)\) marks a vertex \(v\) if and only if there is a path from \(s\) to \(v\).
Proof. (\(\Rightarrow\)) DFS only ever recurses along an edge out of an already-marked vertex, starting from \(s\); so following the recursive calls that led to \(v\) traces an actual path \(s \rightsquigarrow v\). Every mark is genuinely reachable.
(\(\Leftarrow\)) Suppose some reachable vertex is missed. Take a path \(s \rightsquigarrow w\) to a missed vertex \(w\), and let \(q\) be the first missed vertex along it, with \(p\) the vertex just before \(q\) (so \(p\) was visited, and \(p \to q\) is an edge). But \(\texttt{dfs}(p)\) loops over all of \(p\)’s out-edges, including \(p \to q\); it therefore either found \(q\) unvisited and recursed into it (marking \(q\)) or found \(q\) already visited. Either way \(q\) is marked — contradicting “\(q\) missed.” So nothing reachable is missed. ∎

The “first missed vertex on a path” argument is the template for many graph proofs. A single DFS reaches one connected piece; to cover a disconnected graph, restart DFS from every not-yet-seen vertex, and each restart discovers one connected component.

Cost: cashing in the handshake lemma

Proposition (DFS cost) DFS runs in \(\Theta(V + E)\) time on an adjacency list.
Proof. The \(\texttt{seen}\) check guarantees each vertex is processed once. Processing \(v\) does \(O(1)\) work plus a scan of its neighbor list, which is \(\deg(v)\) entries. Summing over vertices, \[ \sum_{v}\bigl(1 + \deg(v)\bigr) = \underbrace{\sum_v 1}_{V} + \underbrace{\sum_v \deg(v)}_{2E} = V + 2E = \Theta(V+E), \] the degree sum being \(2E\) by the handshake lemma (exactly \(E\) for a digraph’s out-degrees). ∎

This is optimal: any correct search must at least look at every vertex and every edge. BFS’s cost proof, next section, is word-for-word the same — the two searches differ only in the frontier container, not in the accounting.

Iterative DFS, and a recursion-depth caveat

Recursion uses the program’s call stack; the same search can manage an explicit stack instead: 

void dfs(Graph& g, int s) {
    stack<int> st; st.push(s);
    vector<bool> seen(g.V, false);
    while (!st.empty()) {
        int u = st.top(); st.pop();
        if (seen[u]) continue;
        seen[u] = true;             // visit u
        for (int v : g.adj[u])
            if (!seen[v]) st.push(v);
    }
}
void dfs(Graph g, int s) {
    Deque<Integer> st = new ArrayDeque<>();
    st.push(s);
    boolean[] seen = new boolean[g.V];
    while (!st.isEmpty()) {
        int u = st.pop();
        if (seen[u]) continue;
        seen[u] = true;             // visit u
        for (int v : g.adj.get(u))
            if (!seen[v]) st.push(v);
    }
}
def dfs(g, s):
    st = [s]
    seen = [False] * g.V
    while st:
        u = st.pop()
        if seen[u]:
            continue
        seen[u] = True              # visit u
        for v in g.adj[u]:
            if not seen[v]:
                st.append(v)
fn dfs(g: &Graph, s: usize) {
    let mut st = vec![s];
    let mut seen = vec![false; g.adj.len()];
    while let Some(u) = st.pop() {
        if seen[u] { continue; }
        seen[u] = true;             // visit u
        for &v in &g.adj[u] {
            if !seen[v] { st.push(v); }
        }
    }
}

The two versions visit the same set (the order can differ, since the stack reverses each neighbor list). The iterative form matters in production: recursive DFS’s depth equals the longest path it explores, and the call stack is small (often about a megabyte), so a graph with a million-vertex path overflows it. The explicit stack lives on the heap, which is far larger — prefer it for arbitrarily deep graphs.

Edge classification and cycle detection

As DFS runs it classifies every edge \(u \to v\) it examines:

  • a tree edge leads to an unvisited vertex (DFS recurses) — these form the DFS tree;
  • a back edge leads to a vertex still on the recursion stack (an ancestor of \(u\)) — it closes a loop back to that ancestor, so it is the signature of a cycle;
  • an edge to a vertex already finished is harmless (no cycle).

Cycle detection in a digraph is therefore exactly “did DFS find a back edge?”, tracked with three states per vertex (unvisited / on-stack / finished). We use this in §6.

5. Breadth-first search

Breadth-first search visits all vertices at distance \(1\) from the start, then all at distance \(2\), and so on — expanding in rings. Swap DFS’s stack for a queue (FIFO) and the search fans out level by level instead of diving. That single change makes BFS compute shortest paths in unweighted graphs, which we now prove. 

void bfs(Graph& g, int s) {
    queue<int> q; q.push(s);
    vector<bool> seen(g.V, false); seen[s] = true;
    while (!q.empty()) {
        int u = q.front(); q.pop();     // visit u
        for (int v : g.adj[u])
            if (!seen[v]) {
                seen[v] = true;         // mark on ENQUEUE
                q.push(v);
            }
    }
}
void bfs(Graph g, int s) {
    Queue<Integer> q = new ArrayDeque<>();
    q.add(s);
    boolean[] seen = new boolean[g.V];
    seen[s] = true;
    while (!q.isEmpty()) {
        int u = q.poll();               // visit u
        for (int v : g.adj.get(u))
            if (!seen[v]) {
                seen[v] = true;         // mark on ENQUEUE
                q.add(v);
            }
    }
}

Mark on enqueue, not on dequeue A vertex must be marked the instant it is enqueued, not when it is later dequeued. Otherwise two different neighbors could each enqueue it before either is processed, so it enters the queue twice, is processed twice, and receives a wrong distance. In the diamond both \(1\) and \(2\) point at \(3\); marking on enqueue guarantees each vertex is queued exactly once — essential for correctness and for the \(\Theta(V+E)\) bound (the same \(V + \sum\deg\) sum as DFS).
Example (a BFS trace) The same diamond \(0\to1,\ 0\to2,\ 1\to3,\ 2\to3\), one row per step — generated from the demo engine’s own trace, so it matches the interactive demo step for step:

Why the distances are shortest

The proof rests on a structural invariant of the queue.

Lemma (the queue is sorted by distance) At every moment, the vertices in the queue have distances that take at most two consecutive values \(d\) and \(d+1\), with all the \(d\)s ahead of all the \(d+1\)s. Consequently vertices are dequeued in non-decreasing order of distance.
Proof (induction on the steps). Initially the queue is \([s]\), one value. Assume it reads \([\,d\ \cdots\ d\mid d{+}1\ \cdots\ d{+}1\,]\). Dequeuing the front, a distance-\(d\) vertex \(u\), we enqueue its undiscovered neighbors; each such neighbor is reached by one edge from a distance-\(d\) vertex, so it is assigned distance \(d+1\) and appended at the back, preserving the shape. Once the last \(d\) is dequeued the queue holds only \((d+1)\)s, and the argument repeats with \(d\) advanced. No third value ever appears before the current ones are gone. ∎
Theorem (BFS shortest paths) For every vertex \(v\) reachable from \(s\), the label \(\texttt{dist}[v]\) computed by BFS equals \(\delta(v)\), the length of a fewest-edge path from \(s\) to \(v\).
Proof (two inequalities). (\(\ge\)) BFS assigns \(\texttt{dist}[v]\) only by following actual edges, so there is a real \(s\to v\) path of exactly \(\texttt{dist}[v]\) edges; no real path can be shorter than the shortest, giving \(\texttt{dist}[v] \ge \delta(v)\).
(\(\le\)) Take a shortest path \(s = v_0 \to v_1 \to \cdots \to v_k = v\) with \(k = \delta(v)\), and show \(\texttt{dist}[v_i] \le i\) by induction. It holds for \(v_0\) (\(\texttt{dist}[s]=0\)). Assume \(\texttt{dist}[v_{i-1}] \le i-1\). When \(v_{i-1}\) is dequeued it scans the edge to \(v_i\); either \(v_i\) is still undiscovered, so it is discovered now with \(\texttt{dist}[v_i] = \texttt{dist}[v_{i-1}] + 1 \le i\), or it was discovered earlier, and by the queue lemma an earlier discovery means a distance no larger, so again \(\texttt{dist}[v_i] \le i\). Thus \(\texttt{dist}[v] = \texttt{dist}[v_k] \le k = \delta(v)\). The two inequalities give \(\texttt{dist}[v] = \delta(v)\). ∎

The argument uses that every edge adds exactly one to the distance; it breaks the moment edges carry different weights, which is why weighted shortest paths need a different algorithm (Dijkstra, Chapter 9).

One detail is load-bearing: a vertex is finalized when it leaves the frontier, not when it enters, and the frontier may briefly hold duplicates that the continue skips. For a stack or queue that is cosmetic; for the priority-queue rows it is the whole game — finalizing a vertex the moment it is added would lock in the first cost seen, which is the classic bug Dijkstra must avoid. When you meet a new graph problem, the question is less “which algorithm?” than “what should the frontier prioritize?”

Grids are implicit graphs

A grid or maze is a graph whose vertices are cells and whose edges join each cell to its four neighbors — but you never build the adjacency list. The neighbors of cell \((r,c)\) are \((r\pm1, c)\) and \((r, c\pm1)\), computed on the fly, skipping walls. The same BFS code then gives the fewest moves from a start cell to every reachable cell — the workhorse behind maze solving, game pathfinding, and image flood-fill.

6. Directed acyclic graphs and topological sort

Definition (DAG, topological order) A directed acyclic graph (DAG) is a digraph with no directed cycle. A topological order of a digraph is a listing of its vertices such that every edge points forward: for each edge \(u \to v\), \(u\) appears before \(v\).

A DAG models “must happen before” relationships — course prerequisites, compilation order, spreadsheet formulas, package installs — and a topological order is a schedule that respects all of them: do the vertices left to right and every prerequisite is already done. The diamond \(0\to1,\ 0\to2,\ 1\to3,\ 2\to3\) has two topological orders, \(0\,1\,2\,3\) and \(0\,2\,1\,3\), since \(1\) and \(2\) are independent. When does an order exist?

A cycle destroys it immediately: \(a \to b \to \cdots \to a\) would demand \(a\) before \(b\) before \(\cdots\) before \(a\), which no linear order satisfies. So a topological order exists only if the graph is a DAG. The converse — every DAG has one — is proved constructively by an algorithm, and the algorithm rests on one lemma.

Lemma (every DAG has a source) A source is a vertex of in-degree \(0\). Every non-empty DAG has at least one source.
Proof. Start at any vertex and, while the current vertex has some incoming edge \(u \to v\), step backward to \(u\). If this walk never got stuck, after \(V\) steps it would have listed \(V+1\) vertices; among only \(V\) distinct vertices, two must coincide (pigeonhole), and a repeat on a backward walk exhibits a directed cycle — impossible in a DAG. So the walk gets stuck, at a vertex with no incoming edge: a source. (Symmetrically, walking forward proves every DAG has a sink, an out-degree-\(0\) vertex.) ∎

Kahn’s algorithm

Repeatedly output a source and remove it, freeing its dependents: 

compute the in-degree of every vertex
queue all vertices with in-degree 0
repeat until the queue is empty:
    dequeue u  →  OUTPUT u
    for each edge u → v:  in-degree(v)--
        if in-degree(v) becomes 0:  enqueue v
Example (Kahn on the diamond) 
  0→1  0→2  1→3  2→3     in-deg: 0:0  1:1  2:1  3:2

  queue [0]                 (only source: 0)
  out 0 → dec 1, 2 → both hit 0 → queue [1, 2]
  out 1 → dec 3 → 3:1
  out 2 → dec 3 → 3:0 → queue [3]
  out 3
  order:  0 1 2 3    (all 4 output ⇒ the graph is acyclic)
Proposition (Kahn is correct, \(\Theta(V+E)\)) On a DAG, Kahn’s algorithm outputs all \(V\) vertices in a valid topological order, in \(\Theta(V+E)\) time. On a graph with a cycle it outputs fewer than \(V\) vertices.
Proof. By the source lemma a DAG always has an in-degree-\(0\) vertex available, so the algorithm never stalls early; deleting a vertex from a DAG leaves a DAG (removing vertices and edges cannot create a cycle), so by induction it proceeds to the end and outputs every vertex. An edge \(u \to v\) is processed — and \(v\)’s in-degree decremented — only when \(u\) is output; \(v\) is not output until its in-degree reaches \(0\), i.e. until all its predecessors are out, so \(u\) precedes \(v\): every edge points forward. Cost: computing in-degrees scans all lists once (\(\Theta(V+E)\)); each vertex is enqueued and dequeued once, and each edge is decremented exactly once, so the loop is \(\Theta(V+E)\). Finally, if the graph has a cycle, the cycle’s vertices each wait on another in the cycle, so none ever reaches in-degree \(0\); the algorithm halts with fewer than \(V\) outputs — which is a ready-made cycle detector. ∎
Example (Kahn on the running DAG) In-degrees: \(0{:}0,\ 1{:}1,\ 2{:}1,\ 3{:}3,\ 4{:}1,\ 5{:}2,\ 6{:}1,\ 7{:}2\). The only source is \(0\); outputting it drops \(1\) to in-degree \(0\) (and \(3\) to \(2\), \(5\) to \(1\)); each output in turn frees the next index, and breaking ties by smaller index gives the order \(0\ 1\ 2\ 3\ 4\ 5\ 6\ 7\). Topological sort via DFS post-order

A second route uses DFS. Run DFS (restarting at every unvisited vertex so all vertices finish); record each vertex’s finish time when its call returns; the answer is the reverse of finish order. 

dfs(u): ... recurse into all of u's unvisited neighbors ...
        on FINISHING u, push u onto a stack
answer = pop the stack   // reverse of finish order
Example (DFS post-order on the diamond) 
  dfs(0) → dfs(1) → dfs(3): 3 finishes   (finish #1)
           1 finishes                   (finish #2)
           dfs(2): 3 seen → 2 finishes  (finish #3)
           0 finishes                   (finish #4)

  finish order:  3 1 2 0
  REVERSE     :  0 2 1 3     ✓ a topological order
Deepest vertices finish first; \(0\) finishes last, because everything depends on it, and lands first when reversed. Compare with Kahn’s \(0\,1\,2\,3\): different tie-breaking, both valid.
Theorem (reverse post-order is topological) In a DAG, for every edge \(u \to v\), \(v\) finishes before \(u\) in DFS. Hence listing vertices in reverse order of finish time places every edge forward.
Proof. When \(\texttt{dfs}(u)\) examines the edge \(u \to v\), vertex \(v\) is in one of three states. (1) Unvisited: then \(\texttt{dfs}(v)\) runs inside \(\texttt{dfs}(u)\) and returns before it, so \(v\) finishes first. (2) Already finished: then \(v\) finished before \(u\) trivially. (3) On the recursion stack (an ancestor of \(u\)): then a path \(v \rightsquigarrow u\) exists, and together with \(u \to v\) that is a directed cycle — impossible in a DAG. So only cases (1) and (2) occur, and in both \(v\) finishes before \(u\). Reversing finish order therefore lists \(u\) before \(v\) for every edge. ∎

Case (3) is exactly the back edge of §4: on a general digraph its occurrence is precisely the cycle detector, so the same three-case analysis does double duty as topological sort and cycle detection. Two free cycle detectors, then: Kahn (fewer than \(V\) vertices come out) and DFS (a back edge appears). A digraph has a topological order if and only if it is acyclic.

Many valid orders, and applications

Whenever two vertices have no path between them (like \(1\) and \(2\) in the diamond), either may come first, so a topological order is not unique unless the DAG is a single chain. A grader — and every real consumer — must therefore check the property (all edges forward), not compare against one specific sequence. Topological sort is DFS’s most practically important application: build systems compile in dependency order, package managers install dependencies first, spreadsheets recompute cells in formula order, and a “circular dependency” or “import cycle” error is the cycle detector speaking.

7. The graph toolkit

Five reductions cover most graph problems you will meet:

  • represent — adjacency list, \(\Theta(V+E)\) space;
  • explore — a stack gives DFS, a queue gives BFS, a priority queue gives Dijkstra (Chapter 9);
  • fewest-edge paths in an unweighted graph — BFS distance labels (proved shortest);
  • schedule a DAG — Kahn’s algorithm or DFS reverse post-order (both proved correct, both \(\Theta(V+E)\));
  • detect a cycle — a DFS back edge, or Kahn stopping short of \(V\).

The frontier table is also the map forward: replace the queue with a priority queue keyed by accumulated path cost and BFS becomes Dijkstra’s shortest-path algorithm — which is why heaps came before graphs in this course, and what the next chapter builds.

8. Exercises (with solutions)

  1. An undirected graph has degree sequence \(4, 3, 3, 2, 2\). How many edges does it have? Could a simple undirected graph have degree sequence \(4, 4, 1, 1\)?
    solution By the handshake lemma \(\sum \deg = 2E\), so \(E = (4+3+3+2+2)/2 = 14/2 = 7\). For \(4,4,1,1\): the sum is \(10\), an even number, so the lemma is satisfied, but a vertex of degree \(4\) needs \(4\) distinct neighbors and there are only \(3\) other vertices — impossible in a simple graph. (The handshake parity test is necessary, not sufficient.)
  2. On the running DAG, give the DFS visit order and the BFS distance from \(0\) to every vertex, starting from \(0\) with adjacency lists sorted ascending.
    solution DFS: \(0\ 1\ 2\ 3\ 4\ 5\ 6\ 7\) — dive \(0\to1\to2\to3\to4\to5\to6\), back up to \(4\), then \(4\to7\) (\(5\) already seen); every remaining edge (\(3\to7\), \(0\to3\), \(0\to5\)) points at an already-visited vertex. BFS distances: \(0{:}0,\ 1{:}1,\ 2{:}2,\ 3{:}1,\ 4{:}2,\ 5{:}1,\ 6{:}2,\ 7{:}2\) — three rings \(\{0\}\), \(\{1,3,5\}\), \(\{2,4,6,7\}\).
  3. Explain why marking a BFS vertex on dequeue rather than on enqueue can produce a wrong distance, using the diamond as your example.
    solution Both \(1\) and \(2\) point at \(3\). With mark-on-dequeue, processing \(1\) enqueues \(3\), and then processing \(2\) sees \(3\) still unmarked and enqueues it again. Vertex \(3\) now sits in the queue twice and is processed twice; worse, on a larger graph a second, longer path could overwrite \(3\)’s distance. Marking on enqueue makes each vertex enter the queue exactly once, preserving both the distance and the \(\Theta(V+E)\) bound.
  4. Give a small digraph on which DFS from some start and BFS from the same start produce different visit orders, and one on which they produce the same order.
    solution Different: the diamond — DFS gives \(0\ 1\ 3\ 2\), BFS gives \(0\ 1\ 2\ 3\). Same: any simple directed path \(0 \to 1 \to 2 \to \cdots\) — both must visit \(0, 1, 2, \dots\) in that one order, since each vertex has a single out-neighbor and there is only one way forward. (More generally the orders coincide whenever the reachable subgraph is a path.)
  5. A digraph has \(V\) vertices and \(E\) edges. You must answer many “is \(u \to v\) an edge?” queries and no neighbor-iteration queries. Which representation do you choose, and how does the answer change if the graph is sparse and you instead iterate neighbors constantly?
    solution For pure edge-existence queries the adjacency matrix answers each in \(O(1)\); if you can afford \(\Theta(V^2)\) space (small or dense graph), it wins. For a sparse graph where the operations iterate neighbors — as DFS, BFS, and topological sort do — the adjacency list wins on both counts: \(\Theta(V+E)\) space and \(O(\deg u)\) neighbor iteration, which is optimal. Representation follows the algorithm’s access pattern.
  6. Prove that a digraph has a topological order if and only if it is acyclic.
    solution (\(\Rightarrow\)) If a cycle \(a \to b \to \cdots \to a\) existed, a topological order would need \(a\) before \(b\) before \(\cdots\) before \(a\), a contradiction; so an order forces acyclicity. (\(\Leftarrow\)) If the graph is a DAG, Kahn’s algorithm outputs all \(V\) vertices in a valid order (source lemma + the correctness proposition of §6), constructing an order. Both directions hold.
  7. You add the edge \(6 \to 2\) to the running DAG. List a cycle it creates, and state what each of the two cycle detectors reports.
    solution The edge closes the cycle \(2 \to 3 \to 4 \to 5 \to 6 \to 2\) (the unique path from \(2\) to \(6\), now looped back). Kahn outputs \(0\), then \(1\), and then no remaining vertex ever reaches in-degree \(0\) (each of \(2,3,4,5,6\) waits on the next around the loop), so it halts with fewer than \(8\) outputs — the leftover vertices contain the cycle. DFS from \(0\) eventually examines an edge into a vertex still on the recursion stack (\(6 \to 2\) while \(2\) is an ancestor), a back edge, and reports the cycle.
  8. A maze is an \(r \times c\) grid of open cells and walls. Argue that BFS from the entrance finds the fewest-moves path to the exit, and give its running time in terms of the number of cells.
    solution Model each open cell as a vertex and each pair of orthogonally adjacent open cells as an unweighted edge; a “move” is one edge. BFS distances equal fewest-edge distances (the shortest-path theorem), so \(\texttt{dist}[\text{exit}]\) is the minimum number of moves, and the BFS tree gives an actual shortest path. There are at most \(rc\) cells and \(O(rc)\) edges (each cell has \(\le 4\) neighbors), so BFS runs in \(\Theta(rc)\) — linear in the grid. The neighbors are computed from coordinates, so no adjacency list is built.
