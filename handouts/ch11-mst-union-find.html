Chapter 11 — Union-Find & Minimum Spanning Trees

CSS 343 — Data Structures, Algorithms & Discrete Mathematics II. Course text, primary reading for Lecture 11.
Two linked topics. First a tiny structure — union-find — that answers “are these two things connected?” under a stream of merges, refined from \(O(N)\) to near-constant time through weighting and path compression. Then the minimum spanning tree: the cheapest set of edges that connects a weighted graph, made greedy-safe by one theorem (the cut property), and computed two ways — Kruskal, which is exactly a sort plus union-find, and Prim, which is the Chapter 6 heap again. The back half of this course is mostly old structures, recombined into new algorithms, and this chapter is the clearest case. It opens with union-find, a structure so small it is two integer arrays, yet it answers dynamic-connectivity queries in essentially constant time after a sequence of clever refinements. It then turns to the minimum spanning tree — the cheapest way to wire a weighted graph so everything is connected — and finds that union-find is precisely the tool Kruskal’s algorithm needs, while the binary heap from Chapter 6 powers Prim’s. 1. Dynamic connectivity: the union-find structure

Definition (dynamic connectivity) Given \(N\) objects and a stream of connections, support two operations in any interleaving: unite(p, q) records that \(p\) and \(q\) are connected, and connected(p, q) asks whether they are connected, directly or transitively. Because connection is transitive, the objects partition into components, and the question is really “are \(p\) and \(q\) in the same component?”

This abstracts pixels merging into regions, computers joining a network, people connecting in a social graph. The whole implementation hinges on one internal operation, find(p), which returns a canonical identifier for \(p\)’s component; then connected(p, q) is just find(p) == find(q). (The operation is classically called union, but that is a reserved word in C++, so the method is named unite.) We build the final structure through three strictly-better versions.

Quick-find, quick-union, and the trade they make

Quick-find stores a component label id[i] for each element; connected is a single comparison, but unite must sweep the whole array to relabel one component: 

  connected(p, q):  id[p] == id[q]              // O(1)
  unite(p, q):      for every i with id[i]==id[p]: id[i] = id[q]   // O(N)

With \(O(N)\) per union, building the structure from \(N\) unions is \(O(N^2)\) — unusable at scale. Quick-union inverts the trade: store parent[i], making each component a tree whose root is its identifier. Now unite is a single link, but find walks to the root and can be \(O(N)\) if the tree degenerates into a path: 

  find(p):     follow parent[] to the root          // O(tree height)
  unite(p, q): parent[ find(p) ] = find(q)          // O(1) link, plus 2 finds

We have only moved the cost from union to find. Two refinements — applied together — drive both operations to near-constant.

Weighting: keep the trees short

The tall-tree problem is self-inflicted: linking a big tree under a small one makes everything in the big tree deeper. So do the opposite — always link the smaller tree under the larger, tracking each tree’s size. This is weighted quick-union, and it bounds every tree’s height by \(\log_2 N\).

Proposition (weighted trees stay shallow) Under union by size, every tree on \(k\) nodes has height at most \(\lfloor \log_2 k \rfloor\); in particular no node is ever deeper than \(\log_2 N\), so find and unite are \(O(\log N)\).
Proof (a doubling argument). Track the depth of a single node \(x\). Its depth increases by one exactly when the tree containing \(x\) is linked under another tree — and, by the union-by-size rule, that only happens when \(x\)’s tree is the smaller of the two, so the combined tree has size at least twice \(x\)’s old tree size. Thus every time \(x\) sinks one level, the size of the tree it belongs to at least doubles. That size starts at \(1\) and can never exceed \(N\), so it can double at most \(\log_2 N\) times; \(x\) sinks at most \(\log_2 N\) levels. ∎

This is the same doubling argument that bounded the number of resizes in hashing (Chapter 7) and the height of a heap (Chapter 6) — its third appearance, and worth recognizing as a standard tool.

Path compression: flatten as you go

The second refinement is nearly free. Every find already walks from a node to the root; on the way back, repoint the visited nodes closer to the root. The one-line version below is path halving — point each node at its grandparent — which flattens the tree as a side effect of ordinary queries, so the structure gets faster the more it is used: 

  find(p):
      while (p != parent[p]) {
          parent[p] = parent[parent[p]];    // halve the path
          p = parent[p];
      }
      return p;
Example (weighted vs. plain, then compression) Apply unite(4,3), unite(3,8), unite(9,4). Plain quick-union links root under root each time, building a chain of height \(2\): 8←3←4 with 9 also under 8. Weighted quick-union instead hangs the smaller tree under the larger (ties keep the first root), producing the flat star 4←{3, 8, 9} of height \(1\). And a later find on any deep node repoints it straight toward the root, so the next find on it is one hop. Weighting prevents tall trees; compression removes the ones that form anyway.

Here is the whole structure — two arrays and three short methods: 

// Weighted quick-union with path compression. Two arrays: parent[i] (who i
// points to) and size[i] (tree size, for union by size). Near-constant per op.
struct UF {
    vector<int> parent, size;
    UF(int n) : parent(n), size(n, 1) {
        for (int i = 0; i < n; i++) parent[i] = i;   // each element its own root
    }
    int find(int x) {                        // walk to the root, compressing
        while (x != parent[x]) {
            parent[x] = parent[parent[x]];    // path halving: point at grandparent
            x = parent[x];
        }
        return x;
    }
    void unite(int a, int b) {               // union by size: smaller under larger
        int ra = find(a), rb = find(b);
        if (ra == rb) return;
        if (size[ra] < size[rb]) swap(ra, rb);
        parent[rb] = ra;                      // rb (smaller) hangs under ra
        size[ra] += size[rb];
    }
    bool connected(int a, int b) { return find(a) == find(b); }
};

# size[i] = tree size (for union by size). Near-constant amortized per op.
class UF:
    def __init__(self, n):
        self.parent = list(range(n))         # each element its own root
        self.size = [1] * n

    def find(self, x):                       # walk to the root, compressing
        while x != self.parent[x]:
            self.parent[x] = self.parent[self.parent[x]]  # path halving
            x = self.parent[x]
        return x

    def unite(self, a, b):                   # union by size: smaller under larger
        ra, rb = self.find(a), self.find(b)
        if ra == rb:
            return
        if self.size[ra] < self.size[rb]:
            ra, rb = rb, ra
        self.parent[rb] = ra
        self.size[ra] += self.size[rb]

    def connected(self, a, b):
        return self.find(a) == self.find(b)

// size[i] = tree size (for union by size). Near-constant amortized per op.
struct UF {
    parent: Vec<usize>,
    size: Vec<usize>,
}
impl UF {
    fn new(n: usize) -> Self {
        UF { parent: (0..n).collect(), size: vec![1; n] }
    }
    fn find(&mut self, mut x: usize) -> usize {   // walk up, compressing
        while x != self.parent[x] {
            self.parent[x] = self.parent[self.parent[x]];  // path halving
            x = self.parent[x];
        }
        x
    }
    fn unite(&mut self, a: usize, b: usize) {     // union by size
        let (mut ra, mut rb) = (self.find(a), self.find(b));
        if ra == rb { return; }
        if self.size[ra] < self.size[rb] {
            std::mem::swap(&mut ra, &mut rb);          // smaller under larger
        }
        self.parent[rb] = ra;
        self.size[ra] += self.size[rb];
    }
    fn connected(&mut self, a: usize, b: usize) -> bool {
        self.find(a) == self.find(b)
    }
}

The cost, and why union-find is everywhere

implementationunitefind
quick-find\(O(N)\)\(O(1)\)
quick-union\(O(N)\)\(O(N)\)
weighted quick-union\(O(\log N)\)\(O(\log N)\)
weighted + path compression\(\approx O(1)\)\(\approx O(1)\)

The final version costs \(O(\alpha(N))\) amortized per operation, where \(\alpha\) is the inverse Ackermann function — a quantity so slow-growing it is at most \(4\) or \(5\) for any \(N\) that could be stored on any computer. For every practical purpose, union and connected are constant time. The design story — find the bottleneck, fix it, repeat — matters as much as the result.

Union-find vs. a DFS for components Chapter 8’s DFS also finds connected components, but it needs the whole graph up front and a fresh traversal per query. Union-find shines when edges arrive online and connectivity queries interleave with them — social links forming, network cables coming up — because each union and each query is near-constant. 2. Spanning trees and the cut property

Definition (spanning tree, MST) A spanning tree of a connected graph is a subset of its edges that connects all \(V\) vertices with no cycle — necessarily exactly \(V-1\) edges. A minimum spanning tree (MST) of a connected, edge-weighted, undirected graph is a spanning tree of minimum total edge weight.

A spanning tree sits exactly at the connectivity boundary: remove any one of its edges and the graph falls into two pieces; add any one non-tree edge and you close exactly one cycle. (These are the “two of three” facts about {connected, acyclic, \(V-1\) edges} from Chapter 8, now load-bearing.) A graph usually has exponentially many spanning trees; the MST is the cheapest.

The minimum weight is unique even when the tree is not If all edge weights are distinct, the MST is unique. If weights tie, several different edge sets may all be minimal — but every one of them has the same total weight, so “the MST weight” is always well-defined. The running graph below has all-distinct weights, so its MST is unique.

The cut property

One theorem stands behind every MST algorithm. A cut is a partition of the vertices into two nonempty sets; an edge crosses the cut if its endpoints fall on opposite sides.

Theorem (cut property) For any cut, a minimum-weight edge crossing it belongs to some MST. (When weights are distinct, the minimum crossing edge belongs to every MST.)
Proof (exchange argument). Let \(e\) be a lightest edge crossing a given cut, and suppose some MST \(T\) does not contain \(e\). Adding \(e\) to \(T\) creates exactly one cycle (Chapter 8), and that cycle starts and ends on the same side of the cut, so it must cross the cut a second time, by some other edge \(f\). Since \(e\) is a lightest crossing edge, \(w(f) \ge w(e)\). Now form \(T' = T - f + e\): removing \(f\) breaks the cycle, and \(e\) reconnects the two pieces, so \(T'\) is again a spanning tree, and its weight \(w(T') = w(T) - w(f) + w(e) \le w(T)\). But \(T\) was minimum, so \(w(T') = w(T)\): \(T'\) is also an MST, and it contains \(e\). ∎

Every MST algorithm is just a disciplined way to pick cuts and take their lightest crossing edge. This exchange argument — add the greedy choice, find the cycle it closes, swap out a no-better edge — is the template for greedy-optimality proofs, and we reuse it for Huffman coding in Chapter 12.

Example (a cut on the running graph) Split the vertices as \(\{0, 1\}\) versus \(\{2,3,4,5,6,7\}\). The crossing edges are \(1\text{-}2\,(1)\), \(1\text{-}3\,(5)\), \(0\text{-}3\,(6)\), and \(0\text{-}5\,(20)\). The lightest is \(1\text{-}2\) at weight \(1\), so it is safe — and indeed it is in the MST. This happens to be the very first edge Kruskal takes.
The cycle property (the dual) The mirror statement is equally useful: with distinct weights, the maximum-weight edge on any cycle is in no MST — you can always route around it more cheaply. (Same exchange, run backwards.) The cut property says what to add; the cycle property says what to skip. Kruskal uses both.

3. Kruskal’s algorithm

Kruskal’s algorithm considers edges from cheapest to most expensive and accepts each one that connects two so-far-separate pieces, skipping any edge whose endpoints are already connected (it would close a cycle). After \(V-1\) acceptances the tree spans everything.

The one non-trivial step is the test “would this edge form a cycle?” — which is exactly “are its endpoints already in the same component?”, a union-find query. This is why we built union-find first: it makes each check near-constant. 

// Kruskal: sort edges cheapest-first, then add each edge whose endpoints are not
// yet connected (union-find is the cycle check). Stop after V-1 edges.
struct Edge { int u, v; long w; };

vector<Edge> kruskal(int V, vector<Edge> edges) {
    sort(edges.begin(), edges.end(),
         [](const Edge& a, const Edge& b) { return a.w < b.w; });
    UF uf(V);                                    // from union-find.cpp
    vector<Edge> mst;
    for (const Edge& e : edges) {
        if (uf.connected(e.u, e.v)) continue;    // cycle — skip (cycle property)
        mst.push_back(e);                        // safe — take it (cut property)
        uf.unite(e.u, e.v);
        if ((int)mst.size() == V - 1) break;     // spanning tree complete
    }
    return mst;
}
// Kruskal: sort edges cheapest-first, then add each edge whose endpoints are not

    int u, v; long w;
    Edge(int u, int v, long w) { this.u = u; this.v = v; this.w = w; }

    UF uf = new UF(V);                           // from union-find.java
    List<Edge> mst = new ArrayList<>();
    for (Edge e : edges) {

        mst.add(e);                              // safe — take it (cut property)

# Kruskal: sort edges cheapest-first, then add each edge whose endpoints are not
# yet connected. Each edge is a tuple (w, u, v). Stop after V-1 edges.
def kruskal(V, edges):
    uf = UF(V)                               # from union-find.py
    mst = []
    for w, u, v in sorted(edges):            # sorts by weight (first tuple field)
        if uf.connected(u, v):               # cycle — skip (cycle property)
            continue
        mst.append((w, u, v))                # safe — take it (cut property)
        uf.unite(u, v)
        if len(mst) == V - 1:                # spanning tree complete
            break
    return mst
// Kruskal: sort edges cheapest-first, then add each edge whose endpoints are not
// yet connected. Each edge is (weight, u, v). Stop after V-1 edges.
fn kruskal(v: usize, mut edges: Vec<(i64, usize, usize)>)
    -> Vec<(i64, usize, usize)>
{
    edges.sort();                            // tuples sort by weight first
    let mut uf = UF::new(v);                 // from union-find.rs
    let mut mst = Vec::new();
    for (w, a, b) in edges {
        if uf.connected(a, b) { continue; }  // cycle — skip (cycle property)
        mst.push((w, a, b));                 // safe — take it (cut property)
        uf.unite(a, b);
        if mst.len() == v - 1 { break; }     // spanning tree complete
    }
    mst
}

Kruskal maintains a forest that merges: it starts with \(V\) one-vertex trees, and every accepted edge fuses two of them, so the graph goes from \(V\) components down to \(1\) over exactly \(V-1\) acceptances. That forest is the union-find structure — each unite a merge, each connected a “same tree?” question.

Example (the full run) On the running graph, edges sorted ascending (all weights distinct): 
  1: 1-2   2: 3-4   3: 4-5   4: 0-1   5: 1-3   6: 0-3
  7: 4-7   8: 2-3   9: 5-6   11: 3-7   20: 0-5

  1-2 (1)  ADD    total  1
  3-4 (2)  ADD    total  3
  4-5 (3)  ADD    total  6     {3,4,5} formed
  0-1 (4)  ADD    total 10     {0,1} formed
  1-3 (5)  ADD    total 15     merges {0,1,2} and {3,4,5}
  0-3 (6)  SKIP  — 0 and 3 already connected (cycle 0-1-3)
  4-7 (7)  ADD    total 22
  2-3 (8)  SKIP  — 2 and 3 already connected
  5-6 (9)  ADD    total 31     7 edges ⇒ STOP
The skips are the algorithm. When \(0\text{-}3\) is examined, \(0\) and \(3\) already share a component (through \(0\text{-}1\), \(1\text{-}2\), \(1\text{-}3\)), so it is rejected by the cycle property; likewise \(2\text{-}3\). The early stop at seven edges means the two heaviest edges (\(11\) and \(20\)) are never even looked at. Total weight \(31\).
(Note \(1\text{-}2\) and \(3\text{-}4\), the two lightest edges, are accepted first even though they lie in different parts of the graph — Kruskal grows several fragments at once, unlike Prim.)

Why every accepted edge is safe

Proposition (Kruskal is correct) Every edge Kruskal accepts is a minimum crossing edge of some cut, hence in an MST by the cut property; so Kruskal outputs an MST.
Proof. Suppose Kruskal accepts \(e = (u,v)\), and let \(C\) be \(u\)’s component at that moment. Since \(u\) and \(v\) were not yet connected, \(v \notin C\), so \(e\) crosses the cut \((C, \text{rest})\). We claim \(e\) is a lightest crossing edge. Any crossing edge \(f\) with \(w(f) < w(e)\) would have been processed earlier (edges come in weight order), and by then it was either accepted or skipped — and in both cases its two endpoints ended up in the same component. But a crossing edge has one endpoint in \(C\) and one outside, i.e. in different components, a contradiction. So no lighter crossing edge exists; \(e\) is a minimum crossing edge and, by the cut property, safe. Accepting \(V-1\) safe edges yields a spanning tree that is an MST. ∎
Proposition (Kruskal cost) Kruskal runs in \(O(E\log E) = O(E\log V)\) time.
Proof. Sorting the \(E\) edges costs \(O(E\log E)\). The single pass performs \(O(E)\) union-find operations, each near-constant, for \(O(E\,\alpha(V))\) total. The sort dominates, and since \(E \le V^2\), \(\log E \le \log V^2 = 2\log V\), so \(O(E\log E) = O(E\log V)\). ∎

Kruskal is, in one phrase, a sort followed by a near-linear pass. If the edges arrive already sorted (or the weights are small integers that admit radix sort), it approaches linear time — a reason to reach for it on sparse graphs and pre-sorted input.

4. Prim’s algorithm

Prim’s algorithm grows a single tree from a start vertex, repeatedly adding the lightest edge that leaves the tree to a vertex not yet in it. Unlike Kruskal, its correctness needs no separate argument: the growing tree versus everything else is a cut, and Prim takes that cut’s minimum crossing edge by construction — safe by the cut property at every step.

To find the lightest crossing edge quickly, keep the candidate crossing edges in a min-heap (Chapter 6 again). When a vertex joins the tree, push its edges to outside vertices; to grow, pop the cheapest edge and use it if it still leads outside, discarding stale entries whose far endpoint has since joined — the same lazy-deletion trick as Dijkstra. 

// Lazy Prim: grow one tree from s. A min-heap holds candidate crossing edges
// keyed by weight; pop the lightest, skip it if it now leads back into the tree,
// otherwise take it. adj[u] = u's weighted edges. Returns the MST total weight.
struct Edge { int to; long w; };

long prim(int V, vector<vector<Edge>>& adj, int s) {
    vector<bool> inTree(V, false);
    priority_queue<pair<long,int>, vector<pair<long,int>>,
                   greater<>> pq;                 // min-heap keyed by edge weight
    long total = 0; int picked = 0;
    inTree[s] = true;
    for (Edge e : adj[s]) pq.push({e.w, e.to});   // edges leaving the start
    while (picked < V - 1 && !pq.empty()) {
        auto [w, u] = pq.top(); pq.pop();
        if (inTree[u]) continue;                  // stale — both ends already in
        inTree[u] = true; total += w; picked++;   // take the crossing edge
        for (Edge e : adj[u])
            if (!inTree[e.to]) pq.push({e.w, e.to});
    }
    return total;
}
// Lazy Prim: grow one tree from s. Min-heap of {weight, vertex}; skip stale pops.
// adj.get(u) = u's weighted edges. Returns the MST total weight.
long prim(int V, List<List<Edge>> adj, int s) {
    boolean[] inTree = new boolean[V];
    PriorityQueue<long[]> pq =
        new PriorityQueue<>((a, b) -> Long.compare(a[0], b[0]));  // by weight
    long total = 0; int picked = 0;
    inTree[s] = true;
    for (Edge e : adj.get(s)) pq.add(new long[]{e.w, e.to});
    while (picked < V - 1 && !pq.isEmpty()) {
        long[] top = pq.poll();
        long w = top[0]; int u = (int) top[1];
        if (inTree[u]) continue;                  // stale — both ends already in
        inTree[u] = true; total += w; picked++;   // take the crossing edge
        for (Edge e : adj.get(u))
            if (!inTree[e.to]) pq.add(new long[]{e.w, e.to});
    }
    return total;
}
import heapq

# Lazy Prim: grow one tree from s. Min-heap of (weight, vertex); skip stale pops.
# adj[u] is a list of (v, w) pairs, as in the weighted graph type. Returns the
# MST total weight.
def prim(V, adj, s):
    in_tree = [False] * V
    total, picked = 0, 0
    in_tree[s] = True
    pq = [(w, v) for v, w in adj[s]]          # edges leaving the start
    heapq.heapify(pq)
    while picked < V - 1 and pq:
        w, u = heapq.heappop(pq)
        if in_tree[u]:                        # stale — both ends already in
            continue
        in_tree[u] = True                     # take the crossing edge
        total += w
        picked += 1
        for v, wt in adj[u]:
            if not in_tree[v]:
                heapq.heappush(pq, (wt, v))
    return total
use std::collections::BinaryHeap;
use std::cmp::Reverse;

// Lazy Prim from s. adj[u] holds (v, weight) pairs. Reverse turns the max-heap
// into a min-heap on weight; skip stale pops. Returns the MST total weight.
fn prim(n: usize, adj: &Vec<Vec<(usize, i64)>>, s: usize) -> i64 {
    let mut in_tree = vec![false; n];
    let mut pq = BinaryHeap::new();
    let (mut total, mut picked) = (0i64, 0usize);
    in_tree[s] = true;
    for &(v, w) in &adj[s] { pq.push(Reverse((w, v))); }
    while picked < n - 1 {
        let Some(Reverse((w, u))) = pq.pop() else { break };
        if in_tree[u] { continue; }           // stale — both ends already in
        in_tree[u] = true;                    // take the crossing edge
        total += w;
        picked += 1;
        for &(v, wt) in &adj[u] {
            if !in_tree[v] { pq.push(Reverse((wt, v))); }
        }
    }
    total
}

  tree +4           4-5 (3)                        → add 5   total 15
  tree +5           4-7 (7)                        → add 7   total 22
  tree +7           5-6 (9)                        → add 6   total 31
Prim adds vertices in the order \(1,2,3,4,5,7,6\) — different from Kruskal’s edge order, which took the two lightest edges \(1\text{-}2\) and \(3\text{-}4\) first — yet the final edge set and the total, \(31\), are identical. Watch the expensive edges \(0\text{-}3\,(6)\) and \(0\text{-}5\,(20)\) keep losing to cheaper options until the tree is complete and they are never used.

5. Trees that are not shortest-path trees, and the toolkit

Prim grows a tree from a source with a priority queue, and so does Dijkstra (Chapter 9) — the two are the same skeleton, differing only in the key they minimize.

Prim (MST)Dijkstra (shortest paths)
priority-queue keyedge weight \(w\)distance \(\texttt{dist}[u] + w\)
minimizesthe single connecting edgethe total distance from the source
resultcheapest spanning treecheapest paths from the source

Because the keys differ, the trees can differ — and the smallest example proves it.

Example (MST \(\ne\) shortest-path tree) The triangle \(0\text{-}1\,(2)\), \(1\text{-}2\,(2)\), \(0\text{-}2\,(3)\): 
  MST:  keep 0-1 and 1-2      total weight 4   (drop the 3-edge)
  SPT from 0:  keep 0-1, 0-2  dist[1]=2, dist[2]=3

  in the MST, the route 0→2 costs 2+2 = 4, not the shortest 3;
  the SPT's total weight is 2+3 = 5, not the minimum 4.
The MST minimizes total edge weight (\(4 < 5\)); the shortest-path tree minimizes each vertex’s distance from the source (the direct \(0\text{-}2\) at \(3\) beats the MST’s \(4\)-cost detour). Neither tree does the other’s job. (On the eight-vertex running graph the two happen to coincide, which is exactly why this three-vertex triangle is the example to remember: it separates the concepts even when larger graphs do not.)

A queue-free Prim, \(O(V^2)\) As with Dijkstra, dropping the heap gives an array version: keep key[v], the weight of \(v\)’s cheapest connection to the tree, and each of \(V\) rounds scans for the minimum-key outside vertex in \(O(V)\), then updates its neighbors. That is \(O(V^2)\), no heap, and on a dense graph (\(E = \Theta(V^2)\)) it matches or beats the heap version — the same dense-versus-sparse trade seen in Chapters 8 and 9.
Aside: articulation points A related reliability question about an undirected graph is which single vertex, if removed, would disconnect it — an articulation point (cut vertex). One DFS finds them all in \(\Theta(V+E)\), using two numbers per vertex: \(\texttt{disc}[v]\), its discovery time, and \(\texttt{low}[v]\), the earliest discovery time reachable from \(v\)’s DFS subtree using tree edges plus at most one back edge. A non-root \(u\) is an articulation point exactly when some DFS child \(c\) has \(\texttt{low}[c] \ge \texttt{disc}[u]\) (that subtree cannot escape above \(u\), so removing \(u\) strands it); the root is one exactly when it has two or more DFS children. It is the same DFS-tree-and-back-edge machinery as Chapter 8’s cycle detection, put to a new use.

Algorithms from known structures

The real lesson of this chapter is that none of its algorithms needed a new data structure — they are recombinations of the toolkit:

algorithmbuilt from
Kruskalunion-find + a sort
Primthe Chapter 6 binary heap
Dijkstra (Ch. 9)the Chapter 6 binary heap
articulation pointsthe Chapter 8 DFS tree

Minimum spanning trees themselves are the workhorse behind cheapest-network design (cabling, roads, pipelines that reach every site) and, with one twist, clustering: build the MST of a set of data points under distance weights, then delete the \(k-1\) heaviest tree edges, and the tree falls into \(k\) clusters — single-linkage clustering, from the same algorithm that wires a network. As the course continues, the leverage comes from combining known structures, not inventing new ones; Chapter 12 pushes exactly this idea into greedy algorithms and Huffman coding.

6. Exercises (with solutions)

  1. Starting from singletons \(0,\dots,5\), apply weighted quick-union (ties keep the first root) to unite(0,1), unite(2,3), unite(0,3). Give the final parent array and the height of the resulting tree.
    solution unite(0,1): sizes equal, so \(1\) hangs under \(0\); \(\texttt{size}[0]=2\). unite(2,3): \(3\) under \(2\); \(\texttt{size}[2]=2\). unite(0,3): roots \(0\) (size \(2\)) and \(2\) (size \(2\)) tie, so \(2\) hangs under \(0\); \(\texttt{size}[0]=4\). Final parent = [0, 0, 0, 2, 4, 5], and the tree rooted at \(0\) has \(3\) at depth \(2\) (\(3 \to 2 \to 0\)), so height \(2\) — within the \(\lfloor\log_2 4\rfloor = 2\) bound. A subsequent find(3) would compress it to depth \(1\).
  2. Explain, using the doubling argument, why union by size guarantees tree height \(\le \log_2 N\) but quick-union without it can reach height \(N-1\).
    solution Under union by size a node sinks a level only when its tree is linked under an at-least-as-large tree, which at least doubles the tree’s size; a size bounded by \(N\) doubles at most \(\log_2 N\) times, so depth \(\le \log_2 N\). Plain quick-union may link a large tree under a single-node tree, adding depth without growing size — do this repeatedly (unite(1,0), unite(2,1), unite(3,2), …) and you build a path of height \(N-1\), making find \(O(N)\).
  3. In Kruskal on the running graph, the edge \(0\text{-}3\) (weight \(6\)) is skipped. Name the cycle it would have closed and the two MST properties that justify skipping it.
    solution By the time \(0\text{-}3\) is examined, \(0\) and \(3\) are already connected through \(0\text{-}1\), \(1\text{-}3\), so adding \(0\text{-}3\) would close the cycle \(0\text{-}1\text{-}3\text{-}0\). It is safe to skip by the cycle property: among the cycle’s edges (\(0\text{-}1{=}4\), \(1\text{-}3{=}5\), \(0\text{-}3{=}6\)) it is the heaviest, so it is in no MST. Equivalently, the union-find query connected(0,3) returns true, which is Kruskal’s implementation of the same test.
  4. Run Prim from vertex \(6\) on the running graph. Do you get the same tree as Prim from \(0\)? Why should you expect that?
    solution Yes — the same seven edges, total \(31\). From \(6\), Prim first takes \(5\text{-}6\,(9)\), then \(4\text{-}5\,(3)\), \(3\text{-}4\,(2)\), then \(1\text{-}3\,(5)\), \(1\text{-}2\,(1)\), \(0\text{-}1\,(4)\), and \(4\text{-}7\,(7)\), assembling the identical edge set in a different order. Because all weights are distinct, the MST is unique (§2), so every correct MST algorithm and every start vertex must produce that one tree.
  5. Give a weighted graph on which Prim’s MST and Dijkstra’s shortest-path tree from the same source are different, and state each tree.
    solution The triangle \(0\text{-}1\,(2)\), \(1\text{-}2\,(2)\), \(0\text{-}2\,(3)\). The MST is \(\{0\text{-}1, 1\text{-}2\}\) (total weight \(4\), dropping the weight-\(3\) edge). Dijkstra from \(0\) keeps \(\{0\text{-}1, 0\text{-}2\}\), reaching \(2\) by the direct edge at distance \(3\) rather than the \(4\)-cost path through \(1\). The two trees share \(0\text{-}1\) but differ on the second edge, because Prim minimizes the connecting edge’s weight while Dijkstra minimizes total distance from the source.
  6. A graph has a unique lightest edge \(e\). Prove \(e\) is in every MST.
    solution Take any cut with \(e\) crossing it — for instance the cut separating one endpoint of \(e\) from the other. Every crossing edge other than \(e\) is strictly heavier (since \(e\) is the unique global minimum), so \(e\) is the minimum crossing edge of that cut, and by the cut property it belongs to every MST. (More generally, the globally lightest edge is always safe to take first, which is exactly Kruskal’s opening move.)
  7. You are handed the edges of a large sparse graph already sorted by weight, and told weights are small integers. Which MST algorithm do you choose, and what is its running time here?
    solution Kruskal. Its only expensive step is the sort, and the input is already sorted (or radix-sortable in \(O(E)\) for small integer weights), so the algorithm reduces to a single near-linear union-find pass: \(O(E\,\alpha(V)) \approx O(E)\). Prim would still pay the heap’s \(\log V\) factor per edge, so on pre-sorted sparse input Kruskal is the better fit.
  8. True or false, with justification: “Union-find can list the actual path of connections between \(p\) and \(q\) when they are connected.”
    solution False. Union-find records only which component each element belongs to (via parent pointers toward an arbitrary root), not which original edges connected them; path compression actively rewrites those pointers, discarding any trace of the connection sequence. To report an actual connecting path you need a graph traversal (BFS or DFS over the stored edges), which is a different tool with a different cost. Union-find answers whether, not how.
