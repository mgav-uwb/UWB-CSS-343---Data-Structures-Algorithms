Chapter 12 — Greedy Algorithms & Huffman Coding

CSS 343 — Data Structures, Algorithms & Discrete Mathematics II. Course text, primary reading for Lecture 12.
The greedy method — make the locally best choice and never reconsider — named as a general pattern, with the two properties that make it correct (greedy-choice and optimal substructure), the exchange argument that proves them, and the counterexamples that kill an unproved greedy. Then its most elegant application: Huffman coding, which merges the two rarest symbols repeatedly to build a provably optimal prefix-free code. With worked builds, both optimality proofs, links to the demos, and exercises.

We have already written three greedy algorithms — Dijkstra, Prim, and Kruskal each make the locally best choice and never look back — without naming the pattern. This chapter names it. A greedy algorithm builds a solution one committed choice at a time, and it is the fastest strategy there is when it is correct; the whole subtlety is that a locally best choice is not always part of a globally best solution. We learn the two properties that license greed, the one proof technique (the exchange argument) that establishes them, and the one-line counterexamples that demolish a greedy rule that lacks them. Then we study Huffman coding, where a single repeated greedy move — merge the two least-frequent symbols — builds an optimal compression code.

1. The greedy method

Definition (greedy algorithm) A greedy algorithm builds a solution incrementally: at each step it makes the choice that looks best right now — by some fixed rule — commits to it irrevocably, and reduces the problem by that choice. No backtracking, no lookahead. 
  greedy(problem):
      solution = {}
      while problem is not solved:
          x = the locally BEST choice available now
          add x to solution              // never undo it
          reduce the problem by x
      return solution

Every greedy algorithm is this loop; the art is entirely in the choice rule and its proof. The irrevocability is what makes greedy fast — one pass, no search tree — and also what makes it risky, because a locally best choice can foreclose a globally better one. We have already relied on the pattern three times:

algorithmthe greedy choice
Dijkstra (Ch. 9)settle the nearest unsettled vertex
Prim (Ch. 11)add the lightest edge leaving the tree
Kruskal (Ch. 11)take the cheapest edge that avoids a cycle

The same rule, opposite fates

Whether greed works is a property of the problem, not of the rule. The sharpest way to see this is a rule that succeeds and fails on inputs that differ only in their numbers.

Example (making change) “Take the largest coin that fits, repeat.” With US denominations \(\{25, 10, 5, 1\}\), making \(41\)¢ gives \(25 + 10 + 5 + 1\) — four coins, and provably the fewest. With the denominations \(\{1, 3, 4\}\), the identical rule makes \(6\)¢ as \(4 + 1 + 1\) — three coins — while the optimum is \(3 + 3\), two coins. The greedy grab of the \(4\) looks best but forces two \(1\)s. Same rule, different denominations, opposite outcome.

The lesson is blunt: every greedy algorithm needs either a proof that it is optimal or a counterexample that kills it. A hunch about one instance does not transfer. The next section supplies both the proof technique and the properties a problem must have for it to succeed.

2. When greedy is optimal — and when it fails

Definition (the two ingredients) A greedy algorithm is correct exactly when its problem has both:
  • the greedy-choice property — some globally optimal solution contains the locally best (greedy) choice, so committing to it is safe; and
  • optimal substructure — after the greedy choice, what remains is a smaller instance of the same problem, and an optimal solution to it combines with the greedy choice to give a global optimum.

The two are different guarantees, and it is worth not confusing them. Optimal substructure is common — shortest paths, knapsack, and edit distance all have it, which is why dynamic programming (Chapter 14) applies to them. The greedy-choice property is the rarer, stronger condition that you can fix the locally best choice and never regret it. A problem with optimal substructure but no greedy-choice property is precisely where greed fails and dynamic programming takes over.

The exchange argument

The greedy-choice property is almost always proved the same way — the same exchange argument that proved the MST cut property in Chapter 11.

Proof template (exchange argument) Let \(g\) be the greedy choice. Take any optimal solution \(O\) that does not use \(g\). Exhibit an element of \(O\) that can be swapped for \(g\), and show the result is still feasible and no worse than \(O\). Then an optimal solution containing \(g\) exists, so \(g\) is safe.
Example (activity selection) Given activities each with a start and finish time, choose the most that can share one room without overlapping. The greedy rule is always take the compatible activity that finishes earliest. It is optimal, by exchange: let \(g\) finish earliest overall, and let \(O\) be any optimal schedule; its first activity finishes no earlier than \(g\), so replacing it with \(g\) keeps the schedule feasible (\(g\) frees the room soonest) and the same size. The activities starting after \(g\) form a smaller instance — optimal substructure — and induction finishes the proof. Sort by finish time and sweep once: \(O(n\log n)\).
Where greed fails: 0/1 knapsack Given items with a value and a weight and a knapsack of capacity \(W\), take each item whole or not to maximize value. The natural greedy — take items by value-to-weight ratio, highest first — can be far from optimal, because one heavy high-ratio item can block two lighter items that together beat it. This problem has optimal substructure but not the greedy-choice property, so it needs dynamic programming (Chapter 15). Tellingly, the fractional knapsack — where you may take part of an item — is greedy-solvable: the fraction is exactly what lets the exchange argument go through without waste. A small change to the problem flips which technique applies.

The practical method, then: first try to break a proposed greedy rule with a small counterexample (usually faster than a proof, and it settles the matter); if it survives real effort, prove it with an exchange argument plus optimal substructure. Scheduling, selection, and encoding problems with a clean ordering are greedy candidates; problems where committing to the local best can foreclose a better whole are dynamic-programming territory. The rest of this chapter is the cleanest greedy success in the course.

When is greedy always optimal? Matroids There is a precise answer to “which problems is greedy right for,” and it is one of the deeper ideas in Cormen et al. A matroid is a ground set with a family of “independent” subsets closed downward and satisfying an exchange axiom (if one independent set is larger than another, some element of the larger can be added to the smaller keeping it independent). The theorem: the greedy algorithm — sort elements by weight, take each that keeps the set independent — finds a maximum-weight independent set for every weighting if and only if the structure is a matroid. Minimum spanning trees are the canonical case: the forests of a graph form the graphic matroid, and Kruskal is exactly this greedy (which is why it needed no problem-specific cleverness, only the cut property). The \(0/1\) knapsack’s feasible sets are not a matroid, which is the structural reason greedy fails on it. Huffman is a greedy that lies outside the matroid frame, so it needs its own exchange proof — a reminder that matroids explain many greedy successes but not all.

3. Huffman coding: an optimal prefix-free code

The problem is data compression: store text in fewer bits. A fixed-length code — \(3\) bits for each of \(6\) symbols — spends the same on a symbol that appears a hundred times as on one that appears once, which is wasteful whenever the frequencies are skewed. A variable-length code fixes this by giving frequent symbols short codes and rare symbols long ones, lowering the average bits per symbol. But variable-length codes raise a hazard:

Example (ambiguity) With \(a = 0\), \(b = 1\), \(c = 01\), the bit string \(01\) could be “\(ab\)” or “\(c\)” — undecodable. The trouble is that \(a\)’s code \(0\) is a prefix of \(c\)’s code \(01\).
Definition (prefix-free code and its trie) A code is prefix-free if no codeword is a prefix of another. Such a code is exactly a binary trie with the symbols at its leaves: the code of a symbol is the sequence of turns from the root (left \(= 0\), right \(= 1\)). Because a leaf is never an ancestor of another leaf, no code is a prefix of another, so decoding is unambiguous.

Decoding a prefix-free code is a walk down the trie: read bits, turn left or right, and emit a symbol the instant you reach a leaf, then restart at the root. The total encoded length of a text is \[ \operatorname{cost}(T) \;=\; \sum_{\text{symbols } s} f_s \cdot d_s, \] where \(f_s\) is \(s\)’s frequency and \(d_s\) its depth in the trie (the length of its code). Minimizing this over all prefix-free codes is the question Huffman answers.

Huffman’s greedy idea

A deep leaf has a long code, so it should hold a rare symbol; the two rarest symbols can afford to be the deepest, as sibling leaves. That is exactly the greedy move: repeatedly merge the two least-frequent nodes into a new parent whose frequency is their sum, until one tree remains. The “two smallest” query is answered by a min-heap — the Chapter 6 structure again. 

// A Huffman tree node. Leaves carry a symbol; internal nodes just a frequency.
struct Node {
    long freq; char sym;            // sym is '\0' for internal nodes
    Node *left, *right;
};

// Build the Huffman tree: repeatedly merge the two lowest-frequency nodes. A
// min-heap of Node* keyed by frequency answers "the two smallest" in O(log n).
Node* buildHuffman(const vector<pair<char,long>>& freqs) {
    auto cmp = [](Node* a, Node* b) { return a->freq > b->freq; };  // min-heap
    priority_queue<Node*, vector<Node*>, decltype(cmp)> pq(cmp);
    for (auto [sym, f] : freqs)
        pq.push(new Node{f, sym, nullptr, nullptr});
    while (pq.size() > 1) {
        Node* x = pq.top(); pq.pop();          // two smallest frequencies
        Node* y = pq.top(); pq.pop();
        pq.push(new Node{x->freq + y->freq, '\0', x, y});   // merge under a parent
    }
    return pq.top();                           // the single remaining root
}
// A Huffman tree node. Leaves carry a symbol; internal nodes have sym = 0.
class Node {
    long freq; char sym;
    Node left, right;
    Node(long f, char s, Node l, Node r) { freq = f; sym = s; left = l; right = r; }
}

// Build the Huffman tree: merge the two lowest-frequency nodes until one remains.
Node buildHuffman(Map<Character, Long> freqs) {
    PriorityQueue<Node> pq =
        new PriorityQueue<>((a, b) -> Long.compare(a.freq, b.freq));  // min-heap
    for (var e : freqs.entrySet())
        pq.add(new Node(e.getValue(), e.getKey(), null, null));
    while (pq.size() > 1) {
        Node x = pq.poll();                    // two smallest frequencies
        Node y = pq.poll();
        pq.add(new Node(x.freq + y.freq, '\0', x, y));   // merge under a parent
    }
    return pq.peek();                          // the single remaining root
}

// A Huffman tree node. Leaves carry a symbol; internal nodes have sym = None.
struct Node {
    freq: u64,
    sym: Option<char>,
    left: Option<Box<Node>>,
    right: Option<Box<Node>>,
}
// Order by frequency, REVERSED so BinaryHeap (a max-heap) yields the smallest.
impl Ord for Node {
    fn cmp(&self, other: &Self) -> Ordering { other.freq.cmp(&self.freq) }
}
impl PartialOrd for Node {
    fn partial_cmp(&self, other: &Self) -> Option<Ordering> { Some(self.cmp(other)) }
}
impl PartialEq for Node { fn eq(&self, o: &Self) -> bool { self.freq == o.freq } }
impl Eq for Node {}

fn build_huffman(freqs: &[(char, u64)]) -> Node {
    let mut pq: BinaryHeap<Node> = freqs.iter()
        .map(|&(sym, freq)| Node { freq, sym: Some(sym), left: None, right: None })
        .collect();
    while pq.len() > 1 {
        let x = pq.pop().unwrap();            // two smallest frequencies
        let y = pq.pop().unwrap();
        pq.push(Node {                        // merge under a new parent
            freq: x.freq + y.freq, sym: None,
            left: Some(Box::new(x)), right: Some(Box::new(y)),
        });
    }
    pq.pop().unwrap()                         // the single remaining root
}

Starting from \(n\) leaf nodes, each merge removes two roots and adds one, so there are exactly \(n-1\) merges, ending with a single tree of \(2n-1\) nodes. Every internal node has two children — the tree is full — which is what guarantees every symbol lands on a leaf.

Example (a worked build, the CLRS frequencies) Symbols with frequencies \(a{:}5,\ b{:}9,\ c{:}12,\ d{:}13,\ e{:}16,\ f{:}45\) (total \(100\)): 
  merge a+b → 14        pool: 12 13 14 16 45
  merge c+d → 25        pool: 14 16 25 45
  merge 14+16 → 30      pool: 25 30 45
  merge 25+30 → 55      pool: 45 55
  merge 45+55 → 100     → the root
The rarest symbols \(a\) and \(b\) merge first and sink deepest; the most frequent, \(f\), waits until the final merge and lands one hop from the root. The internal-node sums \(14, 25, 30, 55, 100\) build up to the total character count.

Seen as the priority queue’s contents — the state the build actually maintains — each step removes the two smallest roots (shaded) and pushes their sum. Leaf roots are written with their symbol; internal roots are bare sums.

Once the tree is built, one traversal collects every code — append 0 going left, 1 going right, and record the path at each leaf: 

// Walk the finished trie once, building each leaf's root-to-leaf path
// (left = '0', right = '1'). Fills code[sym] for every symbol. O(n).
void extractCodes(Node* node, string path,
                  unordered_map<char, string>& code) {
    if (!node) return;
    if (!node->left && !node->right) {         // a leaf: record its code
        code[node->sym] = path;
        return;
    }
    extractCodes(node->left,  path + "0", code);
    extractCodes(node->right, path + "1", code);
}

// (left = '0', right = '1'). Fills `code` for every leaf. O(n).
fn extract_codes(node: &Node, path: String, code: &mut HashMap<char, String>) {
    match (&node.left, &node.right) {
        (None, None) => { code.insert(node.sym.unwrap(), path); }   // a leaf
        _ => {
            if let Some(l) = &node.left  { extract_codes(l, format!("{path}0"), code); }
            if let Some(r) = &node.right { extract_codes(r, format!("{path}1"), code); }
        }
    }
}

Proposition (Huffman cost) Building the tree and extracting the codes takes \(O(n\log n)\) time for \(n\) symbols.
Proof. Heapifying \(n\) leaves is \(O(n)\). The build does \(n-1\) merges, each two delMins and one insert, each \(O(\log n)\), for \(O(n\log n)\). Extracting the codes is a single DFS over \(2n-1\) nodes, \(O(n)\). The heap operations dominate: \(O(n\log n)\). ∎
Breaking ties When two nodes tie for smallest frequency, Huffman may merge either — different tie-breaks yield different trees and even different individual code lengths, but every resulting tree is optimal, so the total encoded length is identical. This is exactly the minimum-spanning-tree situation: the optimal cost is unique, the optimal structure need not be.

4. Why Huffman is optimal

Two lemmas, in the standard greedy shape: the first shows the greedy merge is safe (greedy-choice property), the second shows the rest is the same problem (optimal substructure). Together with induction they prove every Huffman tree minimizes \(\operatorname{cost}(T) = \sum_s f_s d_s\).

Lemma 1 (the two rarest may be deepest siblings) Let \(x, y\) be the two lowest-frequency symbols. Some optimal prefix-code tree has \(x\) and \(y\) as sibling leaves at maximum depth.
Proof (exchange). Take any optimal tree \(T\). An optimal code tree is full (a one-child node could be deleted, shortening its subtree and lowering the cost), so its deepest internal node has two leaf children; call them \(a, b\), sibling leaves at maximum depth. Swap the rarest symbol \(x\) with whatever sits at \(a\). The only costs that change are those of the two symbols moved, so \[ \Delta\operatorname{cost} = (f_x d_a + f_a d_x) - (f_x d_x + f_a d_a) = (f_a - f_x)(d_x - d_a) \le 0, \] because \(f_a \ge f_x\) (\(x\) is a rarest symbol) and \(d_x \le d_a\) (\(a\) is at maximum depth). The swap cannot increase the cost, so the result is still optimal. Swapping \(y\) with the occupant of \(b\) likewise, we reach an optimal tree with \(x, y\) as deepest sibling leaves — which is exactly Huffman’s first merge. ∎
Lemma 2 (merging shrinks the problem) Merge \(x, y\) into a single symbol \(z\) with \(f_z = f_x + f_y\). For any tree \(T\) in which \(x, y\) are siblings, let \(T'\) be \(T\) with that sibling pair replaced by the single leaf \(z\). Then \(\operatorname{cost}(T) = \operatorname{cost}(T') + f_x + f_y\).
Proof. The symbols other than \(x, y\) occupy the same positions in \(T\) and \(T'\), contributing equally. In \(T\), \(x\) and \(y\) sit one level below \(z\)’s position in \(T'\), so each occurrence of \(x\) or \(y\) pays one extra bit compared with \(z\): the difference is \(f_x d_z' + f_y d_z' \) replaced by \(f_x(d_z'+1) + f_y(d_z'+1)\), i.e. an extra \(f_x + f_y\). Hence \(\operatorname{cost}(T) = \operatorname{cost}(T') + f_x + f_y\). ∎

Since \(f_x + f_y\) is a constant independent of the tree’s shape, minimizing \(\operatorname{cost}(T)\) is the same as minimizing \(\operatorname{cost}(T')\) — a Huffman instance on \(n-1\) symbols. By Lemma 1 the first merge is safe; by Lemma 2 the remaining problem is a smaller Huffman instance; by induction on the number of symbols, Huffman’s tree is optimal. This two-lemma structure — greedy choice is safe, then the rest is the same problem — is the canonical greedy-optimality proof, the same shape as the MST cut property and activity selection.

5. Encoding, decoding, and the limits of compression

With the trie in hand, encoding replaces each symbol with its code and concatenates the bits — no separators, because the code is prefix-free. Decoding walks the trie from the root, emitting a symbol at each leaf and restarting.

Example (a round trip) Using the codes \(f{=}0,\ c{=}100,\ d{=}101,\ e{=}111,\ a{=}1100,\ b{=}1101\): 
  encode "face":  f·a·c·e = 0·1100·100·111 = 01100100111   (11 bits)
  decode 1010:    1,0,1 → leaf d;  restart;  0 → leaf f   ⇒ "df"
The decode shows the crux: after emitting \(d\) you jump back to the root, and the lone remaining \(0\) is a complete code. No delimiter was needed — the tree’s leaf boundaries are the delimiters, which is why prefix-free codes are also called instantaneous.

The bits alone do not decode; the decoder needs the same trie, so a compressed file carries either the frequency table (the decoder re-runs the build to get the identical tree) or a serialization of the tree in its header. For a fixed alphabet — say \(256\) byte values — that header is a few hundred bytes, negligible against a large file.

Theorem (Shannon’s limit) No code can beat the entropy \(H = -\sum_s p_s \log_2 p_s\) bits per symbol on average, where \(p_s\) is \(s\)’s probability. Huffman comes within \(1\) bit of \(H\) per symbol, and is exactly optimal among codes that use an integer number of bits per symbol.
Kraft’s inequality: which code lengths are even possible Before asking which prefix code is best, ask which are legal. Kraft’s inequality says a binary prefix code with codeword lengths \(\ell_1, \dots, \ell_n\) exists if and only if \[ \sum_{i=1}^{n} 2^{-\ell_i} \;\le\; 1. \] The reason is exactly the trie: a codeword of length \(\ell\) is a leaf at depth \(\ell\), which claims a \(2^{-\ell}\) fraction of the leaves of a full depth-\(D\) tree, and disjoint leaves’ fractions cannot exceed the whole. Huffman is then the solution to a clean optimization: minimize \(\sum_i f_i \ell_i\) over all integer lengths satisfying Kraft. Relax the integers to real lengths and calculus gives the optimum \(\ell_i = -\log_2 p_i\), whose cost is the entropy \(H\) — the theoretical floor of the theorem above. Huffman pays for rounding those lengths back to whole bits, which is the entire “within one bit” gap, and is what arithmetic coding sidesteps by never rounding.

The residual sub-bit gap — Huffman must spend a whole number of bits per symbol, while \(H\) may be fractional — is why arithmetic and ANS coders exist, squeezing closer to \(H\). And Huffman rarely works alone: it captures frequency skew but not repetition, so real compressors first find repeats (run-length or LZ77) and then Huffman-code the residual. This is DEFLATE, the engine inside ZIP, gzip, and PNG; Huffman also codes the quantized data in JPEG and MP3. Not bad for the answer to a 1952 term-paper assignment.

6. Exercises (with solutions)

  1. Using the greedy “largest coin first” rule with denominations \(\{1, 5, 8\}\), make \(10\)¢. What does greedy return, what is the optimum, and which greedy ingredient fails?
    solution Greedy takes \(8\), then \(1 + 1\): three coins. The optimum is \(5 + 5\): two coins. Greedy fails because the greedy-choice property does not hold for these denominations — committing to the largest coin \(8\) forecloses the better all-\(5\)s solution, so no exchange argument can go through. (This coin system is not “canonical.”) The problem still has optimal substructure, which is why dynamic programming solves coin change in general.
  2. Build the Huffman tree for frequencies \(x{:}1,\ y{:}1,\ z{:}2,\ w{:}4\). Give the codes and the total encoded length \(\sum f_s d_s\).
    solution Merge \(x{+}y \to 2\); pool \(\{2(z),\ 2(xy),\ 4(w)\}\). Merge the two \(2\)s \(\to 4\); pool \(\{4(w),\ 4\}\). Merge \(\to 8\), the root. Codes: \(w = 0\) (frequency \(4\), shallow), \(z = 10\), \(x = 110\), \(y = 111\). Total \(= 4{\cdot}1 + 2{\cdot}2 + 1{\cdot}3 + 1{\cdot}3 = 14\) bits. The rarest symbols \(x, y\) sink deepest, as the greedy merge guarantees.
  3. In Lemma 1’s exchange, the cost change is \((f_a - f_x)(d_x - d_a)\). Explain why each factor has the sign claimed, and what would go wrong if \(x\) were not a lowest-frequency symbol.
    solution \(f_a - f_x \ge 0\) because \(x\) is chosen as a minimum-frequency symbol, so no symbol (including \(a\)) has smaller frequency. \(d_x - d_a \le 0\) because \(a\) sits at maximum depth, so no leaf (including \(x\)) is deeper. Their product is \(\le 0\), so the swap does not raise the cost. If \(x\) were not a rarest symbol, \(f_a - f_x\) could be negative while \(d_x - d_a\) is negative, making the product positive — the swap could increase cost, and the argument that the rarest belongs deepest would fail.
  4. A file uses only two symbols, \(a\) with frequency \(999\) and \(b\) with frequency \(1\). What codes does Huffman assign, how many bits does it use, and why can it not do better?
    solution With two symbols there is one merge, giving a tree with both as leaves at depth \(1\): \(a = 0\), \(b = 1\) (or the reverse), one bit each. Total \(999 + 1 = 1000\) bits. It cannot do better because a prefix-free code for two distinct symbols needs two distinct nonempty codewords, and the shortest such pair is two single bits — you cannot give \(a\) a zero-length code. This is Huffman’s one weakness against entropy: the entropy here is far below \(1\) bit per symbol, but integer-length codes cannot spend a fraction of a bit, which is what arithmetic coding is for.
  5. Give a set of interval activities on which the greedy “pick the activity that starts earliest” rule selects fewer than the optimum, and state the optimum.
    solution Activities \([0, 10]\), \([1, 2]\), \([3, 4]\). Earliest-start greedy takes \([0,10]\) first, which conflicts with both others, yielding \(1\) activity. The optimum takes \([1,2]\) and \([3,4]\): \(2\) activities. Earliest-finish greedy would take \([1,2]\) first and then \([3,4]\), matching the optimum — which is why the correct rule sorts by finish time, not start time.
  6. True or false, with justification: “A prefix-free code always uses no more bits than a fixed-length code on the same text.”
    solution False in general, true for the Huffman prefix code. A prefix-free code can be chosen badly — e.g. giving a frequent symbol a long code — and use more bits than the fixed-length code. What is true is that Huffman’s prefix code is optimal, so it uses no more than any other prefix code, including the fixed-length one (which is itself a prefix code, the balanced tree). When frequencies are uniform, Huffman ties the fixed-length code; when skewed, it beats it.
  7. Why does Huffman’s build produce a full binary tree (every internal node has two children), and why does optimality require fullness?
    solution Every merge creates a parent from exactly two popped nodes, so no internal node is ever built with one child — the tree is full by construction. Fullness is necessary for optimality: a one-child internal node adds a bit to every symbol in its subtree without any branching benefit, so deleting it (promoting its child) lowers every affected code length and strictly reduces \(\sum f_s d_s\). An optimal tree therefore cannot contain a one-child node.
  8. You must compress a stream but cannot afford a first pass to count frequencies. Name the property of static Huffman this violates and the variant that resolves it.
    solution Static Huffman needs the symbol frequencies before it can build the tree, so it requires either a first counting pass over the data or a transmitted frequency table — impossible on a single-pass stream of unknown composition. Adaptive Huffman resolves it by building and updating the tree as symbols arrive, keeping encoder and decoder in lockstep from the same running counts, so no prior pass or transmitted table is needed (at the cost of continual tree maintenance).
