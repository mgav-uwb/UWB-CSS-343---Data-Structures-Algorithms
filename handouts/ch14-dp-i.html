Chapter 14 — Dynamic Programming I

CSS 343 — Data Structures, Algorithms & Discrete Mathematics II. Course text, primary reading for Lecture 14.
Dynamic programming — recursion that remembers. When subproblems overlap and the problem has optimal substructure, caching each subproblem’s answer once collapses an exponential recursion to a polynomial table: memoization fills the cache top-down, tabulation fills it bottom-up. Fibonacci makes the payoff exact (\(177\) calls become \(19\), then \(10\) additions), rod cutting adds a maximization over choices and a traceback, and the longest common subsequence brings the two-dimensional table behind diff and sequence alignment. A five-question recipe generalizes all three. With worked tables, the call-count theorem, links to the demos, and exercises.

Chapter 12’s greedy method commits to one locally-best choice and hopes it is safe; Chapter 13’s divide-and-conquer splits a problem into independent subproblems. This chapter fills the gap between them. Dynamic programming (DP) handles the problems where no single choice is provably safe — so every choice must be tried — and where the resulting subproblems overlap, so a plain recursion would recompute the same answers exponentially often. The cure is almost embarrassingly simple: store each subproblem’s answer the first time you compute it, and look it up every time after. That one idea turns a \(\Theta(\varphi^n)\) Fibonacci into a \(\Theta(n)\) one, solves the rod-cutting problem greedy gets wrong, and fills the table behind every diff you have ever run.

1. Recursion + remembering: what dynamic programming is

Definition (dynamic programming) Dynamic programming (DP) solves a problem by combining the answers to overlapping subproblems, storing each subproblem’s answer so that it is computed once and reused everywhere it is needed. In one line: DP = recursion + remembering.

A plain recursion often re-solves the same subproblem an exponential number of times — the entire blow-up is recomputation. DP eliminates it: compute each distinct subproblem once, cache the answer, and every later request is a constant-time lookup. Two conditions license the method, and both must hold.

Definition (the two hallmarks) A problem is a DP problem when it has both
  1. overlapping subproblems — the recursion needs the same subproblem many times, so caching pays off; and
  2. optimal substructure — an optimal solution is built from optimal solutions to its subproblems, so a recurrence over cached sub-answers is correct.

Overlap is what makes the cache worth having: in the Fibonacci recursion, \(\mathrm{fib}(3)\) is needed by both \(\mathrm{fib}(4)\) and \(\mathrm{fib}(5)\); in rod cutting (§4), the best way to cut a length-\(3\) remainder is asked for after a \(1\)-cut and after a \(2\)-cut. The same subproblem has many “parents,” so computing it once and sharing the result saves all the repeats. If instead every subproblem had a single parent — a tree of all-distinct nodes rather than a directed acyclic graph — a cache would store answers nobody ever asks for again. That is exactly divide-and-conquer’s territory: mergesort’s two halves share no work, and memoizing it buys nothing. Optimal substructure you have met before — it is the property behind greedy proofs and shortest paths — and it is what lets the recurrence build the optimum out of cached sub-optima. Overlap is the new ingredient.

Why “dynamic programming”? Richard Bellman coined the name in the 1950s, and neither word means what it seems to. “Programming” is the operations-research sense of tabular planning (as in “linear programming”), not writing code; “dynamic” refers to multistage decisions over time — and, by Bellman’s own account, was chosen partly because it sounded impressive to research sponsors. The name is opaque; the idea — remember your subproblems — is simple. Our primary source for this material is Jeff Erickson’s Algorithms, chapter 3, with Cormen, Leiserson, Rivest & Stein (CLRS) chapter 14 as an optional reference; DP is not in Sedgewick at all.

The three great design paradigms of this course now sit side by side, and recognizing which one a problem wants is half of algorithm design:

paradigmideawhen it applies
divide-and-conquer (Ch. 13) split into independent subproblems, recurse, combine subproblems do not overlap
greedy (Ch. 12) commit to one locally-best choice, never reconsider the choice is provably safe (exchange argument)
dynamic programming try all choices, cache the overlapping subproblems overlap + optimal substructure

DP is greedy without the leap of faith: where greedy gambles that one locally-best choice is always safe, DP tries every choice at each step and lets the cached subproblem answers sort out the winner. That is why DP solves the problems where greedy has a counterexample — the \(\{1,3,4\}\) coin system, the 0/1 knapsack — at the price of computing more subproblems, a price overlap and caching keep polynomial. The practical test is to draw the recursion tree: repeated nodes mean DP; all-distinct nodes mean divide-and-conquer; and if one branch provably suffices, greedy. Often you try greedy first and fall back to DP the moment you find a counterexample.

2. Memoization and tabulation

There are two ways to implement the “remember” in recursion-plus-remembering, and every DP you will ever write is one of them.

Definition (memoization, top-down) Memoization keeps the natural recursion and adds a cache: before computing, check whether the answer is already stored; after computing, store it. 
  solve(x):
      if x in memo: return memo[x]      // remember
      result = ... solve(smaller) ...   // the natural recursion
      memo[x] = result
      return result
The recursion tree is unchanged, but each distinct node is computed once; every repeat is an \(O(1)\) cache hit that cuts off a whole subtree.
Definition (tabulation, bottom-up) Tabulation replaces the recursion with a loop that fills a table of subproblem answers in dependency order, smallest subproblems first: 
  for subproblems from SMALLEST to largest:
      table[x] = ... table[smaller] ...   // already computed
  return table[goal]
No recursion, no call stack — but you must work out an evaluation order in which every cell’s dependencies are filled before the cell itself, whereas memoization discovers that order automatically.

Either way, the cost analysis is the same one-line product, and it is the single most useful fact in this chapter:

Proposition (the DP cost formula) Once every subproblem is computed at most once, \[ \text{time} \;=\; (\#\text{ distinct subproblems}) \times (\text{work to combine one from its sub-answers}), \] and space is the number of stored subproblems (often reducible, §3 and §5).

Fibonacci has \(n\) subproblems at \(O(1)\) each: \(\Theta(n)\), down from \(\Theta(\varphi^n)\). Rod cutting has \(n\) subproblems, each a max over \(O(n)\) choices: \(\Theta(n^2)\), down from exponential. The longest common subsequence has \(m \cdot n\) subproblems at \(O(1)\) each: \(\Theta(mn)\). The speedup is exactly the recomputation you avoid — more overlap means more avoided repeats and a bigger win, while zero overlap (divide-and-conquer) means caching does nothing but waste memory.

Which to use Same answers, same asymptotic cost — pick by convenience. Memoize first: keeping the natural recursion and adding a cache is the least error-prone way to get a correct DP, and it computes only the subproblems that are actually needed. Switch to tabulation when the recursion overhead matters, when the recursion depth would overflow the stack, or when you want the “keep only the last two rows” space optimization that only the bottom-up form makes natural. Neither is “the” dynamic programming; they are two implementations of the same table.

3. Fibonacci: the whole arc

The Fibonacci numbers \(0, 1, 1, 2, 3, 5, 8, 13, \dots\) — each the sum of the previous two — are the cleanest possible stage for the DP idea, because the naive recursion is obviously correct and spectacularly wasteful. Straight from the definition: 

  long fib(int n) {
      if (n < 2) return n;              // fib(0) = 0, fib(1) = 1
      return fib(n - 1) + fib(n - 2);
  }

Correct — but trace the calls for \(n = 5\) and the overlap is everywhere:

Proposition (the naive call count, exactly) Naive \(\mathrm{fib}(n)\) makes exactly \[ \mathrm{calls}(n) \;=\; 2\,\mathrm{fib}(n+1) - 1 \] calls — the call tree grows at the same golden-ratio rate \(\Theta(\varphi^n)\), \(\varphi = \tfrac{1+\sqrt5}{2} \approx 1.618\), as the value it computes.
Proof (induction). Each call is one node of the tree, so \(\mathrm{calls}(n) = \mathrm{calls}(n-1) + \mathrm{calls}(n-2) + 1\) for \(n \ge 2\), with \(\mathrm{calls}(0) = \mathrm{calls}(1) = 1\). Check the formula at the base: \(2\,\mathrm{fib}(1) - 1 = 1\) and \(2\,\mathrm{fib}(2) - 1 = 1\). Inductive step: assuming it for \(n-1\) and \(n-2\), \[ \mathrm{calls}(n) = \bigl(2\,\mathrm{fib}(n) - 1\bigr) + \bigl(2\,\mathrm{fib}(n-1) - 1\bigr) + 1 = 2\,\mathrm{fib}(n+1) - 1 . \] The same argument shows \(\mathrm{fib}(k)\) is computed exactly \(\mathrm{fib}(n-k+1)\) times for \(1 \le k \le n\) — the repeat counts are themselves Fibonacci numbers, the recursion tree measuring itself. ∎

Concretely: \(\mathrm{fib}(10)\) takes \(2\,\mathrm{fib}(11) - 1 = 2 \cdot 89 - 1 = 177\) calls; \(\mathrm{fib}(40)\) takes \(331{,}160{,}281 \approx 3.3\times10^{8}\); \(\mathrm{fib}(50)\) takes \(\approx 4.1\times10^{10}\) — forty billion calls for a value fifty additions would deliver. Memoized: exponential to linear

Now add the cache. The recursion is unchanged; a table memo[] remembers every value already computed: 

// Memoized Fibonacci: the natural recursion plus a cache. Each fib(k) is
// computed once; every later request is an O(1) hit. Theta(n) time and space.
long fibMemo(int n, vector<long>& memo) {
    if (n < 2) return n;                  // base cases: fib(0)=0, fib(1)=1
    if (memo[n] != 0) return memo[n];     // remember: cache hit, subtree skipped
    memo[n] = fibMemo(n - 1, memo)        // computed at most once per n ...
            + fibMemo(n - 2, memo);
    return memo[n];                       // ... then stored for every caller
}

long fib(int n) {
    vector<long> memo(n + 1, 0);          // 0 = "not yet computed"
    return fibMemo(n, memo);
}

The three added lines collapse the tree. There are now \(n + 1\) distinct subproblems, each combined from its two sub-answers in \(O(1)\), so the cost formula gives \(\Theta(n)\) time. For \(n = 10\) that is \(19\) calls in total (\(2n - 1\): each of \(\mathrm{fib}(2), \dots, \mathrm{fib}(10)\) spawns two calls, and \(8\) of those \(19\) are cache hits that return immediately) — against \(177\) naive calls, of which \(166\) were recomputation.

Tabulated: and then to constant space

Bottom-up, the same table fills left to right — and once it does, a further prize appears: \(\mathrm{fib}(i)\) needs only the previous two values, so the whole table can shrink to two variables. 

// Tabulated Fibonacci: fill fib(0..n) smallest-first -- every dependency is
// already in the table when a cell needs it. Theta(n) time, Theta(n) space.
long fibTab(int n) {
    vector<long> t(n + 2);
    t[0] = 0; t[1] = 1;                   // base cases seed the table
    for (int i = 2; i <= n; i++)
        t[i] = t[i - 1] + t[i - 2];       // dependencies already filled
    return t[n];
}

// Two variables: each value needs only the previous TWO, so keep just those.
// Theta(n) time, Theta(1) space -- the optimization tabulation makes natural.
long fibTwoVar(int n) {
    long a = 0, b = 1;                    // invariant: a = fib(i), b = fib(i+1)
    for (int i = 0; i < n; i++) {
        long c = a + b;                   // one addition per step
        a = b; b = c;                     // slide the two-cell window right
    }
    return a;                             // after n steps, a = fib(n)
}

4. Rod cutting: optimizing over choices

Fibonacci has overlap but nothing to optimize. Rod cutting adds the missing ingredient — a choice at every step — and with it the full shape of an optimization DP.

Problem (rod cutting) Given a rod of integer length \(n\) and a price list where a piece of length \(i\) sells for \(\mathrm{price}[i]\), cut the rod into pieces (zero or more cuts) to maximize the total selling price.
Example (the running instance) Lengths \(1\)–\(4\) priced \(1, 5, 8, 9\), rod length \(n = 4\). Selling the rod whole earns \(9\); cutting \(3 + 1\) earns \(8 + 1 = 9\); but cutting into two length-\(2\) pieces earns \(5 + 5 = 10\) — the optimum, and nothing about the raw prices announces it.

A greedy attempt fails, exactly in Chapter 12’s sense of a counterexample. The natural greedy rule — cut the piece with the best price per unit length first — computes the ratios \(1/1 = 1.0\), \(5/2 = 2.5\), \(8/3 \approx 2.67\), \(9/4 = 2.25\), grabs the length-\(3\) piece, and is left with a length-\(1\) remainder: total \(8 + 1 = 9 < 10\). The locally best ratio is not globally best because taking it constrains what is left. Since the best first cut cannot be identified without solving the rest of the rod, every first cut must be tried — and the subproblems those trials generate overlap, which is DP’s cue.

Proposition (rod-cutting recurrence) Let \(\mathrm{best}(\ell)\) be the maximum revenue for a rod of length \(\ell\). Then \(\mathrm{best}(0) = 0\) and, for \(\ell \ge 1\), \[ \mathrm{best}(\ell) \;=\; \max_{1 \le i \le \ell} \bigl(\mathrm{price}[i] + \mathrm{best}(\ell - i)\bigr). \]
Why this is correct (optimal substructure). Any optimal cutting of a length-\(\ell\) rod has some first piece, of some length \(i\); the rest of that solution cuts the remaining \(\ell - i\). If that remainder were cut suboptimally, replacing it with an optimal cutting of length \(\ell - i\) would raise the total — contradicting optimality. So the optimum equals \(\mathrm{price}[i] + \mathrm{best}(\ell - i)\) for its own first piece \(i\), and taking the max over all \(i\) can only find it. The subproblems overlap because \(\mathrm{best}(k)\) is needed along many different first-cut paths — without a cache the recursion is \(\Theta(2^n)\), with one it is \(n\) subproblems at \(O(n)\) each: \(\Theta(n^2)\). ∎

One new concern appears that Fibonacci never raised: the table stores the optimal value, but we usually want the cuts. The fix is two lines: record, at each length, which first cut won the max, then walk the records backwards. Every optimization DP does this. 

// Rod cutting, tabulated: best[len] = max over first cuts i of
// price[i] + best[len-i], recording the winning i in cut[len] for the
// traceback. n subproblems x O(n) choices each = Theta(n^2) time, Theta(n) space.
int rodCut(const vector<int>& price, int n, vector<int>& cut) {
    vector<int> best(n + 1, 0);           // base case: best[0] = 0 (empty rod)
    cut.assign(n + 1, 0);
    for (int len = 1; len <= n; len++) {
        int m = -1;
        for (int i = 1; i <= len; i++)    // try every first cut
            if (price[i] + best[len - i] > m) {
                m = price[i] + best[len - i];
                cut[len] = i;             // remember the choice that won
            }
        best[len] = m;                    // reuses smaller best[] -- the overlap
    }
    return best[n];
}

// Traceback: repeatedly take the recorded first cut until nothing is left.
vector<int> pieces(const vector<int>& cut, int n) {
    vector<int> out;
    while (n > 0) {
        out.push_back(cut[n]);            // the winning first cut at length n
        n -= cut[n];                      // cut it off, continue on the rest
    }
    return out;                           // prices 1,5,8,9 at n=4: {2, 2}
}

When an exercise asks for “the” cuts, any optimal set is correct unless a tie-breaking rule is stated; when writing such an exercise, state one.

5. Longest common subsequence

The third example moves to two dimensions, and it is the most deployed DP you will meet this term.

Definition (subsequence, LCS) A subsequence of a string keeps the characters’ relative order but may skip any of them: ACT is a subsequence of AGCAT. The longest common subsequence (LCS) of two strings \(A\) and \(B\) is a longest string that is a subsequence of both. For \(A = \texttt{AGCAT}\), \(B = \texttt{GAC}\): the LCS has length \(2\) — AC (or GC, or GA; length is unique, the string need not be).
Subsequence ≠ substring A substring is contiguous (GCA in AGCAT); a subsequence may leave gaps (ACT is not a substring of AGCAT, but is a subsequence). LCS is about subsequences, which is precisely why it tolerates insertions: in a file comparison, lines added in the middle should not break the match around them. Longest common substring is a different problem — also solvable by DP, with a different recurrence.

Brute force — enumerate all \(2^m\) subsequences of \(A\) and test each against \(B\) — is exponential. The DP subproblem is the LCS of two prefixes:

Proposition (LCS recurrence) Let \(L[i][j]\) be the LCS length of the first \(i\) characters of \(A\) and the first \(j\) of \(B\). Then \(L[i][0] = L[0][j] = 0\), and for \(i, j \ge 1\): \[ L[i][j] = \begin{cases} L[i-1][j-1] + 1 & \text{if } A[i{-}1] = B[j{-}1] \quad\text{(match: extend the diagonal)}\\[2pt] \max\bigl(L[i-1][j],\; L[i][j-1]\bigr) & \text{otherwise} \quad\text{(drop one last character)} \end{cases} \]
Why the recurrence is correct. Look at the last characters of the two prefixes. If they match, some LCS of the prefixes ends with that character: an LCS that ends elsewhere can be exchanged, match for match, into one that uses it without getting shorter. Pairing the two last characters therefore costs nothing, and what remains is an LCS of both shorter prefixes — the diagonal cell, plus one. If they mismatch, no common subsequence ends with both of them, so at least one of the two last characters is not in the LCS; drop it — drop \(A\)’s (the cell above) or \(B\)’s (the cell to the left) — and take the better. Every case reduces to a strictly smaller prefix pair, and the empty prefix is the base case: optimal substructure. Overlap is massive — the same prefix pair is reachable along exponentially many drop orders — and caching leaves just \((m{+}1)(n{+}1)\) subproblems at \(O(1)\) each: \(\Theta(mn)\). ∎

// LCS table: L[i][j] = LCS length of A's first i chars and B's first j.
// Fill row by row -- up, left, and diagonal are ready when a cell needs them.
// Theta(mn) time and space.
int lcsLength(const string& A, const string& B, vector<vector<int>>& L) {
    int m = A.size(), n = B.size();
    L.assign(m + 1, vector<int>(n + 1, 0)); // row 0 / col 0 = 0: empty prefix
    for (int i = 1; i <= m; i++)
        for (int j = 1; j <= n; j++)
            if (A[i - 1] == B[j - 1])
                L[i][j] = L[i - 1][j - 1] + 1;             // match: diagonal + 1
            else
                L[i][j] = max(L[i - 1][j], L[i][j - 1]);   // drop one: up / left
    return L[m][n];                         // bottom-right = the answer
}

// Traceback: walk from L[m][n] toward L[0][0]. The matched characters,
// reversed, spell one LCS. O(m + n) steps -- needs the whole table.
string lcsString(const string& A, const string& B,
                 const vector<vector<int>>& L) {
    string out;
    int i = A.size(), j = B.size();
    while (i > 0 && j > 0) {
        if (A[i - 1] == B[j - 1]) {         // match: it's in the LCS
            out += A[i - 1]; i--; j--;      // step diagonally
        }
        else if (L[i - 1][j] >= L[i][j - 1]) i--;   // follow the larger
        else j--;                                   //   of up / left
    }
    reverse(out.begin(), out.end());
    return out;                             // AGCAT / GAC -> "AC"
}

// LCS table: L[i][j] = LCS length of A's first i chars and B's first j.
// Fill row by row -- up, left, and diagonal are ready when a cell needs them.
// Theta(mn) time and space.
fn lcs_length(a: &[u8], b: &[u8]) -> Vec<Vec<usize>> {
    let (m, n) = (a.len(), b.len());
    let mut l = vec![vec![0usize; n + 1]; m + 1]; // row 0 / col 0 = empty prefix
    for i in 1..=m {
        for j in 1..=n {
            l[i][j] = if a[i - 1] == b[j - 1] {
                l[i - 1][j - 1] + 1                    // match: diagonal + 1
            } else {
                l[i - 1][j].max(l[i][j - 1])           // drop one: up / left
            };
        }
    }
    l                                       // l[m][n] = the answer
}

// Traceback: walk from L[m][n] toward L[0][0]. The matched characters,
// reversed, spell one LCS. O(m + n) steps -- needs the whole table.
fn lcs_string(a: &[u8], b: &[u8], l: &[Vec<usize>]) -> String {
    let mut out = Vec::new();
    let (mut i, mut j) = (a.len(), b.len());
    while i > 0 && j > 0 {
        if a[i - 1] == b[j - 1] {           // match: it's in the LCS
            out.push(a[i - 1]); i -= 1; j -= 1;   // step diagonally
        } else if l[i - 1][j] >= l[i][j - 1] {
            i -= 1;                         // follow the larger of up / left
        } else {
            j -= 1;
        }
    }
    out.reverse();
    String::from_utf8(out).unwrap()         // AGCAT / GAC -> "AC"
}

Walk the traceback on the worked table to see it recover AC: start at \(L[5][3] = 2\). T vs C mismatch, up and left tie at \(2\), follow up; A vs C mismatch, up again; row C vs column C match — emit C, step diagonally; G vs A mismatch, up; row A vs column A match — emit A, step diagonally to row \(0\), stop. Emitted C, A; reversed: AC. Where the traceback breaks a tie differently, it finds a different — equally valid — LCS of the same length.

A built-in bug detector The LCS table has structure you can check at a glance, and a hand-filled (or coded) table that violates it is wrong somewhere:
  • values never decrease going right or down — a longer prefix can only keep or grow the LCS;
  • adjacent cells differ by at most \(1\) — adding one character adds at most one match;
  • the bottom-right cell is a maximum of the whole table.
A value that drops, or jumps by \(2\), pinpoints a recurrence or indexing bug to within a cell — check these invariants on a small example before trusting any LCS code (exercise 6 practices exactly this).

Cost, by the formula of §2: \((m+1)(n+1)\) cells at \(O(1)\) each, \(\Theta(mn)\) time and space. Since each row depends only on the row above and the current row’s left neighbor, the space can drop to two rows — \(\Theta(\min(m,n))\) with the shorter string as columns — but the two-row version can report only the length: the traceback needs the whole table. Decide up front which your application needs (§6, step 5).

LCS in the wild The same \(\Theta(mn)\) table powers diff and git diff (LCS of lines: unmatched lines are the insertions and deletions shown), merge tools and plagiarism detectors, and — at industrial scale — DNA and protein sequence alignment, where LCS-style tables measure evolutionary similarity. Its close cousin edit distance, the minimum number of single-character edits between two strings, fills an almost identical table and drives spell-checking; it is the opening example of Chapter 15 (ch15-dp-ii.html).

6. The DP recipe

Three examples, one process. Every DP you will design this term — and most you will ever design — answers the same five questions, in order. This recipe, not any particular example, is the transferable skill.

Definition (the DP recipe)
  1. Subproblem. State exactly what \(dp[i]\) (or \(dp[i][j]\)) means.
  2. Recurrence. Express each subproblem in terms of strictly smaller ones — usually by casing on the last decision.
  3. Base cases. The subproblems small enough to answer directly.
  4. Order. Evaluate so every dependency is computed first (automatic if memoizing; your job if tabulating).
  5. Answer. Which cell holds it — plus a traceback, if the solution and not just its value is wanted.

Applied to this chapter’s three problems:

stepFibonaccirod cuttingLCS
subproblem\(\mathrm{fib}(i)\) \(\mathrm{best}[\ell]\): max revenue, length \(\ell\) \(L[i][j]\): LCS length of the two prefixes
recurrence\(\mathrm{fib}(i{-}1) + \mathrm{fib}(i{-}2)\) max over first cuts \(i\) match? diagonal \(+\,1\) : \(\max\)(up, left)
base cases\(\mathrm{fib}(0){=}0\), \(\mathrm{fib}(1){=}1\) \(\mathrm{best}[0] = 0\) empty prefix \(\to 0\)
order\(i\) ascending\(\ell\) ascending row by row, left to right
answer\(\mathrm{fib}(n)\) \(\mathrm{best}[n]\) (+ cuts via \(\mathrm{cut}[]\)) \(L[m][n]\) (+ traceback)
cost\(n \times O(1) = \Theta(n)\) \(n \times O(n) = \Theta(n^2)\) \(mn \times O(1) = \Theta(mn)\)

Step 1 is the hard one

Defining the subproblem is where DP problems are won or lost. A precise definition — “\(\mathrm{best}[\ell]\) is the maximum revenue obtainable from a rod of length exactly \(\ell\)” — makes the recurrence nearly write itself; a vague one (“\(dp[i]\) is something about the first \(i\) items”) leads nowhere, because the recurrence cannot close: it needs information the state does not carry. The professional vocabulary for this is state (one subproblem, one table cell) and transition (the recurrence linking states), and the whole of DP design is: choose the state, then the transition. The complexity then reads off as \((\#\text{states}) \times (\text{transition cost})\) — the cost formula of §2 restated.

For step 2, the reliable trigger is the last decision: the last first-cut in rod cutting, whether the last two characters pair in LCS, whether the last item goes in the knapsack. Each possible last decision leaves a smaller subproblem; combine over the choices with \(\max\), \(\min\), or \(+\) as the goal demands. And a small pattern library speeds up step 1 itself — most DP tables have one of a few shapes:

shapesubproblemexamples
1-D\(dp[i]\) over a prefix or length Fibonacci, rod cutting, coin change
2-D grid\(dp[i][j]\) over two indices LCS; edit distance and 0/1 knapsack (Chapter 15)
interval\(dp[i][j]\) over a range \(i..j\) matrix-chain ordering, optimal binary search trees
Common DP mistakes
  • A vague subproblem — the recurrence won’t close because the state is missing information. Fix the definition, not the code.
  • Wrong fill order (tabulation only) — reading a cell before it is written. Memoization is immune, which is one reason to prototype top-down.
  • Missing or misplaced base cases — off-by-one at row \(0\) / column \(0\) is the classic LCS bug; the sanity invariants of §5 catch it.
  • Memoizing a non-DP — if the subproblems never repeat (mergesort’s halves), the cache is pure overhead. Check for overlap before reaching for DP.

When to reach for DP, in one breath: an optimum over exponentially many choices, whose choices generate overlapping subproblems, where greedy is not provably safe. Then: subproblem, recurrence, cache — and the exponential collapses to a polynomial. Chapter 15 (ch15-dp-ii.html) runs this recipe on the two problems where Chapter 12 showed greedy failing outright — the 0/1 knapsack and edit distance — and closes the arc: the problems that defeated greedy fall to DP.

  1. For each, say whether memoization would help, and why: (a) mergesort; (b) naive Fibonacci; (c) binary search.
    solution (a) No. Mergesort’s two halves are disjoint subarrays — the recursion tree’s nodes are all distinct subproblems, so a cache stores answers that are never requested again: divide-and-conquer, not DP. (b) Yes — the poster child: \(\mathrm{fib}(k)\) is requested by many parents, and caching collapses \(\Theta(\varphi^n)\) to \(\Theta(n)\). (c) No. Binary search makes a single recursive call per level — there is only one path, \(\Theta(\log n)\) subproblems, each visited once; nothing repeats, nothing to cache.

    solution (a) Two rows of the shorter dimension: \(\Theta(\min(m,n))\), with time still \(\Theta(mn)\).
