Chapter 15 — Dynamic Programming II

Dynamic programming put to work. The 0/1 knapsack problem — take each item whole or not at all — solved by a 2-D take-or-skip table in \(\Theta(nW)\), and the surprising fine print on that bound: pseudo-polynomial time. Edit distance — the fewest insert/delete/replace operations between two strings — by a min-of-three table in \(\Theta(mn)\). Grid and interval DP round out the table shapes, and the two techniques every optimization DP needs: a traceback to recover the solution, not just its value, and a rolling array to cut the space. With full worked tables, complete code, links to the demos, and exercises.

Chapter 14 and the Dynamic Programming handout built the DP recipe — define the subproblem, write the recurrence off the last decision, pin the base cases, fill in dependency order, read the answer cell — and ran it on Fibonacci, rod cutting, and the longest common subsequence (LCS). This chapter applies the recipe to two problems where greedy visibly fails (the Greedy Algorithms handout): the 0/1 knapsack, whose take-or-skip table also exposes the subtle notion of pseudo-polynomial time, and edit distance, the min-of-three string table behind every spell-checker. Grid DP and interval DP then complete the small library of table shapes that covers almost every DP you will meet, and two closing techniques make the method fully usable: the traceback, which recovers the actual items, edits, or path from a filled table, and the rolling array, which trades that traceback away for a dramatic space saving.

1. The 0/1 knapsack problem

Definition (0/1 knapsack) Given \(n\) items, each with a value \(v_i\) and a weight \(w_i\), and a knapsack of capacity \(W\): choose a subset of items of maximum total value whose total weight is at most \(W\). Each item is taken whole or not at all — that all-or-nothing choice is the “0/1.”

The running example, four items and capacity \(W = 5\): 

  item    A   B   C   D
  value   3   4   5   6
  weight  2   3   4   5        W = 5

Which subset fits and is worth the most? Contrast this with the fractional knapsack of Chapter 12, where items could be cut and greedy-by-ratio was provably optimal. Making the items indivisible breaks the greedy argument completely:

Example (greedy fails) Capacity \(W = 6\), three items: \(P\) (value \(5\), weight \(4\), ratio \(1.25\)), \(Q\) (value \(3\), weight \(3\), ratio \(1.00\)), \(R\) (value \(3\), weight \(3\), ratio \(1.00\)). Greedy takes the best-ratio item \(P\) first — and then only \(2\) units of capacity remain, into which nothing fits, for a total of \(5\). The optimum is \(Q + R\): weight \(6\), value \(6\). One high-ratio item blocks two jointly better ones. The best choice for any one item depends on the remaining capacity and on all the other items, so no single choice can be committed to greedily — both including and excluding each item must be considered. (On the A–D set above, greedy happens to get lucky; one passing test is not a proof, which is why this counterexample matters.)

Trying everything naively is hopeless: \(n\) items form \(2^n\) subsets, and for \(n = 40\) that is \(2^{40} \approx 1.1 \times 10^{12}\) — a trillion subsets. DP replaces the enumeration with a table of \(n \cdot W\) cells: for \(n = 40\) and \(W = 1000\), forty thousand cells. From \(10^{12}\) to \(4 \times 10^4\) is the same exponential-to-polynomial collapse as memoized Fibonacci, now on a real optimization problem (with one caveat about that \(W\), the subject of §2).

The subproblem and the recurrence

Definition (knapsack table) Let \(K[i][w]\) be the maximum value achievable using only the first \(i\) items within weight capacity \(w\). The last-decision question — take item \(i\) or not? — gives the recurrence \[ K[i][w] = \max\bigl(\; \underbrace{K[i-1][w]}_{\text{skip item } i},\;\; \underbrace{v_i + K[i-1][w - w_i]}_{\text{take item } i \text{ (only if } w_i \le w\text{)}} \;\bigr), \] with base cases \(K[0][w] = 0\) (no items, no value) and \(K[i][0] = 0\) (no capacity, nothing fits). The answer is \(K[n][W]\).

The state is two-dimensional — how many items have been considered, and how much capacity remains — exactly the “add a dimension when the recurrence needs information the state lacks” move from Chapter 14: the value of taking an item cannot be judged without knowing the capacity left for the rest. Skipping item \(i\) keeps the best of the first \(i-1\) items at the same capacity (the cell directly above); taking it earns \(v_i\) plus the best the earlier items can do in the reduced capacity \(w - w_i\) (one row up, \(w_i\) columns left). The guard \(w_i \le w\) is the only wrinkle this recurrence adds over LCS — an item that does not fit cannot be taken.

The indexing pitfall In the recurrence, “item \(i\)” is 1-based — \(K[i][w]\) covers items \(1\) through \(i\), and row \(0\) is the empty set. A 0-based C++ vector of items therefore reads value[i-1] and weight[i-1] inside the row-\(i\) loop (or carries a dummy slot \(0\), as the code below does). This off-by-one, and its twin at the base-case edges, is the single most common knapsack bug: write row \(0\) and column \(0\) first, then the double loop never reads an uninitialized or misaligned cell.
Example (one cell, both branches) Compute \(K[2][5]\) — items A and B available, capacity \(5\); B has value \(4\), weight \(3\): 
  skip B:  K[1][5] = 3                       (just A)
  take B:  4 + K[1][5-3] = 4 + K[1][2] = 4 + 3 = 7
  K[2][5] = max(3, 7) = 7                    ← take both A and B
Taking B adds its value to the best use of the two units of capacity that remain for the earlier items — \(K[1][2] = 3\), item A alone (weights \(2 + 3 = 5\) exactly fill the sack). Every cell of the table is this one computation. One more, at a tighter capacity: \(K[3][4]\) (items A–C, capacity \(4\); C has value \(5\), weight \(4\)). Skip C: \(K[2][4] = 4\) (B alone — A\(+\)B weigh \(5\), too much for capacity \(4\)). Take C: \(5 + K[2][0] = 5 + 0 = 5\), C using the whole capacity. \(K[3][4] = \max(4, 5) = 5\): C alone beats B alone at capacity \(4\).

// Tabulated 0/1 knapsack. value[i], weight[i] describe item i (1-based, with a
// dummy slot 0). K[i][w] = best value using items 1..i within capacity w.
// Theta(nW) time and space.
int knapsack(int n, int W, const vector<int>& value, const vector<int>& weight,
             vector<vector<int>>& K) {
    K.assign(n + 1, vector<int>(W + 1, 0));           // row 0: no items -> 0
    for (int i = 1; i <= n; i++)
        for (int w = 0; w <= W; w++) {
            K[i][w] = K[i-1][w];                      // skip item i
            if (weight[i] <= w)                       // does it fit?
                K[i][w] = max(K[i][w],
                              value[i] + K[i-1][w - weight[i]]);   // take it
        }
    return K[n][W];                                   // the optimal value
}

// Traceback: recover WHICH items achieve K[n][W]. If a cell differs from the
// cell directly above, item i was taken; drop its weight and continue. O(n).
vector<int> knapsackItems(int n, int W, const vector<int>& weight,
                          const vector<vector<int>>& K) {
    vector<int> taken;
    int w = W;
    for (int i = n; i >= 1; i--)
        if (K[i][w] != K[i-1][w]) {                   // value changed: taken
            taken.push_back(i);
            w -= weight[i];
        }
    reverse(taken.begin(), taken.end());              // forward order
    return taken;
}

// with a dummy slot 0). k[i][w] = best value using items 1..=i within
// capacity w. Theta(nW) time and space. Returns the full table.
fn knapsack(n: usize, cap: usize, value: &[i64], weight: &[usize]) -> Vec<Vec<i64>> {
    let mut k = vec![vec![0i64; cap + 1]; n + 1];     // row 0: no items -> 0
    for i in 1..=n {
        for w in 0..=cap {
            k[i][w] = k[i - 1][w];                    // skip item i
            if weight[i] <= w {                       // does it fit?
                k[i][w] = k[i][w].max(value[i] + k[i - 1][w - weight[i]]);
            }
        }
    }
    k                                                 // k[n][cap] is the optimum
}

// Traceback: recover WHICH items achieve k[n][cap]. If a cell differs from
// the cell directly above, item i was taken; drop its weight. O(n).

            taken.push(i);
            w -= weight[i];
        }
    }
    taken.reverse();                                  // forward order
    taken
}

The traceback deserves a word here (its general pattern is §6). \(K[n][W]\) is only the optimal value; to know which items achieve it, walk from the bottom-right upward, one item per step. If a cell equals the cell directly above, skipping item \(i\) was (one) optimal choice — taking it changed nothing — so the item was not needed; if the values differ, the item was taken: record it and reduce the remaining capacity by its weight. On the table above: 

  K[4][5]=7 == K[3][5]=7  →  D skipped     (w stays 5)
  K[3][5]=7 == K[2][5]=7  →  C skipped     (w stays 5)
  K[2][5]=7 != K[1][5]=3  →  B TAKEN       (w = 5 − 3 = 2)
  K[1][2]=3 != K[0][2]=0  →  A TAKEN       (w = 2 − 2 = 0)
  →  items {A, B}, value 7
Proposition (knapsack cost) The table has \((n+1)(W+1)\) cells, each filled in \(O(1)\), so the DP runs in \(\Theta(nW)\) time and space; the traceback adds \(O(n)\). (Space drops to \(\Theta(W)\) with the rolling array of §6, if only the value is needed.)
The knapsack family Small changes to the transition cover a whole family of problems. In the 0/1 version, the take branch reads row \(i-1\) — item \(i\), once used, is gone. In the unbounded version (unlimited copies of each item), the take branch reads row \(i\) itself — \(K[i][w] = \max(K[i-1][w],\; v_i + K[i][w-w_i])\) — so an item can be taken again. A bounded version (at most \(k\) copies) loops over the count. Coin change (§5) is unbounded knapsack in disguise, minimizing the number of coins instead of maximizing value. One mental model, a one-line tweak per variant — and §6 shows the same 0/1-versus-unbounded switch hiding inside a loop direction.

The chosen subset changing with capacity is the greedy counterexample’s lesson made visible: no per-item score can decide membership alone.

2. Pseudo-polynomial time

\(\Theta(nW)\) looks polynomial — two variables, multiplied. But look at what the two variables are. The \(n\) counts items: an input with more items is physically longer. The \(W\), though, is not a count of anything in the input — it is the numeric value of a single number that the input spells in binary.

Definition (pseudo-polynomial) An algorithm runs in pseudo-polynomial time if its running time is polynomial in the numeric value of the input’s numbers but not in the size (bit-length) of the input. Knapsack’s \(\Theta(nW)\) is the standard example.
Proposition (the bit-length argument) \(\Theta(nW)\) is exponential in the size of the encoded input.
Why. The capacity \(W\) occupies \(b = \lceil \log_2(W{+}1) \rceil\) bits of input — \(W = 1{,}000{,}000\) is written in just \(20\) bits, since \(2^{20} = 1{,}048{,}576\) — yet the DP builds a table with \(W + 1\) columns: a million columns conjured from twenty bits. In general a \(b\)-bit capacity can be as large as \(2^b - 1\), so the running time is \[ \Theta(nW) = \Theta\!\bigl(n \cdot 2^{\,b}\bigr), \] exponential in the bit-length \(b\). The everyday symptom: appending one \(0\) bit to \(W\) — one more input character — doubles the running time. A truly polynomial algorithm would have to run in time polynomial in \(n\) and \(\log W\). ∎

The contrast with edit distance (§3) makes the distinction sharp. Its \(\Theta(mn)\) bound is truly polynomial, because \(m\) and \(n\) are the lengths of the two input strings — the table grows with the physical size of the input, not with the value of a number inside it. Knapsack’s table has a dimension proportional to a value, and values grow exponentially faster than their encodings.

No contradiction with NP-hardness The 0/1 knapsack problem is NP-hard (NP: nondeterministic polynomial time) — among the problems for which no polynomial-time algorithm is known, and none is expected. The \(\Theta(nW)\) DP does not contradict that: it is efficient when the capacity is numerically modest, and exponential (in the input size) when \(W\) is astronomically large — exactly the regime the hardness lives in. In practice this makes the DP a genuinely useful algorithm: capacities in real scheduling, budgeting, and cargo problems are usually small enough that \(nW\) is tiny. Recognizing a pseudo-polynomial bound — asking “is this variable a count or a value?” — is a standard exam and interview point.

3. Edit distance

Definition (edit distance) The edit distance (also Levenshtein distance) between strings \(A\) (length \(m\)) and \(B\) (length \(n\)) is the fewest single-character edits — insert, delete, or replace — that transform \(A\) into \(B\).

The classic instance: kitten \(\to\) sitting takes three edits — replace k\(\to\)s, replace e\(\to\)i, insert g. Edit distance is the engine behind spell-checkers (nearest dictionary word to a typo), “did you mean…?” fuzzy search, record linkage, and — with weighted operations — DNA and protein alignment. Like LCS, the brute force over all edit sequences is exponential; the DP is \(\Theta(mn)\), and truly polynomial (§2).

Definition (edit-distance table) Let \(D[i][j]\) be the edit distance between the first \(i\) characters of \(A\) and the first \(j\) characters of \(B\). If the last characters match, no edit is spent on them; otherwise pay \(1\) for the cheapest of the three operations: \[ D[i][j] = \begin{cases} D[i-1][j-1] & \text{if } A[i{-}1] = B[j{-}1] \text{ (match, free)}\\[4pt] 1 + \min\bigl(D[i-1][j],\; D[i][j-1],\; D[i-1][j-1]\bigr) & \text{otherwise,} \end{cases} \] where the three candidates are delete \(A[i{-}1]\) (up), insert \(B[j{-}1]\) (left), and replace \(A[i{-}1] \to B[j{-}1]\) (diagonal). Base cases: \(D[i][0] = i\) (turn a length-\(i\) prefix into the empty string by \(i\) deletions) and \(D[0][j] = j\) (build a length-\(j\) prefix by \(j\) insertions). The answer is \(D[m][n]\).

The state is again a pair of prefixes — the same 2-D shape as LCS — but the transition reads three neighbors and takes a min. Two cells worked by hand make the fill mechanical:

Example (two worked cells) \(D[1][1]\), turning "k" into "s": 
  'k' != 's'  →  1 + min( D[0][1] = 1,     // delete 'k', then insert 's'
                           D[1][0] = 1,     // insert 's', then delete 'k'
                           D[0][0] = 0 )    // replace k → s
              =  1 + 0  =  1                (one replace)
The delete and insert routes each cost \(2\) in total (the neighbor already holds one edit); the single replace wins. Next, \(D[2][1]\), turning "ki" into "s": 'i' \(\ne\) 's', so \(1 + \min(D[1][1], D[2][0], D[1][0]) = 1 + \min(1, 2, 1) = 2\) — for instance replace k\(\to\)s then delete i. Both values appear in the full table below. 
// Edit (Levenshtein) distance. D[i][j] = fewest edits turning the first i
// characters of a into the first j characters of b. Theta(mn) time and space.
int editDistance(const string& a, const string& b, vector<vector<int>>& D) {
    int m = a.size(), n = b.size();
    D.assign(m + 1, vector<int>(n + 1, 0));
    for (int i = 0; i <= m; i++) D[i][0] = i;         // base: delete all of a
    for (int j = 0; j <= n; j++) D[0][j] = j;         // base: insert all of b
    for (int i = 1; i <= m; i++)
        for (int j = 1; j <= n; j++)
            if (a[i-1] == b[j-1])
                D[i][j] = D[i-1][j-1];                // match: free diagonal
            else
                D[i][j] = 1 + min({D[i-1][j],         // delete a[i-1]
                                   D[i][j-1],         // insert b[j-1]
                                   D[i-1][j-1]});     // replace a[i-1]->b[j-1]
    return D[m][n];
}

// Traceback: read the edit script off the table, bottom-right to origin. Each
// step asks which neighbor produced this cell's value. O(m + n).
vector<string> editScript(const string& a, const string& b,
                          const vector<vector<int>>& D) {
    vector<string> ops;
    int i = a.size(), j = b.size();
    while (i > 0 || j > 0) {
        if (i > 0 && j > 0 && a[i-1] == b[j-1] && D[i][j] == D[i-1][j-1]) {
            ops.push_back(string("match ") + a[i-1]);            // free
            i--; j--;
        } else if (i > 0 && j > 0 && D[i][j] == D[i-1][j-1] + 1) {
            ops.push_back(string("replace ") + a[i-1] + "->" + b[j-1]);
            i--; j--;
        } else if (i > 0 && D[i][j] == D[i-1][j] + 1) {
            ops.push_back(string("delete ") + a[i-1]);
            i--;
        } else {
            ops.push_back(string("insert ") + b[j-1]);
            j--;
        }
    }
    reverse(ops.begin(), ops.end());                  // forward order
    return ops;
}
// Edit (Levenshtein) distance. D[i][j] = fewest edits turning the first i
// characters of a into the first j characters of b. Theta(mn) time and space.
static int editDistance(String a, String b, int[][] D) {
    int m = a.length(), n = b.length();
    for (int i = 0; i <= m; i++) D[i][0] = i;     // base: delete all of a
    for (int j = 0; j <= n; j++) D[0][j] = j;     // base: insert all of b
    for (int i = 1; i <= m; i++)
        for (int j = 1; j <= n; j++)
            if (a.charAt(i-1) == b.charAt(j-1))
                D[i][j] = D[i-1][j-1];            // match: free diagonal
            else
                D[i][j] = 1 + Math.min(D[i-1][j],           // delete
                              Math.min(D[i][j-1],           // insert
                                       D[i-1][j-1]));       // replace
    return D[m][n];
}

# characters of a into the first j characters of b. Theta(mn) time and space.
def edit_distance(a, b):
    m, n = len(a), len(b)
    D = [[0] * (n + 1) for _ in range(m + 1)]
    for i in range(m + 1): D[i][0] = i            # base: delete all of a
    for j in range(n + 1): D[0][j] = j            # base: insert all of b
    for i in range(1, m + 1):
        for j in range(1, n + 1):
            if a[i - 1] == b[j - 1]:
                D[i][j] = D[i - 1][j - 1]         # match: free diagonal
            else:
                D[i][j] = 1 + min(D[i - 1][j],    # delete a[i-1]
                                  D[i][j - 1],    # insert b[j-1]
                                  D[i - 1][j - 1])  # replace a[i-1]->b[j-1]
    return D                                      # D[m][n] is the distance

# Traceback: read the edit script off the table, bottom-right to origin. Each
# step asks which neighbor produced this cell's value. O(m + n).
def edit_script(a, b, D):
    ops, i, j = [], len(a), len(b)
    while i > 0 or j > 0:
        if i > 0 and j > 0 and a[i-1] == b[j-1] and D[i][j] == D[i-1][j-1]:
            ops.append("match " + a[i - 1])       # free
            i, j = i - 1, j - 1
        elif i > 0 and j > 0 and D[i][j] == D[i-1][j-1] + 1:
            ops.append("replace " + a[i - 1] + "->" + b[j - 1])
            i, j = i - 1, j - 1
        elif i > 0 and D[i][j] == D[i-1][j] + 1:
            ops.append("delete " + a[i - 1])
            i -= 1
        else:
            ops.append("insert " + b[j - 1])
            j -= 1
    return ops[::-1]                              # forward order
// Edit (Levenshtein) distance. d[i][j] = fewest edits turning the first i
// characters of a into the first j characters of b. Theta(mn) time and space.
fn edit_distance(a: &[u8], b: &[u8]) -> Vec<Vec<usize>> {
    let (m, n) = (a.len(), b.len());
    let mut d = vec![vec![0usize; n + 1]; m + 1];
    for i in 0..=m { d[i][0] = i; }                   // base: delete all of a
    for j in 0..=n { d[0][j] = j; }                   // base: insert all of b
    for i in 1..=m {
        for j in 1..=n {
            d[i][j] = if a[i - 1] == b[j - 1] {
                d[i - 1][j - 1]                       // match: free diagonal
            } else {
                1 + d[i - 1][j]                       // delete a[i-1]
                    .min(d[i][j - 1])                 // insert b[j-1]
                    .min(d[i - 1][j - 1])             // replace a[i-1]->b[j-1]
            };
        }
    }
    d                                                 // d[m][n] is the distance
}

// Traceback: read the edit script off the table, bottom-right to origin.
// Each step asks which neighbor produced this cell's value. O(m + n).
fn edit_script(a: &[u8], b: &[u8], d: &[Vec<usize>]) -> Vec<String> {
    let mut ops = Vec::new();
    let (mut i, mut j) = (a.len(), b.len());
    while i > 0 || j > 0 {
        if i > 0 && j > 0 && a[i - 1] == b[j - 1] && d[i][j] == d[i - 1][j - 1] {
            ops.push(format!("match {}", a[i - 1] as char));     // free
            i -= 1; j -= 1;
        } else if i > 0 && j > 0 && d[i][j] == d[i - 1][j - 1] + 1 {
            ops.push(format!("replace {}->{}", a[i - 1] as char, b[j - 1] as char));
            i -= 1; j -= 1;
        } else if i > 0 && d[i][j] == d[i - 1][j] + 1 {
            ops.push(format!("delete {}", a[i - 1] as char));
            i -= 1;
        } else {
            ops.push(format!("insert {}", b[j - 1] as char));
            j -= 1;
        }
    }
    ops.reverse();                                    // forward order
    ops
}

Structurally this is LCS’s double loop with the max-of-two swapped for a min-of-three and the base-case edges filled with \(i\) and \(j\) instead of zeros. The traceback, as always, turns the number into the answer users actually want: each backward step names an operation — a diagonal step on matching characters is a free match, a diagonal on differing characters a replace, an up-step a delete, a left-step an insert — and the reversed path is an optimal edit script. For kitten \(\to\) sitting: 

  k → s   replace
  i → i   match        t → t   match        t → t   match
  e → i   replace
  n → n   match
  ε → g   insert
  →  2 replaces + 1 insert  =  3 edits  =  D[6][7]  ✓

This script — not the count — is what a spell-checker’s suggestion ranking, a diff hunk, or a genome aligner outputs.

Edit distance is a metric With unit costs, edit distance satisfies the metric axioms: \(d(A,A) = 0\), symmetry \(d(A,B) = d(B,A)\) (every edit has an inverse of equal cost), and the triangle inequality \(d(A,C) \le d(A,B) + d(B,C)\) (concatenating two edit scripts transforms \(A\) to \(C\), so the shortest script can only be shorter). Strings therefore live in a genuine metric space, which unlocks metric algorithms and data structures — nearest-neighbor search, clustering, and BK-tree (Burkhard–Keller tree) indexes that answer “every dictionary word within \(2\) edits of this typo” without a linear scan. Not every similarity score is a metric; this one is, and the triangle inequality is precisely what the indexes prune with.

If only insert and delete were allowed (no replace), the optimal strategy would be to keep exactly an LCS and edit everything else: \(m + n - 2\cdot\mathrm{LCS}\) edits. Allowing replace can only help (one replace does the work of a delete\(+\)insert pair), so \[ \max(m, n) - \mathrm{LCS} \;\le\; \text{edit distance} \;\le\; m + n - 2\cdot\mathrm{LCS}. \] On kitten/sitting: the LCS is ittn, length \(4\) (check it in the LCS demo), giving bounds \(7 - 4 = 3 \le 3 \le 6 + 7 - 8 = 5\) — the lower bound is tight here because the optimal script uses replaces wherever possible. If you can fill one of these tables, you can fill the other; they are the “same” problem viewed as a max versus a min.

Weighting the edits Nothing in the DP assumes each operation costs \(1\). Replace the constants with per-operation costs — say replace \(= 2\), insert \(=\) delete \(= 1\), to model a typo as cheaper than an omission — or even a per-character-pair cost matrix, and the recurrence, fill order, and traceback are unchanged. That one-line generalization is biological sequence alignment: the Needleman–Wunsch algorithm is weighted edit distance with a substitution matrix that charges likely mutations less. Same table, different constants, Nobel-grade reach.

4. Grid DP and interval DP

Knapsack and edit distance are both “2-D table, few fixed neighbors” DPs. Two more shapes complete the pattern library: grids, where the table is the input, and intervals, where a cell is a range and the transition scans every split point.

Grid DP: paths through a cost grid

Definition (minimum-cost path) Given a grid of nonnegative cell costs, move from the top-left to the bottom-right corner stepping only right or down, minimizing the total cost of cells visited. The DP is \[ \mathit{dp}[i][j] = \mathit{cost}[i][j] + \min\bigl(\mathit{dp}[i-1][j],\; \mathit{dp}[i][j-1]\bigr), \] with the top-left cell as base case and edge cells reading their only available neighbor.
Example (a 3×3 grid)
cost dp
131 145
151 276
421 687
Each \(dp\) cell adds its own cost to the cheaper of the cell above and the cell to the left: \(dp[1][2] = 1 + \min(5, 7) = 6\), and the corner \(dp[2][2] = 1 + \min(6, 8) = 7\) is the minimum path cost. The highlighted cells are the optimal path \(1 \to 3 \to 1 \to 1 \to 1\) (right, right, down, down), total \(7\) — recovered, as always, by tracing back through whichever neighbor produced each cell.

The same table flexes to relatives by changing only the combine step. Counting monotone paths sums instead of minimizing, \(\mathit{dp}[i][j] = \mathit{dp}[i-1][j] + \mathit{dp}[i][j-1]\), with the edges at \(1\): on a \(3\times 3\) grid the rows fill \(1\,1\,1\) / \(1\,2\,3\) / \(1\,3\,6\), and the corner’s \(6 = \binom{4}{2}\) counts the arrangements of two rights and two downs — the DP rediscovers a binomial coefficient. Obstacles force a cell’s count to \(0\) (or its cost to \(\infty\)); extra move directions just add source neighbors. Robot-path planning, seam carving in image resizing, and dynamic-time-warping all ride this one shape.

Interval DP: the state is a range

Definition (interval DP) In interval DP the subproblem is a contiguous range \([i, j]\) of the input, and the transition optimizes over the split point \(k\): \[ \mathit{dp}[i][j] = \min_{i \le k < j}\bigl(\mathit{dp}[i][k] + \mathit{dp}[k+1][j] + \mathit{cost}(i, j, k)\bigr), \] filled in order of increasing range length. With \(\Theta(n^2)\) ranges and \(\Theta(n)\) splits each, interval DPs typically cost \(\Theta(n^3)\).

The canonical instance is matrix-chain multiplication. Matrix multiplication is associative — every parenthesization of \(A_1 A_2 \cdots A_n\) gives the same product — but not equally priced: multiplying a \(p \times q\) by a \(q \times r\) matrix costs \(p\,q\,r\) scalar multiplications, and the order changes the totals wildly. For \(A\ (10{\times}30)\), \(B\ (30{\times}5)\), \(C\ (5{\times}60)\): 

  (AB)C :  10·30·5  +  10·5·60  =  1500 +  3000  =   4,500
  A(BC) :  30·5·60  + 10·30·60  =  9000 + 18000  =  27,000     6× worse

The last-decision question is which multiplication happens last — the outermost split \(k\): \[ \mathit{dp}[i][j] = \min_{i \le k < j}\bigl(\mathit{dp}[i][k] + \mathit{dp}[k+1][j] + \mathit{rows}_i \cdot \mathit{cols}_k \cdot \mathit{cols}_j\bigr), \] the two halves optimally parenthesized by induction plus the cost of the final multiply. Base case: a single matrix costs \(0\). \(\Theta(n^3)\) overall — and the traceback (store the winning \(k\) per range) recovers the parenthesization itself.

The same shape gives the optimal binary search tree (BST): given keys with known access frequencies, \( \mathit{dp}[i][j]\) is the cheapest expected search cost of a BST over keys \(i..j\), and the split is which key becomes the root, with \(\sum_{k=i}^{j}\mathit{freq}(k)\) added because choosing any root sinks every key in the range one level deeper. Frequently-searched keys settle near the root — the reason a balanced tree (Chapter 3) is not optimal under skewed access, and the principle behind frequency-weighted index structures in real databases. Interval DP is the trickiest of the shapes — the fill order is by range length, not row order — but the recipe is unchanged: state, transition, base, order, answer.

Recognizing the shape hands you the state and the fill order — the two hardest steps — for free.

5. Two more one-dimensional DPs: coin change and LIS

Coin change, revisited

Chapter 12 showed greedy coin change failing on the system \(\{1, 3, 4\}\): making \(6\), greedy grabs the \(4\) and finishes \(4 + 1 + 1\), three coins, when two \(3\)s suffice. The DP fix is one line — state = the amount, transition = try every coin as the last coin used: \[ \mathit{dp}[a] = 1 + \min_{\,c \,\le\, a}\; \mathit{dp}[a - c], \qquad \mathit{dp}[0] = 0. \]

Example (coins {1, 3, 4}, amounts 0–6)
a0123456
dp[a] 0121122
\(dp[6]\) tries a last coin of \(1\) (\(1 + dp[5] = 3\)), of \(3\) (\(1 + dp[3] = 2\)), and of \(4\) (\(1 + dp[2] = 3\)), and keeps \(2\) — the \(3 + 3\) that greedy missed. Cost \(\Theta(a \cdot k)\) for \(k\) coin kinds; and note the family resemblance: this is unbounded knapsack (§1’s aside) with values minimized instead of maximized — a coin can be reused, so the transition stays within the same “row.”

Longest increasing subsequence

Definition (LIS) The longest increasing subsequence (LIS) of an array is the longest subsequence (order kept, gaps allowed — the Chapter 14 distinction) whose values strictly increase. The DP state is the subtle part: let \(\mathit{dp}[i]\) be the length of the longest increasing subsequence ending exactly at index \(i\); then \[ \mathit{dp}[i] = 1 + \max\{\, \mathit{dp}[j] : j < i,\; a[j] < a[i] \,\} \qquad (\text{or } 1 \text{ if no such } j), \] and the answer is \(\max_i \mathit{dp}[i]\), anywhere in the array.
Example
a[i] 31415926
dp[i] 11213424
Each \(dp[i]\) extends the best earlier subsequence ending in a smaller value: \(dp[4] = 3\) via \(1, 4, 5\) (or \(3,4,5\)), \(dp[5] = 4\) via \(1, 4, 5, 9\), and \(dp[7] = 4\) via \(1, 4, 5, 6\). The LIS length is \(4\); tracing back from either maximum recovers \(1\ 4\ 5\ 9\) or \(1\ 4\ 5\ 6\).

Why “ending at \(i\)” rather than “within the first \(i\) elements”? Because the recurrence must know whether the subsequence can be extended by \(a[i]\), and only the value it ends on decides that — a prefix-based state discards exactly the information the transition needs. This is Chapter 14’s advice made concrete: when the recurrence needs a fact the state does not carry, redefine (or extend) the state. The double loop is \(\Theta(n^2)\); a refinement that keeps, for each length, the smallest possible tail value and binary-searches each element into place (“patience sorting”) achieves \(\Theta(n \log n)\) — a nice marriage of DP thinking with the searching toolkit, worth knowing by name. LIS itself shows up as longest chains of nested boxes, compatible version upgrades, and one-axis scheduling.

6. Traceback and space-saving

Two techniques appeared in every problem above; they deserve their general statements, because they apply to every optimization DP you will ever write.

The traceback, in general

Pattern (traceback) A filled DP table stores optimal values; to recover the optimal solution:
  1. start at the answer cell;
  2. determine which neighbor (and which decision) produced this cell’s value — by comparing against each candidate of the recurrence;
  3. record the decision and move to that neighbor;
  4. stop at a base case; reverse the recorded decisions.
Cost: \(O(\text{path length})\). Either store the winning choice per cell while filling (extra memory, trivial walk) or recompute it during the walk (no memory, a few comparisons per step) — the knapsack and edit-distance code of §1 and §3 both recompute.

Chapter 14 traced rod cutting and LCS; this chapter traced knapsack (the equal-vs-different test against the cell above), edit distance (which of three neighbors explains the value), and the grid path. Five problems, one pattern — and one universal skeleton around it: 

  1. allocate the table; set the BASE cells
  2. loop over states in dependency ORDER
  3. each cell = combine(smaller cells)        // the recurrence
  4. read the ANSWER cell
  5. (optional) TRACE BACK for the solution

Every DP in these two chapters — Fibonacci, rod cutting, LCS, knapsack, edit distance, grids, intervals, coin change, LIS — is this five-line skeleton with a different state and combine step. The scaffolding never changes; only the recurrence does. That is what makes DP one technique rather than a zoo of tricks.

Space-saving: the rolling array

Look back at the knapsack and edit-distance recurrences: row \(i\) reads only row \(i-1\) (and, to its left, row \(i\) itself). Rows older than the previous one are dead weight — so keep two rows and swap them, or for knapsack even one row updated in place, cutting space from \(\Theta(mn)\) or \(\Theta(nW)\) to \(\Theta(n)\) or \(\Theta(W)\): 

// 0/1 knapsack value in Theta(W) space: ONE row, updated RIGHT-TO-LEFT so that
// K[w - weight[i]] still holds the previous row's value (item i not yet used).
// Iterating forward instead would let item i be taken again -- that variant is
// exactly the UNBOUNDED knapsack. No table survives, so no traceback.
int knapsackValue(int n, int W, const vector<int>& value,
                  const vector<int>& weight) {
    vector<int> K(W + 1, 0);                          // row 0: no items
    for (int i = 1; i <= n; i++)
        for (int w = W; w >= weight[i]; w--)          // BACKWARD: 0/1
            K[w] = max(K[w], value[i] + K[w - weight[i]]);
    return K[W];
}

// Edit-distance value in Theta(n) space: keep TWO rows, the previous and the
// current, swapping after each row. Same fill, same answer -- no traceback.
int editDistanceValue(const string& a, const string& b) {
    int m = a.size(), n = b.size();
    vector<int> prev(n + 1), cur(n + 1);
    for (int j = 0; j <= n; j++) prev[j] = j;         // row 0: insert all of b
    for (int i = 1; i <= m; i++) {
        cur[0] = i;                                   // column 0: delete all
        for (int j = 1; j <= n; j++)
            cur[j] = (a[i-1] == b[j-1])
                       ? prev[j-1]                    // match: free diagonal
                       : 1 + min({prev[j],            // delete
                                  cur[j-1],           // insert
                                  prev[j-1]});        // replace
        swap(prev, cur);                              // this row becomes "previous"
    }
    return prev[n];                                   // the last computed row
}
// 0/1 knapsack value in Theta(W) space: ONE row, updated RIGHT-TO-LEFT so
// K[w - weight[i]] still holds the previous row's value (item i not yet
// used). Iterating forward would let item i be taken again -- that variant
// is exactly the UNBOUNDED knapsack. No table survives, so no traceback.
static int knapsackValue(int n, int W, int[] value, int[] weight) {
    int[] K = new int[W + 1];                     // row 0: no items
    for (int i = 1; i <= n; i++)
        for (int w = W; w >= weight[i]; w--)      // BACKWARD: 0/1
            K[w] = Math.max(K[w], value[i] + K[w - weight[i]]);
    return K[W];
}

// Edit-distance value in Theta(n) space: keep TWO rows, previous and
// current, swapping after each row. Same fill, same answer -- no traceback.
static int editDistanceValue(String a, String b) {
    int m = a.length(), n = b.length();
    int[] prev = new int[n + 1], cur = new int[n + 1];
    for (int j = 0; j <= n; j++) prev[j] = j;     // row 0: insert all of b
    for (int i = 1; i <= m; i++) {
        cur[0] = i;                               // column 0: delete all
        for (int j = 1; j <= n; j++)
            cur[j] = (a.charAt(i-1) == b.charAt(j-1))
                       ? prev[j-1]                          // match
                       : 1 + Math.min(prev[j],              // delete
                             Math.min(cur[j-1],             // insert
                                      prev[j-1]));          // replace
        int[] t = prev; prev = cur; cur = t;      // rows swap roles
    }
    return prev[n];                               // the last computed row
}
# 0/1 knapsack value in Theta(W) space: ONE row, updated RIGHT-TO-LEFT so
# K[w - weight[i]] still holds the previous row's value (item i not yet
# used). Iterating forward instead would let item i be taken again -- that
# variant is exactly the UNBOUNDED knapsack. No table, so no traceback.
def knapsack_value(n, W, value, weight):
    K = [0] * (W + 1)                             # row 0: no items
    for i in range(1, n + 1):
        for w in range(W, weight[i] - 1, -1):     # BACKWARD: 0/1
            K[w] = max(K[w], value[i] + K[w - weight[i]])
    return K[W]

# Edit-distance value in Theta(n) space: keep TWO rows, previous and current,
# swapping after each row. Same fill, same answer -- no traceback.
def edit_distance_value(a, b):
    m, n = len(a), len(b)
    prev = list(range(n + 1))                     # row 0: insert all of b
    for i in range(1, m + 1):
        cur = [i] + [0] * n                       # column 0: delete all
        for j in range(1, n + 1):
            if a[i - 1] == b[j - 1]:
                cur[j] = prev[j - 1]              # match: free diagonal
            else:
                cur[j] = 1 + min(prev[j],         # delete
                                 cur[j - 1],      # insert
                                 prev[j - 1])     # replace
        prev = cur                                # this row becomes "previous"
    return prev[n]                                # the last computed row
// 0/1 knapsack value in Theta(W) space: ONE row, updated RIGHT-TO-LEFT so
// k[w - weight[i]] still holds the previous row's value (item i not yet
// used). Iterating forward would let item i be taken again -- that variant
// is exactly the UNBOUNDED knapsack. No table survives, so no traceback.
fn knapsack_value(n: usize, cap: usize, value: &[i64], weight: &[usize]) -> i64 {
    let mut k = vec![0i64; cap + 1];                  // row 0: no items
    for i in 1..=n {
        for w in (weight[i]..=cap).rev() {            // BACKWARD: 0/1
            k[w] = k[w].max(value[i] + k[w - weight[i]]);
        }
    }
    k[cap]
}

// Edit-distance value in Theta(n) space: keep TWO rows, previous and current,
// swapping after each row. Same fill, same answer -- no traceback.
fn edit_distance_value(a: &[u8], b: &[u8]) -> usize {
    let (m, n) = (a.len(), b.len());
    let mut prev: Vec<usize> = (0..=n).collect();     // row 0: insert all of b
    let mut cur = vec![0usize; n + 1];
    for i in 1..=m {
        cur[0] = i;                                   // column 0: delete all
        for j in 1..=n {
            cur[j] = if a[i - 1] == b[j - 1] {
                prev[j - 1]                           // match: free diagonal
            } else {
                1 + prev[j]                           // delete
                    .min(cur[j - 1])                  // insert
                    .min(prev[j - 1])                 // replace
            };
        }
        std::mem::swap(&mut prev, &mut cur);          // rows swap roles
    }
    prev[n]                                           // the last computed row
}

The loop direction encodes the variant In the one-row knapsack, the backward sweep is not a stylistic choice — it is the 0/1 semantics. Going right-to-left, \(K[w - w_i]\) has not yet been touched this pass, so it still holds the previous item-row’s value: item \(i\) cannot combine with itself. Flip the loop forward and \(K[w - w_i]\) may already include item \(i\) — which is precisely the unbounded knapsack. Smallest possible demonstration: a single item of value \(3\), weight \(2\), capacity \(4\). Backward the row ends \([0, 0, 3, 3, 3]\) — the item once, value \(3\); forward it ends \([0, 0, 3, 3, 6]\) — the item twice, value \(6\). One character of code, a different problem. (A favorite interview probe, for exactly that reason.)

The rolling array destroys the very cells the traceback walks — a genuine functionality trade, not a free optimization. State which side of it an application needs before writing the fill loop. And if recursion feels more natural than loops, Chapter 14’s other implementation style still applies here: memoize the knapsack recurrence top-down — knap(i, w) checking a \((n{+}1)\times(W{+}1)\) cache before recursing on skip and (if it fits) take — and you get the same \(\Theta(nW)\) bound while computing only the cells actually reachable from \((n, W)\), which can be far fewer. Tabulation’s reward is the rolling row; memoization’s is writing exactly the recurrence and no fill-order reasoning. Both are “the DP.”

Enrichment: both at once (Hirschberg) Remarkably, the trade can be beaten for the string DPs. Hirschberg’s algorithm recovers the full LCS or edit solution in linear space by composing DP with divide-and-conquer (Chapter 13’s paradigm): run the two-row fill forward over the top half of the table and backward over the bottom half to find where the optimal path crosses the middle row — linear space suffices for values — then recurse on the two halves the crossing splits off. Total time stays \(\Theta(mn)\) (the half-tables form a geometric series), space stays \(\Theta(n)\), and the full alignment comes out. This is how genome-scale sequence aligners run in memory that the full table could never fit. Not examinable — but a beautiful example of two paradigms composing.

  3. Count the monotone (right/down) paths across a \(3\times 3\) grid when the center cell is blocked. Show the filled table.
    solution Sum-of-neighbors with the blocked cell forced to \(0\): 
       1  1  1
   1  0  1
   1  1  2
    The corner counts \(2\) paths (down the left edge then across, or across the top then down the right edge), versus \(6\) unblocked (§4). Every path through the center — \(6 - 2 = 4\) of them — is erased by that single \(0\), because the zero propagates through every sum that would have used it.
  4. Coins \(\{1, 4, 5\}\), amount \(8\). What does greedy (largest-coin-first) produce? Fill \(\mathit{dp}[0..8]\) and give the optimal count.
    solution Greedy: \(5\), then three \(1\)s — four coins. The DP table: 
      a:      0  1  2  3  4  5  6  7  8
  dp[a]:  0  1  2  3  1  1  2  3  2
    \(\mathit{dp}[8]\) tries last coins \(1\) (\(1 + dp[7] = 4\)), \(4\) (\(1 + dp[4] = 2\)), and \(5\) (\(1 + dp[3] = 4\)): the optimum is \(2\), namely \(4 + 4\). Greedy’s first grab of the \(5\) leaves the amount \(3\), reachable only by ones — the standard non-canonical-coin-system failure, and the DP examines all last coins instead of committing.
  5. A classmate “simplifies” the one-row knapsack of §6 by iterating \(w\) from weight[i] up to \(W\). The code runs and returns plausible numbers. What does it actually compute? Demonstrate on a single item of value \(3\), weight \(2\), with \(W = 4\).
    solution It computes the unbounded knapsack. Sweeping forward, \(K[w - w_i]\) may already have been updated this same pass, so it can already include item \(i\) — the item combines with itself. On the single item: forward, \(K[2] = \max(0, 3 + K[0]) = 3\), then \(K[4] = \max(0, 3 + K[2]) = 6\) — the item taken twice; final row \([0, 0, 3, 3, 6]\). Backward, \(K[4]\) is computed first from the untouched \(K[2] = 0\), giving \(3\); final row \([0, 0, 3, 3, 3]\) — the correct 0/1 answer. The bug is nasty precisely because nothing crashes: the code solves a different problem correctly.
  6. A friend argues: “\(\Theta(nW)\) is a polynomial in two variables, so the knapsack DP is a polynomial-time algorithm, so knapsack cannot be NP-hard.” Locate the error precisely, using the case \(W = 2^{100}\).
    solution “Polynomial time” means polynomial in the input size — the number of bits needed to write the input down. The capacity \(W = 2^{100}\) is written in only \(101\) bits, yet \(\Theta(nW) = \Theta(n \cdot 2^{100})\) table cells would be needed — astronomically exponential in those \(101\) bits. In general a \(b\)-bit capacity permits \(W\) up to \(2^b - 1\), making the DP \(\Theta(n \cdot 2^b)\): each additional input bit doubles the work. That is pseudo-polynomial — polynomial in the value \(W\), exponential in the size \(\log_2 W\) — so there is no contradiction with NP-hardness. (Contrast edit distance: its \(m\) and \(n\) are string lengths, actual input size, so \(\Theta(mn)\) is genuinely polynomial.)
  7. Matrices \(A\ (5{\times}10)\), \(B\ (10{\times}2)\), \(C\ (2{\times}20)\). Compute the scalar-multiplication cost of both parenthesizations of \(ABC\) and verify that the interval-DP recurrence of §4 picks the cheaper one.
    solution \((AB)C\): \(AB\) costs \(5 \cdot 10 \cdot 2 = 100\) and yields \(5{\times}2\); times \(C\) costs \(5 \cdot 2 \cdot 20 = 200\); total \(300\). \(A(BC)\): \(BC\) costs \(10 \cdot 2 \cdot 20 = 400\) and yields \(10{\times}20\); times \(A\) costs \(5 \cdot 10 \cdot 20 = 1000\); total \(1400\). The DP: \(dp[1][2] = 100\), \(dp[2][3] = 400\) (adjacent pairs have one order), and \(dp[1][3] = \min(k{=}1:\; 0 + 400 + 1000,\;\; k{=}2:\; 100 + 0 + 200) = \min(1400, 300) = 300\), split \(k = 2\) — multiply \(AB\) first. A \(4.7\times\) cost gap on three small matrices; on long chains the gap grows unboundedly, which is why the \(\Theta(n^3)\) DP pays.
  8. Compute the LIS length of \([10, 9, 2, 5, 3, 7, 101, 18]\), showing the \(dp\) array. Then explain concretely what breaks if the state is defined as “\(dp[i]\) = LIS length within the first \(i\) elements” instead of “ending at \(i\).”
    solution 
      a[i]:   10   9   2   5   3   7  101  18
  dp[i]:   1   1   1   2   2   3    4   4
    The maximum is \(4\), achieved by \(2, 5, 7, 101\) (or \(2, 3, 7, 18\), or \(2, 5, 7, 18\)). With the prefix state, consider extending by the next element: knowing only “the first \(6\) elements contain an increasing subsequence of length \(3\)” does not say whether that subsequence ends below \(101\) or below \(18\) — the recurrence cannot decide extendability, because the state discarded the ending value. “Ending at \(i\)” carries exactly the needed fact (the last value is \(a[i]\)) at the price of taking a final max over all \(i\). When a recurrence will not close, enrich the state — the §7 checklist’s first line.
