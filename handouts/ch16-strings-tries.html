Chapter 16 — Strings: Tries & Substring Search

CSS 343 — Data Structures, Algorithms & Discrete Mathematics II. Course text, primary reading for Lecture 16.
Keys with exploitable structure: when keys are strings, a trie stores them along paths of characters, so search and insert cost the key length — independent of how many keys are stored — and prefix queries (autocomplete, longest-prefix match) come free. R-way tries, their space cost, and the two fixes (compressed tries and ternary search tries); then the other classic string problem, finding a pattern inside a text: the brute-force baseline, Knuth–Morris–Pratt’s failure function and its never-back-up linear scan, and Boyer–Moore’s skip-ahead heuristics. With worked traces, full implementations, links to the demos, and exercises.

Every symbol table so far has treated its keys as opaque: a binary search tree compares whole keys, a hash table hashes whole keys. This chapter’s keys are strings, and strings have internal structure — characters — that both of those tools ignore. Exploiting it pays twice. A trie stores keys by their character path, so a lookup costs the key’s length \(L\), no matter whether the table holds a hundred keys or a billion, and the structure answers prefix queries (“everything starting with sh”) that a hash table cannot approach. And in substring search — find a pattern inside a text — the pattern’s self-overlap structure lets Knuth–Morris–Pratt (KMP) scan the text once, never re-reading a character, turning the brute-force \(\Theta(nm)\) worst case into a guaranteed \(\Theta(n+m)\). One theme, both halves: strings are not opaque keys, and the algorithms that look inside them win.

1. String keys: what comparing and hashing miss

Store \(n\) string keys, each of length up to \(L\), in the symbol tables of the earlier chapters and count the character work. A balanced binary search tree (BST, Chapter 3) does \(O(\log n)\) key comparisons per search — but each comparison walks the two strings character by character, up to \(L\) steps, so a search really costs \(O(L \log n)\). A hash table (Chapter 7) does better: the polynomial string hash reads every character once, so a lookup is \(\Theta(L)\) — but the price of that speed is that hashing deliberately scatters the keys, so the table is unordered and completely blind to prefixes. Neither tool can answer “list every key starting with sh” without scanning everything. 

  BST:   O(log n) COMPARES, each up to L chars  →  O(L log n)
  hash:  O(1) probes, but hashes the WHOLE key  →  Θ(L), no order, no prefixes

Both treat a string as one atomic value. The trie’s idea is to stop doing that: use the characters themselves as the search path. The result is \(\Theta(L)\) search — matching the hash table’s bound — while keeping the keys ordered and physically grouped by shared prefix, which is exactly the property the prefix operations of §4 exploit. The second half of the chapter (§5 onward) applies the same “look inside the string” principle to a different problem: finding a pattern inside a text.

2. R-way tries

Definition (trie) A trie (from retrieval; pronounced “try”) is a tree that stores string keys along paths of characters from the root: each edge is labeled with one character, and each node carries an is-word flag marking whether some stored key ends there. A key is present exactly when its character path exists and the final node is word-marked. In an R-way trie each node holds an array of \(R\) child pointers, one per character of the alphabet (\(R = 26\) for lowercase letters, \(R = 256\) for bytes).

Keys that share a prefix share the corresponding nodes: storing she and sea creates one s node with two children, not two separate paths. Here is the trie holding \(\{\)she, sea, sells\(\}\):

Search and insert are the same walk

Search follows next[c] for each character of the key in turn. If a needed child is missing, the walk falls off the trie and the key is absent. If the walk survives to the end of the key, the key is present iff the final node is word-marked — the path might exist only as a prefix of a longer key. Insert is the identical walk, except that it creates each missing node along the way and sets the is-word flag on the last one. Neither operation ever compares two keys; the structure has encoded all the comparisons. Following a character is a single array index — t = t->next[c] — so both operations cost \(\Theta(L)\) for a key of length \(L\), independent of \(n\). 

// R-way trie: each node has one child slot per alphabet character plus an
// is-word flag. Operations walk ONE character per step: Theta(L) for a key of
// length L, independent of how many keys are stored.
const int R = 26;                          // alphabet: lowercase a..z

struct Node {
    Node* next[R] = {};                    // one slot per possible character
    bool  isWord = false;
};

// search: follow next[c] for each character. Fall off -> absent; reach the
// end -> present iff the final node is word-marked (a path is NOT a key).
bool search(Node* t, const string& key) {
    for (char ch : key) {
        int c = ch - 'a';
        if (!t->next[c]) return false;     // fell off: no such path
        t = t->next[c];
    }
    return t->isWord;                      // path exists — is it a word?
}

// insert: the same walk, creating only the MISSING nodes; mark the last one.
void insert(Node* t, const string& key) {
    for (char ch : key) {
        int c = ch - 'a';
        if (!t->next[c]) t->next[c] = new Node();   // create the missing suffix
        t = t->next[c];
    }
    t->isWord = true;                      // mark: a key ends here
}

// collect: emit every key stored in the subtree under t (DFS, in order).
void collect(Node* t, string& path, vector<string>& out) {
    if (t->isWord) out.push_back(path);    // a stored key ends here
    for (int c = 0; c < R; c++)
        if (t->next[c]) {
            path.push_back('a' + c);       // extend the path by this character
            collect(t->next[c], path, out);
            path.pop_back();
        }
}

// keysWithPrefix (autocomplete): walk down the prefix, then collect the
// subtree below it. Theta(prefix length + output size).
vector<string> keysWithPrefix(Node* t, const string& prefix) {
    for (char ch : prefix) {               // 1. walk down the prefix path
        if (!t->next[ch - 'a']) return {}; // no key has this prefix
        t = t->next[ch - 'a'];
    }
    vector<string> out;
    string path = prefix;
    collect(t, path, out);                 // 2. gather the whole subtree below
    return out;
}

// R-way trie: each node has one child slot per alphabet character plus an
// is-word flag. Operations walk ONE character per step: O(L) for a key of
// length L, independent of how many keys are stored.
const R: usize = 26;                       // alphabet: lowercase a..z

struct Node {
    next: Vec<Option<Box<Node>>>,          // one slot per possible character
    is_word: bool,
}

fn new_node() -> Box<Node> {
    Box::new(Node { next: (0..R).map(|_| None).collect(), is_word: false })
}

// search: follow next[c] for each character. Fall off -> absent; reach the
// end -> present iff the final node is word-marked (a path is NOT a key).
fn search(mut t: &Node, key: &str) -> bool {
    for ch in key.bytes() {
        match &t.next[(ch - b'a') as usize] {
            None => return false,          // fell off: no such path
            Some(kid) => t = kid,
        }
    }
    t.is_word                              // path exists — is it a word?
}

// insert: the same walk, creating only the MISSING nodes; mark the last one.
fn insert(mut t: &mut Node, key: &str) {
    for ch in key.bytes() {
        let c = (ch - b'a') as usize;
        t = t.next[c].get_or_insert_with(new_node);   // create missing suffix
    }
    t.is_word = true;                      // mark: a key ends here
}

// collect: emit every key stored in the subtree under t (DFS, in order).
fn collect(t: &Node, path: &mut String, out: &mut Vec<String>) {
    if t.is_word {                         // a stored key ends here
        out.push(path.clone());
    }
    for c in 0..R {
        if let Some(kid) = &t.next[c] {
            path.push((b'a' + c as u8) as char);   // extend by this character
            collect(kid, path, out);
            path.pop();
        }
    }
}

// keysWithPrefix (autocomplete): walk down the prefix, then collect the
// subtree below it. O(prefix length + output size).
fn keys_with_prefix(mut t: &Node, prefix: &str) -> Vec<String> {
    for ch in prefix.bytes() {             // 1. walk down the prefix path
        match &t.next[(ch - b'a') as usize] {
            None => return vec![],         // no key has this prefix
            Some(kid) => t = kid,
        }
    }
    let mut out = Vec::new();
    collect(t, &mut prefix.to_string(), &mut out); // 2. gather the subtree
    out
}

Example (a worked search — and the is-word subtlety) Take the trie holding \(\{\)she, shell, shells\(\}\) and search for shell: 
  search "shell":  s → h → e → l → l     (follow 5 characters)
                   final node is word-marked → FOUND
  search "shel":   s → h → e → l           (the path exists…)
                   …but its node is NOT word-marked → not a key
shel is a perfectly good path — it lies on the way to shell and shells — but no stored key ends there, so the flag correctly rejects it. Returning “did I reach the end of the path” instead of isWord is the classic trie-implementation bug. And note the cost: five character steps for shell, whether the trie holds these three keys or three million — adding keys never slows a search, only longer keys do.
Example (a worked insert) Insert sea into a trie holding \(\{\)she, shell\(\}\): 
  s:  exists (shared with she/shell)  → follow it
  e:  's' has no 'e' child            → CREATE a node
  a:  create a node, mark it a word

  "sea" and "she" now share exactly one node: 's'
Insert allocates only the missing suffix of its key — here two nodes — and reuses every shared-prefix node already present. That reuse is what keeps a trie compact on dictionary-like key sets full of common prefixes, and it is why insert, like search, touches at most \(L\) nodes.

The cost, and the catch

operationcost
search / insert\(\Theta(L)\) — the key’s length
search missoften \(< L\) — the walk falls off at the first missing child
spaceup to \(\Theta(R \cdot N \cdot L)\) pointers for \(N\) keys

The headline is the first row: cost bounded by key length, independent of the number of keys. A miss is often cheaper still — searching a trie of English words for xylophone falls off at the first or second character. The catch is the third row: every R-way node reserves \(R\) child pointers whether it uses them or not, and deep in a trie most nodes have one or two children, leaving the other \(24\) (or \(254\)) slots null. For \(R = 256\) the waste dominates the data. Fixing that space cost without giving up the \(\Theta(L)\) walk is the subject of §3.

Deletion, and shared structure Deleting a key takes two steps: unmark its final node (isWord = false), then prune bottom-up — a node with no children that is not itself a word is useless, so remove it and reconsider its parent, stopping at the first node still needed. The care is in the stopping: never remove a node shared by other keys. Deleting shell from \(\{\)shell, shells\(\}\) only clears a flag — every node on its path is still on shells’s path. This shared-structure trap is deletion’s version of the is-word subtlety.

3. Saving space: compressed tries and TSTs

Two standard fixes attack the \(R\)-pointers-per-node cost from different directions: store fewer nodes, or store cheaper nodes.

Compressed tries (radix trees)

Deep in a trie, long runs of single-child nodes are common — the tail of any key that shares nothing beyond its prefix. A compressed trie (also radix tree) collapses every such chain into one edge labeled with the whole substring: 

  {she, shells}:   "sh" ──▶ "e" (word) ──▶ "lls" (word)
  (no separate s, h, e, l, l, s nodes — three nodes total)

Now nodes exist only where keys branch or end, so the node count is bounded by the number of keys rather than the number of characters — far fewer nodes and far fewer pointer hops, with the same trie semantics. Patricia tries apply this idea to binary keys; they are the structure inside IP (Internet Protocol) routing tables and many key–value stores.

Ternary search tries

Definition (ternary search trie) A ternary search trie (TST) replaces each R-way node’s child array with a small binary search tree over only the characters actually used at that position. Each node stores one character \(c\) and three links: left (query character \(< c\)), right (\(> c\)), and mid (\(= c\) — advance to the key’s next character), plus the usual is-word flag. 
  struct Node {
      char c;
      Node *left, *mid, *right;   // < c,   == c (next char),   > c
      bool isWord;
  };

A lookup, at each position of the key, walks left/right links to find the current character among the node’s siblings — a little BST search over the characters used there — and only on an exact character match descends mid and advances the query. Only mid edges consume characters, and the key takes \(L\) of them; each is preceded by an \(O(\log R)\) sibling search. Worst case \(\Theta(L \log R)\), and in practice close to \(\Theta(L)\), because most nodes have very few children. In exchange, a node is three pointers instead of \(R\) — no null-slot waste at all. TSTs are Sedgewick’s default string symbol table: trie-like speed at BST-like space, the practical choice once the alphabet is large.

The trie’s unique powers are the two things hashing destroys by design: order, and prefix locality. If the workload ever asks “what starts with…” or “what is the longest stored prefix of…” — the operations of the next section — the trie is the only tool on the table that answers without a full scan.

4. Prefix operations: autocomplete and longest-prefix

The trie’s structure already groups keys by shared prefix: all keys with prefix \(P\) are exactly the subtree under \(P\)’s node. No extra index, no sorting — the tree is the prefix index. Two operations fall out of that observation for free, and they are the reason to choose a trie at all.

keysWithPrefix — autocomplete

To find every key starting with a prefix: walk the trie down the prefix path, then collect every word-marked node in the subtree below, by depth-first traversal (the keysWithPrefix/collect pair in §2’s code). In a hash table the keys sharing a prefix are scattered across buckets with no way to enumerate them short of scanning every key; in the trie they are literally one subtree.

Example (keysWithPrefix, worked) Trie holding \(\{\)she, shell, shells, sea\(\}\); query prefix she: 
  1. walk s → h → e            (reach the "she" node)
  2. DFS the subtree below it, emitting word-marked nodes:
     she (marked itself), shell, shells
  → { she, shell, shells }     ("sea" branched off at 's' — never visited)
The cost is \(\Theta(\text{prefix length} + \text{output size})\): the walk touches three nodes, and the collection touches only the keys that actually share the prefix. That output-sensitivity is why autocomplete feels instant even over huge dictionaries.

longestPrefixOf

The mirror-image query: the longest stored key that is a prefix of the query. Walk the query string down the trie, remembering the most recent word-marked node passed; when the walk falls off or the query ends, the last remembered word is the answer — a single \(\Theta(L)\) descent.

Example (longestPrefixOf, worked) Keys \(\{\)she, shells\(\}\); query shellsort: 
  s h e     → passed "she", a word     (remember "she")
  l l s     → passed "shells", a word  (remember it — longer)
  o         → no child → stop
  → longestPrefixOf("shellsort") = "shells"
This is exactly IP longest-prefix match: a router stores network prefixes in a (Patricia) trie, walks a destination address bit by bit, and forwards along the most specific — longest — matching prefix it passed. It is also how word segmenters and lexers peel the longest recognized token off the front of the input.

From prefix queries to real autocomplete

Production autocomplete is keysWithPrefix plus a ranking: store a weight (frequency, recency) at each word-marked node, and while collecting the subtree keep only the top few completions by weight — a small heap (Chapter 6) does it in one pass. That is why suggestions arrive curated rather than alphabetical: the trie supplies the candidate set in \(\Theta(\text{prefix} + \text{output})\), and the weights supply the intelligence. The branching structure supports approximate matching too: a wildcard character (c?t) means “try every child at this position,” and a bounded edit-distance search (“did you mean?”) explores paths within an edit budget — the same tree, searched more forgivingly.

5. Substring search: the problem, and brute force

Definition (substring search) Given a text of length \(n\) and a pattern of length \(m\) (usually \(m \ll n\)), find the index of the first occurrence of the pattern as a contiguous substring of the text (or report that none exists).

This is text.indexOf(pattern), grep, find-in-editor, DNA motif search — one of the most-executed operations in computing. The running example for the rest of the chapter: 

  text:     A B A B A B C A B A B A B C A B      (n = 16)
  pattern:      A B A B C                        (m = 5)
  → first match at index 2   (text[2..6] = ABABC)

Brute force: try every alignment

The obvious algorithm tries the pattern at every start position \(i = 0, 1, \dots, n-m\): compare left to right, and on any mismatch slide the pattern by one and restart the comparison. 

// Brute-force substring search: try the pattern at EVERY start position; on a
// mismatch, slide by ONE and restart the comparison. Worst case Theta(nm) —
// text characters get re-examined after a partial match.
int bruteSearch(const string& t, const string& p) {
    int n = t.size(), m = p.size();
    for (int i = 0; i <= n - m; i++) {     // each start position
        int j = 0;
        while (j < m && t[i + j] == p[j]) j++;   // compare left to right
        if (j == m) return i;              // all m matched: found at i
    }
    return -1;                             // no occurrence
}

# Brute-force substring search: try the pattern at EVERY start position; on a
# mismatch, slide by ONE and restart the comparison. Worst case Theta(nm) —
# text characters get re-examined after a partial match.
def brute_search(t, p):
    n, m = len(t), len(p)
    for i in range(n - m + 1):             # each start position
        j = 0
        while j < m and t[i + j] == p[j]:  # compare left to right
            j += 1
        if j == m:                         # all m matched: found at i
            return i
    return -1                              # no occurrence
// Brute-force substring search: try the pattern at EVERY start position; on a
// mismatch, slide by ONE and restart the comparison. Worst case O(nm) —
// text characters get re-examined after a partial match.
fn brute_search(t: &str, p: &str) -> i32 {
    let (t, p) = (t.as_bytes(), p.as_bytes());
    let (n, m) = (t.len(), p.len());
    if m > n { return -1; }
    for i in 0..=(n - m) {                 // each start position
        let mut j = 0;
        while j < m && t[i + j] == p[j] { j += 1; }   // compare left to right
        if j == m { return i as i32; }     // all m matched: found at i
    }
    -1                                     // no occurrence
}

Shift 0 matches AB, then A falls on the text’s C; shifts 1 and 2 die on their first comparison; shift 3 matches all five — found at index \(3\), \(10\) comparisons in all. Note the waste at shift 0 → 1: having matched AB, brute force forgets it and re-reads text[1] from scratch.
Proposition (brute-force cost) Brute force makes at most \((n-m+1)\,m = \Theta(nm)\) character comparisons, and the bound is achieved: on text \(\texttt{AA}\cdots\texttt{A}\) (all \(n\) characters A) with pattern \(\texttt{AA}\cdots\texttt{AB}\) (\(m-1\) As then a B), every one of the \(n-m+1\) alignments matches \(m-1\) characters before failing on the last, costing \(m\) comparisons each.

The instance is worth internalizing with real numbers: text AAAAAAAAAAAB (\(n = 12\)) and pattern AAAB (\(m = 4\)) drive brute force through all \(9\) alignments at \(4\) comparisons each — \(36\) comparisons, with the match finally at index \(8\). The deep inefficiency is not the sliding but the re-examining: after a partial match, brute force throws away everything it learned about the text and reads the same characters again. Knuth–Morris–Pratt’s insight is that a partial match already tells you exactly how far the pattern can safely jump.

When brute force is the right call Do not over-engineer. On typical natural-language text, a mismatch occurs within the first character or two of most alignments, so brute force runs nearly linearly — which is why simple library find routines are essentially brute force (or Boyer–Moore) and are fine in practice. Reach for the fancier algorithms when the input is long and repetitive (DNA, binary data), adversarial, streamed, or searched so often that pattern preprocessing amortizes. Match the tool to the workload.

6. Knuth–Morris–Pratt: never back up

The Knuth–Morris–Pratt (KMP) algorithm eliminates the re-reading. The key observation: when a mismatch occurs after matching \(j\) characters, those last \(j\) text characters are not unknown — they equal the pattern’s first \(j\) characters, which we know perfectly well. So instead of rewinding the text, slide the pattern forward to the next alignment consistent with what was just matched, and resume. The text pointer \(i\) never decreases; on a mismatch only the pattern pointer \(j\) falls back.

Example (the key insight) 
  text:     A B A B A B C …
  pattern:  A B A B C
                    ↑ mismatch at pattern[4]
We matched ABAB and failed at pattern[4]. Brute force would slide by one and re-read from text[1]. But look at what was matched: AB is a prefix of the pattern that is also a suffix of the matched ABAB. Slide the pattern so that prefix re-aligns under the text’s last two matched characters, resume comparing at pattern[2] — and never move \(i\) back. The “how much of my partial match survives” number is what KMP precomputes, for every mismatch position, before ever touching the text.
Definition (failure function) For a pattern \(P\) of length \(m\), the failure function is the array \[ \mathit{fail}[j] = \text{length of the longest }\textit{proper}\text{ prefix of } P[0..j] \text{ that is also a suffix of } P[0..j], \] for \(j = 0, \dots, m-1\), where \(P[0..j]\) means the first \(j+1\) characters (inclusive). “Proper” excludes the whole string, so \(\mathit{fail}[0] = 0\) always. Reading: if my partial match ends at \(P[j]\) and then breaks, \(\mathit{fail}[j]\) matched characters survive. After matching \(j\) characters, a mismatch sends the pattern pointer to \(j = \mathit{fail}[j-1]\) — and the text pointer stays put.
Example (failure function, worked) Pattern AABAA:
j01234
P[j]AABAA
fail[j]01012
Walk it: \(\mathit{fail}[1] = 1\) (prefix A = suffix A of AA); \(\mathit{fail}[2] = 0\) (AAB ends in B, no prefix survives); \(\mathit{fail}[3] = 1\) (AABA ends with the prefix A); \(\mathit{fail}[4] = 2\) (AABAA ends with the prefix AA). So a mismatch after matching all of AABAA falls back to \(j = \mathit{fail}[4] = 2\): the AA the text just supplied re-serves as the pattern’s first two characters.
Example (a second table — the exam classic) Pattern ABABAC:
j012345
P[j]ABABAC
fail[j]001230
The ABAB self-overlap grows the entries \(1, 2, 3\) (prefixes A, AB, ABA re-appear as suffixes), and the final C matches no prefix, resetting to \(0\). A mismatch at \(j = 5\) — after matching ABABA — falls back to \(j = \mathit{fail}[4] = 3\): the ABA prefix re-aligns under the text’s last three matched characters. Fill in one of these tables by hand; it is the standard exam question on KMP.

Building the table, and the search

The failure function is computed by matching the pattern against itself: a running index \(k\) tracks the current longest prefix-suffix, growing by one when the next character extends it and falling back through the (partially built) table when it does not — the same never-back-up logic, applied to the pattern. The build is \(\Theta(m)\); then the scan reads each text character with the same extend-or-fall-back step. Both loops below are four lines long, and getting the build loop right is the crux of implementing KMP — trace it once on AABAA. 

// Failure function: fail[j] = length of the longest proper prefix of the
// pattern's first j+1 characters that is also a suffix of them. Built by
// matching the pattern against ITSELF — k falls back through the very table
// under construction. Theta(m).
vector<int> failureFn(const string& p) {
    int m = p.size();
    vector<int> fail(m, 0);
    int k = 0;                             // length of the current prefix-suffix
    for (int j = 1; j < m; j++) {
        while (k > 0 && p[j] != p[k]) k = fail[k - 1];  // shrink until it extends
        if (p[j] == p[k]) k++;             // extend the prefix-suffix
        fail[j] = k;
    }
    return fail;
}

// KMP search: the text pointer i NEVER decreases — on a mismatch only the
// pattern pointer j falls back, via the table. Theta(n) scan after the
// Theta(m) build: Theta(n + m) total, guaranteed.
int kmpSearch(const string& t, const string& p) {
    vector<int> fail = failureFn(p);
    int j = 0;                             // characters matched so far
    for (int i = 0; i < (int)t.size(); i++) {
        while (j > 0 && t[i] != p[j]) j = fail[j - 1];  // fall back in the pattern
        if (t[i] == p[j]) j++;             // extend the partial match
        if (j == (int)p.size()) return i - j + 1;       // full match: report start
    }
    return -1;                             // no occurrence
}

// Failure function: fail[j] = length of the longest proper prefix of the
// pattern's first j+1 characters that is also a suffix of them. Built by
// matching the pattern against ITSELF — k falls back through the very table
// under construction. O(m).
fn failure_fn(p: &str) -> Vec<usize> {
    let p = p.as_bytes();
    let mut fail = vec![0usize; p.len()];
    let mut k = 0;                         // length of the current prefix-suffix
    for j in 1..p.len() {
        while k > 0 && p[j] != p[k] { k = fail[k - 1]; }  // shrink until extends
        if p[j] == p[k] { k += 1; }        // extend the prefix-suffix
        fail[j] = k;
    }
    fail
}

// KMP search: the text pointer i NEVER decreases — on a mismatch only the
// pattern pointer j falls back, via the table. O(n) scan after the O(m)
// build: O(n + m) total, guaranteed.
fn kmp_search(t: &str, p: &str) -> i32 {
    let fail = failure_fn(p);
    let (t, p) = (t.as_bytes(), p.as_bytes());
    let mut j = 0;                         // characters matched so far
    for i in 0..t.len() {
        while j > 0 && t[i] != p[j] { j = fail[j - 1]; }  // fall back in pattern
        if t[i] == p[j] { j += 1; }        // extend the partial match
        if j == p.len() { return (i + 1 - j) as i32; }    // full match: start
    }
    -1                                     // no occurrence
}

Proposition (KMP cost) KMP makes at most \(2n\) character comparisons in the scan, after a \(\Theta(m)\) preprocessing pass: \(\Theta(n + m)\) total, guaranteed, on every input.
Proof. Bound the scan’s comparisons by watching the quantity \(i - j\) (the pattern’s current alignment) and \(i\) itself. Each comparison ends in one of two ways. A match advances \(i\) by one — and \(i\) only ever increases, so at most \(n\) comparisons are matches. A mismatch with \(j > 0\) drops \(j\) to \(\mathit{fail}[j-1] < j\) while \(i\) stays, so the alignment \(i - j\) strictly increases; a mismatch at \(j = 0\) advances \(i\), which also increases \(i - j\). The alignment starts at \(0\), never decreases, and cannot exceed \(n\), so at most \(n\) comparisons are mismatches. Total: at most \(2n\) comparisons, and the text pointer never moved backward. The same argument applied to the build loop (pattern against itself) gives \(2m\). ∎

Ten comparisons, one fall-back, and the payoff visible in row 6: after matching AABAA, the C-versus-B mismatch sets \(j = \mathit{fail}[4] = 2\) — the matched suffix AA re-serves as the pattern’s prefix — and \(i\) stays at \(5\). text[0..4] is never read again. (Brute force on this same pair tries four alignments and makes \(15\) comparisons — its six extra are all re-examinations of text characters KMP read once.)
Sedgewick’s DFA view Sedgewick presents the same precomputation as a full deterministic finite automaton (DFA) over the pattern: a 2-D table \(\mathit{dfa}[c][j]\) giving, for every alphabet character \(c\) and pattern state \(j\), the next state directly. The scan then has no fall-back loop at all — one array lookup per text character, \(j = \mathit{dfa}[t[i]][j]\). The build costs \(\Theta(Rm)\) (one entry per character–state pair) instead of \(\Theta(m)\), buying a perfectly branch-free \(\Theta(n)\) scan. It is the identical algorithm with the fall-backs tabulated in advance — and it is this chapter’s bridge to Chapter 17, where the “pattern” itself generalizes from one string to a whole language and the automaton becomes the star.
First match, all matches The code above returns the first occurrence. Find-and-replace-all, grep -c, and hit-highlighting need every occurrence, and the change is one line: on a full match, instead of returning, record the index, set \(j = \mathit{fail}[m-1]\), and keep scanning. That fall-back (rather than \(j = 0\)) correctly finds overlapping occurrences — the two AAs inside AAA — and since \(i\) still never backs up, reporting all matches is still \(\Theta(n + m)\).
Interactive Substring-search race. Brute force and KMP run the same text/pattern pair in lockstep, one comparison per tick, with live compare counters. On the default pair — this chapter’s running example ABABABCABABABCAB / ABABC — both find the match at index \(2\), but watch the pointers: brute force’s text index jumps backward after each partial match (\(11\) compares), while KMP’s only ever advances (\(8\) compares, one fall-back \(j: 4 \to 2\), exactly the ABAB insight above). Then type the adversarial pair A^11B A^3B (repeat shorthand for AAAAAAAAAAAB AAAB): brute force pays \(36\) comparisons to KMP’s \(20\), and KMP finishes and freezes while brute force is still re-scanning. The failure table rides along in the KMP panel’s status line.

Scaled up, the gap is not academic: on an all-As text with \(n = 10^6\) and \(m = 1000\), brute force makes about \(n \cdot m = 10^9\) comparisons and KMP about \(n + m \approx 10^6\) — a thousandfold difference on a megabyte of data. And because the text pointer never retreats, KMP reads its input strictly left to right, once: it runs happily on a stream it cannot rewind — network traffic, a pipe, a file too big to buffer — which no algorithm that backs up or looks ahead can do.

7. Boyer–Moore and the wider family

KMP guarantees linearity; Boyer–Moore goes after raw speed, and on typical text it examines fewer than \(n\) characters — sublinear — which is why it (or a variant) sits inside grep and most editors’ find. This section is a survey — the idea and one trace; this course’s implementations are brute force and KMP.

Two reversals produce the speed. Compare each alignment right to left; then, on a mismatch, use the mismatched text character — the bad character — to decide the jump. If that character does not occur in the pattern at all, no alignment overlapping it can possibly match, so the pattern leaps past it entirely: up to \(m\) positions skipped for one comparison. If it does occur, slide the pattern so its last occurrence lines up under the text character (never sliding backward — the shift is at least 1).

Example (Boyer–Moore, worked) Search for SEARCH (\(m = 6\)) in STRING SEARCH (\(n = 13\)). Cells show the pattern’s placement; only green (compared, matched) and amber (compared, mismatched) cells were actually examined — plain cells were skipped without a look:
text →STRING␣SEARCHcompares
align 0 SEARCH 1
align 6 SEARCH1
align 7 SEARCH6
Alignment 0: the rightmost comparison pits pattern[5] = H against text[5] = G — and G occurs nowhere in SEARCH, so the pattern jumps past it: shift by \(6\), one comparison for six positions. Alignment 6: H against text[11] = C — C does occur in the pattern (last occurrence at index \(4\)), so slide until they line up: shift by \(5 - 4 = 1\). Alignment 7: the right-to-left comparison matches all six — found at index \(7\). Total: \(8\) comparisons to search a \(13\)-character text; six of the text’s characters were never examined at all. That is the sublinear effect, and it grows with \(m\): a longer pattern skips further.
The full algorithm: two rules Industrial Boyer–Moore precomputes two shift tables from the pattern and jumps by whichever is larger. The bad-character rule is the one traced above (as a lone heuristic it is the Boyer–Moore–Horspool variant, a popular simplification). The good-suffix rule is KMP’s idea pointed the other way: the alignment just matched a suffix of the pattern before failing, so slide to the next place that suffix (or a prefix of the pattern matching its tail) re-occurs. Combined, they make Boyer–Moore the practical champion on natural text; its plain worst case is \(\Theta(nm)\) (e.g. all-A texts), with known fixes that restore a linear guarantee at some constant-factor cost.

Rabin–Karp: hashing the windows

One more classic completes the family, and it is Chapter 7 applied to substrings. Rabin–Karp hashes the pattern once, then slides a length-\(m\) window across the text keeping a rolling hash: the polynomial string hash can be updated in \(O(1)\) per shift (subtract the leaving character’s term, multiply, add the entering character), so hashing all \(n-m+1\) windows costs \(\Theta(n)\), not \(\Theta(nm)\). When a window’s hash equals the pattern’s, verify character by character — equal hashes do not prove equal strings (collisions, Chapter 7) — giving \(\Theta(n)\) expected time. Its superpower is many patterns at once: hash all of them into a set and test each window’s hash against the set in the same single pass — the engine inside plagiarism detectors and signature scanners.

The common thread: every fast searcher preprocesses the pattern — failure function, skip tables, hash — and then reads the huge text once (or less). The asymmetry is the point: \(m\) is small and the pattern is reused, so work invested there is amortized across the \(n\)-character scan; it is the same precompute-to-answer-fast principle as a hash table’s function or a dynamic-programming table, applied to search.

9. Exercises (with solutions)

  1. Insert to, tea, ten into an empty R-way trie. How many nodes does it have (besides the root)? Which are word-marked? Is a search for te successful?
    solution Five nodes: t with children o and e, and e with children a and n (paths t–o, t–e–a, t–e–n). The three word-marked nodes are the ends o, a, n; t and e are shared, unmarked path nodes. Searching te walks a perfectly valid path — but its final node is not word-marked, so the search correctly fails: a reachable path is not a stored key.
  2. A trie holds \(\{\)to, tea, ted, ten, in, inn\(\}\). Compute keysWithPrefix("te") and longestPrefixOf("innards"), and state each operation’s cost in general.
    solution keysWithPrefix("te"): walk t–e, then collect the subtree: \(\{\)tea, ted, ten\(\}\) (neither to nor the in branch is visited). Cost: \(\Theta(\text{prefix length} + \text{output size})\). longestPrefixOf("innards"): walk i → n (passing the word in, remember it) → n (passing inn, remember it — longer) → a has no child → stop and return the last word passed: inn. Cost: one descent, \(\Theta(\text{query length})\) at most.
  3. From a trie holding \(\{\)she, shell, shells\(\}\), delete shell, then delete shells. Describe exactly what each deletion changes.
    solution Deleting shell only clears the is-word flag on its final l node: that node still has a child (the s of shells), so nothing can be pruned — every node on the path is still shared. Deleting shells then unmarks its final s node and prunes bottom-up: the s (childless, unmarked) goes, then the second l (now childless, and its mark was cleared by the first deletion), then the first l (childless, unmarked) — three nodes removed — and pruning stops at e, which is word-marked (she survives). Forgetting the stop condition and pruning into shared/marked nodes is the classic deletion bug.
  4. Compute the failure function of AAAA and of ABCABD.
    solution AAAA: \([0, 1, 2, 3]\) — each added A extends the self-overlap by one (the longest proper prefix of \(j+1\) As that is also a suffix is \(j\) As). ABCABD: \([0, 0, 0, 1, 2, 0]\) — B and C match no prefix; then A, B rebuild the overlap to \(1, 2\) (AB is both prefix and suffix of ABCAB); the final D fails against \(P[2] = \texttt{C}\), falls back to \(\mathit{fail}[1] = 0\), fails against \(P[0] = \texttt{A}\), and settles at \(0\).
  5. KMP is searching for ABABC (\(\mathit{fail} = [0, 0, 1, 2, 0]\)) in ABABABCABABABCAB. The first four comparisons match ABAB; comparison five finds \(\texttt{text}[4] = \texttt{A} \ne \texttt{C} = P[4]\). What exactly happens next, and why is it safe not to move \(i\)?
    solution The mismatch after \(j = 4\) matched characters sets \(j = \mathit{fail}[3] = 2\) while \(i\) stays at \(4\): the next comparison is text[4] = A against \(P[2] = \texttt{A}\) — a match — and the scan proceeds to find the full match at index \(2\) after \(8\) total comparisons. It is safe because the matched text text[0..3] = ABAB is known: its suffix AB equals the pattern’s prefix AB, so the only alignment worth trying next is the one where that prefix re-uses the last two matched text characters — everything the fall-back skipped provably cannot match. The text characters before \(i = 4\) never need re-reading.
  6. Exactly how many character comparisons does brute force make searching for \(\texttt{A}^{m-1}\texttt{B}\) (that is, \(m-1\) As then B) in the text \(\texttt{A}^{n-1}\texttt{B}\)? Evaluate at \(n = 12\), \(m = 4\), and compare with KMP’s guarantee.
    solution Every alignment \(i = 0, \dots, n-m\) compares \(m-1\) matching As and then one more character (a mismatching A under B at every alignment except the last, where the B matches and the search ends): \(m\) comparisons per alignment, \((n-m+1)\,m\) in all. At \(n = 12, m = 4\): \(9 \times 4 = 36\) comparisons, matching at index \(8\) — the demo’s adversarial pair. KMP’s scan is bounded by \(2n\) (each comparison either advances \(i\) or strictly increases the alignment \(i - j\)); on this instance it makes \(20\).
  7. Boyer–Moore is matching SEARCH right to left, and the very first comparison of an alignment mismatches at \(P[5] = \texttt{H}\). Using the bad-character rule, give the shift when the mismatched text character is (a) X, (b) S, (c) R.
    solution The rule slides the pattern so its last occurrence of the text character lands under it: shift \(= j - \mathit{last}[c]\) with \(\mathit{last}[c] = -1\) for an absent character (taking at least \(1\)). Last occurrences in SEARCH: S:0, E:1, A:2, R:3, C:4, H:5. (a) X is absent: \(5 - (-1) = 6\) — leap the whole pattern past it. (b) S: \(5 - 0 = 5\). (c) R: \(5 - 3 = 2\). One comparison can thus discard up to \(m\) alignments — the sublinear engine.
  8. A firewall must scan an unbounded, non-seekable network stream for a fixed 64-byte signature, with a hard guarantee that scanning keeps up with line rate. Which algorithm fits, and why do the other three family members fail at least one requirement?
    solution KMP. Its text pointer never decreases, so it reads each streamed byte exactly once with \(O(1)\) work amortized (at most \(2n\) comparisons total) and \(\Theta(m)\) memory for the failure function — both requirements met. Brute force re-examines earlier text characters after a partial match (needs buffering and loses the linear guarantee — \(\Theta(nm)\) on repetitive traffic). Boyer–Moore compares right to left inside a sliding window, so it needs an \(m\)-byte lookahead buffer (workable), but its bad-character heuristic alone has a \(\Theta(nm)\) worst case — an adversary can craft traffic that breaks the line-rate guarantee. Rabin–Karp streams fine, but its \(\Theta(n)\) is expected: hash-collision verifications re-read window bytes, so the hard guarantee again fails against adversarial input (though it becomes the right answer the moment there are many signatures at once).
