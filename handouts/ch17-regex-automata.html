Chapter 17 — Regular Expressions & Automata

CSS 343 — Data Structures, Algorithms & Discrete Mathematics II. Course text, primary reading for Lecture 17.
A regular expression is a pattern that describes a language — a whole set of strings — using concatenation, | (or), * (closure), and parentheses. A finite automaton is a machine that recognizes such a language, and the two are equivalent: Thompson’s construction turns any regex into a nondeterministic finite automaton (NFA) with \(\varepsilon\)-transitions in linear time, and tracking the set of reachable states runs that automaton in guaranteed \(O(mn)\) time. Kleene’s theorem says the correspondence is exact. Companion reading: Sedgewick & Wayne §5.4. With worked machines, verified simulation traces, links to the demos, and exercises.

Chapter 16 searched a text for one fixed pattern. This chapter generalizes the pattern itself: a regular expression describes not one string but a whole set of strings — all binary numerals, all valid phone numbers, all strings ending in B — and the matching question becomes “is this string in the set?” The beautiful result is that every such pattern can be compiled into a machine: a finite automaton that reads the string once and answers yes or no. We build that machine (Thompson’s construction), run it (by tracking a set of states — graph reachability from Chapter 8, reused), and close with Kleene’s theorem, which says the pattern and the machine have exactly the same expressive power. Pattern in, machine out — theory meets practice.

1. Regular expressions and the languages they describe

Definition (regular expression; language) A regular expression (regex) over an alphabet is a pattern built from single characters and four operations: concatenation (AB — an A then a B), or (A|B — an A or a B), closure (A* — zero or more A’s), and grouping ((…)). A regex denotes a language \(L\) — the set of all strings it matches — and a string matches the regex if and only if it is in that language.

You use regexes daily — grep, editor find, input validation — and the four operations generate everything: 

  AB       matches exactly "AB"
  A|B      matches "A" or "B"
  A*       matches "", "A", "AA", "AAA", …
  (AB)*    matches "", "AB", "ABAB", …

Thinking in terms of the set of accepted strings — not the character-by-character syntax — is what connects regexes to automata, which are membership testers for these sets: 

  L(A*B)     = { B, AB, AAB, AAAB, … }    (any run of A's, then a B)
  L((A|B)C)  = { AC, BC }
  L((0|1)*)  = every binary string (any length, including "")
  L(A(B|C)*D) = A, then any mix of B's and C's, then D

The reading skill is to decompose at the outermost operator first — break the regex at its top-level | and *, then recurse into the groups. That recursive structure is exactly what the machine construction of §3 will follow.

Example (does it match?) Which of "ABAB", "ABA", and "" does (AB)* match? "ABAB" is two copies of AB — match. "ABA" has a dangling A that no whole number of copies covers — no match. And the empty string matches, because * allows zero copies — the case students forget. Read * as “any number, including none.”
Example (write the regex) Over the alphabet \(\{\texttt{A},\texttt{B}\}\), describe all strings that end in B. Answer: (A|B)*B — anything, then one final B. Check it against boundary strings: "B" matches (the star contributes zero characters), "AB" and "AABB" match; "BA" fails (ends in A) and "" fails (there must be at least the final B). The common wrong answer is A*B — it describes “A’s then a B,” which rejects "BAB" even though that string does end in B. Testing a proposed pattern against the empty string, single characters, and one deliberately tricky case is exactly how you should test real-world regexes too.
Precedence — the #1 source of regex bugs Binding tightest to loosest: *, then concatenation, then |. So AB|C = (AB)|C, A|BC = A|(BC), and AB* = A(B*) — which is not (AB)*: the first is one A followed by any number of B’s, the second any number of AB pairs. When in doubt, parenthesize. This is the same operator-precedence discipline as arithmetic expressions, and the machine builder in §3 will use a stack to respect it, exactly as an expression evaluator does.
Everything else is sugar The rest of practical regex syntax is shorthand over the four operations: A+ = AA* (one or more), A? = (A|ε) (optional), the character class [abc] = (a|b|c), . = any single character, and named classes like \d (digit). A realistic pattern decodes accordingly: the phone validator \(?\d{3}\)?[- ]?\d{3}-?\d{4} matches (206) 555-1234, 206-555-1234, and 2065551234 — digit runs (\d{3} is a repeated class), optional literal parentheses (\(?), an optional space-or-dash ([- ]?). Because it is all sugar, machinery for the four core operations handles every regex; it also illustrates the readability tax — regexes get cryptic fast, which is why understanding the underlying automaton (and testing thoroughly) matters.

Regular languages — and what a regex cannot say

Definition (regular language) The languages describable by regular expressions are the regular languages — a fundamental class in the theory of computation. Regular: \(L(\texttt{A*B})\), \(L(\texttt{(AB)*})\), “an even number of A’s.” Not regular: balanced parentheses; \(\texttt{A}^n\texttt{B}^n\) (\(n\) A’s followed by exactly \(n\) B’s).

Why the ceiling? To match \(\texttt{A}^n\texttt{B}^n\) you must count \(n\), and (as §2 makes precise) a finite automaton has finite memory — no fixed number of states can count to arbitrary \(n\). The pumping lemma turns this intuition into a proof. Anything with unbounded counting or nesting — balanced brackets, HTML tags, arithmetic expressions — is inherently beyond regex and needs more machinery (a stack, a grammar; §5 places these on the ladder). The famous advice “don’t parse HTML with a regex” is this theorem in disguise. Knowing the limits of regex is as important as knowing its power.

2. Finite automata: machines that recognize languages

Definition (finite automaton) A finite automaton is a machine with a fixed set of states, a start state, one or more accept states, and transitions that say “in state \(s\), reading character \(c\), go to state \(t\).” It reads a string character by character, following transitions; if it ends in an accept state, it accepts the string, otherwise it rejects. The set of strings it accepts is the language it recognizes.

This is the simplest computer — and it is the machine counterpart to a regex: one describes a language, the other decides membership in it. A concrete example first, in the easy deterministic flavor.

Example (a DFA for (0|1)*0) A deterministic finite automaton (DFA) has exactly one transition per (state, character) pair. This two-state DFA recognizes binary strings ending in 0 — the same “ends in…” language you wrote a regex for in §1, one alphabet over:

Run "110": \(A \xrightarrow{1} A \xrightarrow{1} A \xrightarrow{0} B\) — accept. Run "0110": \(A \xrightarrow{0} B \xrightarrow{1} A \xrightarrow{1} A \xrightarrow{0} B\) — accept. Run "111": \(A \to A \to A \to A\) — reject. And the empty string never leaves \(A\) — reject (it does not end in 0). Each step follows exactly one arrow, so a DFA processes an \(n\)-character string in \(O(n)\) — and the invariant “state \(B\) \(\iff\) the input so far ends in 0” is the correctness argument, a loop invariant like every other one this course. It is also the finite-memory point of §1 made concrete: two states can remember “ends in 0 or not,” but no fixed number of states can remember an unbounded count.

Nondeterminism and \(\varepsilon\)-transitions

Definition (NFA; \(\varepsilon\)-transition) A nondeterministic finite automaton (NFA) may have several transitions — or none — for a (state, character) pair, plus \(\varepsilon\)-transitions: moves between states that consume no input (“free” moves, after \(\varepsilon\), the empty string). An NFA accepts a string if some sequence of choices consumes the whole string and ends in an accept state.

“Nondeterministic” does not mean random. Picture the machine cloning itself to pursue every option in parallel: at any moment it occupies a set of states, not one, and it accepts if any clone ends accepting. That picture is not just intuition — §4 turns “track the set of states” into the actual, deterministic algorithm that runs an NFA on a real computer. The \(\varepsilon\)-transitions, meanwhile, are the construction’s superpower: they let you glue small machines together — offer a choice (|), loop back (*) — without reading a character.

Why tolerate the extra complexity? Because each flavor is easy where the other is hard:

NFADFA
build from a length-\(m\) regexeasy, \(O(m)\) states (§3)can need \(2^m\) states
run on a length-\(n\) text\(O(mn)\) (track a set, §4)\(O(n)\) (follow one arrow)
\(\varepsilon\)-transitionsyesno

Build small or run fast — the classic space/time trade. Sedgewick’s route, and this chapter’s: build the (small) NFA, then simulate it. §6 returns to when you would instead pay to determinize. Either way, the bridge that makes everything cohere: the automaton is a membership tester for the regex’s language — 

  the NFA ACCEPTS a string   ⇔   the string is IN the regex's language
  "does it match?"  =  "does the NFA accept?"

3. Thompson’s construction: from regex to NFA

Thompson’s construction turns any regex into an NFA compositionally: each of the four operations has a fixed little machine — a gadget — with a single entry and a single exit, and gadgets plug together with \(\varepsilon\)-transitions, following the regex’s recursive structure. Because every piece has one way in and one way out, they compose without special cases — the same discipline as a recursive-descent parser or an expression-tree evaluation.

The atom: a single character

Concatenation AB: a series connection

Or A|B: a parallel connection

Closure A*: skip it, or loop it

Four operators, four wirings, each preserving the single-entry/single-exit invariant — so they nest freely, and building an NFA from a regex is just applying them along the parse. Note the skip edge in the closure gadget: it is the machine-level face of “(AB)* matches the empty string” from §1.

Sedgewick’s encoding: one state per character

The implementation insight that fits the whole construction in a page: don’t allocate gadget states — reuse the regex string itself.

Encoding invariant For a regex re of length \(m\), the NFA has states \(0, 1, \dots, m\), where state \(m\) is the accept state and state \(i\) “is” the character re[i]:
  • a letter state consumes its own character, stepping \(i \xrightarrow{\texttt{re}[i]} i+1\) — and the only way to consume input is \(i \to i+1\);
  • a metacharacter state ((, ), |, *) never consumes input — it has only \(\varepsilon\)-edges.
The whole machine is the regex string plus an \(\varepsilon\)-digraph — nothing else.
Example (build A*B, the chapter’s main machine) 
  re:  A   *   B         states 0 1 2 3   (3 = accept)

  i=0 'A', next is '*' →  ε 0→1  (skip the A)
                          ε 1→0  (repeat the A)
                          match edge 0 —A→ 1
  i=1 '*'              →  ε 1→2  (fall through the '*')
  i=2 'B'              →  match edge 2 —B→ 3

  ε-edges: 0→1, 1→0, 1→2      match edges: 0—A→1, 2—B→3
The star wires two \(\varepsilon\)-edges around its operand — \(0 \to 1\) skips the A (zero copies), \(1 \to 0\) loops back (more copies) — and the * state falls through to the B. Three \(\varepsilon\)-edges, two match edges, four states: Thompson’s gadgets, compressed onto the string. This exact machine (same state numbers, same edges) is the one the in-class activity (ICA 17) hands you.

Where do the \(\varepsilon\)-edges go? A stack, and an outer wrap

Each operator needs to know its operands, which requires parsing — and Sedgewick does it in one left-to-right scan with a stack, exactly like evaluating an arithmetic expression: 

  '(' or '|'  →  push its index
  ')'         →  pop; if a '|' comes off, wire the alternation:
                   ε from the '(' to just after the '|'   (enter branch 2)
                   ε from the '|' to the ')'             (skip to the join)
  '*' after an atom or group starting at lp:
                   ε lp→star  (skip)   and   ε star→lp  (repeat)
  '(', '*', ')'  →  ε i→i+1  (fall through the metacharacter)
  first of all:    wrap the regex in implicit parens — "A|B" → "(A|B)"

The outer wrap is a real implementation detail worth understanding: a top-level | or * needs an enclosing group to anchor its \(\varepsilon\)-edges — without it, A|B has no ( to pop when the ) logic fires. So the machine actually built for A*B is the machine for (A*B): six states instead of four, the letters shifted up by one, and structural \(\varepsilon\)-edges \(0\to1\) and \(4\to5\) at the ends. The demos (and the code below) build the wrapped machine; the worked figures above drop the wrap for clarity, and every state number simply shifts by one.

Proposition (the construction is linear) A regex of (wrapped) length \(m\) yields an NFA with \(m+1\) states and \(O(m)\) edges, built in \(\Theta(m)\) time.
Proof (count edges per character). Each letter contributes exactly one match edge. Each (, ), and * contributes one fall-through \(\varepsilon\)-edge; each * adds its skip/repeat pair; each | adds its branch-in/skip-out pair. So every character contributes at most one match edge and at most three \(\varepsilon\)-edges, and the single left-to-right scan does \(O(1)\) stack work per character. Check against the machine above — ((A*B|AC)D): \(9\) \(\varepsilon\)-edges \(= 5\) fall-throughs (from ( ( * ) )) \(+\ 2\) for the star \(+\ 2\) for the |, plus \(5\) match edges from the \(5\) letters — exactly the tally the demo reports. Contrast \(2^m\): that is what determinizing can cost (§5); the NFA itself never blows up. ∎

Type a regex over letters and ( ) | * (up to 12 characters, balanced parens) and press Build: the construction replays edge by edge — states laid out one per character, then each \(\varepsilon\)-edge with the rule that added it, then the match edges. 4. Simulating the NFA: track a set of states

An NFA is easy to build and awkward to run — it makes choices. The resolution is the idea from §2: instead of guessing which branch to take, follow all branches at once by maintaining the set of states the machine could possibly be in. That turns nondeterminism into bookkeeping: 

  states = ε-closure({ start })
  for each character c of the text:
      next   = { i+1 : i ∈ states and re[i] == c }     // advance
      states = ε-closure(next)                          // close
  accept  iff  the accept state ∈ states

Two phases per character — advance every letter state whose character matches (its one \(i \to i+1\) edge; the encoding invariant doing its job), then \(\varepsilon\)-close to add the free moves. No backtracking, no exponential guessing.

Definition (\(\varepsilon\)-closure) The \(\varepsilon\)-closure of a set \(S\) of states is \(S\) together with every state reachable from \(S\) by following \(\varepsilon\)-edges — a graph-reachability computation: a depth-first or breadth-first search over the \(\varepsilon\)-digraph, seeded with \(S\). This is the Chapter 8 graph search, reused on the automaton.
Example (\(\varepsilon\)-closure, worked) On the A*B machine (\(\varepsilon\)-edges \(0\to1\), \(1\to0\), \(1\to2\)), compute \(\varepsilon\)-closure(\(\{0\}\)): 
  expand 0:  0→1                 add 1
  expand 1:  1→0 (already seen),  1→2   add 2
  expand 2:  no ε-edges
  → {0, 1, 2}
Read the result: before consuming anything, the machine is simultaneously ready to match an A (state 0) and to match the B directly (state 2) — the skip edge has already opened both the zero-A’s and the some-A’s futures. This exact set is ICA 17’s first test.

Here is the entire engine — construction, closure, and the set-of-states loop. It is a graph search per character, built from pieces you already own: 

// Regex -> NFA (Sedgewick's one-state-per-character encoding) + set-of-states
// simulation. Syntax: letters, concatenation, | (one per group), *, ( ).
struct NFA {
    string re;                    // wrapped regex: "(" + pattern + ")"
    int m;                        // re.length(); state m is the accept state
    vector<vector<int>> eps;      // eps[u] = the e-edge out-neighbors of u

    NFA(const string& pattern) {
        re = "(" + pattern + ")"; // implicit wrap anchors a top-level | or *
        m = re.size();
        eps.assign(m + 1, {});
        vector<int> ops;          // stack of '(' and '|' positions
        for (int i = 0; i < m; i++) {
            int lp = i;           // left end of the atom/group ending at i
            if (re[i] == '(' || re[i] == '|') ops.push_back(i);
            else if (re[i] == ')') {
                int orPos = ops.back(); ops.pop_back();
                if (re[orPos] == '|') {           // wire the alternation
                    lp = ops.back(); ops.pop_back();
                    eps[lp].push_back(orPos + 1); // '(' skips into branch 2
                    eps[orPos].push_back(i);      // '|' skips to the ')'
                } else lp = orPos;                // plain group: lp = the '('
            }
            if (i < m - 1 && re[i + 1] == '*') {  // closure around lp..i
                eps[lp].push_back(i + 1);         //   skip the operand
                eps[i + 1].push_back(lp);         //   repeat the operand
            }
            if (re[i] == '(' || re[i] == '*' || re[i] == ')')
                eps[i].push_back(i + 1);          // fall through the meta char
        }
    }

    // e-closure(S): DFS over the e-digraph -- all states reachable for free
    set<int> closure(const set<int>& s) const {
        set<int> seen = s;
        vector<int> stack(s.begin(), s.end());
        while (!stack.empty()) {
            int u = stack.back(); stack.pop_back();
            for (int v : eps[u])
                if (seen.insert(v).second) stack.push_back(v);
        }
        return seen;
    }

    // does the NFA accept text? -- track the reachable-state set: O(mn)
    bool matches(const string& text) const {
        set<int> pc = closure({0});               // states reachable on ""
        for (char c : text) {
            set<int> next;
            for (int u : pc)                      // advance every letter state
                if (u < m && re[u] == c) next.insert(u + 1);
            pc = closure(next);                   // then close over e-edges
            if (pc.empty()) return false;         // dead: nothing can revive
        }
        return pc.count(m) > 0;                   // accept state in final set?
    }
};
// Regex -> NFA (Sedgewick's one-state-per-character encoding) + set-of-states
// simulation. Syntax: letters, concatenation, | (one per group), *, ( ).
class NFA {
    final String re;              // wrapped regex: "(" + pattern + ")"
    final int m;                  // re.length(); state m is the accept state
    final List<List<Integer>> eps = new ArrayList<>();

    NFA(String pattern) {
        re = "(" + pattern + ")"; // implicit wrap anchors a top-level | or *
        m = re.length();
        for (int i = 0; i <= m; i++) eps.add(new ArrayList<>());
        Deque<Integer> ops = new ArrayDeque<>();  // '(' and '|' positions
        for (int i = 0; i < m; i++) {
            int lp = i;           // left end of the atom/group ending at i
            char c = re.charAt(i);
            if (c == '(' || c == '|') ops.push(i);
            else if (c == ')') {
                int orPos = ops.pop();
                if (re.charAt(orPos) == '|') {    // wire the alternation
                    lp = ops.pop();
                    eps.get(lp).add(orPos + 1);   // '(' skips into branch 2
                    eps.get(orPos).add(i);        // '|' skips to the ')'
                } else lp = orPos;                // plain group: lp = the '('
            }
            if (i < m - 1 && re.charAt(i + 1) == '*') {  // closure lp..i
                eps.get(lp).add(i + 1);           //   skip the operand
                eps.get(i + 1).add(lp);           //   repeat the operand
            }
            if (c == '(' || c == '*' || c == ')')
                eps.get(i).add(i + 1);            // fall through the meta char
        }
    }

    // e-closure(S): DFS over the e-digraph -- all states reachable for free
    Set<Integer> closure(Set<Integer> s) {
        Set<Integer> seen = new TreeSet<>(s);
        Deque<Integer> stack = new ArrayDeque<>(s);
        while (!stack.isEmpty()) {
            int u = stack.pop();
            for (int v : eps.get(u))
                if (seen.add(v)) stack.push(v);
        }
        return seen;
    }

    // does the NFA accept text? -- track the reachable-state set: O(mn)
    boolean matches(String text) {
        Set<Integer> pc = closure(new TreeSet<>(Set.of(0)));
        for (char c : text.toCharArray()) {
            Set<Integer> next = new TreeSet<>();
            for (int u : pc)                      // advance every letter state
                if (u < m && re.charAt(u) == c) next.add(u + 1);
            pc = closure(next);                   // then close over e-edges
            if (pc.isEmpty()) return false;       // dead: nothing can revive
        }
        return pc.contains(m);                    // accept state in final set?
    }
}
# Regex -> NFA (Sedgewick's one-state-per-character encoding) + set-of-states
# simulation. Syntax: letters, concatenation, | (one per group), *, ( ).
class NFA:
    def __init__(self, pattern):
        self.re = "(" + pattern + ")"   # implicit wrap anchors top-level | or *
        self.m = len(self.re)           # state m is the accept state
        self.eps = [[] for _ in range(self.m + 1)]
        ops = []                        # stack of '(' and '|' positions
        for i, c in enumerate(self.re):
            lp = i                      # left end of the atom/group ending at i
            if c in "(|":
                ops.append(i)
            elif c == ")":
                orpos = ops.pop()
                if self.re[orpos] == "|":          # wire the alternation
                    lp = ops.pop()
                    self.eps[lp].append(orpos + 1) # '(' skips into branch 2
                    self.eps[orpos].append(i)      # '|' skips to the ')'
                else:
                    lp = orpos                     # plain group: lp = the '('
            if i < self.m - 1 and self.re[i + 1] == "*":   # closure lp..i
                self.eps[lp].append(i + 1)         # skip the operand
                self.eps[i + 1].append(lp)         # repeat the operand
            if c in "(*)":
                self.eps[i].append(i + 1)          # fall through the meta char

    def closure(self, s):
        """e-closure(S): DFS over the e-digraph -- all states reachable free."""
        seen, stack = set(s), list(s)
        while stack:
            u = stack.pop()
            for v in self.eps[u]:
                if v not in seen:
                    seen.add(v)
                    stack.append(v)
        return seen

    def matches(self, text):
        """Does the NFA accept text? Track the reachable-state set: O(mn)."""
        pc = self.closure({0})                     # states reachable on ""
        for c in text:
            nxt = {u + 1 for u in pc               # advance every letter state
                   if u < self.m and self.re[u] == c}
            pc = self.closure(nxt)                 # then close over e-edges
            if not pc:
                return False                       # dead: nothing can revive
        return self.m in pc                        # accept state in final set?
// Regex -> NFA (Sedgewick's one-state-per-character encoding) + set-of-states
// simulation. Syntax: letters, concatenation, | (one per group), *, ( ).
struct Nfa {
    re: Vec<char>,           // wrapped regex: "(" + pattern + ")"
    m: usize,                // re.len(); state m is the accept state
    eps: Vec<Vec<usize>>,    // eps[u] = the e-edge out-neighbors of u
}

impl Nfa {
    fn new(pattern: &str) -> Nfa {
        let re: Vec<char> = format!("({})", pattern).chars().collect();
        let m = re.len();
        let mut eps = vec![Vec::new(); m + 1];
        let mut ops: Vec<usize> = Vec::new();   // '(' and '|' positions
        for i in 0..m {
            let mut lp = i;  // left end of the atom/group ending at i
            match re[i] {
                '(' | '|' => ops.push(i),
                ')' => {
                    let or = ops.pop().unwrap();
                    if re[or] == '|' {           // wire the alternation
                        lp = ops.pop().unwrap();
                        eps[lp].push(or + 1);    // '(' skips into branch 2
                        eps[or].push(i);         // '|' skips to the ')'
                    } else { lp = or; }          // plain group: lp = the '('
                }
                _ => {}
            }
            if i + 1 < m && re[i + 1] == '*' {   // closure around lp..i
                eps[lp].push(i + 1);             //   skip the operand
                eps[i + 1].push(lp);             //   repeat the operand
            }
            if matches!(re[i], '(' | '*' | ')') {
                eps[i].push(i + 1);              // fall through the meta char
            }
        }
        Nfa { re, m, eps }
    }

    // e-closure(S): DFS over the e-digraph -- all states reachable for free
    fn closure(&self, s: &BTreeSet<usize>) -> BTreeSet<usize> {
        let mut seen = s.clone();
        let mut stack: Vec<usize> = s.iter().copied().collect();
        while let Some(u) = stack.pop() {
            for &v in &self.eps[u] {
                if seen.insert(v) { stack.push(v); }
            }
        }
        seen
    }

    // does the NFA accept text? -- track the reachable-state set: O(mn)
    fn matches(&self, text: &str) -> bool {
        let mut pc = self.closure(&BTreeSet::from([0]));
        for c in text.chars() {
            let next: BTreeSet<usize> = pc.iter()
                .filter(|&&u| u < self.m && self.re[u] == c)  // letter states
                .map(|&u| u + 1)                             // ... advance
                .collect();
            pc = self.closure(&next);            // then close over e-edges
            if pc.is_empty() { return false; }   // dead: nothing can revive
        }
        pc.contains(&self.m)                     // accept state in final set?
    }
}

Two more one-step traces test real understanding. "B": the start closure \(\{0,1,2\}\) already contains state 2, so the B advances to \(\{3\}\) — accept, with zero A’s: the skip edge made concrete, the same subtlety as “(AB)* matches the empty string” in §1. "BA": after the B the set is \(\{3\}\), and the accept state matches nothing — the A empties the set: reject. Acceptance is checked after the last character, not on first touch; reaching the accept state early only helps if the input ends there.

The full machine, both branches

The same simulation on the wrapped 12-state (A*B|AC)D machine of §3 shows nondeterminism earning its keep — one start set covers both alternatives, and the input decides which branch survives:

Proposition (simulation cost) Simulating a length-\(m\) regex’s NFA on a length-\(n\) text takes \(O(mn)\) time, worst case, guaranteed.
Proof. The NFA has \(m+1\) states and \(O(m)\) edges (§3). Each input character does at most \(m\) match checks (the set never exceeds \(m+1\) states) plus one graph search over the \(O(m)\)-edge \(\varepsilon\)-digraph — \(O(m)\) work — for \(n\) characters. There is no input that makes this engine blow up, because the set is bounded by the state count no matter how the branches multiply. Compare Knuth–Morris–Pratt (KMP) from Chapter 16: the same flavor of promise — linear, worst-case, never backs up over the text — one level more general. ∎
Theorem (the simulation is exact) After reading the prefix \(t[0..i)\), the tracked set equals exactly the set of states reachable from the start state by consuming \(t[0..i)\). Hence the final accept test decides NFA acceptance.
Proof (induction on \(i\) — a loop invariant, like every other one this course). Base: before any input the reachable states are those \(\varepsilon\)-reachable from the start — exactly the initial closure. Step: a path consuming one more character \(t[i]\) must take some matching edge out of a state reachable on the shorter prefix, then any number of \(\varepsilon\)-moves — exactly advance-then-close, no more and no less. So the set is never an approximation. At the end, “accept state \(\in\) set” is literally the definition of NFA acceptance: some path consumes the whole string and ends at the accept state. Nondeterminism handled by bookkeeping, proved by induction. ∎
Why set-simulation is safe: ReDoS Many production engines (Perl, Java, Python, JavaScript) instead backtrack: try one path, undo, try another — which supports fancy features but risks exponential time. The pattern (a*)* on "aaaa…aaaX" triggers exactly that — catastrophic backtracking, a real denial-of-service vector (regular-expression denial of service, ReDoS) that has taken down production services. The set-of-states simulation explores all paths simultaneously in one forward pass, so it cannot blow up — \(O(mn)\), full stop. grep and Google’s RE2 use it for precisely this reason. The theory has direct engineering consequences.

5. Kleene’s theorem and the limits of regularity

Theorem (Kleene) Regular expressions and finite automata describe exactly the same class of languages — the regular languages. Every regex has an equivalent finite automaton, and every finite automaton has an equivalent regex.
Why, in both directions (sketch). Regex \(\to\) automaton is §3: Thompson’s construction is itself the proof, an explicit \(O(m)\) recipe. Automaton \(\to\) regex is state elimination: remove states one at a time, and whenever a state goes, merge the paths through it into a single edge labeled by a regex — concatenate what came in and out, star any self-loop, union parallel edges. When only one start\(\to\)accept edge remains, its label is a regex for the whole machine. Both directions constructive, so the two formalisms have exactly equal power. ∎

This is the deep result the chapter has been building: a rare, beautiful equivalence between a syntax (patterns you write) and a machine model (automata you run). It is why “regular expression” and “finite automaton” are two views of one idea — and why we can answer “does it match?” by running a machine.

Determinizing: the subset construction The simulation’s “current set of states” can be made into an explicit DFA state — one DFA state per reachable subset of NFA states, with the advance-and-close step precomputed into its transitions. That is the subset construction, and it is where the \(2^m\) in §2’s table comes from: up to one DFA state per subset. Once built, the DFA matches in pure \(O(n)\), no closure per step. A follow-up, minimization, merges DFA states that accept the same set of continuations, yielding the unique smallest DFA for the language — a canonical form, so two regexes are equivalent iff their minimal DFAs coincide. Compilers minimize their lexer DFAs to save space.

Above the regular languages

Kleene’s theorem also sharpens §1’s ceiling: since regexes and finite automata are the same thing, whatever a finite-memory machine cannot do, no regex can say. \(\texttt{A}^n\texttt{B}^n\) needs an unbounded counter; balanced parentheses need unbounded nesting. Those sit on the next rungs of the Chomsky hierarchy, where each class adds power by adding memory:

Regex sits at the bottom rung — which explains both its speed (finite memory means one linear pass) and its limits (no counting, no nesting). Programming-language syntax lives one rung up, recognized with a stack: that is why a compiler’s lexer is a regex engine but its parser is not.

6. Choosing an engine in practice

Everything a real regex engine does is some point on the build-cost versus run-cost trade the NFA/DFA table of §2 set up:

you match…usewhy
a pattern oncesimulate the NFA\(O(mn)\), no build cost worth amortizing
one pattern, many texts (a lexer)compile to a DFApay the subset construction once, then \(O(n)\) per text
untrusted patterns or inputsNFA simulationguaranteed \(O(mn)\) — immune to ReDoS

Precompute or pay per query — the same decision as indexes, hash tables, and dynamic programming. And the ideas land everywhere: grep and RE2 are the set-of-states simulator of §4; every compiler tokenizes source with a (minimized) DFA generated by tools like flex; protocol validators run automata over byte streams. One more callback: KMP’s precomputed failure-function table from Chapter 16 is a DFA, specialized to a single fixed pattern — you were doing automata theory a chapter before the vocabulary arrived. Pattern \(\to\) machine threads through the whole strings unit.

7. Exercises (with solutions)

  1. Which of "ABAB", "ABA", "" does (AB)* match? And write out the language \(L(\texttt{(A|B)C})\) as an explicit set.
    solution "ABAB" — yes (two copies of AB); "ABA" — no (a dangling A that no whole number of copies covers); "" — yes (* allows zero copies). \(L(\texttt{(A|B)C}) = \{\texttt{AC}, \texttt{BC}\}\): the group offers one choice, then a C.
  2. Over \(\{\texttt{A},\texttt{B}\}\), give a regex for all strings that end in B, and explain precisely why A*B is wrong.
    solution (A|B)*B — anything, then one final B. A*B describes only “a run of A’s then a single B”: it is a subset of ends-in-B that misses every string with an earlier B, e.g. "BAB" (which ends in B but is rejected — after the first B the pattern is finished, and the trailing AB has nowhere to go). Boundary checks on the right answer: "B" ✓, "BA" ✗, "" ✗.
  3. Precedence. Fully parenthesize AB|C*. Then classify each of "A", "", "ABAB" as matching AB*, (AB)*, both, or neither.
    solution Star binds tightest, then concatenation, then |: AB|C* = (AB)|(C*). For the classification, AB* is “one A, then zero or more B’s” while (AB)* is “zero or more AB pairs”: "A" matches AB* only (zero B’s); "" matches (AB)* only (zero pairs — AB* requires the A); "ABAB" matches (AB)* only (AB* allows nothing after the B’s start). No string here matches both, though e.g. "AB" would.
  4. Build the NFA for AB* under Sedgewick’s encoding (unwrapped, like the worked A*B): list the states, the \(\varepsilon\)-edges, and the match edges — then compute \(\varepsilon\)-closure(\(\{0\}\)). It differs from A*B’s in an instructive way.
    solution States \(0\,\texttt{'A'}\), \(1\,\texttt{'B'}\), \(2\,\texttt{'*'}\), \(3 =\) accept. The star’s operand is now the B, so the \(\varepsilon\)-edges wrap state 1: \(1\to2\) (skip the B), \(2\to1\) (repeat it), and \(2\to3\) (fall through). Match edges \(0\,\xrightarrow{\texttt{A}}\,1\) and \(1\,\xrightarrow{\texttt{B}}\,2\). And \(\varepsilon\)-closure(\(\{0\}\)) \(= \{0\}\) — state 0 has no \(\varepsilon\)-edges out, so unlike A*B nothing is free before the first character: the A is mandatory. (Check: "A" accepts — \(\{0\}\to\{1,2,3\}\); "B" rejects — the set dies immediately.)
  5. Simulate the worked A*B machine on "AB" and on the empty string "", giving every state set. What general fact does the second trace illustrate?
    solution "AB": start \(\{0,1,2\}\); read A — 0 advances to \(\{1\}\), closure \(\{0,1,2\}\); read B — 2 advances to \(\{3\}\), closure \(\{3\}\); \(3\) is the accept state → accept. "": no characters are read, so the final set is the start closure \(\{0,1,2\}\), which does not contain \(3\) → reject. The general fact: acceptance is evaluated only after the last character, on whatever set is active then — the machine being “near” the accept state, or passing through it mid-string, counts for nothing.
  6. Trace the wrapped 12-state (A*B|AC)D machine on "BD". Which branch of the alternation accepts it, and what does that reveal about the string?
    solution Start closure \(\{0,1,2,3,4,6\}\). Read B: only state 4 matches, advance \(\{5\}\), closure \(\{5,8,9\}\) (the | skips to the ), which falls through to the D). Read D: state 9 advances to \(\{10\}\), closure \(\{10,11\}\) — contains the accept state \(11\) → accept. The left branch accepted, with the star contributing zero A’s: "BD" is \(\texttt{A*B}\) (as just B) followed by D. The right branch (AC) died on the first character — state 6 does not match B.
  7. A compiler’s lexer will match one fixed token pattern against millions of source lines; a web form will run patterns supplied by users against their own input. For each, choose NFA simulation or DFA compilation, with costs.
    solution Lexer: compile to a DFA. The subset construction’s build cost (potentially large, up to \(2^m\) states, though rarely near it) is paid once and amortized over millions of \(O(n)\) matches; per-text there is no set tracking and no closure. User-supplied patterns: NFA simulation. The build is \(\Theta(m)\) and matching is guaranteed \(O(mn)\), so no adversarial pattern can trigger exponential behavior — whereas handing untrusted patterns to a backtracking engine invites ReDoS, and determinizing an adversarial pattern could itself blow up in space. This is the build-cost/run-cost trade of §6: amortize when the pattern is reused, stay linear when it is not trusted or not reused.
  8. Classify each language over \(\{\texttt{A},\texttt{B}\}\) (or the bracket alphabet) as regular or not, with justification: (a) strings with an even number of A’s; (b) \(\texttt{A}^n\texttt{B}^n\); (c) balanced parentheses.
    solution (a) Regular: a two-state automaton whose state is the parity of A’s seen decides it — finite memory (one bit) suffices. A regex: (B|AB*A)* — reading left to right, each block is either a lone B or a pair of A’s with any B’s between, so A’s are consumed strictly in pairs. (b) Not regular: matching the B-count to the A-count requires remembering an unbounded integer \(n\), and a finite automaton’s only memory is which of its fixed states it occupies — the pumping lemma formalizes the contradiction. (c) Not regular, same reason: nesting depth is an unbounded count. Both (b) and (c) are context-free — one rung up the hierarchy, recognized with a stack.
