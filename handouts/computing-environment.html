Computing Environment:
The CSS Linux Lab & How to Use It

CSS 343 - Data Structures, Algorithms & Discrete Mathematics II. Required setup.
How to reach the school's Linux lab, transfer files, and compile & run C++ there - including the rule for compiling template classes, and how to check for memory leaks with Valgrind. Every programming assignment is graded on this environment.

C and C++ programs are OS-dependent: code that builds and runs on your laptop can behave differently - or fail to compile - on another system. To keep grading fair and reproducible, this course standardizes on Linux. You may write code in any editor or IDE you like, but all submitted code must compile and run on the CSS Linux lab with g++ -std=c++17, because that is where we grade it.

The one rule that costs the most points "It compiles on my machine" is not enough. Before every submission, do a final build and test on a csslab machine. If your code does not compile there, you receive 0 for correctness (see the assignment rubric).

2. Connecting from your computer

You connect with SSH (Secure Shell), which gives you a terminal on the lab machine. Every modern OS ships with an ssh command built in - no extra software required.

macOS / Linux

Open Terminal and run:

# replace "netid" with your UW NetID
ssh netid@csslab1.uwb.edu

The first time you connect to a host it asks you to confirm a fingerprint - type yes. Then enter your UW NetID password (the cursor does not move as you type - that's normal). To leave, type exit.

Windows

Windows 10/11 also has ssh built in. Open Windows Terminal (or PowerShell) and run the same command:

ssh netid@csslab1.uwb.edu

If you prefer a graphical client, MobaXterm (terminal + file transfer in one) or PuTTY (terminal) both work; set Host = csslab1.uwb.edu, Port = 22.

3. Transferring files

Two good workflows - pick whichever you prefer:

Command line: scp (works on Mac, Linux, Windows)

Run these from a terminal on your computer (not the lab):

# copy files UP to the lab (into ~/css343/pa1/)
scp NDLList.h NDLList.cpp CDLList.h CDLList.cpp driver.cpp netid@csslab1.uwb.edu:~/css343/pa1/

# copy a whole folder up
scp -r pa1 netid@csslab1.uwb.edu:~/css343/

# copy a result back DOWN to the current folder
scp netid@csslab1.uwb.edu:~/css343/pa1/result.txt .

Graphical clients

FileZilla (all platforms), WinSCP (Windows), or MobaXterm let you drag and drop. Use protocol SFTP, Host csslab1.uwb.edu, Port 22, and your UW NetID credentials.

Typical loop Edit locally in your IDE → scp the files up → build & test on the lab. Or skip the round-trip and edit directly over SSH with nano (beginner-friendly) or vim.

4. Compiling & running C++

The compiler is g++. This course builds with the C++17 standard:

# one source file with main()
g++ -std=c++17 main.cpp -o main
./main

Always pass -std=c++17 and name your program with -o (rather than relying on the default a.out). If a program has several .cpp files and exactly one main(), compile them together:

g++ -std=c++17 *.cpp -o prog     # or list them explicitly
./prog

Template classes - the rule to memorize

A template's full definition must be visible to the compiler at the point where it is instantiated. The course convention (and what the assignments require) is: put the template's implementation in a .cpp, and #include that .cpp at the bottom of its header. Then you compile only the file that contains main() - never the template .cpp on its own.

Template compile rule Header ends with the implementation: List.h → last line #include "List.cpp".
Compile only the driver (the file with main()). Do not put the template .cpp on the g++ line.
Example Files: List.h (class spec; last line #include "List.cpp"), List.cpp (the template implementation), and driver.cpp (your main(), which does #include "List.h"). Build and run: 
g++ -std=c++17 driver.cpp -o driver
./driver
Compiling List.cpp by itself, or adding it to the g++ line, causes duplicate-definition / linker errors. Compile the driver only.

Two flags you'll want on assignments: -O2 turns on optimization (use it for timing experiments such as the doubling experiment), and -g adds debug symbols (use it with Valgrind and debuggers).

5. Checking for memory leaks (Valgrind)

Assignments that use new/delete are graded for leaks with Valgrind, which is installed on the lab. Build with -g, then run your program under it:

g++ -std=c++17 -g driver.cpp -o driver
valgrind --leak-check=full ./driver

A clean run ends with All heap blocks were freed -- no leaks are possible and ERROR SUMMARY: 0 errors. Anything else means your destructor, clear(), copy constructor, or assignment operator is not releasing everything it allocated. For how to read the full report - direct vs. indirect loss, and the file:line that pinpoints the leak - see the Valgrind handout.

Faster alternative while developing The sanitizers catch many of the same bugs with less output: g++ -std=c++17 -fsanitize=address,undefined -g driver.cpp -o driver && ./driver. Still do a final Valgrind check, since that is what grading uses.

7. First session: a quick walkthrough

Try this end-to-end once so the workflow is familiar before an assignment is due.

# 1. log in (off campus: connect Husky OnNet first)
ssh netid@csslab1.uwb.edu

# 2. make a place to work and go there
mkdir -p css343/lab0
cd css343/lab0

# 3. create a tiny program (nano opens an editor; paste, then Ctrl-O, Enter, Ctrl-X)
nano hello.cpp

# 4. build and run it
g++ -std=c++17 hello.cpp -o hello
./hello

A minimal hello.cpp:

#include <iostream>
int main() { std::cout << "Hello from the CSS lab\n"; }

To practice the template build: put a List.h (ending in #include "List.cpp"), a List.cpp, and a driver.cpp in a folder, then compile only the driver: g++ -std=c++17 driver.cpp -o driver && ./driver. (Ask yourself: why is only driver.cpp on the command line, and not List.cpp? That is the template rule from §4.)
