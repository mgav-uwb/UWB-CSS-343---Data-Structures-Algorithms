Greedy Algorithms

CSS 343 — Data Structures, Algorithms & Discrete Mathematics II. Course text — companion reading for Lectures 9, 11, and 12.
The greedy method in general: what it is (make the locally best choice and never reconsider), the change-making pair that shows the same rule succeeding and failing, the two properties that license it (greedy-choice and optimal substructure), the exchange argument that proves them, the classic failure cases, and where greedy sits among the design paradigms. The course’s specific greedy algorithms live in their own chapters: Dijkstra (Ch. 9), Prim and Kruskal (Ch. 11), Huffman (Ch. 12).

Three of this course’s algorithms — Dijkstra, Prim, and Kruskal — share one strategy: build the answer one committed choice at a time, always taking whatever looks best right now. This handout is about that strategy in general — the fastest one there is when it is correct, and the whole subtlety is that a locally best choice is not always part of a globally best solution. We define the method, watch one rule succeed and fail on nearly identical inputs, name the two properties that license it, learn the one proof technique (the exchange argument) that establishes them, and place greedy among its alternatives.

1. What a greedy algorithm is

Definition (greedy algorithm) A greedy algorithm builds a solution incrementally: at each step it makes the choice that looks best right now — by some fixed rule — commits to it irrevocably, and reduces the problem by that choice. No backtracking, no lookahead. 
  greedy(problem):
      while not solved:
          x = the locally BEST choice available
          commit to x               // never undone
          shrink the problem

Every greedy algorithm is this loop; the art is entirely in the choice rule and its proof. The irrevocability is what makes greedy fast — one pass, no search tree — and also what makes it risky, because a locally best choice can foreclose a globally better one. Whether it does depends on the problem, not the strategy.

2. The same rule, opposite fates

The sharpest way to see that greed’s correctness belongs to the problem is a rule that succeeds and fails on inputs that differ only in their numbers. Making change with the fewest coins, rule: take the largest coin that fits, repeat. 

  US coins {25, 10, 5, 1},  make 41¢:
      41 → 25 + 10 + 5 + 1  =  4 coins     ✓ optimal (always)

  coins {1, 3, 4},  make 6:
      greedy:   4 + 1 + 1  =  3 coins
      optimal:  3 + 3      =  2 coins        ✗ greedy loses!

The rule is identical in both runs; only the denominations changed. For US denominations the rule is provably optimal for every amount; for \(\{1,3,4\}\) grabbing the 4 forces two 1s, while the optimum starts with a choice that did not look best. Nothing on the surface distinguishes the two cases. The lesson is blunt: every greedy algorithm needs either a proof that it is optimal or a counterexample that kills it; a hunch about one instance does not transfer.

3. The two licensing ingredients

Definition (the two ingredients) A greedy algorithm is correct exactly when its problem has both:
  • the greedy-choice property — some globally optimal solution contains the locally best (greedy) choice, so committing to it is safe; and
  • optimal substructure — after the greedy choice, what remains is a smaller instance of the same problem, and an optimal solution to it combines with the greedy choice to give a global optimum.

The two are different guarantees, and it is worth not confusing them. Optimal substructure is common — shortest paths, knapsack, and edit distance all have it, which is why dynamic programming (the Dynamic Programming handout) applies to them. The greedy-choice property is the rarer, stronger condition that you can fix the locally best choice and never regret it. A problem with optimal substructure but no greedy-choice property is precisely where greed fails and dynamic programming takes over — \(\{1,3,4\}\) change-making is exactly such a problem.

4. The exchange argument

The greedy-choice property is almost always proved the same way — the same exchange argument that proves the MST cut property (Chapter 11).

Proof template (exchange argument) Let \(g\) be the greedy choice. Take any optimal solution \(O\) that does not use \(g\). Exhibit an element of \(O\) that can be swapped for \(g\), and show the result is still feasible and no worse than \(O\). Then an optimal solution containing \(g\) exists, so \(g\) is safe.
Example (activity selection) Given activities each with a start and finish time, choose the most that can share one room without overlapping. The greedy rule is always take the compatible activity that finishes earliest. It is optimal, by exchange: let \(g\) finish earliest overall, and let \(O\) be any optimal schedule; its first activity finishes no earlier than \(g\), so replacing it with \(g\) keeps the schedule feasible (\(g\) frees the room soonest) and the same size. The activities starting after \(g\) form a smaller instance — optimal substructure — and induction finishes the proof. Sort by finish time and sweep once: \(O(n\log n)\).

5. Where greed fails

The 0/1 knapsack Given items with a value and a weight and a knapsack of capacity \(W\), take each item whole or not to maximize value. The natural greedy — take items by value-to-weight ratio, highest first — can be far from optimal, because one heavy high-ratio item can block two lighter items that together beat it. This problem has optimal substructure but not the greedy-choice property, so it needs dynamic programming (Chapter 15). Tellingly, the fractional knapsack — where you may take part of an item — is greedy-solvable: the fraction is exactly what lets the exchange argument go through without waste. A small change to the problem flips which technique applies.
When is greedy always optimal? Matroids There is a precise answer to “which problems is greedy right for,” and it is one of the deeper ideas in Cormen et al. A matroid is a ground set with a family of “independent” subsets closed downward and satisfying an exchange axiom (if one independent set is larger than another, some element of the larger can be added to the smaller keeping it independent). The theorem: the greedy algorithm — sort elements by weight, take each that keeps the set independent — finds a maximum-weight independent set for every weighting if and only if the structure is a matroid. Minimum spanning trees are the canonical case: the forests of a graph form the graphic matroid, and Kruskal is exactly this greedy (which is why it needed no problem-specific cleverness, only the cut property). The \(0/1\) knapsack’s feasible sets are not a matroid, which is the structural reason greedy fails on it. Huffman is a greedy that lies outside the matroid frame, so it needs its own exchange proof — a reminder that matroids explain many greedy successes but not all.

Kruskaltake the cheapest edge that avoids a cycle the cut property Ch. 11
Huffmanmerge the two least-frequent symbols its own exchange lemma Ch. 12

The practical method for a new problem: first try to break a proposed greedy rule with a small counterexample (usually faster than a proof, and it settles the matter); if it survives real effort, prove it with an exchange argument plus optimal substructure. Scheduling, selection, and encoding problems with a clean ordering are greedy candidates; problems where committing to the local best can foreclose a better whole are dynamic-programming territory. Never ship a greedy you have neither proved nor stress-tested.

7. Greedy among the paradigms

paradigmideawhen it applies
brute force try every solution, keep the best always correct; usually exponential
greedy commit to one locally-best choice, never reconsider the choice is provably safe (exchange argument)
divide-and-conquer (Ch. 13) split into independent subproblems, recurse, combine subproblems do not overlap
dynamic programming (handout) try all choices, cache the overlapping subproblems overlap + optimal substructure

Dynamic programming is greedy without the leap of faith: where greedy gambles that one locally-best choice is always safe, DP tries every choice at each step and lets cached subproblem answers sort out the winner — which is why it solves the problems where greedy has a counterexample. For \(\{1,3,4\}\) change-making, \(\texttt{best}[n] = 1 + \min(\texttt{best}[n{-}1], \texttt{best}[n{-}3], \texttt{best}[n{-}4])\) recovers the optimal \(3+3\) that greedy missed. The Dynamic Programming handout is this handout’s mirror image, and the two together are the decision procedure for optimization problems.

8. Exercises (with solutions)

  1. Using the greedy “largest coin first” rule with denominations \(\{1, 5, 8\}\), make \(10\)¢. What does greedy return, what is the optimum, and which greedy ingredient fails?
    solution Greedy takes \(8\), then \(1 + 1\): three coins. The optimum is \(5 + 5\): two coins. Greedy fails because the greedy-choice property does not hold for these denominations — committing to the largest coin \(8\) forecloses the better all-\(5\)s solution, so no exchange argument can go through. (This coin system is not “canonical.”) The problem still has optimal substructure, which is why dynamic programming solves coin change in general.
  2. Activities with (start, finish) times: \(A(0,10)\), \(B(1,2)\), \(C(3,4)\), \(D(5,6)\). Run the earliest-finish greedy. Which activities are chosen? Then run the plausible-sounding earliest-start rule. What goes wrong, and what does the contrast tell you about greedy choice rules?
    solution Earliest-finish sorts \(B(1,2), C(3,4), D(5,6), A(0,10)\) and takes \(B\), then \(C\) (starts \(3 \ge 2\)), then \(D\) (starts \(5 \ge 4\)) — three activities, which is optimal. Earliest-start takes \(A(0,10)\) first; it overlaps everything, so the schedule ends at one activity. Both rules are “greedy”; only one has the greedy-choice property (the exchange argument works for earliest-finish, and \(A\) is a counterexample that kills earliest-start). The lesson: greedy is a family of rules, and the proof certifies a specific choice rule, not the strategy.
