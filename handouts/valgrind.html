Finding Memory Leaks with Valgrind

CSS 343 - Data Structures, Algorithms & Discrete Mathematics II. Tooling handout.
How to build for Valgrind, run it, and read its report - a clean run versus a real leak - with annotated transcripts. Every assignment that uses new/delete is graded leak-free with Valgrind. (Transcripts below are real output, captured on Ubuntu 24.04, g++ 13.3.0, Valgrind 3.22.)

When you manage memory by hand, two questions decide your grade on the Memory criterion: did every new get a matching delete? and did you ever touch memory you shouldn't? Valgrind answers both. It runs your program on a simulated CPU, tracking every heap block and every access, and at exit tells you exactly how much leaked and where it was allocated.

1. Build it & run it

Compile with -g (debug symbols, so Valgrind can name your file and line), then run your program under Valgrind with full leak checking:

g++ -std=c++17 -g driver.cpp -o driver
valgrind --leak-check=full ./driver

Your program runs normally - just 10-50× slower, since every memory access is checked. --leak-check=full prints a stack trace for each leak. (Add -s to also list error details, or --track-origins=yes to trace uninitialized values back to their source.)

2. A clean report (what passing looks like)

This is the linked-list builder that frees every node. Two lines decide it - the HEAP SUMMARY and the ERROR SUMMARY:

==584== Memcheck, a memory error detector
==584== Command: ./clean
==584==
sum = 6
==584==
==584== HEAP SUMMARY:
==584==     in use at exit: 0 bytes in 0 blocks
==584==   total heap usage: 5 allocs, 5 frees, 77,872 bytes allocated
==584==
==584== All heap blocks were freed -- no leaks are possible
==584==
==584== ERROR SUMMARY: 0 errors from 0 contexts (suppressed: 0 from 0)
A pass is exactly this in use at exit: 0 bytes in 0 blocks,  All heap blocks were freed -- no leaks are possible,  and  ERROR SUMMARY: 0 errors. (The number after == is just the process id. 5 allocs, 5 frees - allocations and frees balance.)

3. A leak report, line by line

Now the same program with the delete loop removed - it builds a 3-node list and never frees it:

==583== HEAP SUMMARY:
==583==     in use at exit: 48 bytes in 3 blocks
==583==   total heap usage: 5 allocs, 2 frees, 77,872 bytes allocated
==583==
==583== 48 (16 direct, 32 indirect) bytes in 1 blocks are definitely lost in loss record 2 of 2
==583==    at 0x4846FA3: operator new(unsigned long) (vg_replace_malloc.c)
==583==    by 0x1091AF: main (leak.cpp:8)
==583==
==583== LEAK SUMMARY:
==583==    definitely lost: 16 bytes in 1 blocks
==583==    indirectly lost: 32 bytes in 2 blocks
==583==      possibly lost: 0 bytes in 0 blocks
==583==    still reachable: 0 bytes in 0 blocks
==583==         suppressed: 0 bytes in 0 blocks
==583==
==583== ERROR SUMMARY: 1 errors from 1 contexts (suppressed: 0 from 0)

Read it top to bottom:

  • in use at exit: 48 bytes in 3 blocks - 3 heap blocks were still allocated when the program ended. Those are leaked.
  • 5 allocs, 2 frees - you allocated 5 times but freed only 2, so 3 blocks were never freed. (The extra 2 allocs/frees are the C++ I/O library's, not yours.)
  • The loss record + stack trace - by 0x1091AF: main (leak.cpp:8) points at the allocation that leaked: line 8, head = new Node{...}. That is where the lost memory was created; your missing delete is the fix. Build with -g or you get raw addresses instead of file:line.
  • 48 (16 direct, 32 indirect) - the teaching detail for linked structures (see below).
  • ERROR SUMMARY: 1 errors - not clean. Grading requires 0.
Direct vs indirect loss Forgetting to free a linked list loses the head node directly (16 bytes - you dropped the last pointer to it) and the two nodes it chains to indirectly (32 bytes - they were only reachable through the head). One missing destructor leaks the whole chain. Fix the definitely lost blocks (free the structure) and the indirect losses disappear with them.

4. The five leak categories

CategoryMeaningAction
definitely lostNo pointer to the block survives - a true leak you caused.Fix it.
indirectly lostLost only because a block that pointed to it was lost (e.g. chained nodes).Fixing the definite leaks clears these.
possibly lostA pointer into the middle of the block survives, but not to its start.Usually your bug - investigate.
still reachableNot freed, but a valid pointer still exists at exit (often library/global state).Usually benign; clean it up if it's yours.
suppressedKnown, deliberately ignored (system suppression files).Ignore.

5. Where leaks come from in this course

For the pointer-based structures you build (linked lists, trees), almost every leak is a Rule-of-Three gap:

  • Destructor that doesn't walk the whole structure and delete every node (and, for a list with a sentinel, the dummy header too).
  • clear() that resets head = nullptr without deleting the nodes first - it abandons them.
  • Copy constructor / operator= that do a shallow copy: both objects share the same nodes, so you get a double free (an error, not a leak) - or you reassign without freeing the old contents first, which leaks.
  • Self-assignment (a = a) that frees this's nodes before copying from the same object.
The rule Every new has exactly one matching delete. The destructor frees the entire structure; the copy constructor and assignment operator make a deep copy (and assignment frees the old contents first, after guarding against self-assignment).

6. Other errors Valgrind catches

Leaks are only half of it. Memcheck also flags, each with a stack trace and a non-zero ERROR SUMMARY:

  • Invalid read / write - Invalid read of size 8: reading or writing outside a block, e.g. walking one node past the end, or using retrieve(index) out of bounds.
  • Use after free / double free - touching or deleting a node you already deleted (classic shallow-copy fallout).
  • Uninitialized values - Conditional jump or move depends on uninitialised value(s): e.g. a next pointer you forgot to set to nullptr. Add --track-origins=yes to find where it came from.
  • Mismatched new/delete - freeing array new[] with scalar delete (or vice versa).

Read every stack trace top-down and stop at the first line that names your file - that's where to look.

7. Where Valgrind runs (and the macOS substitute)

Valgrind runs on Linux - the CSS lab, or a Linux container/VM. It is not supported on current macOS. While developing on a Mac, get most of the same coverage from the compiler's sanitizers, then do the authoritative Valgrind check on the lab:

# macOS / quick local check (AddressSanitizer + UB sanitizer)
clang++ -std=c++17 -fsanitize=address,undefined -g driver.cpp -o driver
./driver

# authoritative check on the CSS Linux lab (this is what grading uses)
g++ -std=c++17 -g driver.cpp -o driver
valgrind --leak-check=full ./driver

See the Computing Environment handout for connecting to the lab.

8. Checklist

  • Built with -g.
  • Ran valgrind --leak-check=full ./prog on every code path (have your driver exercise insert/remove/copy/assign/clear).
  • All heap blocks were freed -- no leaks are possible and ERROR SUMMARY: 0 errors ⇒ pass.
  • Any definitely lost, Invalid read/write, or ERROR SUMMARY: ≥1 ⇒ fix using the file:line in the trace, then rerun.
